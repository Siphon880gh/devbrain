Not all blobs are equal. A photo your users see every day is "hot." A monthly invoice you might fetch once a year is "cool" or "cold." A backup you legally must keep for 7 years is "archive."

Azure Blob Storage lets you assign each blob an **access tier** that trades off:

```text
- Per-GB storage cost   (lower for cooler tiers)
- Read/write cost       (higher for cooler tiers)
- Retrieval latency     (instant → up to 15 hours for archive)
- Minimum storage time  (longer for cooler tiers)
```

Used right, tiers can cut storage bills by 80%+ with zero code changes.

---

## 1. The Four Tiers at a Glance

| Tier    | Per-GB stored | Per-GB read | Min. storage time | Read latency       |
|---------|---------------|-------------|-------------------|--------------------|
| Hot     | Highest       | Lowest      | none              | Milliseconds       |
| Cool    | ~50% of Hot   | Higher      | 30 days           | Milliseconds       |
| Cold    | ~25% of Hot   | Even higher | 90 days           | Milliseconds       |
| Archive | ~5% of Hot    | Highest     | 180 days          | Up to 15 hours (rehydrate) |

> Exact prices vary by region — always confirm with the Pricing Calculator. The proportions above are typical for the Standard_LRS Block Blob tier.

The trade-off in plain language:

```text
Hotter tier   = pay more to STORE, less to READ      (frequent access)
Colder tier   = pay less to STORE, more to READ      (rare access)
Archive       = nearly free to store, but slow + costly to read
```

---

## 2. The Mental Model

```text
Hot     = front shelf of a store          (grab any time)
Cool    = stockroom in the back            (still fast, slightly farther)
Cold    = warehouse off-site               (you can drive there)
Archive = sealed container in deep storage (call ahead, wait hours)
```

Pick the tier that matches **how often the file is actually opened**, not how important it is.

---

## 3. Setting the Tier on a New Blob

In the **portal**, when uploading: expand "Advanced" → "Access tier" → pick.

In the **CLI**:

```bash
az storage blob upload \
  --account-name mystorage \
  --container-name backups \
  --name 2026-05-backup.tar.gz \
  --file ./backup.tar.gz \
  --tier Cool
```

In the **Python SDK**:

```python
from azure.storage.blob import StandardBlobTier

container.upload_blob(
    name="2026-05-backup.tar.gz",
    data=open("./backup.tar.gz", "rb"),
    standard_blob_tier=StandardBlobTier.Cool,
    overwrite=True,
)
```

---

## 4. Changing the Tier of an Existing Blob

```bash
az storage blob set-tier \
  --account-name mystorage \
  --container-name backups \
  --name 2026-05-backup.tar.gz \
  --tier Cold
```

Tier changes are **instant** for Hot ↔ Cool ↔ Cold. Moving **to** Archive is instant; moving **from** Archive requires rehydration (next section).

---

## 5. Archive Rehydration

You cannot read an Archive blob directly. You must first "rehydrate" it back to Hot, Cool, or Cold:

```bash
az storage blob set-tier \
  --account-name mystorage \
  --container-name old-backups \
  --name big-blob.tar \
  --tier Hot \
  --rehydrate-priority Standard
```

```text
Standard priority   → up to 15 hours
High priority       → typically under 1 hour, costs more
```

While rehydration is in progress, the blob's `archiveStatus` property is set; reads still fail until rehydration finishes.

---

## 6. Minimum Storage Duration Penalty

Each cooler tier has a minimum storage time. Delete or move the blob earlier and you pay an "early deletion" fee equal to the rest of the period.

```text
Tier      Minimum days   Early deletion charge
-------   ------------   -----------------------------------------
Hot       0              none
Cool      30             charged for the remaining days up to 30
Cold      90             charged for the remaining days up to 90
Archive   180            charged for the remaining days up to 180
```

> Don't drop a blob into Archive "just in case" if you might need to delete or read it again next month. The early-delete penalty often costs more than just leaving it in Hot.

---

## 7. Default Tier of the Storage Account

Storage accounts have a **default access tier** (Hot or Cool). New blobs inherit it unless you specify otherwise.

```bash
az storage account update \
  --name mystorage \
  --resource-group rg-x \
  --access-tier Cool
```

For a backup-heavy account, set the default to Cool so you don't have to remember to set it on every upload.

## 8. Lifecycle Management Policies

Manually re-tiering blobs at scale is painful. **Lifecycle management policies** automate it: rules like "if not modified in 30 days, move to Cool" or "delete after 365 days."

A policy is a JSON file with one or more rules. Each rule has:

```text
- A name
- A "filter" (which blobs the rule applies to)
- An "action" (what to do when conditions are met)
```

### Example: typical "ages out" lifecycle

```json
{
  "rules": [
    {
      "enabled": true,
      "name": "ageOutBlobs",
      "type": "Lifecycle",
      "definition": {
        "filters": {
          "blobTypes": ["blockBlob"],
          "prefixMatch": ["backups/"]
        },
        "actions": {
          "baseBlob": {
            "tierToCool":    { "daysAfterModificationGreaterThan":  30 },
            "tierToCold":    { "daysAfterModificationGreaterThan":  90 },
            "tierToArchive": { "daysAfterModificationGreaterThan": 180 },
            "delete":        { "daysAfterModificationGreaterThan": 730 }
          },
          "snapshot": {
            "delete":        { "daysAfterCreationGreaterThan":      90 }
          }
        }
      }
    }
  ]
}
```

In English: for blobs under `backups/`,

```text
- after 30 days unmodified  → move to Cool
- after 90 days unmodified  → move to Cold
- after 180 days unmodified → move to Archive
- after 730 days unmodified → delete
- snapshots older than 90 days → delete
```

### Apply via CLI

Save the JSON above as `lifecycle.json`, then:

```bash
az storage account management-policy create \
  --account-name mystorage \
  --resource-group rg-x \
  --policy @lifecycle.json
```

### Apply via Portal

```text
Storage account → Data management → Lifecycle management → "Add a rule"
```

The portal has a friendly wizard that builds the JSON for you.

> Lifecycle policies run **once a day**, not in real time. A blob can sit a few hours past the threshold before being moved.

---

## 9. Other Lifecycle Triggers

Besides "days since modification," you can match on:

```text
- daysAfterCreationGreaterThan
- daysAfterLastAccessTimeGreaterThan   (requires "Last access time tracking" enabled)
- daysAfterLastTierChangeGreaterThan
- daysAfterCreationGreaterThan          (for snapshots/versions)
```

Combined with `prefixMatch` and `blobIndexMatch` (tags), you can build very targeted rules:

```text
"Move only invoices older than 2 years from container client-files
 whose blob index tag dept=accounting is set."
```

## 10. Practical Tier Strategies

| Workload                       | Suggested tier setup                                  |
|--------------------------------|-------------------------------------------------------|
| Website images / CDN origin    | Hot                                                   |
| User uploads (recent)          | Hot, age to Cool after 30 days                        |
| Application logs               | Hot for 7 days → Cool 30 → Cold 180 → delete 365      |
| Daily SQL backups              | Cool, age to Archive at 90 days, delete at 7 years    |
| Monthly snapshots              | Cold immediately                                      |
| Compliance / WORM archive      | Archive, with immutability policy                     |
| Anything you might delete soon | Stay on Hot — early deletion fees outweigh savings    |

---

## 11. Costs to Watch

Cooler tiers reduce **storage** cost, but each operation is more expensive:

```text
Operation                          Hot       Cool      Cold      Archive
-------------------                ----      ----      ----      -------
Storage per GB / month             baseline  ~50%      ~25%      ~5%
Write transactions                 baseline  higher    higher    higher
Read transactions                  baseline  higher    higher    much higher
Data retrieval (per GB read)       free      ~$0.01    ~$0.03    ~$0.02 + rehydrate fee
Early deletion                     none      pro-rated pro-rated pro-rated
```

Common pitfall: dumping millions of small files into Cold and then scanning them often. The per-operation cost wipes out any storage savings.

## 12. Verifying What Tier a Blob Is In

```bash
az storage blob show \
  --account-name mystorage \
  --container-name backups \
  --name 2026-05-backup.tar.gz \
  --query "{tier:properties.blobTier, modified:properties.lastModified}" \
  -o table
```

The portal also shows the tier as a column in the blob list.

## Quick Recap

| Concept              | Key Point                                                       |
|----------------------|-----------------------------------------------------------------|
| Hot / Cool / Cold    | Same speed, different storage vs. read price trade-offs         |
| Archive              | Cheapest storage, requires rehydration to read                  |
| Min. storage time    | Cool 30d, Cold 90d, Archive 180d — early delete is charged      |
| Set tier on upload   | `--tier` flag (CLI) or "Advanced" panel (Portal)                |
| Re-tier later        | `az storage blob set-tier`                                      |
| Default tier         | Set on the storage account so new blobs inherit it              |
| Lifecycle policies   | JSON rules to automate tier moves and deletion                  |
| Watch out            | Many small reads on Cold/Archive can outcost the savings        |
