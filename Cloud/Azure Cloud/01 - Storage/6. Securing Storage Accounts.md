By default, a brand-new storage account is reachable from anywhere on the internet, and anyone with one of its **storage account keys** has full god-mode access to all four services (Blob, Files, Queue, Table). This lesson covers the four layers you should actually use:

```text
Layer 1: Authentication      — who is asking?
Layer 2: Authorization       — what can they do?
Layer 3: Network             — where are they coming from?
Layer 4: Data protection     — what if something goes wrong?
```

By the end you will know how to lock a storage account down to "only my app, only from my network, with no permanent secrets."

---

## 1. Threat Model — What Are We Protecting Against?

```text
- Leaked storage key in a Git repo                   → Layer 1 (kill keys)
- Compromised dev laptop with `az login` token       → Layer 2 (least privilege)
- Public internet attackers scanning blob.core.*     → Layer 3 (network rules)
- Accidental delete by an admin or buggy script      → Layer 4 (soft delete)
- Ransomware that overwrites blobs                   → Layer 4 (versioning + immutability)
```

Most beginner setups have none of these in place. Adding even two or three closes 90% of the risk.

## 2. Layer 1 — Authentication

Azure Storage has four ways to prove who you are:

```text
Method                          Strength    When to use
-----------------------------   ---------   ---------------------------------
Storage account keys (key1/2)   Weakest     Quick local tests only
Shared Access Signature (SAS)   Medium      Time-limited share to one resource
Microsoft Entra ID (RBAC)       Strong      All apps, all users, all the time
Managed Identity                Strongest   Apps running INSIDE Azure
```

### Storage account keys

Every storage account ships with two 512-bit keys (`key1`, `key2`). Anyone with a key has full access. Keys can be regenerated:

```bash
az storage account keys renew \
  --resource-group rg-x --account-name mystorage --key key1
```

Two keys exist so you can rotate without downtime: rotate apps to `key2`, regenerate `key1`, then swap back later.

> **Best practice:** disable shared key access entirely once your apps use Entra ID:
>
> ```bash
> az storage account update \
>   --name mystorage --resource-group rg-x \
>   --allow-shared-key-access false
> ```

### SAS URLs (recap)

Lesson 2 covered SAS in detail. Key reminders:

```text
- Treat SAS like passwords
- Use SHORT expiration (minutes/hours, not years)
- Use READ-ONLY (sp=r) unless you really need write
- Scope to a specific blob, not the whole container
- Prefer "user delegation SAS" (signed by Entra ID) over "account SAS"
```

User delegation SAS:

```bash
az storage blob generate-sas \
  --account-name mystorage \
  --container-name files \
  --name report.pdf \
  --permissions r \
  --expiry 2026-05-01T15:00:00Z \
  --auth-mode login \
  --as-user
```

`--as-user` makes the SAS signed by your Entra ID — if your account is disabled, the SAS stops working immediately.

### Microsoft Entra ID (Azure AD)

Sign in once with your identity, then `az` and the SDKs use that identity automatically:

```bash
az login
```

In SDKs:

```python
from azure.identity import DefaultAzureCredential
from azure.storage.blob import BlobServiceClient

cred = DefaultAzureCredential()
svc  = BlobServiceClient(account_url="https://mystorage.blob.core.windows.net", credential=cred)
```

`DefaultAzureCredential` tries: environment variables → managed identity → Azure CLI → VS Code → interactive. The same code works on a laptop and in production.

### Managed Identity (for apps in Azure)

When your app runs in App Service, Functions, AKS, or a VM, give it a **system-assigned managed identity** instead of storing a key or secret anywhere:

```bash
az webapp identity assign \
  --name myapp --resource-group rg-app
```

Then assign it the storage role (next section). The app's code uses `DefaultAzureCredential()` and "just works." No secrets to leak.

## 3. Layer 2 — Authorization (RBAC Roles)

Once Azure knows **who** is asking, RBAC decides **what** they can do.

The five roles you will actually use on Storage:

| Role                                       | What it grants                                  |
|--------------------------------------------|-------------------------------------------------|
| Storage Blob Data Reader                   | Read blobs                                      |
| Storage Blob Data Contributor              | Read + write + delete blobs                     |
| Storage Blob Data Owner                    | Above + manage POSIX ACLs (Data Lake Gen2)      |
| Storage File Data SMB Share Reader/Contributor | Read/write Azure Files via SMB              |
| Storage Account Contributor                | Manage the account itself (NOT data)            |

> "Owner" and "Contributor" on the **management plane** can manage settings and keys, but do NOT automatically grant data access. You must assign one of the **Data** roles above to read/write blobs.

### Assign a role

```bash
USER_OBJECT_ID=$(az ad signed-in-user show --query id -o tsv)
ACCT_ID=$(az storage account show -g rg-x -n mystorage --query id -o tsv)

az role assignment create \
  --assignee $USER_OBJECT_ID \
  --role "Storage Blob Data Contributor" \
  --scope $ACCT_ID
```

Now your `az login` identity (and any code using `DefaultAzureCredential()`) can read and write blobs without keys.

### Scope the role narrowly

Roles can be assigned at:

```text
Subscription           ← too broad
Resource group         ← still broad
Storage account        ← good for most apps
Single container       ← best for least privilege
```

```bash
CONTAINER_ID="$ACCT_ID/blobServices/default/containers/uploads"

az role assignment create \
  --assignee $APP_OBJECT_ID \
  --role "Storage Blob Data Contributor" \
  --scope $CONTAINER_ID
```

## 4. Layer 3 — Network Restrictions

By default the public endpoint accepts traffic from anywhere on the internet. Most production accounts should restrict this.

### Option A — Allow only specific IP ranges

```bash
az storage account update \
  --name mystorage --resource-group rg-x \
  --default-action Deny

az storage account network-rule add \
  --account-name mystorage --resource-group rg-x \
  --ip-address 203.0.113.42

az storage account network-rule add \
  --account-name mystorage --resource-group rg-x \
  --ip-address 198.51.100.0/24
```

Now only those IPs can reach the storage account on the public endpoint.

### Option B — Allow only specific virtual networks

For apps inside an Azure VNet:

```bash
az storage account network-rule add \
  --account-name mystorage --resource-group rg-x \
  --vnet-name vnet-app --subnet subnet-web
```

### Option C — Private Endpoints (the strong option)

A **private endpoint** gives the storage account a private IP inside your VNet. Public DNS still resolves the storage account, but to a private IP that is only reachable from your VNet. The public endpoint can be turned off entirely.

```bash
az network private-endpoint create \
  --name pe-mystorage \
  --resource-group rg-x \
  --vnet-name vnet-app \
  --subnet subnet-data \
  --private-connection-resource-id $ACCT_ID \
  --group-id blob \
  --connection-name blob-conn
```

Then disable public network access:

```bash
az storage account update \
  --name mystorage --resource-group rg-x \
  --public-network-access Disabled
```

Now the storage account is unreachable from the public internet — only from your VNet (and via the private endpoint's private DNS zone).

### "Allow trusted Microsoft services"

Even with `Deny`, you usually want this on so things like Azure Functions deployment, Backup, and AzureMonitor still work:

```bash
az storage account update \
  --name mystorage --resource-group rg-x \
  --bypass AzureServices
```

## 5. Layer 4 — Data Protection

These features protect against accidents and attacks, **after** authentication and authorization.

### Soft delete for blobs

A deleted blob is recoverable for N days:

```bash
az storage blob service-properties delete-policy update \
  --account-name mystorage \
  --enable true --days-retained 14
```

### Soft delete for containers

Recover a container that was deleted by mistake:

```bash
az storage account blob-service-properties update \
  --resource-group rg-x --account-name mystorage \
  --enable-container-delete-retention true \
  --container-delete-retention-days 14
```

### Versioning

Every overwrite creates a new immutable version:

```bash
az storage account blob-service-properties update \
  --resource-group rg-x --account-name mystorage \
  --enable-versioning true
```

This is great against ransomware that overwrites everything — older versions remain.

### Change feed

An append-only log of every change to every blob:

```bash
az storage account blob-service-properties update \
  --resource-group rg-x --account-name mystorage \
  --enable-change-feed true
```

Useful for audits and incremental backups.

### Immutable storage (legal hold / time-based retention)

For compliance scenarios (SEC 17a-4, GDPR, HIPAA):

```text
- Time-based retention: cannot delete or overwrite for N years
- Legal hold: cannot delete or overwrite until legal hold lifted
```

Set per container in the portal or via CLI. Blobs become genuinely undeletable for the period — even by an account owner.

### Encryption

Always-on:

```text
- Encryption at rest with Microsoft-managed keys (AES-256)   ← default, free
- Encryption in transit via HTTPS                            ← default, free
```

Optional:

```text
- Customer-managed keys (CMK) backed by Azure Key Vault
  for compliance scenarios
- Infrastructure encryption (double encryption)
- Encryption scopes (different keys for different containers)
```

Require HTTPS only:

```bash
az storage account update \
  --name mystorage --resource-group rg-x \
  --https-only true \
  --min-tls-version TLS1_2
```

## 6. The "Good Default" Storage Account

If you want one command to start with sensible security:

```bash
az storage account create \
  --name mystorage --resource-group rg-x --location eastus \
  --sku Standard_LRS --kind StorageV2 \
  --https-only true \
  --min-tls-version TLS1_2 \
  --allow-blob-public-access false \
  --allow-shared-key-access false \
  --default-action Deny \
  --bypass AzureServices \
  --public-network-access Enabled
```

Then:

```text
1. Add IP / VNet / private-endpoint rules (Layer 3)
2. Assign data RBAC roles to your user / app (Layer 2)
3. Enable soft delete + versioning (Layer 4)
4. Use Entra ID auth in code (Layer 1)
```

## 7. Auditing — Who Did What?

```text
Activity log         → control-plane changes (key rotation, RBAC, deletes of containers)
Diagnostic settings  → data-plane access (per-blob reads, writes, deletes)
                       sent to Log Analytics, Storage, or Event Hubs
Defender for Storage → threat detection (anomalous access, malware in uploads)
                       paid feature, ~$10/account/month
```

Enable diagnostics for blob:

```text
Storage account → Monitoring → Diagnostic settings → Add
   - Categories: StorageRead, StorageWrite, StorageDelete
   - Destination: Send to Log Analytics workspace
```

You can then query:

```kql
StorageBlobLogs
| where TimeGenerated > ago(1h) and OperationName == "DeleteBlob"
| project TimeGenerated, AccountName, ObjectKey, CallerIpAddress, Identity
```

## 8. Anti-Patterns Checklist

```text
[ ] Storage key committed to GitHub
[ ] SAS with full permissions and 1-year expiration
[ ] Storage account allows shared key access AND public network access
[ ] Public anonymous container with sensitive files
[ ] App on Azure using a connection string instead of Managed Identity
[ ] No budget alerts (the bill goes through the roof during attacks)
[ ] No soft delete  (one DROP TABLE-equivalent and data is gone)
[ ] Diagnostic settings disabled (no audit trail)
```

If any of those are checked, fix it.

## 9. Mental Model

```text
Authentication      = "I am Alice."
Authorization       = "Alice may read /uploads, but not /backups."
Network             = "Alice may only ask from inside the office VPN."
Data protection     = "Even if Alice deletes a blob, we can recover it for 14 days."
Audit               = "We logged that Alice deleted it at 03:14 AM."
```

Add these layers in order. Each one alone helps, but together they form proper defense in depth.

## Quick Recap

| Goal                                       | Mechanism                                            |
|--------------------------------------------|------------------------------------------------------|
| Stop using account keys                    | Disable shared key + use Entra ID + Managed Identity |
| Limit what an identity can do              | RBAC data roles, scoped to container if possible     |
| Block the public internet                  | Network rules, private endpoints, public-access off  |
| Survive an accidental delete               | Soft delete (blob + container), versioning           |
| Survive ransomware                         | Versioning + change feed + (optional) immutability   |
| Audit who did what                         | Diagnostic settings → Log Analytics                  |
| Detect attacks                             | Microsoft Defender for Storage                       |
