In the previous lessons we learned what storage accounts and blobs are, and how access is controlled with private/public/SAS URLs. Now we actually move bytes.

There are four common ways to upload and download blobs:

```text
1. Azure Portal       — easiest, no install, good for one-off files
2. Azure CLI (az)     — best for scripts, automation, CI/CD
3. AzCopy             — fastest for bulk transfers (millions of files / TBs)
4. SDK (Python/JS/C#) — for use inside your own applications
```

This lesson shows all four with copy-pasteable examples.

---

## 1. Setup: One-Time Resources

Before uploading anything, create the boxes from lesson 1:

```bash
RG=rg-storage-demo
LOC=eastus
ACCT=mystoragelearner$RANDOM     # must be globally unique, lowercase
CONTAINER=demo-uploads

az group create -n $RG -l $LOC

az storage account create \
  --name $ACCT \
  --resource-group $RG \
  --location $LOC \
  --sku Standard_LRS \
  --kind StorageV2

az storage container create \
  --account-name $ACCT \
  --name $CONTAINER \
  --auth-mode login
```

> `--auth-mode login` uses your signed-in Azure identity instead of the storage account key. It is the safer default once your user has the **Storage Blob Data Contributor** role on the account (covered in lesson 6).

For these examples, simpler is fine — fall back to a key:

```bash
KEY=$(az storage account keys list -g $RG -n $ACCT --query "[0].value" -o tsv)
```

## 2. Method 1: Upload via the Azure Portal

```text
1. Portal → search "Storage accounts" → click your account
2. Left menu → Data storage → Containers → click your container
3. Click "Upload"
4. Select files (drag & drop or browser picker)
5. (Optional) Expand "Advanced":
     - Blob type: Block blob (default; correct for normal files)
     - Access tier: Hot / Cool / Cold / Archive (lesson 4)
     - Encryption scope, metadata, tags, etc.
6. Click "Upload"
```

Downloading is just as click-y:

```text
1. Open the container
2. Click the blob name
3. Click "Download"
```

Best for: one or two files, ad-hoc uploads, quickly testing access settings.

## 3. Method 2: Upload with the Azure CLI

### Single file

```bash
az storage blob upload \
  --account-name $ACCT \
  --account-key  $KEY \
  --container-name $CONTAINER \
  --name "hello.txt" \
  --file "./hello.txt"
```

### Whole folder (recursive)

```bash
az storage blob upload-batch \
  --account-name $ACCT \
  --account-key  $KEY \
  --destination $CONTAINER \
  --source ./local-folder
```

### Single file download

```bash
az storage blob download \
  --account-name $ACCT \
  --account-key  $KEY \
  --container-name $CONTAINER \
  --name "hello.txt" \
  --file "./hello-downloaded.txt"
```

### Whole folder download

```bash
az storage blob download-batch \
  --account-name $ACCT \
  --account-key  $KEY \
  --destination ./downloaded-folder \
  --source $CONTAINER
```

### List blobs

```bash
az storage blob list \
  --account-name $ACCT \
  --account-key  $KEY \
  --container-name $CONTAINER \
  --output table
```

### Delete a blob

```bash
az storage blob delete \
  --account-name $ACCT \
  --account-key  $KEY \
  --container-name $CONTAINER \
  --name "hello.txt"
```

## 4. Method 3: AzCopy for Bulk Transfers

When you have **gigabytes or millions of files**, the CLI is slow. Use AzCopy.

### Install

```bash
# macOS
brew install azcopy

# Windows
winget install -e --id Microsoft.AzCopy
```

### Authenticate

```bash
azcopy login
```

### Upload an entire folder

```bash
azcopy copy "./big-folder" \
  "https://${ACCT}.blob.core.windows.net/${CONTAINER}" \
  --recursive
```

### Download an entire container

```bash
azcopy copy \
  "https://${ACCT}.blob.core.windows.net/${CONTAINER}" \
  "./big-folder-downloaded" \
  --recursive
```

### Sync one-way (like rsync)

```bash
azcopy sync "./big-folder" \
  "https://${ACCT}.blob.core.windows.net/${CONTAINER}" \
  --recursive
```

AzCopy uses parallel HTTP connections and is dramatically faster than the CLI for large jobs.

## 5. Method 4: SDK Examples

When the upload/download happens **inside your application**, use an SDK. Azure has SDKs for Python, Node.js, .NET, Java, Go, and more.

### Python

Install:

```bash
pip install azure-storage-blob azure-identity
```

Upload:

```python
from azure.identity import DefaultAzureCredential
from azure.storage.blob import BlobServiceClient

account_url = "https://mystoragelearner1234.blob.core.windows.net"
credential  = DefaultAzureCredential()
service     = BlobServiceClient(account_url=account_url, credential=credential)

container = service.get_container_client("demo-uploads")

with open("hello.txt", "rb") as f:
    container.upload_blob(name="hello.txt", data=f, overwrite=True)
```

Download:

```python
data = container.download_blob("hello.txt").readall()
with open("hello-downloaded.txt", "wb") as f:
    f.write(data)
```

### Node.js

Install:

```bash
npm install @azure/storage-blob @azure/identity
```

Upload:

```javascript
import { DefaultAzureCredential } from "@azure/identity";
import { BlobServiceClient } from "@azure/storage-blob";

const accountUrl = "https://mystoragelearner1234.blob.core.windows.net";
const service = new BlobServiceClient(accountUrl, new DefaultAzureCredential());

const container = service.getContainerClient("demo-uploads");
const blob      = container.getBlockBlobClient("hello.txt");

await blob.uploadFile("./hello.txt");
```

Download:

```javascript
await blob.downloadToFile("./hello-downloaded.txt");
```

### C# (.NET)

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var accountUrl = new Uri("https://mystoragelearner1234.blob.core.windows.net");
var service    = new BlobServiceClient(accountUrl, new DefaultAzureCredential());
var container  = service.GetBlobContainerClient("demo-uploads");
var blob       = container.GetBlobClient("hello.txt");

await blob.UploadAsync("./hello.txt", overwrite: true);
await blob.DownloadToAsync("./hello-downloaded.txt");
```

> `DefaultAzureCredential()` automatically uses your `az login` identity in development and your Managed Identity in production. We cover that pattern in `05 - Identity and Security`.

## 6. Choosing the Right Method

| Scenario                                | Best Tool         |
|-----------------------------------------|-------------------|
| One file, occasional                    | Portal            |
| Repeatable scripts, CI/CD               | Azure CLI         |
| Bulk transfer (GBs / many files)        | AzCopy            |
| Application code (web app, API, worker) | SDK               |
| Mass migration to/from S3 / GCS         | AzCopy or Storage Mover |
| Visual browse + drag & drop             | Storage Explorer  |

---

## 7. Performance Tips

```text
- Block blobs are uploaded in parallel chunks. SDKs and AzCopy do this for you.
- Larger files split into more parallel parts; very small files are network-overhead bound.
- AzCopy "--cap-mbps" can throttle to leave room for other traffic.
- For uploads from outside Azure, the bottleneck is usually your local upload bandwidth.
- Inside Azure (VM → Storage), expect line-speed transfers and very low cost (no egress).
```

## 8. Common Errors and Fixes

| Error                                  | Likely Cause                                    |
|----------------------------------------|-------------------------------------------------|
| `AuthorizationPermissionMismatch`      | Your identity lacks RBAC on the account/container |
| `AuthenticationFailed`                 | Wrong key, expired SAS, or clock skew           |
| `BlobAlreadyExists`                    | Pass `--overwrite true` or a new blob name      |
| `ContainerNotFound`                    | Typo, or container was created in another account |
| `ContentLengthMismatch`                | Network interruption — retry; AzCopy resumes    |
| `RequestBodyTooLarge`                  | Use Block Blob upload (default), not single PUT |

## 9. Quick End-to-End Round Trip

Copy and run this whole block to prove your setup works:

```bash
RG=rg-storage-demo
ACCT=mystoragelearner$RANDOM
CONTAINER=demo-uploads
LOC=eastus

az group create -n $RG -l $LOC

az storage account create -n $ACCT -g $RG -l $LOC \
  --sku Standard_LRS --kind StorageV2

KEY=$(az storage account keys list -g $RG -n $ACCT --query "[0].value" -o tsv)

az storage container create --account-name $ACCT --name $CONTAINER --account-key $KEY

echo "hello azure" > hello.txt

az storage blob upload \
  --account-name $ACCT --account-key $KEY \
  --container-name $CONTAINER --name "hello.txt" --file "./hello.txt"

az storage blob list \
  --account-name $ACCT --account-key $KEY \
  --container-name $CONTAINER -o table

az storage blob download \
  --account-name $ACCT --account-key $KEY \
  --container-name $CONTAINER --name "hello.txt" --file "./hello-back.txt"

cat ./hello-back.txt

# Clean up everything
az group delete -n $RG --yes --no-wait
```

If `cat ./hello-back.txt` prints `hello azure`, your end-to-end pipeline is working.
