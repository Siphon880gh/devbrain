## Definitions

**Jenkins**  
Jenkins is a CI/CD tool used to run builds, tests, and deployments. Teams use it to see whether a build passed, what version was deployed, and which environment received the deployment.

**Datadog**  
Datadog is an observability platform used to monitor applications and infrastructure. It helps teams inspect metrics, logs, traces, alerts, and service health when investigating failures or performance issues.

**Instances**  
An instance usually means a running compute resource that hosts part of an application, such as a virtual machine, cloud server, container, or service replica. In modern environments, teams often use more specific terms like **host**, **container**, **pod**, or **service instance** instead of the generic word _instance_.

**Pods**  
A pod is a Kubernetes unit that runs one or more containers together. In Kubernetes environments, pods are often the runtime unit engineers inspect when troubleshooting crashes, restarts, or service failures.

## Frontend Access vs. Backend and DevOps Access

In some organizations, the frontend team does not have direct access to Kubernetes or deeper server-side infrastructure. That is normal. They may still have access to tools like **Jenkins** for build and deployment visibility, and **Datadog** for observability.

Through those tools, the frontend team may be able to find useful identifiers such as:

- a **pod name**
    
- a **host**
    
- a **service name**
    
- a **trace ID**
    
- a **request ID**
    
- or another runtime tag that helps narrow down an issue
    

Because of that, the frontend team can often see:

- which frontend build or deployment went out
    
- whether API calls are failing in the browser
    
- HTTP errors such as **500, 502, 503, and 504**
    
- timeouts, CORS failures, bad gateway errors, or upstream connection failures
    
- request IDs, trace IDs, pod names, hostnames, service names, or related tags in Datadog, if those are exposed
    

But they may **not** have access to:

- Kubernetes cluster access through `kubectl`
    
- pod or container logs
    
- server console output
    
- backend application logs
    
- infrastructure-level error details
    
- secrets, configs, or shell access into running systems
    

So from the frontend side, the team may only be able to say something like:

> This API request is failing on pod X, host Y, or trace Z, but the browser response does not contain enough detail to explain the root cause.

In other words, the frontend team can often confirm that a request is failing and may be able to point to a pod, host, service, request ID, or trace ID. But they usually do not have the backend log access needed to determine the exact cause.

`host Y` in Datadog does **not** usually map to a 1-to-1 record in Jenkins. Jenkins is mainly used to show build and deployment activity. The connection is indirect: Jenkins can help show **which version was deployed** and **when it was deployed**, while Datadog can help show **where the failing workload is running**.

If the environment is Kubernetes-based, then `host Y` in Datadog may refer to:

- the **Kubernetes node hostname**
    
- the **container host**
    
- the **underlying VM or cloud host**
    
- or a Datadog infrastructure tag associated with the workload
    

At that point, the backend, DevOps, platform, or SRE team usually takes over, because they have the deeper access needed to inspect:

- server logs
    
- backend application error logs
    
- pod and container logs
    
- Kubernetes events
    
- deployments, rollouts, crashes, and restarts
    
- infrastructure health, networking, and configuration issues

## Example Scenario

Datadog is not properly showing information about our website, and I am on the frontend team. I may be able to tell that telemetry is missing or incomplete, but I usually cannot see the server-side setup that controls Datadog ingestion, agents, log forwarding, trace configuration, environment tags, or service mapping.

A likely conclusion would be:

> Datadog is missing or incomplete for this website. From the frontend side, I can see the visibility gap, but I cannot verify the server-side configuration. This likely needs the backend or DevOps team to check whether the Datadog integration, agent setup, logging, tracing, or environment tagging was configured correctly.
