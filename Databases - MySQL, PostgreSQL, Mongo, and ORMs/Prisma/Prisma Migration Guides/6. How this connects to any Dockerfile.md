A common production Docker workflow looks like this:

```dockerfile
# syntax=docker/dockerfile:1

FROM node:22-alpine
WORKDIR /app

RUN apk add --no-cache libc6-compat openssl

COPY package.json package-lock.json* ./
RUN npm ci

COPY . .
RUN npx prisma generate && npm run build

ENV NODE_ENV=production
ENV NEXT_TELEMETRY_DISABLED=1
EXPOSE 3000

CMD ["sh", "-c", "npx prisma migrate deploy && npm run start"]
```

## What it means

### During build

```bash
RUN npx prisma generate && npm run build
```

**Breakdown:**
`npx prisma generate` reads your `schema.prisma` file and generates the Prisma Client for your project based on your models and configured database provider. This is a code generation step, not a database migration step.

That matters because the application build may import Prisma Client or depend on its generated types. Running `prisma generate` first makes sure the Prisma Client already exists before `npm run build` compiles the app.

In other words:

- `prisma generate` = create the Prisma Client code for this schema
- `npm run build` = build the application using that generated client

This makes sure Prisma Client exists before the application is built.

### At runtime

```bash
npx prisma migrate deploy && npm run start
```

**Breakdown:**
`npx prisma migrate deploy` applies any existing committed Prisma migration files to the target database in production or deployment environments. It does **not** create new migrations. It only runs the migrations that already exist in your `prisma/migrations` folder so the database schema matches what the application expects.

After that, `npm run start` launches the application. At that point, when the app creates and uses Prisma Client, Prisma connects to the actual database using the configured connection string such as `DATABASE_URL`.

In other words:

- `prisma migrate deploy` = bring the production database schema up to date
- `npm run start` = start the app against that updated database

This helps prevent cases where the app starts with code that expects tables, columns, or relations that do not exist yet in the database.

That is the normal production pattern.

## Important distinction

A team member should clearly understand:

- `migrate dev` = create and apply migrations during development
    
- `migrate deploy` = apply already-created migrations in staging or production
    
- `db seed` = explicitly populate baseline data
