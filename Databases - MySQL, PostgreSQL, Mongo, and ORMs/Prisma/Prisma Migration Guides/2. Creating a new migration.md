
This is the workflow for someone actively changing the schema.

## Step 1: Install dependencies

```bash
npm install
```

Many teams use `npm ci` inside containers for reproducible installs, while local development may use either `npm install` or `npm ci` depending on workflow.

## Step 2: Make sure the environment is configured

Prisma needs a database connection string.

Example:

```env
DATABASE_URL="postgresql://user:password@localhost:5432/appdb"
```

If the seed script also depends on environment variables, those should be available too before running Prisma commands.

## Step 3: Edit `prisma/schema.prisma`

Example: add a nullable `bio` field to `User`.

```prisma
model User {
  id           String  @id @default(cuid())
  email        String  @unique
  passwordHash String
  role         UserRole
  name         String
  bio          String?
}
```

## Step 4: Create and apply the migration

```bash
npx prisma migrate dev --name add_user_bio
```

What this does:

- compares the current schema with migration history
    
- creates a new migration folder under `prisma/migrations/`
    
- writes SQL into `migration.sql`
    
- applies the migration to the development database
    
- regenerates Prisma Client
    

After that, the structure may look like this:

```text
prisma/
├── migrations
│   ├── 20250321000000_initial_setup
│   ├── 20250321120000_add_external_reference
│   ├── 20250322100000_add_record_metadata
│   └── 20260325183000_add_user_bio
│       └── migration.sql
```

## Step 5: Review the SQL

Open the generated file:

```text
prisma/migrations/20260325183000_add_user_bio/migration.sql
```

Reviewing the SQL helps confirm the change is safe and correct before it is merged.

## Step 6: Regenerate Prisma Client if needed

Often `migrate dev` already does this, but if needed:

```bash
npx prisma generate
```

## Step 7: Update seed files if necessary

If the schema change affects required seed data, update:

- `prisma/seed.ts`
    
- `prisma/seed-data.ts`
    

Example: if a new required field is added to a model, the seed definitions and any `upsert()` logic may need to be updated.

## Step 8: Commit all related changes

That usually includes:

- `prisma/schema.prisma`
    
- the new migration folder
    
- any seed changes
    
- any application code using the new field
