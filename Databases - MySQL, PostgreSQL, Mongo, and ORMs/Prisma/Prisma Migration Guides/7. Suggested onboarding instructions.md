
## Onboarding
### Local setup

```bash
npm install
cp .env.example .env
npx prisma generate
npx prisma migrate dev
npx prisma db seed
npm run dev
```

### When changing the schema

```bash
# 1. edit prisma/schema.prisma

# 2. create a migration
npx prisma migrate dev --name your_change_name

# 3. reseed if needed
npx prisma db seed
```

### When deploying

Do not edit the schema on the server.

Only run:

```bash
npx prisma migrate deploy
```

## Related: Clean version to hand to the team

### Prisma in this project

This project uses Prisma with a migration-based workflow.

- `prisma/schema.prisma` defines the database structure
    
- `prisma/migrations/` stores the history of schema changes
    
- `prisma/seed.ts` inserts baseline data
    
- `prisma/seed-data.ts` provides reusable seed definitions
    

### For a new developer setting up locally

```bash
npm install
npx prisma generate
npx prisma migrate dev
npx prisma db seed
```

This will:

- generate Prisma Client
    
- apply existing migrations to the local database
    
- seed default data
    

### For a developer making a schema change

1. Edit `prisma/schema.prisma`
    
2. Create a migration:
    

```bash
npx prisma migrate dev --name describe_change
```

3. Review the generated SQL
    
4. Update seed files if needed
    
5. Commit both the schema change and the migration folder
    

### For staging or production

Use:

```bash
npx prisma migrate deploy
```

That applies only pending committed migrations and matches the expected production workflow.

## Related: Common mistakes to warn new team members about

### 1. Using `migrate deploy` when a new migration is needed

`migrate deploy` only applies existing migrations.

It does not create a new migration.

### 2. Editing the database manually without creating a migration

That can create drift between:

- `schema.prisma`
    
- migration history
    
- the actual database
    

### 3. Forgetting to commit the migration folder

Changing only `schema.prisma` is not enough in a team project.

The migration directory must also be committed.

### 4. Assuming seed data runs automatically

Seeding should be treated as an explicit step:

```bash
npx prisma db seed
```

### 5. Forgetting to update seeds after adding required fields

If a required field is added to a model, the seed files may fail until they are updated too.
