This project uses Prisma in a standard migration-driven setup:

- `schema.prisma` defines the database structure
    
- `migrations/` stores versioned SQL changes
    
- `seed.ts` inserts baseline data
    
- `seed-data.ts` holds reusable seed definitions
    
- the container startup process runs `prisma migrate deploy` before starting the app
    

That setup reflects a common Prisma workflow: create migrations during development, then apply already-committed migrations in staging or production.

## What each file does

### `prisma/schema.prisma`

This is the main Prisma schema file.

It defines:

- models
    
- enums
    
- relations
    
- datasource settings
	- Tells prisma if this is postgresql, mysql, mongodb, etc
	- Example:
```
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```
    
- generator settings
    

When the database structure needs to change, this is usually the first file to edit.

### `prisma/migrations/`

Each subfolder represents one migration.

A migration is a saved database change, usually generated after updating `schema.prisma`.

Examples:

- `20250321000000_initial_setup`
    
- `20250321120000_add_external_reference`
    
- `20250322100000_add_record_metadata`
    

These names suggest a progression like:

- initial schema setup
    
- a later change to add an external reference field
    
- another later change to add metadata fields
    

Prisma uses this migration history to track what has already been applied to a database.

### `prisma/migration_lock.toml`

This is Prisma’s migration lock file.

It helps Prisma track migration state and should normally stay committed in the repository.

### `prisma/seed-data.ts`

This is a project seed-data module.

It stores reusable structured seed objects separately from the database write logic. That keeps larger seed datasets organized and easier to maintain.

### `prisma/seed.ts`

This is the seed runner.

It typically:

- creates a `PrismaClient`
    
- loads environment variables
    
- hashes any seed passwords
    
- inserts or updates default records
    
- calls `upsert()` so seeding can be rerun without creating duplicates
    

That makes onboarding easier because a developer can seed a fresh local database repeatedly without having to manually clean it up each time.

## How to think about Prisma in this setup

A helpful mental model is:

### In development

You edit `schema.prisma`, then create a migration.

Typical command:

```bash
npx prisma migrate dev --name your_change_name
```

This is the development workflow.

### In staging or production

You do not create new migrations there.

You only apply migrations that were already created and committed:

```bash
npx prisma migrate deploy
```

### For baseline data

You run the seed script explicitly:

```bash
npx prisma db seed
```
