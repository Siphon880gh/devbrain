### 1. Connect your AWS account to Datadog first

Start with the **Datadog AWS integration**, because this is what lets Datadog read AWS service-level data from your account. Datadog’s current getting-started flow recommends setting this up with its **CloudFormation template**. If you prefer manual setup, Datadog’s manual guide uses an **IAM role + external ID** so Datadog can assume a role into your AWS account and pull data safely. ([Datadog Monitoring](https://docs.datadoghq.com/getting_started/integrations/aws/ "Getting Started with AWS"))

At this stage, enable the AWS services you care about and confirm the account is connected. Datadog notes that AWS metrics may take **up to about 10 minutes** to begin appearing after setup. ([Datadog Monitoring](https://docs.datadoghq.com/integrations/guide/aws-manual-setup/ "AWS Manual Setup Guide"))

### 2. Deploy the Datadog Agent into Amazon EKS

For EKS, you have two normal choices:
- **Helm** install
- **Datadog Operator / EKS add-on**

Both are supported. The simplest portable approach for many teams is Helm, while AWS users who want a more EKS-native workflow may choose the **Datadog Operator as an Amazon EKS add-on**. ([Datadog Monitoring](https://docs.datadoghq.com/containers/kubernetes/installation/ "Install the Datadog Agent on Kubernetes"))

A standard Helm-based install looks like this:

```bash
helm repo add datadog https://helm.datadoghq.com
helm repo update
kubectl create secret generic datadog-secret --from-literal api-key=<DATADOG_API_KEY>
```

Create a values file like this:

```yaml
datadog:
  apiKeyExistingSecret: datadog-secret
  clusterName: <CLUSTER_NAME>
  site: <DATADOG_SITE>
```

Then install:

```bash
helm install datadog-agent -f datadog-values.yaml datadog/datadog
```

That flow is straight from Datadog’s Kubernetes install docs. ([Datadog Monitoring](https://docs.datadoghq.com/containers/kubernetes/installation/ "Install the Datadog Agent on Kubernetes"))

If you use the Operator path instead, Datadog’s Kubernetes docs use a `DatadogAgent` custom resource and the EKS add-on docs show installing the Operator add-on first, then applying the `DatadogAgent` manifest. ([Datadog Monitoring](https://docs.datadoghq.com/containers/guide/operator-eks-addon/ "Installing the Datadog Agent on Amazon EKS with the Datadog Operator add-on"))

After deployment, verify that Agent pods show up in Datadog’s Containers view. Datadog says they should appear within a few minutes. ([Datadog Monitoring](https://docs.datadoghq.com/containers/kubernetes/installation/ "Install the Datadog Agent on Kubernetes"))

### 3. Watch for the EKS Fargate caveat

If any of your workloads run on **EKS with Fargate**, do not assume the normal node Agent setup covers everything the same way it does on EC2-backed worker nodes. Datadog calls out separate Fargate guidance, and its EKS add-on docs note important differences for Fargate-based collection. ([Datadog Monitoring](https://docs.datadoghq.com/containers/guide/operator-eks-addon/ "Installing the Datadog Agent on Amazon EKS with the Datadog Operator add-on"))

### 4. Decide how AWS logs and metrics should reach Datadog

For **AWS service logs**, Datadog documents two main paths:

- **Amazon Data Firehose destination** — recommended for **high-volume CloudWatch logs**
    
- **Datadog Forwarder Lambda** — useful for **S3 logs**, and required in some Lambda-related cases such as traces, enhanced metrics, or custom metrics from Lambda functions ([Datadog Monitoring](https://docs.datadoghq.com/getting_started/integrations/aws/ "Getting Started with AWS"))
    

For **AWS metrics**, Datadog supports:

- **Default API polling** through the AWS integration, which pulls CloudWatch metrics about every **10 minutes on average**
    
- **CloudWatch Metric Streams + Amazon Data Firehose**, which lowers metric latency to roughly **2–3 minutes** and requires separate setup ([Datadog Monitoring](https://docs.datadoghq.com/integrations/amazon-web-services/ "Amazon Web Services"))
    

A good starting pattern is:

- use **default polling** first
    
- add **Metric Streams** later if you need faster AWS metric updates
    
- use **Firehose** for heavy CloudWatch log volume
    
- use the **Forwarder Lambda** where Firehose is not the right fit ([Datadog Monitoring](https://docs.datadoghq.com/getting_started/integrations/aws/ "Getting Started with AWS"))
    

### 5. Integrate Jenkins with Datadog

Install the **Datadog Plugin** in Jenkins under:

`Manage Jenkins -> Manage Plugins`

Then configure it under:

`Manage Jenkins -> Configure System -> Datadog Plugin` ([Datadog Monitoring](https://docs.datadoghq.com/continuous_integration/pipelines/jenkins/ "Jenkins Setup for CI Visibility"))

You have two reporting modes:

#### Option A: report through a Datadog Agent

This is the mode Datadog labels as **recommended**. In Jenkins, set the Agent host, set the traces collection port if needed (default is **8126**), and enable **CI Visibility**. You can also enable optional Jenkins log collection if your Agent is configured for it. ([Datadog Monitoring](https://docs.datadoghq.com/continuous_integration/pipelines/jenkins/ "Jenkins Setup for CI Visibility"))

#### Option B: agentless reporting

If Jenkins cannot easily reach a Datadog Agent, Jenkins can report directly to Datadog using your **Datadog site** and **API key**. In that mode, choose the correct site, enter the API key, and enable **CI Visibility**. ([Datadog Monitoring](https://docs.datadoghq.com/continuous_integration/pipelines/jenkins/ "Jenkins Setup for CI Visibility"))

Datadog’s Jenkins docs also state that the plugin forwards **metrics, events, and service checks**, and CI Visibility adds pipeline execution visibility. ([Datadog Monitoring](https://docs.datadoghq.com/integrations/jenkins/ "Jenkins"))

### 6. What to monitor first

Once the plumbing is in place, the first useful monitors are usually:

- Jenkins build failures
    
- unusually long build durations
    
- EKS pod restarts / crash loops
    
- node resource pressure
    
- high container CPU or memory
    
- missing Agent data from nodes or Jenkins ([Datadog Monitoring](https://docs.datadoghq.com/continuous_integration/pipelines/jenkins/ "Jenkins Setup for CI Visibility"))
    

That monitor list is a practical recommendation based on the telemetry these integrations expose.

## Recommended implementation order

For most teams, this order is the cleanest:

1. **Connect AWS account to Datadog**
    
2. **Deploy Datadog Agent to EKS**
    
3. **Choose log path**: Firehose or Forwarder Lambda
    
4. **Add Jenkins Datadog plugin**
    
5. **Enable CI Visibility**
    
6. **Create dashboards and monitors** ([Datadog Monitoring](https://docs.datadoghq.com/getting_started/integrations/aws/ "Getting Started with AWS"))
    

## Bottom line

Your original text is directionally correct, but I would not describe the setup as only “install Agent on EKS + configure Jenkins plugin.” A more complete and accurate Amazon implementation is:

- **AWS integration** for AWS-native service data
    
- **Datadog Agent in EKS** for Kubernetes and container telemetry
    
- **Jenkins Datadog plugin** for CI/CD metrics and CI Visibility
    
- **Firehose and/or Forwarder Lambda** for AWS logs
    
- optional **Metric Streams** for lower-latency AWS metrics ([Datadog Monitoring](https://docs.datadoghq.com/integrations/amazon-web-services/ "Amazon Web Services"))
