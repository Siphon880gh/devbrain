# Aurora Basics

MySQL and PostgreSQL-compatible relational database with up to 5x throughput, automatic storage scaling, and a serverless option.

## What You'll Learn

- What makes Aurora fundamentally different from standard RDS MySQL/PostgreSQL
- How Aurora's storage architecture works (6 copies, 3 AZs)
- Aurora Serverless v2 and when it makes sense
- How storage auto-scaling works and what it costs
- When to choose Aurora over plain RDS (and when not to)
- Pricing differences and why Aurora has no free tier

## What Makes Aurora Different from RDS

Aurora runs MySQL and PostgreSQL — your application code, SQL queries, and tools all work the same. The difference is under the hood. AWS rebuilt the storage layer from scratch to be faster, more durable, and more scalable.

Here's what changes:

| Feature | Standard RDS | Aurora |
|---|---|---|
| **Storage** | EBS volume attached to one instance | Distributed across 3 AZs, 6 copies |
| **Max storage** | 64 TB | 128 TB, auto-grows |
| **Replication** | Asynchronous (read replicas lag) | Synchronous within storage layer |
| **Failover time** | 60-120 seconds | ~30 seconds (often under 15s) |
| **Read replicas** | Up to 5 | Up to 15 |
| **Throughput** | Baseline MySQL/PostgreSQL | Up to 5x MySQL, 3x PostgreSQL |
| **Backtrack** | Not available | Rewind database to a point in time without restoring |

The key insight: with standard RDS, your database writes to a single EBS volume. If that volume fails, you rely on a snapshot to recover. With Aurora, every write is replicated 6 times across 3 Availability Zones as it happens. Aurora can lose 2 copies and still handle writes, or lose 3 copies and still handle reads.

## Aurora Architecture

Aurora separates the **compute layer** (the database instances that process your queries) from the **storage layer** (a shared, distributed cluster volume).

```
┌─────────────────┐     ┌─────────────────┐
│  Writer Instance │     │  Reader Instance │  (up to 15 readers)
│  (Primary)       │     │  (Read Replica)  │
└────────┬────────┘     └────────┬────────┘
         │                       │
         └───────────┬───────────┘
                     │
         ┌───────────▼───────────┐
         │  Shared Storage Layer  │
         │  (Aurora Cluster Vol.) │
         │                       │
         │  ┌─────┐ ┌─────┐     │
         │  │Copy1│ │Copy2│     │  AZ-a
         │  └─────┘ └─────┘     │
         │  ┌─────┐ ┌─────┐     │
         │  │Copy3│ │Copy4│     │  AZ-b
         │  └─────┘ └─────┘     │
         │  ┌─────┐ ┌─────┐     │
         │  │Copy5│ │Copy6│     │  AZ-c
         │  └─────┘ └─────┘     │
         └───────────────────────┘
```

This architecture means:
- **Adding a read replica doesn't duplicate the storage** — all replicas share the same cluster volume
- **Replica lag is minimal** (typically under 20ms) because replicas read from the same shared storage
- **Failover is fast** because a reader can be promoted to writer without copying any data

### Cluster Endpoints

Aurora gives you two DNS endpoints:

- **Cluster endpoint** (writer): Points to the current primary instance. Use this for all writes and reads that need the latest data.
- **Reader endpoint**: Load-balances across all read replicas. Use this for read-heavy queries that can tolerate a few milliseconds of lag.

```
Writer: my-cluster.cluster-abc123.us-east-1.rds.amazonaws.com
Reader: my-cluster.cluster-ro-abc123.us-east-1.rds.amazonaws.com
```

Your application connects to these endpoints, and Aurora routes traffic automatically — even during failover.

## Aurora Serverless v2

Standard Aurora requires you to pick an instance size (e.g., db.r6g.large). Aurora Serverless v2 scales compute capacity automatically based on your workload.

**How it works:**
- You set a **minimum** and **maximum** ACU (Aurora Capacity Units)
- 1 ACU ≈ 2 GB of memory
- Minimum is 0.5 ACU; maximum can go up to 256 ACU
- Aurora scales up in seconds when traffic increases and scales back down when traffic drops

**When Serverless v2 makes sense:**
- Development and staging environments (scales down when no one's using it)
- Applications with variable traffic (busy during business hours, quiet at night)
- New applications where you don't know the traffic pattern yet

**When it doesn't make sense:**
- Applications with constant, predictable high traffic (provisioned is cheaper)
- Extremely latency-sensitive workloads (scaling events add a brief delay)

```bash
# Create an Aurora Serverless v2 cluster (PostgreSQL)
aws rds create-db-cluster \
  --db-cluster-identifier my-serverless-cluster \
  --engine aurora-postgresql \
  --engine-version 15.4 \
  --master-username postgres \
  --master-user-password YourSecurePassword123 \
  --serverless-v2-scaling-configuration MinCapacity=0.5,MaxCapacity=8

# Add a Serverless v2 instance to the cluster
aws rds create-db-instance \
  --db-instance-identifier my-serverless-instance \
  --db-cluster-identifier my-serverless-cluster \
  --db-instance-class db.serverless \
  --engine aurora-postgresql
```

**"Scales to zero-ish":** Aurora Serverless v2 can scale down to 0.5 ACU but cannot scale to true zero. At 0.5 ACU, you're paying about $44/month even with no traffic. AWS has mentioned Aurora Serverless v2 scaling to zero as a future capability, but it's not available as of this writing.

## Storage Auto-Scaling

Aurora storage grows automatically as you add data:

- Starts at 10 GB
- Grows in 10 GB increments up to 128 TB
- You **never** need to pre-allocate storage or worry about running out of disk space
- You only pay for what you use

This is a real operational advantage over standard RDS, where you have to set a storage size and monitor it (or enable storage auto-scaling, which has a lag).

One important detail: **Aurora storage does not shrink** when you delete data. If your database grows to 50 GB and you delete half the data, you still pay for 50 GB. The freed space is reused for future writes, but the allocated storage doesn't decrease. This is a common surprise.

## Aurora vs Standard RDS: When to Choose Which

### Choose Standard RDS When:

- **You're on free tier** — Aurora has no free tier, RDS gives you 750 hours/month of db.t3.micro
- **Your workload fits on a small instance** — If a db.t3.small handles your traffic, Aurora's premium isn't worth paying
- **You need Oracle or SQL Server** — Aurora only supports MySQL and PostgreSQL
- **Cost is the primary concern** — Aurora is 20-30% more expensive than equivalent RDS instances
- **You're learning or prototyping** — Start with RDS, upgrade to Aurora if you outgrow it

### Choose Aurora When:

- **You need high availability** — 6 copies of data, 30-second failover, and up to 15 read replicas
- **Read replica lag matters** — Aurora replicas lag by ~20ms vs 100ms+ for standard RDS
- **You need more than 5 read replicas** — Aurora supports 15
- **Your database is growing fast** — Auto-scaling storage up to 128 TB with no downtime
- **You want fast failover** — Aurora failover completes in about 30 seconds vs 60-120 for Multi-AZ RDS
- **You need Backtrack** — The ability to rewind your database to a specific point in time without restoring from a snapshot

### Pricing Comparison

Here's a rough monthly cost comparison for a moderately sized database:

| Component | RDS MySQL (db.r6g.large) | Aurora MySQL (db.r6g.large) |
|---|---|---|
| **Instance** | ~$140/mo | ~$175/mo |
| **100 GB storage** | ~$11.50/mo (gp3) | ~$10/mo |
| **I/O** | Included in gp3 | ~$0.20 per million requests |
| **Read replica** | ~$140/mo (full instance cost) | ~$175/mo |
| **Multi-AZ** | ~$280/mo (doubles instance cost) | Built into storage (6 copies) |
| **Rough total** | ~$150-300/mo | ~$185-360/mo |

Aurora's I/O charges can add up on write-heavy workloads. In January 2024, AWS introduced **Aurora I/O-Optimized**, which eliminates I/O charges for an additional 30% on compute cost — worth it if your I/O bill exceeds 25% of your database cost.

## Hands-On: Create an Aurora Serverless Cluster

**Step 1:** Create the cluster:

```bash
aws rds create-db-cluster \
  --db-cluster-identifier practice-aurora \
  --engine aurora-postgresql \
  --engine-version 15.4 \
  --master-username postgres \
  --master-user-password PracticePass2024 \
  --serverless-v2-scaling-configuration MinCapacity=0.5,MaxCapacity=4
```

**Step 2:** Add an instance:

```bash
aws rds create-db-instance \
  --db-instance-identifier practice-aurora-instance \
  --db-cluster-identifier practice-aurora \
  --db-instance-class db.serverless \
  --engine aurora-postgresql
```

**Step 3:** Wait for it to be available:

```bash
aws rds wait db-instance-available \
  --db-instance-identifier practice-aurora-instance
```

**Step 4:** Get the cluster endpoint:

```bash
aws rds describe-db-clusters \
  --db-cluster-identifier practice-aurora \
  --query 'DBClusters[0].Endpoint' \
  --output text
```

**Step 5:** Connect (same as any PostgreSQL connection):

```bash
psql -h <cluster-endpoint> -U postgres -d postgres
```

**Step 6:** Clean up (delete instance first, then cluster):

```bash
aws rds delete-db-instance \
  --db-instance-identifier practice-aurora-instance \
  --skip-final-snapshot

aws rds wait db-instance-deleted \
  --db-instance-identifier practice-aurora-instance

aws rds delete-db-cluster \
  --db-cluster-identifier practice-aurora \
  --skip-final-snapshot
```

**Cost warning:** Even at minimum capacity (0.5 ACU), Aurora Serverless v2 costs about $44/month. Delete your practice cluster immediately after experimenting.

## Key Takeaways

- Aurora is MySQL/PostgreSQL-compatible — same SQL, same tools, redesigned storage layer
- Data is replicated **6 times across 3 AZs** automatically — no Multi-AZ checkbox needed
- Aurora supports **up to 15 read replicas** with very low replication lag (~20ms)
- **Serverless v2** auto-scales compute capacity but doesn't scale to true zero (minimum 0.5 ACU ≈ $44/month)
- Storage auto-grows up to 128 TB but **does not shrink** when you delete data
- **Aurora has no free tier** — start with standard RDS for learning, upgrade to Aurora when your workload demands it
- Choose Aurora over RDS when you need higher availability, faster failover, more read replicas, or auto-scaling storage
