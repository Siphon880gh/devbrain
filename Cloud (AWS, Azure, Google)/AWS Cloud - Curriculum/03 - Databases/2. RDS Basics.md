# RDS Basics

Managed relational databases on AWS — MySQL, PostgreSQL, and MariaDB with automated backups, patching, and scaling.

## What You'll Learn

- What RDS manages for you vs what you still handle
- The six supported database engines and when to use each
- How to launch an RDS instance via the console and CLI
- How to connect to your database from EC2 or your local machine
- Multi-AZ deployments, read replicas, and backup strategies
- Free tier limits and how to avoid unexpected charges

## What RDS Manages for You

Running a database on your own EC2 instance means you handle everything: installing the database software, applying security patches, setting up backups, configuring replication, and monitoring disk space. RDS takes all of that off your plate.

**AWS handles:**
- Provisioning the underlying server (you never SSH into the database host)
- Operating system and database engine patching
- Automated daily backups with point-in-time recovery
- Automatic failover to a standby instance (if Multi-AZ is enabled)
- Monitoring via CloudWatch (CPU, memory, connections, disk I/O)
- Storage scaling (if auto-scaling is enabled)

**You still handle:**
- Schema design (creating tables, indexes, relationships)
- Query optimization (slow queries are still your problem)
- Application-level connection management
- Security group rules (who can connect)
- Choosing the right instance size

Think of RDS like renting a managed apartment vs owning a house. You don't fix the plumbing or roof, but you still arrange your own furniture.

## Supported Engines

RDS supports six database engines. They all use the same RDS management layer but differ in SQL dialect, features, and licensing costs.

| Engine | Best For | License Cost | Notes |
|---|---|---|---|
| **MySQL** | General purpose web apps | Free (open source) | Most popular, huge community |
| **PostgreSQL** | Complex queries, GIS data, JSON | Free (open source) | More features than MySQL, stricter SQL compliance |
| **MariaDB** | MySQL alternative | Free (open source) | Fork of MySQL, community-driven |
| **Oracle** | Enterprise apps already on Oracle | License required | Bring-your-own-license or pay included |
| **SQL Server** | .NET/Windows shops | License included | Express edition available for free tier |
| **Aurora (MySQL)** | High-performance MySQL | AWS pricing | Covered in the Aurora lesson |
| **Aurora (PostgreSQL)** | High-performance PostgreSQL | AWS pricing | Covered in the Aurora lesson |

**For beginners:** Start with **PostgreSQL** or **MySQL**. Both are free, well-documented, and supported by every programming language. PostgreSQL is the more modern choice; MySQL has a larger install base.

## Creating an RDS Instance

### Via the AWS Console

1. Go to the **RDS Dashboard** → **Create database**
2. Choose **Standard create** (not Easy create — you want to see all options)
3. Select your engine (e.g., PostgreSQL)
4. Under **Templates**, choose **Free tier** — this locks the instance to db.t3.micro
5. Set your **DB instance identifier** (a name like `my-app-db`)
6. Set **Master username** (default is `postgres`) and a **Master password**
7. Under **Connectivity**, choose your VPC and whether the database should be publicly accessible
8. Leave storage at the default 20 GB with auto-scaling enabled
9. Click **Create database** and wait 5-10 minutes

### Via the AWS CLI

This single command creates a free-tier-eligible PostgreSQL instance:

```bash
aws rds create-db-instance \
  --db-instance-identifier my-app-db \
  --db-instance-class db.t3.micro \
  --engine postgres \
  --engine-version 15.4 \
  --master-username postgres \
  --master-user-password YourSecurePassword123 \
  --allocated-storage 20 \
  --storage-type gp3 \
  --no-multi-az \
  --backup-retention-period 7 \
  --publicly-accessible
```

Check the status (it takes a few minutes to become "available"):

```bash
aws rds describe-db-instances \
  --db-instance-identifier my-app-db \
  --query 'DBInstances[0].DBInstanceStatus'
```

Get the endpoint once it's ready:

```bash
aws rds describe-db-instances \
  --db-instance-identifier my-app-db \
  --query 'DBInstances[0].Endpoint.Address' \
  --output text
```

This returns something like `my-app-db.abc123xyz.us-east-1.rds.amazonaws.com`.

### Connecting to Your Database

**From an EC2 instance in the same VPC** (recommended for production):

```bash
psql -h my-app-db.abc123xyz.us-east-1.rds.amazonaws.com \
     -U postgres \
     -d postgres
```

**From your local machine** (requires the instance to be publicly accessible and your IP in the security group):

```bash
psql -h my-app-db.abc123xyz.us-east-1.rds.amazonaws.com \
     -U postgres \
     -d postgres \
     -p 5432
```

For MySQL, replace `psql` with `mysql`:

```bash
mysql -h my-app-db.abc123xyz.us-east-1.rds.amazonaws.com \
      -u admin \
      -p
```

If the connection hangs or times out, it's almost always a **security group issue** — covered in lesson 5 of this module.

## Backups, Snapshots, and Restore

RDS gives you two backup mechanisms:

### Automated Backups
- Enabled by default with a 7-day retention period (configurable up to 35 days)
- Taken during your preferred backup window (a 30-minute window you choose)
- Support **point-in-time recovery** — you can restore to any second within the retention period
- Transaction logs are uploaded to S3 every 5 minutes
- Stored in S3 (you don't see them in your S3 console — they're managed by RDS)

### Manual Snapshots
- You trigger these yourself, and they persist until you delete them
- Not subject to the retention period — they stick around forever
- Use these before major changes (schema migrations, version upgrades)

```bash
# Create a manual snapshot
aws rds create-db-snapshot \
  --db-instance-identifier my-app-db \
  --db-snapshot-identifier my-app-db-before-migration

# List your snapshots
aws rds describe-db-snapshots \
  --db-instance-identifier my-app-db \
  --query 'DBSnapshots[*].[DBSnapshotIdentifier,Status,SnapshotCreateTime]' \
  --output table
```

### Restoring from a Snapshot

Restoring creates a **new** RDS instance — it doesn't overwrite the existing one.

```bash
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier my-app-db-restored \
  --db-snapshot-identifier my-app-db-before-migration
```

After restoring, update your application's connection string to point to the new instance, then delete the old one when you're confident.

## Multi-AZ and Read Replicas

These are two different features that solve different problems.

### Multi-AZ (High Availability)

Multi-AZ creates a **standby copy** of your database in a different Availability Zone. If the primary instance fails (hardware issue, AZ outage), AWS automatically fails over to the standby — typically within 60-120 seconds.

- The standby is **not** readable — it's only there for failover
- Your application connects to the same DNS endpoint; the failover is transparent
- Doubles your cost (you're paying for two instances)
- Enable it for any production database

```bash
# Enable Multi-AZ on an existing instance
aws rds modify-db-instance \
  --db-instance-identifier my-app-db \
  --multi-az \
  --apply-immediately
```

### Read Replicas (Performance)

Read replicas are **readable copies** of your database that handle SELECT queries. Your application writes to the primary and reads from the replica(s).

- You can create up to 15 read replicas (for Aurora) or 5 (for standard RDS)
- Replication is asynchronous — there's a small delay (usually under a second)
- Each replica has its own endpoint
- Great for read-heavy workloads (e.g., 90% reads, 10% writes)

```bash
# Create a read replica
aws rds create-db-instance-read-replica \
  --db-instance-identifier my-app-db-read1 \
  --source-db-instance-identifier my-app-db
```

| Feature | Multi-AZ | Read Replica |
|---|---|---|
| **Purpose** | High availability (failover) | Performance (read scaling) |
| **Readable?** | No | Yes |
| **Automatic failover?** | Yes | No (manual promotion) |
| **Replication** | Synchronous | Asynchronous |
| **Cross-region?** | No (same region) | Yes |

## Hands-On: Launch a PostgreSQL RDS Instance

Follow this end-to-end exercise to get a working database:

**Step 1:** Create the instance (free tier eligible):

```bash
aws rds create-db-instance \
  --db-instance-identifier practice-db \
  --db-instance-class db.t3.micro \
  --engine postgres \
  --engine-version 15.4 \
  --master-username postgres \
  --master-user-password MyPractice2024 \
  --allocated-storage 20 \
  --storage-type gp3 \
  --no-multi-az \
  --publicly-accessible \
  --backup-retention-period 7
```

**Step 2:** Wait for it to become available (5-10 minutes):

```bash
aws rds wait db-instance-available \
  --db-instance-identifier practice-db
```

**Step 3:** Get the endpoint:

```bash
aws rds describe-db-instances \
  --db-instance-identifier practice-db \
  --query 'DBInstances[0].Endpoint.Address' \
  --output text
```

**Step 4:** Connect and create a table:

```bash
psql -h <your-endpoint> -U postgres -d postgres
```

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  email VARCHAR(255) UNIQUE NOT NULL,
  name VARCHAR(100),
  created_at TIMESTAMP DEFAULT NOW()
);

INSERT INTO users (email, name) VALUES ('jane@example.com', 'Jane Doe');
INSERT INTO users (email, name) VALUES ('john@example.com', 'John Smith');

SELECT * FROM users;
```

**Step 5:** Clean up when done (to avoid charges):

```bash
aws rds delete-db-instance \
  --db-instance-identifier practice-db \
  --skip-final-snapshot
```

## Key Takeaways

- RDS handles patching, backups, and failover — you handle schema design, queries, and security groups
- Start with **PostgreSQL** or **MySQL** on a **db.t3.micro** to stay in free tier (750 hours/month for 12 months)
- **Automated backups** give you point-in-time recovery; **manual snapshots** persist until you delete them
- **Multi-AZ** is for high availability (automatic failover), **read replicas** are for read performance
- Always delete practice instances when done — a forgotten db.t3.micro costs about $13/month after free tier expires
- If connections time out, check your security group rules first
