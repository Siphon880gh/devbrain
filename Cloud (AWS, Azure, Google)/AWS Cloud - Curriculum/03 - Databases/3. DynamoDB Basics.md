# DynamoDB Basics

AWS's fully managed NoSQL database — key-value and document storage with single-digit millisecond performance at any scale.

## What You'll Learn

- Core NoSQL concepts: tables, items, attributes, and keys
- The difference between partition keys and sort keys
- On-demand vs provisioned capacity and when to use each
- How to create tables and perform CRUD operations with the CLI and Python (boto3)
- Query vs Scan and why it matters for performance
- Global and Local Secondary Indexes
- Free tier details and using DynamoDB Local for development

## How DynamoDB Works

DynamoDB is fundamentally different from a relational database. There are no servers to manage, no SQL JOINs, and no schema to define upfront. You create a table, define a primary key, and start writing items immediately.

Behind the scenes, AWS spreads your data across many servers automatically. You never see these servers, and you never worry about disk space, replication, or failover. DynamoDB handles all of it.

The trade-off: DynamoDB is incredibly fast for simple access patterns (get item by ID, query items by category) but **terrible** for ad-hoc queries like "find all users who signed up last month and ordered more than 3 items." If you need complex queries, use a relational database.

## Tables, Items, and Attributes

DynamoDB uses different terminology than SQL databases:

| SQL Concept | DynamoDB Equivalent |
|---|---|
| Table | Table |
| Row | Item |
| Column | Attribute |
| Schema (all rows same shape) | Schemaless (items can differ) |

A DynamoDB **table** holds **items**. Each item is a collection of **attributes** (key-value pairs). Items in the same table don't need to have the same attributes — only the primary key is required.

```json
// Item 1: a book
{
  "PK": "BOOK#978-0-13-468599-1",
  "title": "The Pragmatic Programmer",
  "author": "David Thomas",
  "pages": 352
}

// Item 2: also a book, but with different attributes
{
  "PK": "BOOK#978-1-09-813417-0",
  "title": "Clean Code",
  "author": "Robert C. Martin",
  "rating": 4.5,
  "tags": ["programming", "best-practices"]
}
```

Notice that the first item has `pages` and the second has `rating` and `tags`. Both are valid in the same table.

## Primary Keys: Partition Key and Sort Key

Every DynamoDB table must have a primary key, and there are two options:

### Simple Primary Key (Partition Key Only)

A single attribute uniquely identifies each item. Like a user ID or product SKU.

```
Table: Users
Partition Key: userId

| userId (PK) | name        | email              |
|-------------|-------------|--------------------|
| user-001    | Jane Doe    | jane@example.com   |
| user-002    | John Smith  | john@example.com   |
```

### Composite Primary Key (Partition Key + Sort Key)

Two attributes together uniquely identify each item. The partition key groups related items; the sort key orders them within the group.

```
Table: Orders
Partition Key: customerId
Sort Key: orderDate

| customerId (PK) | orderDate (SK)       | total  | status    |
|-----------------|---------------------|--------|-----------|
| cust-001        | 2024-01-15T10:30:00 | 49.99  | shipped   |
| cust-001        | 2024-02-20T14:15:00 | 129.50 | delivered |
| cust-002        | 2024-01-18T09:00:00 | 25.00  | pending   |
```

With this design, you can efficiently query "all orders for customer cust-001" or "all orders for cust-001 after January 2024."

**Rule of thumb:** If you only need to look up items by one attribute, use a simple key. If you need to look up groups of related items, use a composite key.

## Read and Write Capacity Modes

DynamoDB offers two pricing/capacity modes:

### On-Demand Mode

- Pay per request — literally per read and write operation
- No capacity planning needed
- Scales automatically to handle any traffic spike
- Costs about 5-7x more per request than provisioned mode at steady traffic
- **Best for:** New applications, unpredictable traffic, development/testing

### Provisioned Mode

- You specify how many reads/writes per second you need (RCU and WCU)
- 1 RCU = 1 strongly consistent read/sec for items up to 4 KB
- 1 WCU = 1 write/sec for items up to 1 KB
- You can enable **auto-scaling** to adjust capacity based on traffic
- Cheaper at predictable, steady workloads
- **Best for:** Production applications with predictable traffic patterns

```bash
# Create a table with on-demand mode
aws dynamodb create-table \
  --table-name Users \
  --attribute-definitions AttributeName=userId,AttributeType=S \
  --key-schema AttributeName=userId,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST

# Create a table with provisioned mode
aws dynamodb create-table \
  --table-name Orders \
  --attribute-definitions \
    AttributeName=customerId,AttributeType=S \
    AttributeName=orderDate,AttributeType=S \
  --key-schema \
    AttributeName=customerId,KeyType=HASH \
    AttributeName=orderDate,KeyType=RANGE \
  --billing-mode PROVISIONED \
  --provisioned-throughput ReadCapacityUnits=5,WriteCapacityUnits=5
```

**Free tier:** 25 GB of storage, 25 WCU, and 25 RCU of provisioned capacity. This is **always free** (not just 12 months), enough for about 200 million requests per month.

## Querying vs Scanning

This is the most important performance concept in DynamoDB.

### Query

- Retrieves items by **partition key** (required) and optionally filters by **sort key**
- Fast and efficient — DynamoDB goes directly to the right partition
- Only reads the items that match

```bash
# Query: get all orders for customer cust-001
aws dynamodb query \
  --table-name Orders \
  --key-condition-expression "customerId = :cid" \
  --expression-attribute-values '{":cid": {"S": "cust-001"}}'
```

### Scan

- Reads **every item in the entire table**, then filters
- Slow and expensive on large tables
- Consumes read capacity proportional to the entire table size

```bash
# Scan: find all orders with status "pending" (reads entire table!)
aws dynamodb scan \
  --table-name Orders \
  --filter-expression "#s = :status" \
  --expression-attribute-names '{"#s": "status"}' \
  --expression-attribute-values '{":status": {"S": "pending"}}'
```

**The rule:** Design your keys and indexes so you can always use Query instead of Scan. If you find yourself needing Scan in production, your table design probably needs rethinking.

### Secondary Indexes (GSI and LSI)

What if you need to query by an attribute that isn't your primary key? That's what indexes are for.

**Global Secondary Index (GSI):**
- A completely new primary key (different partition key and/or sort key)
- Can be added to an existing table at any time
- Has its own provisioned throughput (separate from the base table)

```bash
# Add a GSI to query orders by status
aws dynamodb update-table \
  --table-name Orders \
  --attribute-definitions AttributeName=status,AttributeType=S \
  --global-secondary-index-updates '[{
    "Create": {
      "IndexName": "StatusIndex",
      "KeySchema": [{"AttributeName": "status", "KeyType": "HASH"}],
      "Projection": {"ProjectionType": "ALL"},
      "ProvisionedThroughput": {"ReadCapacityUnits": 5, "WriteCapacityUnits": 5}
    }
  }]'
```

**Local Secondary Index (LSI):**
- Same partition key as the base table, but a different sort key
- Must be defined at table creation time (cannot be added later)
- Shares throughput with the base table

## Hands-On: Create a Table and Perform CRUD Operations

### Using the AWS CLI

**Create the table:**

```bash
aws dynamodb create-table \
  --table-name Movies \
  --attribute-definitions \
    AttributeName=year,AttributeType=N \
    AttributeName=title,AttributeType=S \
  --key-schema \
    AttributeName=year,KeyType=HASH \
    AttributeName=title,KeyType=RANGE \
  --billing-mode PAY_PER_REQUEST
```

**Put an item:**

```bash
aws dynamodb put-item \
  --table-name Movies \
  --item '{
    "year": {"N": "2023"},
    "title": {"S": "Oppenheimer"},
    "director": {"S": "Christopher Nolan"},
    "rating": {"N": "8.5"}
  }'
```

**Get an item:**

```bash
aws dynamodb get-item \
  --table-name Movies \
  --key '{
    "year": {"N": "2023"},
    "title": {"S": "Oppenheimer"}
  }'
```

**Query all movies from 2023:**

```bash
aws dynamodb query \
  --table-name Movies \
  --key-condition-expression "#y = :year" \
  --expression-attribute-names '{"#y": "year"}' \
  --expression-attribute-values '{":year": {"N": "2023"}}'
```

**Delete an item:**

```bash
aws dynamodb delete-item \
  --table-name Movies \
  --key '{
    "year": {"N": "2023"},
    "title": {"S": "Oppenheimer"}
  }'
```

### Using Python (boto3)

```python
import boto3

dynamodb = boto3.resource('dynamodb')
table = dynamodb.Table('Movies')

# Put an item
table.put_item(Item={
    'year': 2023,
    'title': 'Oppenheimer',
    'director': 'Christopher Nolan',
    'rating': 8.5
})

# Get an item
response = table.get_item(Key={
    'year': 2023,
    'title': 'Oppenheimer'
})
movie = response['Item']
print(f"{movie['title']} directed by {movie['director']}")

# Query all movies from 2023
from boto3.dynamodb.conditions import Key

response = table.query(
    KeyConditionExpression=Key('year').eq(2023)
)
for movie in response['Items']:
    print(f"  {movie['title']} - {movie.get('rating', 'N/A')}")

# Delete an item
table.delete_item(Key={
    'year': 2023,
    'title': 'Oppenheimer'
})
```

### DynamoDB Local for Development

You can run DynamoDB on your own machine for free during development. No AWS account needed.

```bash
# Pull and run the Docker image
docker run -p 8000:8000 amazon/dynamodb-local

# Use --endpoint-url to point CLI commands at your local instance
aws dynamodb create-table \
  --endpoint-url http://localhost:8000 \
  --table-name Movies \
  --attribute-definitions \
    AttributeName=year,AttributeType=N \
    AttributeName=title,AttributeType=S \
  --key-schema \
    AttributeName=year,KeyType=HASH \
    AttributeName=title,KeyType=RANGE \
  --billing-mode PAY_PER_REQUEST
```

In Python, point boto3 at the local endpoint:

```python
dynamodb = boto3.resource('dynamodb', endpoint_url='http://localhost:8000')
```

**Clean up the table when done:**

```bash
aws dynamodb delete-table --table-name Movies
```

## Key Takeaways

- DynamoDB is schemaless NoSQL — items are like JSON documents, accessed by primary key
- Use a **partition key** for simple lookups, add a **sort key** when you need to query groups of related items
- **On-demand** mode is simplest (pay per request); **provisioned** mode is cheaper for steady traffic
- **Always use Query over Scan** — design your keys and indexes to make this possible
- **GSIs** let you query by different attributes; add them when your access patterns require it
- Free tier is **always free** (not just 12 months): 25 GB storage, 25 RCU, 25 WCU
- Use **DynamoDB Local** (Docker) for development to avoid any AWS charges
