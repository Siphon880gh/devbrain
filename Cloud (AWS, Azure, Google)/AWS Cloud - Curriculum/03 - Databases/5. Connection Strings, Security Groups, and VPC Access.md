# Connection Strings, Security Groups, and VPC Access

Connect your applications to AWS databases securely — configuring endpoints, security groups, subnets, and secrets management.

## What You'll Learn

- How to find your database endpoint and construct connection strings
- Connection string formats for MySQL and PostgreSQL
- How security groups control database access (and how to configure them)
- Public vs private subnets for databases and when to use each
- RDS Proxy for managing connections from Lambda
- How to store and retrieve credentials with Secrets Manager

## Finding Your Database Endpoint

Every RDS and Aurora instance has a DNS endpoint — a hostname that your application uses to connect. You never connect by IP address because the IP can change during failover.

**Via the Console:**
Go to RDS → Databases → click your instance → look under **Connectivity & security** for the **Endpoint** field.

**Via the CLI:**

```bash
# RDS instance endpoint
aws rds describe-db-instances \
  --db-instance-identifier my-app-db \
  --query 'DBInstances[0].Endpoint.[Address,Port]' \
  --output text

# Aurora cluster endpoints (writer + reader)
aws rds describe-db-clusters \
  --db-cluster-identifier my-cluster \
  --query 'DBClusters[0].[Endpoint,ReaderEndpoint,Port]' \
  --output text
```

RDS endpoints look like:
```
my-app-db.abc123xyz.us-east-1.rds.amazonaws.com
```

Aurora cluster endpoints look like:
```
my-cluster.cluster-abc123xyz.us-east-1.rds.amazonaws.com       (writer)
my-cluster.cluster-ro-abc123xyz.us-east-1.rds.amazonaws.com    (reader)
```

## Connection String Formats by Engine

A connection string tells your application where the database is, what credentials to use, and which database to connect to. The format varies by engine.

### PostgreSQL

```
postgresql://username:password@hostname:5432/database_name
```

Concrete example:
```
postgresql://postgres:MyPassword123@my-app-db.abc123xyz.us-east-1.rds.amazonaws.com:5432/myapp
```

In application code (Python with psycopg2):
```python
import psycopg2

conn = psycopg2.connect(
    host="my-app-db.abc123xyz.us-east-1.rds.amazonaws.com",
    port=5432,
    dbname="myapp",
    user="postgres",
    password="MyPassword123"
)
```

In Node.js (pg):
```javascript
const { Pool } = require('pg');

const pool = new Pool({
  host: 'my-app-db.abc123xyz.us-east-1.rds.amazonaws.com',
  port: 5432,
  database: 'myapp',
  user: 'postgres',
  password: 'MyPassword123',
  ssl: { rejectUnauthorized: false }
});
```

### MySQL

```
mysql://username:password@hostname:3306/database_name
```

Concrete example:
```
mysql://admin:MyPassword123@my-app-db.abc123xyz.us-east-1.rds.amazonaws.com:3306/myapp
```

In Python (pymysql):
```python
import pymysql

conn = pymysql.connect(
    host="my-app-db.abc123xyz.us-east-1.rds.amazonaws.com",
    port=3306,
    database="myapp",
    user="admin",
    password="MyPassword123"
)
```

**Default ports:** PostgreSQL uses **5432**, MySQL/MariaDB/Aurora-MySQL use **3306**, SQL Server uses **1433**, Oracle uses **1521**.

## Security Groups for Database Access

A security group acts as a virtual firewall for your database. By default, a new RDS instance blocks all incoming connections. You must add **inbound rules** that specify who can connect and on which port.

### The Basics

- Security groups are **stateful** — if you allow inbound traffic, the response is automatically allowed outbound
- Rules are **allow-only** — you can't create deny rules
- You specify rules by **port**, **protocol**, and **source** (IP range or another security group)

### Common Rule Configurations

**Allow your local machine to connect (for development):**

```bash
# Get your public IP
curl -s https://checkip.amazonaws.com

# Add a rule allowing your IP on PostgreSQL port
aws ec2 authorize-security-group-ingress \
  --group-id sg-0abc123def456 \
  --protocol tcp \
  --port 5432 \
  --cidr 203.0.113.42/32
```

The `/32` means "this exact IP address only." Never use `0.0.0.0/0` (all IPs) for a database security group in production.

**Allow EC2 instances to connect (recommended for production):**

Instead of allowing an IP range, reference the EC2 instance's security group. This way, any EC2 instance in that security group can connect, regardless of its IP.

```bash
# Allow any instance in sg-ec2group to connect on port 5432
aws ec2 authorize-security-group-ingress \
  --group-id sg-rdsgroup \
  --protocol tcp \
  --port 5432 \
  --source-group sg-ec2group
```

**Allow Lambda functions to connect:**

Lambda functions running in a VPC get a security group. Reference it the same way:

```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-rdsgroup \
  --protocol tcp \
  --port 5432 \
  --source-group sg-lambdagroup
```

### Viewing Current Rules

```bash
aws ec2 describe-security-groups \
  --group-ids sg-rdsgroup \
  --query 'SecurityGroups[0].IpPermissions' \
  --output table
```

### Common Mistakes

- **Timeout on connect** → Your security group doesn't have a rule for your IP/port
- **Using `0.0.0.0/0`** → You've opened your database to the entire internet — never do this in production
- **Wrong port** → PostgreSQL is 5432, MySQL is 3306 — using the wrong port silently fails
- **Forgot to set Publicly Accessible** → Even with the right security group rule, if the instance isn't flagged as publicly accessible, you can't connect from outside the VPC

## Public vs Private Subnets

Where you place your database in the VPC determines who can reach it.

### Private Subnet (Recommended for Production)

- Database has no public IP address
- Only reachable from within the VPC (EC2, Lambda, ECS in the same VPC)
- The default and most secure option
- Your application talks to the database over the private network

```
Internet → Load Balancer → EC2 (public subnet) → RDS (private subnet)
```

### Public Subnet (Development Only)

- Database gets a public IP/DNS
- Reachable from your laptop if the security group allows your IP
- Convenient for development but risky for production

To make an RDS instance publicly accessible:

```bash
aws rds modify-db-instance \
  --db-instance-identifier my-app-db \
  --publicly-accessible \
  --apply-immediately
```

The database also needs to be in a **subnet group** that includes public subnets, and the associated security group must allow your IP.

### DB Subnet Groups

RDS requires a **DB subnet group** — a collection of subnets (in at least 2 AZs) where the database can be placed. When you create an RDS instance in the console, AWS creates a default subnet group with all subnets in your default VPC.

```bash
# Create a custom subnet group with private subnets
aws rds create-db-subnet-group \
  --db-subnet-group-name my-private-db-subnets \
  --db-subnet-group-description "Private subnets for databases" \
  --subnet-ids subnet-abc123 subnet-def456
```

## RDS Proxy for Lambda Connections

Lambda functions create a new database connection every time they execute. Under high concurrency (hundreds of Lambda invocations), this can overwhelm your database with connection requests — each connection consumes memory on the database server.

**RDS Proxy** solves this by pooling and sharing connections:

```
Lambda invocations → RDS Proxy (connection pool) → RDS/Aurora
         500 concurrent                                50 connections
```

Instead of 500 Lambda functions each opening their own connection, RDS Proxy maintains a pool of ~50 connections and multiplexes them.

**Setting up RDS Proxy:**

1. Store your database credentials in **Secrets Manager** (required — RDS Proxy reads credentials from there)
2. Create an IAM role that lets RDS Proxy access the secret
3. Create the proxy:

```bash
aws rds create-db-proxy \
  --db-proxy-name my-app-proxy \
  --engine-family POSTGRESQL \
  --auth '[{
    "AuthScheme": "SECRETS",
    "SecretArn": "arn:aws:secretsmanager:us-east-1:123456789012:secret:my-db-creds",
    "IAMAuth": "DISABLED"
  }]' \
  --role-arn arn:aws:iam::123456789012:role/rds-proxy-role \
  --vpc-subnet-ids subnet-abc123 subnet-def456
```

4. Register the target (your RDS instance):

```bash
aws rds register-db-proxy-targets \
  --db-proxy-name my-app-proxy \
  --db-instance-identifiers my-app-db
```

5. In your Lambda function, connect to the **proxy endpoint** instead of the RDS endpoint:

```python
conn = psycopg2.connect(
    host="my-app-proxy.proxy-abc123.us-east-1.rds.amazonaws.com",
    port=5432,
    dbname="myapp",
    user="postgres",
    password="MyPassword123"
)
```

**When to use RDS Proxy:**
- Lambda functions connecting to RDS/Aurora (the primary use case)
- Applications with many short-lived connections
- When you see "too many connections" errors

**When you don't need it:**
- EC2 or ECS applications that manage their own connection pool
- Low-concurrency applications

## Secrets Manager for Credentials

Hardcoding database passwords in your application code or environment variables is a security risk. AWS Secrets Manager stores credentials securely and lets your application retrieve them at runtime.

### Storing a Secret

```bash
aws secretsmanager create-secret \
  --name my-app/database \
  --description "Production database credentials" \
  --secret-string '{
    "username": "postgres",
    "password": "MySecurePassword123",
    "engine": "postgres",
    "host": "my-app-db.abc123xyz.us-east-1.rds.amazonaws.com",
    "port": 5432,
    "dbname": "myapp"
  }'
```

### Retrieving a Secret in Python

```python
import json
import boto3

def get_db_credentials():
    client = boto3.client('secretsmanager')
    response = client.get_secret_value(SecretId='my-app/database')
    return json.loads(response['SecretString'])

creds = get_db_credentials()
conn = psycopg2.connect(
    host=creds['host'],
    port=creds['port'],
    dbname=creds['dbname'],
    user=creds['username'],
    password=creds['password']
)
```

### Automatic Rotation

Secrets Manager can automatically rotate your database password on a schedule (e.g., every 30 days). For RDS and Aurora, AWS provides built-in rotation Lambda functions.

```bash
aws secretsmanager rotate-secret \
  --secret-id my-app/database \
  --rotation-lambda-arn arn:aws:lambda:us-east-1:123456789012:function:SecretsManagerRotation \
  --rotation-rules AutomaticallyAfterDays=30
```

**Pricing:** Secrets Manager costs $0.40 per secret per month, plus $0.05 per 10,000 API calls.

## Hands-On: Connect an EC2 Instance to an RDS Database

This exercise walks through the full connection flow — creating security groups, launching an instance, and connecting.

**Step 1:** Create a security group for the database:

```bash
aws ec2 create-security-group \
  --group-name rds-sg \
  --description "Security group for RDS" \
  --vpc-id vpc-abc123
```

**Step 2:** Create a security group for EC2:

```bash
aws ec2 create-security-group \
  --group-name ec2-sg \
  --description "Security group for EC2" \
  --vpc-id vpc-abc123
```

**Step 3:** Allow EC2 to reach the database on port 5432:

```bash
aws ec2 authorize-security-group-ingress \
  --group-id <rds-sg-id> \
  --protocol tcp \
  --port 5432 \
  --source-group <ec2-sg-id>
```

**Step 4:** Allow SSH to EC2 from your IP:

```bash
MY_IP=$(curl -s https://checkip.amazonaws.com)
aws ec2 authorize-security-group-ingress \
  --group-id <ec2-sg-id> \
  --protocol tcp \
  --port 22 \
  --cidr "${MY_IP}/32"
```

**Step 5:** Launch an EC2 instance with the ec2-sg, SSH in, install the PostgreSQL client, and connect:

```bash
ssh -i mykey.pem ec2-user@<ec2-public-ip>

# On the EC2 instance:
sudo dnf install -y postgresql15

psql -h my-app-db.abc123xyz.us-east-1.rds.amazonaws.com \
     -U postgres \
     -d postgres
```

If the connection succeeds, you'll see the `postgres=#` prompt.

## Troubleshooting Common Connection Issues

| Symptom | Likely Cause | Fix |
|---|---|---|
| Connection times out | Security group missing inbound rule | Add a rule for your IP/SG on the correct port |
| Connection refused | Database not running or wrong port | Check instance status; verify port number |
| Can't resolve hostname | Endpoint is wrong or DNS issue | Copy endpoint directly from RDS console |
| "too many connections" | Connection pool exhausted | Use RDS Proxy or increase `max_connections` |
| "password authentication failed" | Wrong credentials | Check username/password; try resetting master password |
| Timeout from Lambda | Lambda not in VPC, or wrong SG | Place Lambda in VPC with SG that allows RDS access |
| Works from EC2, not from laptop | Instance not publicly accessible | Set Publicly Accessible = Yes and add your IP to SG |

**Debugging checklist:**
1. Is the RDS instance in "available" state?
2. Is your source (EC2, Lambda, laptop) in the security group's inbound rules?
3. Are you using the right port (5432 for PostgreSQL, 3306 for MySQL)?
4. Is the instance publicly accessible (if connecting from outside the VPC)?
5. Can you ping the endpoint? (RDS doesn't respond to ping, but DNS resolution confirms the hostname works: `nslookup <endpoint>`)

## Key Takeaways

- Always connect to the **DNS endpoint**, never by IP address — the IP changes during failover
- Security groups are the #1 cause of connection failures — check them first when troubleshooting
- Use **private subnets** for production databases, **public** only for development
- Use **security group references** (sg-to-sg) instead of IP ranges for EC2/Lambda access
- **RDS Proxy** is essential for Lambda-to-database connections under high concurrency
- Store credentials in **Secrets Manager**, not in code or environment variables
- Connection string format: `engine://user:pass@host:port/dbname`
