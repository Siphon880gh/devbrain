# Choosing a Database

A guide to selecting the right AWS database service — RDS, DynamoDB, Aurora, ElastiCache, or Redshift — based on your data model and access patterns.

## What You'll Learn

- The five main AWS database services and what each one does
- How to pick the right database for your workload
- When to use relational (SQL) vs NoSQL vs in-memory vs data warehouse
- Common multi-database patterns used in real applications

## Database Types on AWS

AWS offers purpose-built databases rather than a one-size-fits-all solution. Each service is optimized for a specific data model and access pattern. Here's a plain-English breakdown of each.

### RDS (Relational Database Service)

RDS gives you a traditional SQL database — the same MySQL, PostgreSQL, or MariaDB you'd install on your own server — but AWS handles the infrastructure. You define tables with rows and columns, write SQL queries, and use foreign keys to relate data.

**Best for:** Structured data with relationships. User accounts, orders, inventory, anything where you need JOINs and transactions.

```
Example: An e-commerce site storing users, products, and orders
- users table → products table → orders table
- SQL JOINs link them together
- ACID transactions ensure an order isn't created without payment
```

### DynamoDB

DynamoDB is a fully managed NoSQL database. There are no servers to provision, no SQL to write (in the traditional sense), and it scales automatically. You store data as items (like JSON documents) in tables, accessed by a primary key.

**Best for:** High-speed, high-scale workloads with simple access patterns. Session stores, shopping carts, gaming leaderboards, IoT sensor data.

```
Example: A mobile app storing user profiles
- Each profile is a single item: { userId: "abc", name: "Jane", preferences: {...} }
- Retrieved by userId in single-digit milliseconds
- Scales to millions of users with no performance degradation
```

### Aurora

Aurora is AWS's own relational database engine, compatible with MySQL and PostgreSQL. It uses the same SQL you already know but runs on a completely redesigned storage layer. Aurora stores 6 copies of your data across 3 Availability Zones and can handle up to 5x the throughput of standard MySQL.

**Best for:** Production workloads that need relational SQL but outgrow standard RDS performance. SaaS platforms, financial applications, high-traffic web apps.

### ElastiCache

ElastiCache provides managed Redis or Memcached — in-memory data stores that respond in microseconds. You don't use ElastiCache as your primary database. Instead, you put it *in front of* your database to cache frequently accessed data.

**Best for:** Caching database query results, session storage, real-time leaderboards, rate limiting.

```
Example: A product page that gets 10,000 views per minute
- Without cache: 10,000 SQL queries per minute to RDS
- With ElastiCache: 1 SQL query, result cached in Redis, 9,999 reads from memory
```

### Redshift

Redshift is a data warehouse designed for analytics. While RDS handles your day-to-day transactions (OLTP), Redshift handles your reporting and analysis (OLAP). It's columnar storage optimized for aggregating millions of rows.

**Best for:** Business intelligence, reporting dashboards, analyzing historical data. Queries like "What were total sales by region for the past 3 years?"

## Comparison Matrix

| Feature | RDS | DynamoDB | Aurora | ElastiCache | Redshift |
|---|---|---|---|---|---|
| **Type** | Relational (SQL) | NoSQL (key-value/document) | Relational (SQL) | In-memory cache | Data warehouse |
| **Query Language** | SQL | PartiQL / API calls | SQL | Redis commands / Memcached | SQL |
| **Scaling** | Vertical (bigger instance) | Automatic / horizontal | Auto-scaling storage + read replicas | Vertical + clustering | Add nodes |
| **Latency** | Low ms | Single-digit ms | Low ms | Sub-millisecond | Seconds (complex queries) |
| **Max Storage** | 64 TB | Unlimited | 128 TB (auto-grows) | Up to 340 GB per node | Petabytes |
| **Managed Patching** | Yes | Fully serverless | Yes | Yes | Yes |
| **Multi-AZ** | Optional | Built-in (3 AZs) | Built-in (3 AZs, 6 copies) | Optional | Optional |
| **Free Tier** | 750 hrs db.t3.micro/mo | 25 GB + 25 RCU/WCU always free | No | No | 2-month trial |
| **Starting Cost** | ~$13/mo (db.t3.micro) | Pay-per-request from $0 | ~$30/mo | ~$12/mo | ~$180/mo |

## Decision Criteria

Use this quick decision tree to narrow down your choice:

**1. What kind of data do you have?**
- Structured with relationships (users → orders → products) → **RDS or Aurora**
- Simple key-value lookups (userId → profile) → **DynamoDB**
- Analytics over huge datasets → **Redshift**
- Frequently repeated reads of the same data → **ElastiCache** (as a layer on top)

**2. How much traffic do you expect?**
- Low to moderate, predictable → **RDS** (cheapest relational option)
- High, variable, or unpredictable spikes → **DynamoDB** (on-demand mode) or **Aurora Serverless**
- Extreme read-heavy traffic → Add **ElastiCache** in front of any database

**3. Do you need SQL?**
- Yes, complex queries with JOINs → **RDS** or **Aurora**
- No, simple get/put by key → **DynamoDB**
- Yes, but for analytics → **Redshift**

**4. What's your budget?**
- Free tier / minimal cost → **RDS db.t3.micro** or **DynamoDB on-demand**
- Cost-sensitive production → **RDS** (cheaper than Aurora for small workloads)
- Performance-critical production → **Aurora** (worth the premium)

**5. How much do you want to manage?**
- Nothing at all → **DynamoDB** (fully serverless, no capacity planning needed)
- Some (pick instance size, handle connections) → **RDS** or **Aurora**
- Minimal caching layer → **ElastiCache**

## Common Patterns and Combinations

Real applications often use multiple databases together. Here are patterns you'll see in production:

### Pattern 1: RDS + ElastiCache
The most common combo. Your application writes to RDS and reads from ElastiCache. When data changes, you invalidate the cache.

```
User Request → Check Redis Cache → Cache Hit? → Return data
                                  → Cache Miss? → Query RDS → Store in Redis → Return data
```

### Pattern 2: DynamoDB + RDS
Use DynamoDB for high-speed, simple access patterns (sessions, user preferences) and RDS for complex relational data (orders, inventory).

### Pattern 3: RDS/Aurora + Redshift
Your application database (RDS or Aurora) handles live transactions. Periodically, you export data to Redshift for business analytics and reporting dashboards.

### Pattern 4: DynamoDB + ElastiCache (DAX)
DynamoDB has its own caching layer called DAX (DynamoDB Accelerator) that sits transparently in front of your table — microsecond reads without changing your application code.

### Anti-Patterns to Avoid

- **Don't use Redshift for your web app's primary database** — it's designed for analytics queries, not transactional workloads
- **Don't use DynamoDB when you need complex JOINs** — restructuring relational data into DynamoDB is painful and often slower
- **Don't skip caching and just upgrade your RDS instance** — a cache is almost always cheaper and faster than a bigger database
- **Don't choose Aurora by default** — if your workload fits on a db.t3.small RDS instance, Aurora's premium isn't worth it

## Key Takeaways

- **RDS** is your default choice for relational data — familiar SQL, managed infrastructure, free tier available
- **DynamoDB** is best for simple, high-speed access patterns with automatic scaling — and it has an always-free tier
- **Aurora** is RDS on steroids — same SQL, better performance and durability, but costs more and has no free tier
- **ElastiCache** is a caching layer, not a primary database — use it to reduce load on your main database
- **Redshift** is for analytics and reporting, not for serving your application's live traffic
- Most production apps use 2+ database services together
