# Setting Billing Alerts, Budgets, and Cost Anomaly Detection

Protecting yourself from surprise charges with proactive cost monitoring.

## What You'll Learn

- How to navigate the AWS Billing Dashboard
- How to create a monthly cost budget with email alerts
- How to set up a billing alarm using CloudWatch
- How to enable Cost Anomaly Detection for automatic monitoring
- How to use Cost Explorer to analyze spending
- CLI commands for creating budgets programmatically
- Why this should be the very first thing you do after creating your account

> **Do this immediately.** Before launching a single EC2 instance, before creating a single S3 bucket — set up billing alerts. A forgotten resource running for a month can cost $30–100+. Five minutes of setup here can save you real money.

## AWS Billing Dashboard

The Billing Dashboard is your central view for all cost-related information.

### How to Get There

1. Click your **account name** in the top-right of the console
2. Click **Billing and Cost Management**
3. Or search "Billing" in the console search bar

> **IAM user access:** By default, IAM users cannot see billing information — only the root user can. To grant access, sign in as root, go to **Account** settings, scroll to **IAM User and Role Access to Billing Information**, and click **Edit** → **Activate IAM Access**.

### What You'll See

- **Month-to-date charges** — your running total for the current billing cycle
- **Forecasted charges** — AWS's prediction for the end of the month
- **Top services** — which services are generating the most cost
- **Free Tier usage** — a breakdown showing how much of each Free Tier allowance you've consumed

### Free Tier Usage Tracking

In the Billing console, click **Free Tier** in the left sidebar. This page shows a bar chart for each Free Tier service you're using, with your current usage vs. the free limit. Check this weekly while you're learning to catch anything approaching its limit.

## Creating a Budget

AWS Budgets lets you set a dollar threshold and receive email alerts when your spending approaches or exceeds it. This is the single most important guardrail for learners.

### Console Method (Step-by-Step)

1. Go to **Billing and Cost Management** → **Budgets** (left sidebar)
2. Click **Create a budget**
3. Choose **Use a template (simplified)**
4. Select **Zero spend budget** (alerts you on any charge) or **Monthly cost budget** (alerts at a threshold you set)

For the **Zero spend budget:**
- Enter your email address for notifications
- Click **Create budget**
- You'll get an email the moment any charge appears on your account

For a **Monthly cost budget** (recommended alongside the zero-spend budget):
1. Select **Monthly cost budget**
2. Set a **budget amount** — $10 is a good starting point for learners
3. Enter your **email address**
4. Click **Create budget**

This creates alerts at **85%** ($8.50) and **100%** ($10) of your budget by default.

### Understanding Alert Thresholds

You can create budgets with multiple alert thresholds:

| Threshold | When It Triggers | Purpose |
|-----------|-----------------|---------|
| 50% ($5) | Halfway through budget | Early warning |
| 85% ($8.50) | Approaching limit | Time to investigate |
| 100% ($10) | Budget exceeded | Take immediate action |
| 120% ($12) | Over budget | Something may be running unintended |

> **AWS Budgets does not stop your resources.** It only sends notifications. If you exceed your budget, resources keep running and charges keep accumulating. You must manually stop or terminate resources.

### CLI Method

You can create a budget from the command line. First, create a JSON file defining the budget:

```json
{
  "BudgetName": "Monthly-10-Dollar-Budget",
  "BudgetLimit": {
    "Amount": "10",
    "Unit": "USD"
  },
  "BudgetType": "COST",
  "TimeUnit": "MONTHLY",
  "TimePeriod": {
    "Start": "2024-01-01T00:00:00Z",
    "End": "2087-06-15T00:00:00Z"
  }
}
```

Save this as `budget.json`. Then create a notifications file:

```json
[
  {
    "Notification": {
      "NotificationType": "ACTUAL",
      "ComparisonOperator": "GREATER_THAN",
      "Threshold": 85,
      "ThresholdType": "PERCENTAGE"
    },
    "Subscribers": [
      {
        "SubscriptionType": "EMAIL",
        "Address": "your-email@example.com"
      }
    ]
  },
  {
    "Notification": {
      "NotificationType": "ACTUAL",
      "ComparisonOperator": "GREATER_THAN",
      "Threshold": 100,
      "ThresholdType": "PERCENTAGE"
    },
    "Subscribers": [
      {
        "SubscriptionType": "EMAIL",
        "Address": "your-email@example.com"
      }
    ]
  }
]
```

Save this as `notifications.json`. Now create the budget:

```bash
aws budgets create-budget \
  --account-id $(aws sts get-caller-identity --query Account --output text) \
  --budget file://budget.json \
  --notifications-with-subscribers file://notifications.json
```

Verify it was created:

```bash
aws budgets describe-budgets \
  --account-id $(aws sts get-caller-identity --query Account --output text) \
  --output table
```

## Setting Up Billing Alerts

Billing alarms through CloudWatch are an older but still useful mechanism that complements AWS Budgets.

### Prerequisites

Billing metrics are only available in `us-east-1`. You must enable them first:

1. Go to **Billing and Cost Management** → **Billing preferences** (left sidebar)
2. Under **Alert preferences**, ensure **Receive CloudWatch billing alerts** is checked
3. Click **Update**

It takes about 15 minutes for billing data to start flowing to CloudWatch after enabling this.

### Creating a Billing Alarm in CloudWatch

1. Switch your region to **US East (N. Virginia)** — `us-east-1`
2. Go to **CloudWatch** (search in the console)
3. Click **Alarms** → **All alarms** → **Create alarm**
4. Click **Select metric**
5. Click **Billing** → **Total Estimated Charge**
6. Check the box next to **USD** → click **Select metric**
7. Under **Conditions**:
   - Threshold type: **Static**
   - Whenever EstimatedCharges is: **Greater than**
   - Threshold value: `5` (or whatever dollar amount you want)
8. Click **Next**
9. Under **Notification**:
   - Select **Create new topic**
   - Name it `billing-alerts`
   - Enter your email address
   - Click **Create topic**
10. Click **Next**, give the alarm a name (e.g., "Billing Over $5"), click **Next** again, then **Create alarm**

> **Confirm the SNS subscription!** Check your email for a message from AWS Notifications and click the **Confirm subscription** link. The alarm won't send emails until you confirm.

### CLI Method for CloudWatch Billing Alarm

```bash
# Create an SNS topic for billing alerts
aws sns create-topic --name billing-alerts --region us-east-1

# Subscribe your email to the topic (replace with your email and topic ARN)
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:billing-alerts \
  --protocol email \
  --notification-endpoint your-email@example.com \
  --region us-east-1

# Create the billing alarm
aws cloudwatch put-metric-alarm \
  --alarm-name "Billing-Over-5-USD" \
  --metric-name EstimatedCharges \
  --namespace AWS/Billing \
  --statistic Maximum \
  --period 21600 \
  --threshold 5 \
  --comparison-operator GreaterThanThreshold \
  --dimensions Name=Currency,Value=USD \
  --evaluation-periods 1 \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:billing-alerts \
  --region us-east-1
```

Replace `123456789012` with your actual account ID and update the email address.

## Cost Anomaly Detection

Cost Anomaly Detection uses machine learning to detect unusual spending patterns. It learns your normal usage and alerts you when something deviates. This catches scenarios that a fixed-dollar budget might miss — like a service that normally costs $0.50 suddenly costing $15.

### Setting It Up

1. Go to **Billing and Cost Management** → **Cost Anomaly Detection** (left sidebar)
2. Click **Create monitor**
3. Choose a monitor type:
   - **AWS services** — monitors all services (recommended for beginners)
   - **Linked account** — for multi-account setups
   - **Cost category** — for specific cost allocations
4. Select **AWS services** and click **Next**
5. Set up an alert subscription:
   - **Alert threshold:** Individual alerts when anomaly impact exceeds **$1** (the lowest meaningful threshold for learners)
   - **Email recipients:** your email address
   - **Alert frequency:** Individual alerts (sends an email for each anomaly)
6. Click **Create monitor**

Cost Anomaly Detection is free — there's no charge for using it. It typically takes 24–48 hours to establish a baseline before it starts detecting anomalies.

### What Anomaly Alerts Look Like

When an anomaly is detected, you receive an email with:
- The service causing the anomaly
- The expected cost vs. actual cost
- The root cause (e.g., which usage type increased)
- A link to investigate further in the Billing console

## Cost Explorer

Cost Explorer provides detailed visualizations and breakdowns of your AWS spending. It helps you answer questions like "what's costing me the most?" and "is my spending trending up?"

### Enabling Cost Explorer

1. Go to **Billing and Cost Management** → **Cost Explorer** (left sidebar)
2. Click **Launch Cost Explorer** if it's not already enabled
3. It takes up to 24 hours for data to become available

### Useful Views

**Daily cost breakdown:**
- Set the time range to the last 30 days
- Group by **Service** to see which services cost the most
- Switch to **Daily** granularity to spot spikes

**Monthly trend:**
- Set the time range to the last 6 months
- Use **Monthly** granularity
- Look for patterns — is your spending growing month-over-month?

**Filtering by tag:**
- If you tag your resources (e.g., `Project: learning`), you can filter Cost Explorer to show only costs for that project

### CLI Access to Cost Data

```bash
# Get month-to-date costs grouped by service
aws ce get-cost-and-usage \
  --time-period Start=$(date -u +%Y-%m-01),End=$(date -u +%Y-%m-%d) \
  --granularity MONTHLY \
  --metrics BlendedCost \
  --group-by Type=DIMENSION,Key=SERVICE \
  --output table
```

```bash
# Get daily costs for the last 7 days
aws ce get-cost-and-usage \
  --time-period Start=$(date -u -v-7d +%Y-%m-%d),End=$(date -u +%Y-%m-%d) \
  --granularity DAILY \
  --metrics BlendedCost \
  --output table
```

> On Linux, replace `-v-7d` with `-d '7 days ago'`:
> ```bash
> --time-period Start=$(date -u -d '7 days ago' +%Y-%m-%d),End=$(date -u +%Y-%m-%d)
> ```

## Hands-On: Configure a Zero-Spend Alert

Complete these steps right now — it takes less than 5 minutes:

### Step 1: Create a Zero-Spend Budget
1. Go to **Billing and Cost Management** → **Budgets**
2. Click **Create a budget**
3. Select **Use a template** → **Zero spend budget**
4. Enter your email
5. Click **Create budget**

### Step 2: Create a $10 Monthly Budget
1. Click **Create a budget** again
2. Select **Use a template** → **Monthly cost budget**
3. Enter `10` as the budget amount
4. Enter your email
5. Click **Create budget**

### Step 3: Enable Cost Anomaly Detection
1. Go to **Cost Anomaly Detection** in the left sidebar
2. Create a monitor for **AWS services**
3. Set alert threshold to $1
4. Enter your email
5. Click **Create monitor**

### Step 4: Verify Everything

Check your email for confirmation messages from AWS. You should receive:
- Budget creation confirmations
- An SNS subscription confirmation (if you set up the CloudWatch alarm) — click to confirm

Run this CLI command to verify your budgets exist:

```bash
aws budgets describe-budgets \
  --account-id $(aws sts get-caller-identity --query Account --output text) \
  --query 'Budgets[*].[BudgetName,BudgetLimit.Amount,BudgetType]' \
  --output table
```

### Quick-Check Routine

Get in the habit of running these checks at the end of each study session:

```bash
# Check for any running EC2 instances across common regions
for region in us-east-1 us-west-2 eu-west-1; do
  echo "=== $region ==="
  aws ec2 describe-instances \
    --region $region \
    --query 'Reservations[*].Instances[*].[InstanceId,State.Name,InstanceType]' \
    --output table
done

# Check for any RDS instances
aws rds describe-db-instances \
  --query 'DBInstances[*].[DBInstanceIdentifier,DBInstanceStatus,DBInstanceClass]' \
  --output table

# Check for NAT Gateways (expensive if left running)
aws ec2 describe-nat-gateways \
  --query 'NatGateways[?State==`available`].[NatGatewayId,State]' \
  --output table

# Check for Elastic IPs not attached to anything
aws ec2 describe-addresses \
  --query 'Addresses[?AssociationId==null].[PublicIp,AllocationId]' \
  --output table
```

## Key Takeaways

- **Set up billing alerts before doing anything else** — this is non-negotiable
- Create both a **zero-spend budget** (catches any charge) and a **$10 monthly budget** (catches larger issues)
- Enable **Cost Anomaly Detection** — it's free and uses ML to catch unusual patterns
- AWS Budgets only sends notifications — it does **not** stop resources or prevent charges
- **CloudWatch billing alarms** only work in `us-east-1` and require enabling billing metrics first
- **Cost Explorer** shows detailed breakdowns of your spending by service, time period, and tags
- Always confirm the **SNS email subscription** — alerts won't work until you click the confirmation link
- Run resource checks after each study session to catch anything you left running
