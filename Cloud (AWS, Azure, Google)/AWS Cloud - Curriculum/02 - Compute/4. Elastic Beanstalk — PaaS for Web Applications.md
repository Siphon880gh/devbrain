# Elastic Beanstalk — PaaS for Web Applications

Deploy web applications quickly using AWS's platform-as-a-service without managing the underlying infrastructure.

## What You'll Learn

- What Elastic Beanstalk is and what it manages for you
- Supported platforms and languages
- How to deploy an application via the console and CLI
- Environment tiers: web server vs worker
- How to customize your environment with `.ebextensions`
- Monitoring, scaling, and deployment strategies

## How Elastic Beanstalk Works

Elastic Beanstalk (EB) is AWS's **Platform-as-a-Service (PaaS)**. You upload your application code, and Beanstalk automatically provisions and manages the infrastructure: EC2 instances, load balancers, auto-scaling groups, security groups, and CloudWatch monitoring.

The key idea: **you focus on your code, Beanstalk handles the infrastructure.**

Here's what Beanstalk creates for a typical web application:

```
Internet
  │
  ▼
Application Load Balancer (ALB)
  │
  ▼
Auto Scaling Group
  ├── EC2 Instance (your app)
  ├── EC2 Instance (your app)
  └── EC2 Instance (your app)
  │
  ▼
CloudWatch (monitoring + alarms)
```

The important thing to know is that Beanstalk is not a separate service with its own compute — it's an orchestration layer that creates and configures standard AWS resources. You can still access the underlying EC2 instances, modify the load balancer, and adjust auto-scaling rules. Beanstalk just sets it all up for you.

There is **no additional charge** for Elastic Beanstalk itself. You only pay for the resources it creates (EC2 instances, load balancers, etc.).

## Supported Platforms and Languages

Beanstalk supports most popular web application platforms:

| Platform | Versions | Example Frameworks |
|---|---|---|
| **Node.js** | 18, 20 | Express, Fastify, Nest.js |
| **Python** | 3.9, 3.11, 3.12 | Flask, Django, FastAPI |
| **Java** | Corretto 11, 17, 21 | Spring Boot, Tomcat |
| **Ruby** | 3.1, 3.2 | Rails, Sinatra |
| **.NET** | 6, 8 | ASP.NET Core |
| **PHP** | 8.1, 8.2, 8.3 | Laravel, Symfony |
| **Go** | 1.21 | Standard library, Gin |
| **Docker** | Single and Multi-container | Any language/framework |

The Docker platform is especially useful — if your language isn't listed or you need a custom environment, just package your app in a Docker container.

## Deploying an Application

### Key Concepts

- **Application** — A logical collection of environments and versions. Think of it as your project.
- **Environment** — A running instance of your application (e.g., "production", "staging"). Each environment has its own URL, load balancer, and set of EC2 instances.
- **Application Version** — A specific labeled version of your code (stored as a zip in S3).

### Deploy via the Console

1. Go to **Elastic Beanstalk** in the AWS Console
2. Click **Create application**
3. Application name: `my-web-app`
4. Platform: Select your language (e.g., Node.js)
5. Application code: Upload your code as a zip file, or use the **Sample application** to test
6. Choose **Single instance (Free Tier eligible)** for learning, or **High availability** for production (adds a load balancer)
7. Click through the configuration steps (defaults are fine for learning)
8. Click **Submit**

Beanstalk will take 5–10 minutes to create all the resources. When it's done, you'll get a URL like `my-web-app-env.us-east-1.elasticbeanstalk.com`.

### Deploy via the EB CLI

The EB CLI is a dedicated command-line tool for Elastic Beanstalk (separate from the AWS CLI). Install it first:

```bash
pip install awsebcli
```

Then initialize and deploy your project:

```bash
# Navigate to your project directory
cd my-web-app

# Initialize Elastic Beanstalk (interactive prompts)
eb init

# During init, you'll choose:
# - Region (e.g., us-east-1)
# - Application name
# - Platform (e.g., Node.js 20)
# - Whether to set up SSH access to instances

# Create an environment and deploy
eb create my-web-app-env

# This creates all resources and deploys your code
# Takes 5-10 minutes the first time
```

After the initial setup, deploying new code is a single command:

```bash
# Make changes to your code, then:
eb deploy
```

Other useful EB CLI commands:

```bash
eb status          # Check environment health and info
eb open            # Open the app URL in your browser
eb logs            # View recent logs
eb ssh             # SSH into an EC2 instance
eb config          # View/edit environment configuration
eb terminate       # Delete the environment and all resources
```

### Sample Node.js App

Here's a minimal Express app that works with Elastic Beanstalk:

```javascript
const express = require("express");
const app = express();
const port = process.env.PORT || 8080;

app.get("/", (req, res) => {
    res.json({
        message: "Hello from Elastic Beanstalk!",
        environment: process.env.NODE_ENV || "development",
        timestamp: new Date().toISOString()
    });
});

app.get("/health", (req, res) => {
    res.status(200).json({ status: "healthy" });
});

app.listen(port, () => {
    console.log(`Server running on port ${port}`);
});
```

Beanstalk sets the `PORT` environment variable, so make sure your app reads from it.

Your `package.json` should include a `start` script:

```json
{
  "name": "my-web-app",
  "version": "1.0.0",
  "scripts": {
    "start": "node app.js"
  },
  "dependencies": {
    "express": "^4.18.0"
  }
}
```

## Environment Tiers

Elastic Beanstalk has two environment tiers:

### Web Server Tier

This is the default. It creates a load balancer and EC2 instances that serve HTTP requests from the internet. Use this for web applications and APIs.

```
Internet → Load Balancer → EC2 Instances (your web app)
```

### Worker Tier

A worker environment processes background jobs from an SQS queue. It doesn't have a load balancer or serve HTTP traffic. Use this for tasks like sending emails, processing images, or generating reports.

```
SQS Queue → EC2 Instances (your worker code)
```

A common pattern is to have a web server tier handling user requests and a worker tier processing background jobs. The web tier pushes messages to an SQS queue, and the worker tier processes them.

## Environment Configuration

### `.ebextensions`

Create a `.ebextensions` folder in the root of your project to customize your Beanstalk environment. Configuration files in this folder must have a `.config` extension and use YAML format.

**Example: Install packages and configure Nginx**

`.ebextensions/01-packages.config`:
```yaml
packages:
  yum:
    git: []
    jq: []

option_settings:
  aws:elasticbeanstalk:application:environment:
    NODE_ENV: production
    LOG_LEVEL: info
```

**Example: Run commands during deployment**

`.ebextensions/02-setup.config`:
```yaml
container_commands:
  01_migrate_db:
    command: "npm run db:migrate"
    leader_only: true
  02_seed_data:
    command: "npm run db:seed"
    leader_only: true
```

`leader_only: true` ensures the command runs on only one instance during a deployment (important for database migrations).

**Example: Set up HTTPS**

`.ebextensions/03-https.config`:
```yaml
option_settings:
  aws:elb:listener:443:
    SSLCertificateId: arn:aws:acm:us-east-1:123456789012:certificate/abc-123
    ListenerProtocol: HTTPS
    InstancePort: 80
```

### Environment Variables

Set environment variables through the console, CLI, or `.ebextensions`:

```bash
# Via EB CLI
eb setenv DB_HOST=mydb.cluster.us-east-1.rds.amazonaws.com DB_NAME=myapp

# Via AWS CLI
aws elasticbeanstalk update-environment \
  --environment-name my-web-app-env \
  --option-settings \
    Namespace=aws:elasticbeanstalk:application:environment,OptionName=DB_HOST,Value=mydb.cluster.us-east-1.rds.amazonaws.com
```

## Monitoring and Scaling

### Health Monitoring

Beanstalk has built-in health monitoring. The console shows your environment health as a color:

| Color | Meaning |
|---|---|
| **Green** | All instances are healthy |
| **Yellow** | One or more health checks are warning |
| **Red** | Health checks are failing |
| **Grey** | Environment is updating |

Beanstalk performs HTTP health checks on your app. By default, it hits the root URL (`/`). To customize, set the health check path to a dedicated endpoint like `/health`.

### Auto Scaling

Beanstalk manages auto-scaling for you. Configure it through the console or `.ebextensions`:

```yaml
option_settings:
  aws:autoscaling:asg:
    MinSize: 2
    MaxSize: 6
  aws:autoscaling:trigger:
    MeasureName: CPUUtilization
    Statistic: Average
    Unit: Percent
    UpperThreshold: 70
    LowerThreshold: 30
```

This scales between 2 and 6 instances based on average CPU utilization.

### Deployment Strategies

Beanstalk supports several deployment strategies:

| Strategy | Downtime | Speed | Risk |
|---|---|---|---|
| **All at once** | Yes (brief) | Fastest | Highest — all instances update simultaneously |
| **Rolling** | No | Medium | Medium — updates in batches |
| **Rolling with additional batch** | No | Slower | Low — maintains full capacity during update |
| **Immutable** | No | Slowest | Lowest — deploys to new instances, swaps when healthy |

For production, use **Rolling with additional batch** or **Immutable** to avoid downtime.

## Clean Up

Elastic Beanstalk environments cost money because of the underlying EC2 instances and load balancers. Terminate when done:

```bash
eb terminate my-web-app-env
```

Or in the console: Actions → Terminate environment.

This deletes the EC2 instances, load balancer, auto-scaling group, and security groups. If you attached an RDS database, be careful — it may be deleted too depending on your configuration.

## Key Takeaways

- Elastic Beanstalk is a PaaS that manages EC2 instances, load balancers, and auto-scaling for you
- There's no extra charge — you only pay for the underlying AWS resources
- Supports Node.js, Python, Java, Ruby, .NET, PHP, Go, and Docker
- Use the EB CLI (`eb init`, `eb create`, `eb deploy`) for fast command-line workflows
- The web server tier handles HTTP traffic; the worker tier processes SQS messages
- Customize environments with `.ebextensions` config files
- Use rolling or immutable deployments in production to avoid downtime
- Always terminate environments when you're done to avoid charges
