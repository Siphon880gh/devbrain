## Containers on AWS: The Big Picture

### What is a Container?

A container is a lightweight, standalone package that includes your application code, runtime, libraries, and system tools — everything needed to run your app. Unlike a virtual machine, a container shares the host operating system's kernel, making it much smaller and faster to start.

Think of it this way:
- **Virtual machine** = a full computer with its own operating system (heavy, slow to boot)
- **Container** = just your app and its dependencies in an isolated process (lightweight, boots in seconds)

### Docker Basics

Docker is the tool you use to build and run containers. Here's the workflow:

1. **Write a Dockerfile** — instructions to build your container image
2. **Build the image** — `docker build` creates a portable image from the Dockerfile
3. **Push to a registry** — upload the image somewhere accessible (like Amazon ECR)
4. **Run the container** — AWS pulls the image and runs it

Here's a simple Dockerfile for a Node.js application:

```dockerfile
FROM node:20-slim
WORKDIR /app
COPY package*.json ./
RUN npm ci --production
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]
```

### Amazon ECR (Elastic Container Registry)

ECR is AWS's private Docker image registry. It's where you store your container images so that ECS can pull them.

```bash
# Create a repository
aws ecr create-repository --repository-name my-app

# Get the login command (authenticates Docker with ECR)
aws ecr get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin \
  123456789012.dkr.ecr.us-east-1.amazonaws.com

# Build your image
docker build -t my-app .

# Tag it for ECR
docker tag my-app:latest \
  123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:latest

# Push to ECR
docker push \
  123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:latest
```

Replace `123456789012` with your AWS account ID.

## ECS Core Concepts: Tasks, Services, and Clusters

ECS (Elastic Container Service) has four key concepts. Understanding these is essential:

### Cluster

A cluster is a logical grouping of tasks and services. It's the top-level container for running your workloads. Think of it as a namespace — you might have a `production` cluster and a `staging` cluster.

```bash
aws ecs create-cluster --cluster-name my-cluster
```

### Task Definition

A task definition is a **blueprint** for your container(s). It specifies:
- Which container image to use
- How much CPU and memory to allocate
- Which ports to expose
- Environment variables
- Logging configuration
- IAM roles

A task definition can describe one or more containers that run together (e.g., your app container plus a logging sidecar).

### Task

A task is a **running instance** of a task definition. When ECS launches your container, that's a task. Tasks are ephemeral — they run, complete (or crash), and are replaced.

### Service

A service ensures that a specified number of tasks are running at all times. If a task crashes, the service automatically launches a replacement. Services also integrate with load balancers to distribute traffic across multiple tasks.

```
Cluster
  └── Service (maintains 3 running tasks)
        ├── Task (running container)
        ├── Task (running container)
        └── Task (running container)
```

## Fargate vs EC2 Launch Type

When running ECS, you choose how the underlying compute works:

| Feature | Fargate | EC2 Launch Type |
|---|---|---|
| **Server management** | None — AWS manages everything | You manage EC2 instances |
| **Scaling** | Per-task — each task gets its own resources | Per-instance — tasks share EC2 capacity |
| **Pricing** | Per vCPU-second + memory-second | Per EC2 instance (hourly/second) |
| **Startup time** | ~30 seconds | Depends on EC2 instance availability |
| **Control** | Limited (no SSH, no host access) | Full (SSH into instances, install agents) |
| **Best for** | Most workloads, getting started | GPU tasks, specific instance types, cost optimization at scale |

**Use Fargate** unless you have a specific reason to manage EC2 instances. Fargate is simpler and the right default for most teams.

## Task Definitions

A task definition is written in JSON. Here's a complete example:

```json
{
  "family": "my-web-app",
  "networkMode": "awsvpc",
  "requiresCompatibilities": ["FARGATE"],
  "cpu": "256",
  "memory": "512",
  "executionRoleArn": "arn:aws:iam::123456789012:role/ecsTaskExecutionRole",
  "containerDefinitions": [
    {
      "name": "web",
      "image": "123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:latest",
      "portMappings": [
        {
          "containerPort": 3000,
          "protocol": "tcp"
        }
      ],
      "environment": [
        { "name": "NODE_ENV", "value": "production" },
        { "name": "PORT", "value": "3000" }
      ],
      "logConfiguration": {
        "logDriver": "awslogs",
        "options": {
          "awslogs-group": "/ecs/my-web-app",
          "awslogs-region": "us-east-1",
          "awslogs-stream-prefix": "web"
        }
      },
      "essential": true
    }
  ]
}
```

**Key fields explained:**

- `family` — The name of the task definition. New versions are created under the same family.
- `cpu` / `memory` — Resources for the task. Fargate has specific valid combinations (e.g., 256 CPU / 512 MB memory, 512 CPU / 1024 MB memory).
- `networkMode: "awsvpc"` — Required for Fargate. Each task gets its own elastic network interface (ENI) with a private IP.
- `executionRoleArn` — IAM role that ECS uses to pull images from ECR and write logs to CloudWatch.
- `containerDefinitions` — Array of containers in this task. Most tasks have one; use multiple for sidecar patterns.
- `essential: true` — If this container stops, the entire task stops.

**Valid Fargate CPU/memory combinations:**

| CPU (units) | Memory Options |
|---|---|
| 256 (0.25 vCPU) | 512 MB, 1 GB, 2 GB |
| 512 (0.5 vCPU) | 1 GB – 4 GB (in 1 GB increments) |
| 1024 (1 vCPU) | 2 GB – 8 GB (in 1 GB increments) |
| 2048 (2 vCPU) | 4 GB – 16 GB (in 1 GB increments) |
| 4096 (4 vCPU) | 8 GB – 30 GB (in 1 GB increments) |

## Hands-On: Deploy a Container with Fargate

This walkthrough uses the AWS CLI to deploy a simple Nginx container.

### Step 1: Create a Cluster

```bash
aws ecs create-cluster --cluster-name demo-cluster
```

### Step 2: Create the Task Execution Role

This role lets ECS pull images and write logs. If you've used ECS before, this role may already exist:

```bash
aws iam create-role \
  --role-name ecsTaskExecutionRole \
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {"Service": "ecs-tasks.amazonaws.com"},
      "Action": "sts:AssumeRole"
    }]
  }'

aws iam attach-role-policy \
  --role-name ecsTaskExecutionRole \
  --policy-arn arn:aws:iam::aws:policy/service-role/AmazonECSTaskExecutionRolePolicy
```

### Step 3: Create a CloudWatch Log Group

```bash
aws logs create-log-group --log-group-name /ecs/demo-nginx
```

### Step 4: Register the Task Definition

Save this as `task-def.json`:

```json
{
  "family": "demo-nginx",
  "networkMode": "awsvpc",
  "requiresCompatibilities": ["FARGATE"],
  "cpu": "256",
  "memory": "512",
  "executionRoleArn": "arn:aws:iam::123456789012:role/ecsTaskExecutionRole",
  "containerDefinitions": [
    {
      "name": "nginx",
      "image": "nginx:latest",
      "portMappings": [
        { "containerPort": 80, "protocol": "tcp" }
      ],
      "logConfiguration": {
        "logDriver": "awslogs",
        "options": {
          "awslogs-group": "/ecs/demo-nginx",
          "awslogs-region": "us-east-1",
          "awslogs-stream-prefix": "nginx"
        }
      },
      "essential": true
    }
  ]
}
```

Register it:

```bash
aws ecs register-task-definition --cli-input-json file://task-def.json
```

### Step 5: Create a Service

You need a VPC, subnets, and a security group. Use your default VPC for learning:

```bash
# Get your default VPC ID
VPC_ID=$(aws ec2 describe-vpcs \
  --filters "Name=isDefault,Values=true" \
  --query "Vpcs[0].VpcId" --output text)

# Get subnet IDs
SUBNETS=$(aws ec2 describe-subnets \
  --filters "Name=vpc-id,Values=$VPC_ID" \
  --query "Subnets[*].SubnetId" --output text | tr '\t' ',')

# Create a security group allowing HTTP
SG_ID=$(aws ec2 create-security-group \
  --group-name ecs-demo-sg \
  --description "Allow HTTP" \
  --vpc-id $VPC_ID \
  --query "GroupId" --output text)

aws ec2 authorize-security-group-ingress \
  --group-id $SG_ID \
  --protocol tcp --port 80 --cidr 0.0.0.0/0

# Create the service
aws ecs create-service \
  --cluster demo-cluster \
  --service-name demo-nginx-service \
  --task-definition demo-nginx \
  --desired-count 2 \
  --launch-type FARGATE \
  --network-configuration "awsvpcConfiguration={subnets=[$SUBNETS],securityGroups=[$SG_ID],assignPublicIp=ENABLED}"
```

This launches 2 Nginx containers. Each gets a public IP. Find them with:

```bash
aws ecs list-tasks --cluster demo-cluster --service-name demo-nginx-service

# Get the task details including the public IP
aws ecs describe-tasks --cluster demo-cluster --tasks <TASK_ARN> \
  --query "tasks[0].attachments[0].details[?name=='networkInterfaceId'].value" --output text
```

## Logging and Monitoring

### CloudWatch Logs

The `awslogs` log driver in the task definition sends container stdout/stderr to CloudWatch Logs. View them in the console under CloudWatch → Log groups, or via CLI:

```bash
aws logs tail /ecs/demo-nginx --follow
```

### Container Insights

Enable Container Insights on your cluster for detailed metrics (CPU, memory, network, storage) per task and per service:

```bash
aws ecs update-cluster-settings \
  --cluster demo-cluster \
  --settings name=containerInsights,value=enabled
```

### Service Events

Check service events to troubleshoot deployment issues:

```bash
aws ecs describe-services \
  --cluster demo-cluster \
  --services demo-nginx-service \
  --query "services[0].events[:5]"
```

## Clean Up

Delete the service, cluster, and security group when you're done:

```bash
aws ecs update-service --cluster demo-cluster \
  --service demo-nginx-service --desired-count 0

aws ecs delete-service --cluster demo-cluster \
  --service demo-nginx-service

aws ecs delete-cluster --cluster demo-cluster

aws ec2 delete-security-group --group-id $SG_ID
```

## Key Takeaways

- Containers package your app with all its dependencies — they run the same everywhere
- ECR is AWS's container registry — push your Docker images here for ECS to pull
- ECS organizes containers into clusters, task definitions, tasks, and services
- Task definitions are JSON blueprints that define images, CPU, memory, ports, and logging
- Fargate is serverless — no EC2 instances to manage, AWS handles the infrastructure
- Services keep a desired number of tasks running and replace crashed containers
- Use CloudWatch Logs with the `awslogs` driver for container logging
- Fargate is more expensive per-unit than EC2 at high utilization, but eliminates operational overhead
