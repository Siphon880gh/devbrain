# Choosing the Right Compute Service

A decision framework for selecting between EC2, Lambda, Fargate, Elastic Beanstalk, and App Runner based on your workload requirements.

## What You'll Learn

- The five main AWS compute services and what makes each one different
- How to choose the right service based on your workload
- Trade-offs between control, simplicity, and cost
- Which service fits common scenarios like web apps, APIs, batch jobs, and scheduled tasks

## The Five Compute Services at a Glance

AWS gives you multiple ways to run code. Each service sits on a spectrum from "you manage everything" to "AWS manages everything."

**EC2 (Elastic Compute Cloud)** — Virtual machines you fully control. You pick the operating system, install software, configure networking, and manage scaling yourself. Maximum flexibility, maximum responsibility.

**Lambda** — Serverless functions. You upload code, and AWS runs it when triggered by events. No servers to manage. You pay only when your code executes. Best for event-driven, short-running tasks.

**Elastic Beanstalk** — Platform-as-a-service. You upload your application code and Beanstalk provisions EC2 instances, load balancers, and auto-scaling groups for you. You still have access to the underlying resources but don't need to set them up manually.

**ECS with Fargate** — Serverless containers. You package your application in a Docker container and Fargate runs it without you managing any servers. Good middle ground between Lambda's simplicity and EC2's flexibility.

**App Runner** — The simplest container hosting. Point it at a container image or a source code repository and it builds, deploys, and scales your app automatically. Minimal configuration needed.

## Service Comparison Matrix

| Feature | EC2 | Lambda | Elastic Beanstalk | ECS/Fargate | App Runner |
|---|---|---|---|---|---|
| **You manage servers** | Yes | No | No (but you can access them) | No (Fargate) | No |
| **Max execution time** | Unlimited | 15 minutes | Unlimited | Unlimited | Unlimited |
| **Startup time** | Minutes | Milliseconds–seconds | Minutes | Seconds | Seconds |
| **Scaling** | Manual or Auto Scaling Groups | Automatic (instant) | Automatic (managed ASG) | Automatic (Service auto-scaling) | Automatic (built-in) |
| **Pricing model** | Per hour/second (running) | Per request + duration | Per EC2 instance hours | Per vCPU + memory per second | Per vCPU + memory per second |
| **Container support** | Yes (install Docker yourself) | Container images supported | Yes (Docker platform) | Native | Native |
| **Custom OS/software** | Full control | No | Limited | Via Dockerfile | Via Dockerfile |
| **Learning curve** | High | Medium | Low | Medium–High | Low |
| **Free tier** | 750 hrs/month t2.micro (12 months) | 1M requests/month (always free) | No (pays for underlying EC2) | No | No |

## When to Use Each Service

### EC2 — When You Need Full Control

Choose EC2 when you need to:

- Run software that requires a specific OS configuration (e.g., a Windows Server application)
- Install custom drivers, kernel modules, or low-level system packages
- Run GPU workloads for machine learning training
- Maintain long-running processes like game servers or database servers
- Have full control over networking, storage volumes, and instance placement

**Typical use cases:** Legacy enterprise applications, custom database servers, high-performance computing, machine learning training, applications that need GPU access.

### Lambda — When You Have Event-Driven, Short Tasks

Choose Lambda when you need to:

- Run code in response to events (file uploaded, HTTP request, message in a queue)
- Process workloads that complete in under 15 minutes
- Handle unpredictable or spiky traffic (scales to zero when idle)
- Keep costs near zero for low-traffic applications

**Typical use cases:** REST APIs (with API Gateway), image/video processing, real-time file processing, scheduled cron jobs, webhook handlers, chat bots.

### Elastic Beanstalk — When You Want Managed Infrastructure for Traditional Web Apps

Choose Elastic Beanstalk when you need to:

- Deploy a standard web application without learning infrastructure
- Use familiar frameworks (Express, Flask, Django, Spring Boot, Rails)
- Get load balancing, auto-scaling, and health monitoring out of the box
- Still have the option to SSH into instances and tweak configurations

**Typical use cases:** Web applications, REST APIs, internal tools, applications being migrated from traditional hosting.

### ECS with Fargate — When You Run Containers and Want Flexibility

Choose Fargate when you need to:

- Run Docker containers without managing EC2 instances
- Have fine-grained control over networking, service discovery, and load balancing
- Run sidecar containers (e.g., log collectors alongside your app)
- Integrate tightly with other AWS services via IAM task roles

**Typical use cases:** Microservices architectures, containerized APIs, background workers, applications that need multiple containers working together.

### App Runner — When You Want the Fastest Path to Production

Choose App Runner when you need to:

- Deploy a container or source code with minimal configuration
- Get automatic HTTPS, scaling, and deployments
- Avoid learning ECS task definitions, services, and clusters
- Ship a prototype or small production service quickly

**Typical use cases:** Simple web apps, REST APIs, internal tools, prototypes, startups that want to ship fast.

## Cost Considerations

**Lowest cost for low/no traffic:** Lambda. It scales to zero — you pay nothing when no requests come in. This makes it ideal for side projects, development environments, and applications with unpredictable usage.

**Lowest cost for steady, high traffic:** EC2 with Reserved Instances or Savings Plans. When you know your workload is constant, committing to 1-year or 3-year terms can save 30–70% compared to on-demand pricing.

**Middle ground:** Fargate and App Runner charge per vCPU-second and GB-second of memory. More expensive than a dedicated EC2 instance at full utilization, but you avoid paying for idle capacity.

**Hidden costs to watch:**

- **EC2:** You pay even when the instance is idle (stopped instances still incur EBS storage costs)
- **Lambda:** At high request volumes (millions/day), Lambda can become more expensive than a dedicated server
- **Elastic Beanstalk:** Free itself, but you pay for the EC2 instances, load balancers, and other resources it creates
- **Fargate:** No free tier — minimum cost for a single small task is roughly $0.01/hour
- **App Runner:** Pauses to a lower cost when idle, but doesn't scale to zero completely

## Decision Flowchart

Ask yourself these questions in order:

**1. Does your code run for more than 15 minutes at a time?**
- No → Lambda is a strong candidate
- Yes → Continue to question 2

**2. Do you need full control over the operating system?**
- Yes → EC2
- No → Continue to question 3

**3. Is your application containerized (or can it be)?**
- No → Elastic Beanstalk (supports raw source code in many languages)
- Yes → Continue to question 4

**4. Do you need advanced container features (service mesh, sidecar containers, custom networking)?**
- Yes → ECS with Fargate (or EKS for Kubernetes)
- No → App Runner (simplest container hosting)

**5. Still unsure?**
- Starting a new project and want to move fast → App Runner or Elastic Beanstalk
- Building microservices → ECS with Fargate
- Building an event-driven system → Lambda
- Migrating a legacy application → EC2 or Elastic Beanstalk

## Real-World Scenario Examples

| Scenario | Best Fit | Why |
|---|---|---|
| A REST API with 100 requests/day | Lambda + API Gateway | Scales to zero, costs nearly nothing |
| A Node.js web app with steady traffic | Elastic Beanstalk | Quick deploy, managed scaling |
| A microservices platform with 10 services | ECS/Fargate | Fine-grained container orchestration |
| A batch job that processes files for 2 hours | EC2 (Spot Instance) | Long-running, cost-effective with Spot |
| An internal admin dashboard | App Runner | Simple deploy, auto HTTPS |
| A cron job that runs every 5 minutes | Lambda + EventBridge | No server idle costs between runs |
| A legacy Java app with custom JVM settings | EC2 | Needs full OS-level control |
| A containerized API for a startup MVP | App Runner | Fastest path to production URL |

## Key Takeaways

- There is no single "best" compute service — the right choice depends on your workload, team experience, and budget
- Lambda is the cheapest option for low-traffic and event-driven workloads because it scales to zero
- EC2 gives you maximum control but requires you to manage everything (patching, scaling, monitoring)
- Elastic Beanstalk is the fastest way to deploy a traditional web app without learning infrastructure
- Fargate is the best choice when you need container flexibility without managing servers
- App Runner is the simplest option for getting a container or source code into production
- You can (and often will) use multiple compute services in the same project — Lambda for background jobs, Fargate for APIs, EC2 for specialized workloads
