# App Runner — Simple Container Hosting

Deploy containerized web apps and APIs without cluster management — the simplest path from container to production URL.

## What You'll Learn

- What App Runner is and what problems it solves
- How App Runner compares to ECS, Elastic Beanstalk, and Lambda
- How to deploy from a container image in ECR
- How to deploy directly from source code in a GitHub repository
- Auto-scaling, custom domains, and configuration options
- Pricing and when App Runner makes sense
- Limitations and when to move to a more powerful service

## What is App Runner?

App Runner is the **simplest way to run a containerized web application on AWS**. You point it at a container image or a source code repository, and it handles everything else: building, deploying, scaling, load balancing, TLS certificates, and health monitoring.

There is no cluster to create, no task definition to write, no load balancer to configure, and no VPC to set up. App Runner abstracts all of that away.

Here's how it works:

```
You provide:                    App Runner handles:
─────────────                   ───────────────────
Container image (ECR)           Build and deploy
  OR                            Load balancing
Source code (GitHub)            TLS/HTTPS certificates
  +                             Auto-scaling (to zero-ish)
A few settings                  Health checks
                                Logging
                                ↓
                                https://your-app.awsapprunner.com
```

Think of it as **Heroku on AWS** — you push code, it runs.

## App Runner vs ECS vs Elastic Beanstalk

| Feature | App Runner | ECS/Fargate | Elastic Beanstalk |
|---|---|---|---|
| **Setup complexity** | Minimal (5 min) | Medium (30+ min) | Medium (15 min) |
| **Infrastructure you see** | None | Tasks, services, clusters | EC2 instances, ALB, ASG |
| **VPC required** | No (optional) | Yes | Yes |
| **Load balancer** | Built-in (hidden) | You configure ALB/NLB | Auto-configured ALB |
| **Source types** | ECR image or GitHub repo | ECR image | Source code zip, Docker |
| **Auto-scaling** | Built-in, request-based | You configure | You configure |
| **SSH access** | No | No (Fargate) | Yes |
| **Sidecar containers** | No | Yes | No |
| **Custom networking** | Optional VPC connector | Full VPC control | Full VPC control |
| **Best for** | Simple web apps, APIs | Microservices, complex apps | Traditional web apps |

**When App Runner wins:** You have a single container or source repo and want to deploy it with minimal configuration. No sidecar containers, no custom networking, no complex service discovery needed.

**When App Runner isn't enough:** You need multiple containers per task, service mesh, custom VPC networking, GPU support, or integration with ECS Service Connect/Cloud Map.

## Deploying from Source Code or Container Image

App Runner supports two source types:

### Source Type 1: Container Image from ECR

You build your own Docker image, push it to Amazon ECR, and point App Runner at it.

**Prerequisites:**
- A Docker image pushed to ECR (see the ECS lesson for how to push to ECR)
- An App Runner ECR access role (so App Runner can pull your image)

### Source Type 2: Source Code from GitHub

You connect a GitHub repository, and App Runner builds the container for you using a managed build environment. Supported runtimes:

- Python 3
- Node.js 16, 18
- Java 11, 17 (Corretto)
- .NET 6
- Go 1.18+
- PHP 8.1
- Ruby 3.1

You provide build and start commands in the console or in an `apprunner.yaml` config file:

```yaml
version: 1.0
runtime: nodejs18
build:
  commands:
    build:
      - npm ci
run:
  command: node server.js
  network:
    port: 3000
  env:
    - name: NODE_ENV
      value: production
```

Place this file in your repo root and App Runner uses it automatically.

## Hands-On: Deploy a Container with App Runner

### Deploy from ECR via the Console

1. Go to **App Runner** in the AWS Console
2. Click **Create service**
3. Source: **Container registry** → Amazon ECR
4. Container image URI: paste your ECR image URI (e.g., `123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:latest`)
5. ECR access role: Create a new role or select an existing one
6. Deployment trigger: **Automatic** (redeploys when you push a new image) or **Manual**
7. Click **Next**
8. Service name: `my-app-runner-service`
9. CPU: 1 vCPU, Memory: 2 GB (adjust based on your app)
10. Port: Enter the port your app listens on (e.g., 3000)
11. Environment variables: Add any your app needs
12. Click **Next**, then **Create & deploy**

In about 2–5 minutes, your service is live with a URL like `https://abc123.us-east-1.awsapprunner.com`.

### Deploy from ECR via the CLI

```bash
# Create the App Runner service
aws apprunner create-service \
  --service-name my-web-api \
  --source-configuration '{
    "AuthenticationConfiguration": {
      "AccessRoleArn": "arn:aws:iam::123456789012:role/AppRunnerECRAccessRole"
    },
    "AutoDeploymentsEnabled": true,
    "ImageRepository": {
      "ImageIdentifier": "123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:latest",
      "ImageConfiguration": {
        "Port": "3000",
        "RuntimeEnvironmentVariables": {
          "NODE_ENV": "production",
          "LOG_LEVEL": "info"
        }
      },
      "ImageRepositoryType": "ECR"
    }
  }' \
  --instance-configuration '{
    "Cpu": "1024",
    "Memory": "2048"
  }'
```

Check the deployment status:

```bash
aws apprunner describe-service \
  --service-arn arn:aws:apprunner:us-east-1:123456789012:service/my-web-api/abc123 \
  --query "Service.{Status:Status,URL:ServiceUrl}"
```

### Deploy from GitHub via the Console

1. Go to **App Runner** → **Create service**
2. Source: **Source code repository**
3. Connect to GitHub (you'll authorize AWS to access your repos)
4. Select your repository and branch
5. Build settings: Choose your runtime (e.g., Node.js 18)
6. Build command: `npm ci`
7. Start command: `node server.js`
8. Port: `3000`
9. Click through and deploy

App Runner creates a build pipeline that automatically redeploys when you push to the selected branch.

## Auto Scaling and Custom Domains

### Auto Scaling

App Runner scales automatically based on incoming request traffic. You configure:

- **Max concurrency** — how many concurrent requests each instance handles before scaling out (default: 100)
- **Max size** — maximum number of instances (default: 25)
- **Min size** — minimum instances to keep warm (default: 1, set to 0 for scale-to-zero-ish)

```bash
aws apprunner create-auto-scaling-configuration \
  --auto-scaling-configuration-name my-scaling \
  --max-concurrency 50 \
  --min-size 1 \
  --max-size 10
```

When traffic drops to zero, App Runner pauses your instances to a lower-cost state. It's not true scale-to-zero like Lambda (there's still a small charge), but it's significantly cheaper than running idle.

### Custom Domains

App Runner assigns a default URL (`*.awsapprunner.com`). To use your own domain:

```bash
aws apprunner associate-custom-domain \
  --service-arn arn:aws:apprunner:us-east-1:123456789012:service/my-web-api/abc123 \
  --domain-name api.example.com
```

App Runner provides CNAME records you need to add to your DNS. It automatically provisions and renews TLS certificates via AWS Certificate Manager.

### Health Checks

App Runner performs health checks on your application. Configure the path and thresholds:

```bash
aws apprunner create-service \
  ... \
  --health-check-configuration '{
    "Protocol": "HTTP",
    "Path": "/health",
    "Interval": 10,
    "Timeout": 5,
    "HealthyThreshold": 1,
    "UnhealthyThreshold": 3
  }'
```

Make sure your application responds with a 200 status code on the health check path.

## Pricing

App Runner pricing is based on compute and memory, charged per second:

| Resource | Active (processing requests) | Paused (no traffic) |
|---|---|---|
| **vCPU** | $0.064 per vCPU-hour | $0.007 per GB-hour (provisioned memory only) |
| **Memory** | $0.007 per GB-hour | $0.007 per GB-hour |

**Automatic deployments from ECR:** $1/month per service for the build pipeline.

**Example monthly cost:**

A service with 1 vCPU / 2 GB memory running 8 hours/day, paused 16 hours/day:
- Active: 8h × 30d × ($0.064 + 2 × $0.007) = $18.72
- Paused: 16h × 30d × (2 × $0.007) = $6.72
- **Total: ~$25/month**

Compare to Fargate running 24/7 with the same specs: ~$45/month.

**When App Runner saves money:**
- Applications with variable traffic (scales down when idle)
- Services that don't need to run 24/7
- Small teams that value reduced operational overhead over per-unit cost

**When App Runner costs more:**
- High-traffic services running at constant load (Fargate or EC2 may be cheaper)
- Multiple interconnected services (ECS service discovery is included at no extra cost)

## Limitations and When to Graduate

App Runner is intentionally simple. That means limitations:

**What you can't do with App Runner:**
- Run sidecar containers (one container per service)
- Access a GPU
- Use advanced networking (service mesh, service discovery)
- Run non-HTTP workloads (no TCP, UDP, or gRPC listeners)
- Execute background tasks or cron jobs (it's designed for request/response)
- Access the underlying compute (no SSH, no host metrics)
- Use persistent storage (no EBS volumes)

**When to graduate to ECS/Fargate:**
- You need multiple containers (sidecars for logging, proxies, etc.)
- You need to connect to resources in a private VPC without a VPC connector
- You need service-to-service communication with service discovery
- You need more than 4 vCPU or 12 GB memory per instance
- You need advanced deployment strategies (blue/green, canary)

**When to graduate to EKS:**
- You need Kubernetes-native tooling (Helm, Istio, ArgoCD)
- You need multi-cloud portability
- You have a platform team managing infrastructure for many application teams

## Clean Up

Delete your App Runner service when you're done:

```bash
aws apprunner delete-service \
  --service-arn arn:aws:apprunner:us-east-1:123456789012:service/my-web-api/abc123
```

Or in the console: select your service → Actions → Delete.

## Key Takeaways

- App Runner is the simplest way to run a web app or API on AWS — no clusters, no VPCs, no load balancers to configure
- Deploy from a container image in ECR or directly from source code in GitHub
- Auto-scaling is built in — it scales based on request concurrency and pauses when idle
- Custom domains and TLS certificates are handled automatically
- App Runner is ideal for simple web apps, APIs, and prototypes where speed of deployment matters more than infrastructure control
- Graduate to ECS/Fargate when you need sidecars, advanced networking, or more compute resources
- Pricing is per-second for CPU and memory, with reduced rates when paused — cost-effective for variable-traffic workloads
