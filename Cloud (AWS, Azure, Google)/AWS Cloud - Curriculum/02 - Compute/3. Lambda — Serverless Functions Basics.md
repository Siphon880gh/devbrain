# Lambda — Serverless Functions Basics

Write and deploy serverless functions that run in response to events without provisioning or managing servers.

## What You'll Learn

- What serverless computing means and how Lambda works under the hood
- The Lambda execution model: cold starts, execution environments, and concurrency
- How to write handler functions in Python and Node.js
- Common triggers: API Gateway, S3, SQS, and EventBridge
- How to deploy functions via the console and CLI
- Lambda pricing: requests + duration
- Configuration options: environment variables, layers, memory, and timeouts

## How Lambda Works

### What is Serverless?

"Serverless" doesn't mean there are no servers — it means **you don't manage them**. AWS handles provisioning, scaling, patching, and maintaining the infrastructure. You just write a function and upload it.

With Lambda:
- You write a function (a single piece of code that does one thing)
- You define what triggers it (an HTTP request, a file upload, a timer, etc.)
- AWS runs your function when the trigger fires
- You pay only for the time your function is executing

When no requests are coming in, you pay nothing. There are no idle servers.

### The Execution Model

When a trigger invokes your Lambda function, here's what happens:

1. **AWS creates an execution environment** — a lightweight container with your code, runtime (Python, Node.js, etc.), and any dependencies
2. **Your handler function runs** — it receives an event object (the trigger data) and a context object (metadata about the invocation)
3. **The function returns a response** — or throws an error
4. **The execution environment stays warm** — for a short time (usually 5–15 minutes), AWS keeps the container alive in case another request comes in

### Cold Starts

A **cold start** happens when there's no warm execution environment available. AWS must create a new one, which adds latency (typically 100ms–1s for Python/Node.js, longer for Java/.NET).

Cold starts occur when:
- Your function is invoked for the first time
- All existing environments are busy handling other requests
- AWS has recycled idle environments after a period of inactivity

For most use cases, cold starts are barely noticeable. If sub-100ms latency is critical, you can use **Provisioned Concurrency** to keep environments warm (at additional cost).

## Triggers and Event Sources

Lambda functions are event-driven — they run when something happens. Here are the most common triggers:

| Trigger | What Fires It | Typical Use Case |
|---|---|---|
| **API Gateway** | HTTP request (GET, POST, etc.) | REST APIs, webhooks |
| **S3** | File uploaded, deleted, or modified | Image processing, data pipeline |
| **SQS** | Message arrives in a queue | Background job processing |
| **EventBridge** | Scheduled rule or custom event | Cron jobs, event routing |
| **DynamoDB Streams** | Item created/updated/deleted in a table | Real-time data processing |
| **SNS** | Notification published to a topic | Fan-out processing |
| **CloudWatch Logs** | New log entry matches a filter | Log analysis, alerts |

A single Lambda function can have multiple triggers. For example, a function could process files from S3 and also respond to HTTP requests from API Gateway.

## Writing Your First Function

### Python Handler

Every Lambda function has a **handler** — the function that AWS calls when the trigger fires. It receives two arguments:

```python
import json

def lambda_handler(event, context):
    """
    event  — the trigger data (e.g., HTTP request body, S3 event details)
    context — metadata (function name, memory limit, remaining time)
    """
    name = event.get("name", "World")

    return {
        "statusCode": 200,
        "headers": {"Content-Type": "application/json"},
        "body": json.dumps({"message": f"Hello, {name}!"})
    }
```

### Node.js Handler

```javascript
export const handler = async (event) => {
    const name = event.name || "World";

    return {
        statusCode: 200,
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({ message: `Hello, ${name}!` }),
    };
};
```

The `event` object shape depends on the trigger. An API Gateway event looks very different from an S3 event. AWS documentation provides the event structure for every trigger type.

### S3 Trigger Example (Python)

This function runs whenever a file is uploaded to an S3 bucket:

```python
import json

def lambda_handler(event, context):
    for record in event["Records"]:
        bucket = record["s3"]["bucket"]["name"]
        key = record["s3"]["object"]["key"]
        size = record["s3"]["object"]["size"]

        print(f"New file: s3://{bucket}/{key} ({size} bytes)")

    return {"statusCode": 200, "body": "Processed"}
```

## Configuration: Memory, Timeout, and Environment Variables

### Memory and CPU

Lambda lets you configure memory from **128 MB to 10,240 MB** (10 GB). CPU power scales proportionally with memory — at 1,769 MB you get one full vCPU.

More memory = more CPU = faster execution = higher cost per millisecond. For many functions, the sweet spot is 256–512 MB.

### Timeout

The maximum timeout is **15 minutes** (900 seconds). The default is 3 seconds, which is too short for most real work. Set it based on your expected execution time, plus a buffer:

- API handlers: 10–30 seconds
- File processing: 1–5 minutes
- Data pipelines: 5–15 minutes

If your function hits the timeout, it's forcefully killed with no graceful shutdown.

### Environment Variables

Store configuration outside your code using environment variables:

```bash
# Set via CLI when creating the function
aws lambda create-function \
  --function-name my-function \
  --runtime python3.12 \
  --handler lambda_function.lambda_handler \
  --role arn:aws:iam::123456789012:role/lambda-role \
  --zip-file fileb://function.zip \
  --environment "Variables={DB_HOST=mydb.cluster.us-east-1.rds.amazonaws.com,STAGE=production}"
```

Access them in your code:

```python
import os

db_host = os.environ["DB_HOST"]
stage = os.environ["STAGE"]
```

For sensitive values (API keys, database passwords), use AWS Secrets Manager or Parameter Store instead of plain environment variables.

### Lambda Layers

Layers let you package libraries and dependencies separately from your function code. This is useful when multiple functions share the same dependencies.

```bash
# Create a layer from a zip file containing your dependencies
aws lambda publish-layer-version \
  --layer-name my-python-libs \
  --compatible-runtimes python3.12 \
  --zip-file fileb://python-libs.zip
```

A function can use up to 5 layers. The total unzipped size (function + all layers) must be under 250 MB.

## Hands-On: Deploy a Lambda Function

### Create via the Console

1. Go to the **Lambda** service in the AWS Console
2. Click **Create function**
3. Choose **Author from scratch**
4. Function name: `hello-lambda`
5. Runtime: Python 3.12
6. Architecture: x86_64
7. Click **Create function**
8. In the code editor, replace the default code with:

```python
import json

def lambda_handler(event, context):
    name = event.get("queryStringParameters", {}).get("name", "World")
    return {
        "statusCode": 200,
        "body": json.dumps({"message": f"Hello, {name}!"})
    }
```

9. Click **Deploy**
10. Click **Test** → Create a test event → Use the default JSON → Click **Test**

### Create via the CLI

Write your function code to a file, zip it, and deploy:

```bash
# Create the function file
cat > lambda_function.py << 'EOF'
import json

def lambda_handler(event, context):
    name = event.get("name", "World")
    return {
        "statusCode": 200,
        "body": json.dumps({"message": f"Hello, {name}!"})
    }
EOF

# Zip it
zip function.zip lambda_function.py

# Create the function (you need a Lambda execution role — see IAM lesson)
aws lambda create-function \
  --function-name hello-lambda \
  --runtime python3.12 \
  --handler lambda_function.lambda_handler \
  --role arn:aws:iam::123456789012:role/lambda-execution-role \
  --zip-file fileb://function.zip

# Test it
aws lambda invoke \
  --function-name hello-lambda \
  --payload '{"name": "AWS Learner"}' \
  --cli-binary-format raw-in-base64-out \
  response.json

cat response.json
```

### Add an API Gateway Trigger

To make your function accessible via HTTP:

1. In the Lambda console, click **Add trigger**
2. Select **API Gateway**
3. Choose **Create a new API** → HTTP API
4. Security: Open (for testing)
5. Click **Add**

You'll get a public URL like `https://abc123.execute-api.us-east-1.amazonaws.com/default/hello-lambda`. Visit it in your browser to see the response.

## Pricing Model

Lambda pricing has two components:

**1. Requests:** $0.20 per 1 million requests (first 1 million per month are free)

**2. Duration:** Charged per GB-second of memory allocated
- $0.0000166667 per GB-second
- A function with 512 MB memory running for 1 second costs ~$0.000008

**Free Tier (always free, not just 12 months):**
- 1 million requests per month
- 400,000 GB-seconds of compute per month

**Example cost calculation:**
- Function: 256 MB memory, 200ms average duration
- Traffic: 10 million requests/month
- Compute: 10M × 0.2s × 0.25 GB = 500,000 GB-seconds
- Cost: (9M requests × $0.20/1M) + (100,000 GB-seconds × $0.0000166667) = $1.80 + $1.67 = **~$3.47/month**

That same workload on a `t3.small` EC2 instance would cost ~$15/month running 24/7.

## Key Takeaways

- Lambda runs your code without any server management — you upload a function and AWS handles everything else
- Cold starts add latency on the first invocation but are usually under 1 second for Python and Node.js
- The maximum execution time is 15 minutes — if your task takes longer, use EC2, Fargate, or Step Functions
- Lambda scales automatically from zero to thousands of concurrent executions
- Pricing is per-request and per-duration — extremely cost-effective for low-to-moderate traffic
- The Free Tier includes 1 million requests/month permanently (not just 12 months)
- Use environment variables for configuration and Secrets Manager for sensitive data
- Layers let you share dependencies across multiple functions
