# EKS Overview

Managed Kubernetes on AWS — when you need Kubernetes-native orchestration at scale.

## What You'll Learn

- What Kubernetes is and what problems it solves
- What EKS provides on top of Kubernetes
- EKS architecture: control plane and data plane
- When to choose EKS over ECS
- How to create a cluster with `eksctl`
- Fargate profiles for EKS
- Cost considerations and why EKS may not be ideal for beginners

## What is Kubernetes?

Kubernetes (often shortened to "K8s") is an open-source system for automating the deployment, scaling, and management of containerized applications. Originally built by Google, it's now the industry standard for container orchestration.

At its core, Kubernetes does the same job as ECS — it runs containers, keeps them healthy, scales them up and down, and handles networking between them. The difference is that Kubernetes is:

- **Open source** — not locked to any cloud provider
- **Highly extensible** — a massive ecosystem of add-ons and tools
- **Portable** — the same Kubernetes manifests work on AWS, GCP, Azure, or on-premises
- **Complex** — a significantly steeper learning curve than ECS

Kubernetes uses a declarative model. You write YAML files that describe your desired state ("I want 3 replicas of my web app"), and Kubernetes continuously works to make reality match your description.

### Key Kubernetes Concepts

| Concept | What It Is | ECS Equivalent |
|---|---|---|
| **Pod** | Smallest deployable unit — one or more containers running together | Task |
| **Deployment** | Manages a set of identical pods with rolling updates | Service |
| **Service** | Stable network endpoint for accessing pods | Service + Load Balancer |
| **Namespace** | Logical partition within a cluster | No direct equivalent |
| **Node** | A machine (EC2 instance) that runs pods | EC2 instance in ECS cluster |

A simple Kubernetes Deployment looks like this:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-web-app
  template:
    metadata:
      labels:
        app: my-web-app
    spec:
      containers:
        - name: web
          image: 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:latest
          ports:
            - containerPort: 3000
          resources:
            requests:
              cpu: "250m"
              memory: "256Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"
```

## EKS vs Self-Managed Kubernetes

Running Kubernetes yourself means setting up and maintaining the control plane — the API server, etcd database, scheduler, and controller manager. This is complex, error-prone, and time-consuming.

**EKS manages the control plane for you.** AWS runs the Kubernetes API server and etcd across multiple Availability Zones, handles upgrades, patches, and ensures high availability.

| Aspect | Self-Managed Kubernetes | EKS |
|---|---|---|
| **Control plane** | You install, configure, and patch | AWS manages it |
| **Upgrades** | You perform manually | AWS provides managed upgrades |
| **Availability** | You configure HA | Multi-AZ by default |
| **etcd** | You manage backups and recovery | AWS handles it |
| **Worker nodes** | You manage | You manage (or use Fargate) |
| **Cost** | EC2 cost only | EC2 + $0.10/hour for control plane |

## EKS Architecture

An EKS cluster has two parts:

### Control Plane (Managed by AWS)

- Kubernetes API server — where `kubectl` commands go
- etcd — distributed database storing cluster state
- Scheduler — decides which node runs each pod
- Controller manager — ensures desired state matches actual state

You never access the control plane instances directly. AWS runs them across 3 Availability Zones for high availability.

### Data Plane (Managed by You)

The data plane is where your containers actually run. You have three options:

**1. Managed Node Groups** — AWS provisions and manages EC2 instances for you. You choose the instance type and count, AWS handles node setup, registration, and updates.

```bash
eksctl create nodegroup \
  --cluster my-cluster \
  --name my-nodes \
  --node-type t3.medium \
  --nodes 3
```

**2. Self-Managed Nodes** — You create and manage EC2 instances yourself. Maximum control, maximum effort.

**3. Fargate** — Serverless pods. No nodes to manage at all — each pod runs in its own isolated environment. Best for getting started and for workloads that don't need node-level access.

## When to Choose EKS Over ECS

**Choose EKS when:**

- Your team already knows Kubernetes
- You need to run the same workloads across multiple clouds (AWS + GCP + on-premises)
- You rely on Kubernetes-specific tools (Helm, Istio, ArgoCD, Prometheus)
- You need advanced scheduling (affinity rules, taints, tolerations)
- You're running a large platform with many teams sharing one cluster (namespace isolation)

**Choose ECS when:**

- You're an AWS-focused team and don't need multi-cloud portability
- You want simpler operations with less overhead
- You're a small team that doesn't need the Kubernetes ecosystem
- You want tighter native integration with AWS services (simpler IAM, service discovery)
- You're new to containers and want a gentler learning curve

**The honest reality:** If you're asking "should I use EKS or ECS?" — the answer is almost always ECS. EKS makes sense when you have Kubernetes expertise on your team or a specific need for Kubernetes-native tooling.

## Getting Started with EKS

### Install Prerequisites

You need three tools:

```bash
# AWS CLI (if not already installed)
# See: https://docs.aws.amazon.com/cli/latest/userguide/getting-started-install.html

# eksctl — CLI tool for creating EKS clusters
brew install eksctl        # macOS
# Or: choco install eksctl  # Windows

# kubectl — CLI tool for managing Kubernetes
brew install kubectl        # macOS
# Or: choco install kubernetes-cli  # Windows
```

### Create a Cluster with eksctl

`eksctl` is the easiest way to create an EKS cluster. A single command creates the control plane, node group, VPC, and all networking:

```bash
eksctl create cluster \
  --name my-cluster \
  --region us-east-1 \
  --nodegroup-name my-nodes \
  --node-type t3.medium \
  --nodes 2 \
  --nodes-min 1 \
  --nodes-max 4
```

This takes 15–20 minutes. When it's done, `eksctl` automatically configures `kubectl` to connect to your cluster.

Verify the connection:

```bash
kubectl get nodes
```

You should see your 2 worker nodes listed as "Ready."

### Deploy a Simple Application

```bash
# Create a deployment
kubectl create deployment nginx --image=nginx:latest --replicas=2

# Expose it with a load balancer
kubectl expose deployment nginx --port=80 --type=LoadBalancer

# Wait a minute, then get the external URL
kubectl get service nginx
```

The `EXTERNAL-IP` column shows the load balancer URL. Visit it in your browser to see the Nginx welcome page.

### Clean Up a Deployment

```bash
kubectl delete service nginx
kubectl delete deployment nginx
```

## Fargate Profiles for EKS

Just like ECS, EKS can use Fargate to run pods without managing nodes. You define a **Fargate profile** that tells EKS which pods should run on Fargate (based on namespace and labels).

```bash
eksctl create fargateprofile \
  --cluster my-cluster \
  --name my-fargate-profile \
  --namespace my-app
```

Now any pod deployed to the `my-app` namespace runs on Fargate instead of EC2 nodes. This is great for specific workloads where you don't want to manage nodes, but note that Fargate pods have some limitations:

- No DaemonSets (can't run agents on every node — there are no nodes)
- No privileged containers
- No GPU support
- Slightly slower pod startup (~30 seconds)
- Maximum 4 vCPU and 30 GB memory per pod

## Cost Considerations

EKS is one of the more expensive container options on AWS:

| Cost Component | Price |
|---|---|
| **EKS control plane** | $0.10/hour = **$73/month per cluster** |
| **Worker nodes (EC2)** | Standard EC2 pricing (e.g., t3.medium = ~$30/month) |
| **Fargate pods** | Per vCPU-second + memory-second (same as ECS Fargate) |
| **Load balancer** | ~$16/month + data transfer |

**Minimum monthly cost for a basic cluster:**
- Control plane: $73
- 2x t3.medium nodes: $60
- Load balancer: $16
- **Total: ~$149/month before any workloads**

Compare this to ECS, which has no control plane charge — the ECS control plane is free.

### Cost Optimization Tips

- Use **Spot Instances** for non-critical node groups (up to 90% savings)
- Use **Fargate** for bursty workloads to avoid paying for idle nodes
- Use **Karpenter** (node auto-provisioning) to right-size nodes automatically
- Shut down development clusters when not in use
- Consider ECS if you don't specifically need Kubernetes

## Delete the Cluster

When you're done experimenting, delete everything:

```bash
eksctl delete cluster --name my-cluster --region us-east-1
```

This deletes the control plane, node groups, VPC, and all associated resources. It takes about 10 minutes.

## Key Takeaways

- Kubernetes is the industry standard for container orchestration, but it's complex
- EKS manages the Kubernetes control plane — you manage the worker nodes (or use Fargate)
- EKS costs $0.10/hour ($73/month) just for the control plane, on top of compute costs
- Use `eksctl` to create clusters quickly — it sets up VPC, nodes, and networking in one command
- Choose EKS when your team knows Kubernetes or needs multi-cloud portability
- Choose ECS for simpler operations, lower cost, and tighter AWS integration
- Fargate profiles let you run specific pods serverlessly, but with limitations
- **For beginners: Start with ECS.** Learn EKS later when you have a specific need for it.
