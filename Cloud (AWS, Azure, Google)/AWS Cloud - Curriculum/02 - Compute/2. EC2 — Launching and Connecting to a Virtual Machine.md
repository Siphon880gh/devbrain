# EC2 — Launching and Connecting to a Virtual Machine

Launch your first EC2 instance, configure security groups, and connect via SSH or Session Manager.

## What You'll Learn

- What EC2 is and how virtual machines work in the cloud
- How to choose an instance type and Amazon Machine Image (AMI)
- How to create key pairs and configure security groups
- How to launch an instance via the console and CLI
- How to connect via SSH and run commands on your instance
- The EC2 instance lifecycle (running, stopped, terminated)
- How to use user data scripts to automate setup on launch

## What is EC2?

EC2 (Elastic Compute Cloud) is AWS's virtual machine service. When you launch an EC2 instance, AWS allocates a physical server in one of its data centers and runs a virtual machine on it for you. You get full root/admin access to this virtual machine — you can install software, run services, and configure it however you want.

Think of it as renting a computer in the cloud. You pick the size (CPU, memory, storage), choose an operating system, and it's ready in about 60 seconds.

## Instance Types and Sizing

Every EC2 instance has a **type** that determines its hardware profile. Instance types follow a naming pattern:

```
t2.micro
│ │  └── Size (nano, micro, small, medium, large, xlarge, 2xlarge...)
│ └──── Generation (higher = newer)
└────── Family (t = burstable, m = general purpose, c = compute optimized...)
```

**Common instance families for beginners:**

| Family | Optimized For | Example Use Case |
|---|---|---|
| **t2/t3** | Burstable general purpose | Dev/test, small web apps, learning |
| **m5/m6i** | Balanced compute + memory | Production web apps, app servers |
| **c5/c6i** | Compute-intensive | Data processing, batch jobs, gaming |
| **r5/r6i** | Memory-intensive | Databases, in-memory caching |

**For this tutorial, use `t2.micro`** — it has 1 vCPU, 1 GB of memory, and is included in the AWS Free Tier (750 hours/month for 12 months).

## Launching an Instance

### Choosing an AMI (Amazon Machine Image)

An AMI is a pre-built template that contains the operating system and sometimes pre-installed software. When you launch an instance, you choose an AMI as your starting point.

**Recommended for beginners: Amazon Linux 2023**
- Built by AWS, optimized for EC2
- Lightweight, fast boot times
- Pre-installed AWS CLI and common tools
- Long-term support with security updates

Other popular AMIs: Ubuntu Server, Debian, Red Hat Enterprise Linux, Windows Server.

### Step-by-Step: Launch via the AWS Console

1. Open the **EC2 Dashboard** in the AWS Console
2. Click **Launch Instance**
3. **Name:** Give your instance a name (e.g., "my-first-server")
4. **AMI:** Select "Amazon Linux 2023" (should be the default, Free Tier eligible)
5. **Instance type:** Select `t2.micro` (Free Tier eligible)
6. **Key pair:** Click "Create new key pair"
   - Name it (e.g., "my-key")
   - Key pair type: RSA
   - Format: `.pem` for Mac/Linux, `.ppk` for PuTTY on Windows
   - Download and save the file — you cannot download it again
7. **Network settings:** Click "Edit" and verify:
   - Auto-assign public IP: **Enable**
   - Create a security group or select existing
   - Add rule: SSH (port 22) from "My IP" (not 0.0.0.0/0 for security)
8. **Storage:** Default 8 GB gp3 is fine for learning
9. Click **Launch Instance**

Your instance will be in "Pending" state for about 30–60 seconds, then transition to "Running."

## Key Pairs and Security Groups

### Key Pairs

A key pair is how you authenticate when connecting to your instance via SSH. AWS stores the **public key** on the instance, and you keep the **private key** (the `.pem` file) on your local machine.

After downloading your key pair, restrict its permissions immediately:

```bash
chmod 400 my-key.pem
```

Without this, SSH will refuse to use the key because the permissions are too open.

### Security Groups

A security group acts as a virtual firewall for your instance. It controls what traffic can reach your instance (inbound rules) and what traffic can leave (outbound rules).

**Minimum security group rules for SSH access:**

| Type | Protocol | Port | Source | Purpose |
|---|---|---|---|---|
| SSH | TCP | 22 | Your IP (e.g., 203.0.113.50/32) | Allow SSH from your machine |

**Important security rules:**
- Never open SSH (port 22) to `0.0.0.0/0` (the entire internet) in production
- Use "My IP" in the console to automatically fill in your current IP address
- If your IP changes (common with home internet), you'll need to update the rule
- For web servers, you'll add HTTP (port 80) and HTTPS (port 443) rules later

## Connecting to Your Instance

### Connect via SSH (Mac/Linux/WSL)

Once your instance is running, find its **Public IPv4 address** in the EC2 console and connect:

```bash
ssh -i my-key.pem ec2-user@<PUBLIC_IP_ADDRESS>
```

For Amazon Linux 2023, the default username is `ec2-user`. For Ubuntu, it's `ubuntu`. For Debian, it's `admin`.

If you see "The authenticity of host can't be established" — type `yes` to continue. This is normal for the first connection.

### Connect via EC2 Instance Connect (Browser)

If you don't want to use SSH from your terminal:

1. Select your instance in the EC2 console
2. Click **Connect**
3. Choose the **EC2 Instance Connect** tab
4. Click **Connect**

This opens a browser-based terminal session — no key pair needed.

### Connect via Session Manager (No SSH Required)

Session Manager is part of AWS Systems Manager. It lets you connect without opening port 22 at all. This is the most secure option, but requires the SSM Agent (pre-installed on Amazon Linux 2023) and an IAM role attached to the instance.

## Hands-On: Launch and SSH into an EC2 Instance

### Launch via the AWS CLI

First, find the latest Amazon Linux 2023 AMI ID:

```bash
aws ec2 describe-images \
  --owners amazon \
  --filters "Name=name,Values=al2023-ami-2023*-x86_64" \
            "Name=state,Values=available" \
  --query "sort_by(Images, &CreationDate)[-1].ImageId" \
  --output text
```

Create a key pair:

```bash
aws ec2 create-key-pair \
  --key-name my-key \
  --query "KeyMaterial" \
  --output text > my-key.pem

chmod 400 my-key.pem
```

Create a security group:

```bash
aws ec2 create-security-group \
  --group-name my-sg \
  --description "Allow SSH from my IP"

aws ec2 authorize-security-group-ingress \
  --group-name my-sg \
  --protocol tcp \
  --port 22 \
  --cidr $(curl -s https://checkip.amazonaws.com)/32
```

Launch the instance:

```bash
aws ec2 run-instances \
  --image-id ami-0abcdef1234567890 \
  --instance-type t2.micro \
  --key-name my-key \
  --security-groups my-sg \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=my-first-server}]'
```

Replace `ami-0abcdef1234567890` with the AMI ID from the first command.

Get the public IP:

```bash
aws ec2 describe-instances \
  --filters "Name=tag:Name,Values=my-first-server" \
  --query "Reservations[0].Instances[0].PublicIpAddress" \
  --output text
```

Connect:

```bash
ssh -i my-key.pem ec2-user@<PUBLIC_IP>
```

### User Data Scripts

User data lets you run a script automatically when an instance first launches. This is how you automate initial setup.

Example: launch an instance that automatically installs and starts a web server:

```bash
aws ec2 run-instances \
  --image-id ami-0abcdef1234567890 \
  --instance-type t2.micro \
  --key-name my-key \
  --security-groups my-sg \
  --user-data '#!/bin/bash
yum update -y
yum install -y httpd
systemctl start httpd
systemctl enable httpd
echo "<h1>Hello from EC2</h1>" > /var/www/html/index.html'
```

In the console, you paste this script into the "User data" field under "Advanced details" during launch.

User data scripts run as root, and only on the **first boot** by default. You can view the script's output log at `/var/log/cloud-init-output.log` on the instance.

## Instance Lifecycle

EC2 instances have three main states:

| State | What It Means | You Pay For... |
|---|---|---|
| **Running** | Instance is on and accessible | Compute + storage |
| **Stopped** | Instance is off, data on EBS preserved | Storage only (EBS volumes) |
| **Terminated** | Instance deleted permanently | Nothing (EBS deleted by default) |

**Key behaviors:**
- **Stopping** an instance is like shutting down a computer. The EBS (disk) data is preserved, but the public IP address changes when you start it again (unless you use an Elastic IP).
- **Terminating** an instance deletes it permanently. The root EBS volume is deleted by default. This cannot be undone.
- **Rebooting** keeps the instance running, preserves the public IP, and is equivalent to a restart.

Manage instance state via CLI:

```bash
# Stop an instance
aws ec2 stop-instances --instance-ids i-0abc123def456

# Start a stopped instance
aws ec2 start-instances --instance-ids i-0abc123def456

# Terminate (delete) an instance
aws ec2 terminate-instances --instance-ids i-0abc123def456
```

## Clean Up to Avoid Charges

When you're done practicing, terminate your instance:

1. Go to the EC2 console → Instances
2. Select your instance → Instance state → **Terminate instance**
3. Also delete any Elastic IPs, unused EBS volumes, and key pairs you no longer need

The t2.micro Free Tier gives you 750 hours/month — roughly one instance running 24/7 for a month. If you launch multiple instances, the hours add up across all of them.

## Key Takeaways

- EC2 gives you full virtual machine access — choose it when you need OS-level control
- `t2.micro` is Free Tier eligible and perfect for learning
- Amazon Linux 2023 is the recommended AMI for beginners on AWS
- Always restrict your key pair file permissions with `chmod 400`
- Never open SSH to `0.0.0.0/0` — restrict security group rules to your IP
- User data scripts let you automate instance setup on first launch
- Stop instances when not in use to save money; terminate when you're done
- The public IP changes when you stop/start — use an Elastic IP if you need a fixed address
