# AWS SAM CLI — Run Lambda and API Gateway on Your Laptop

Use the AWS Serverless Application Model CLI to invoke Lambda functions locally, start a local API Gateway, and test event-driven workflows without deploying.

## What You'll Learn

- What AWS SAM is and why it exists
- How to install the SAM CLI
- How `template.yaml` defines your serverless infrastructure
- How to invoke Lambda functions locally with `sam local invoke`
- How to run a local API Gateway with `sam local start-api`
- How to build and deploy to real AWS when ready
- Hot reloading for faster iteration

## What Is AWS SAM?

AWS SAM (Serverless Application Model) is two things:

1. **A template specification** — An extension of CloudFormation that makes it simpler to define Lambda functions, API Gateways, DynamoDB tables, and event sources
2. **A CLI tool** — `sam` commands that let you build, test, and deploy serverless applications

The key benefit: SAM lets you run your Lambda functions and API Gateway **on your laptop** using Docker. You write code, test it locally, and only deploy to AWS when everything works.

SAM is free and open source, maintained by AWS.

## Installing the SAM CLI

**macOS (Homebrew):**

```bash
brew install aws-sam-cli
```

**Linux:**

```bash
wget https://github.com/aws/aws-sam-cli/releases/latest/download/aws-sam-cli-linux-x86_64.zip
unzip aws-sam-cli-linux-x86_64.zip -d sam-installation
sudo ./sam-installation/install
```

**Windows:**

Download the MSI installer from the [SAM CLI releases page](https://github.com/aws/aws-sam-cli/releases).

**Verify installation:**

```bash
sam --version
# SAM CLI, version 1.x.x
```

**Requirements:**
- Docker Desktop must be installed and running (SAM uses Docker to simulate the Lambda runtime)
- AWS CLI configured (for deployment)

## Project Structure and template.yaml

Scaffold a new SAM project:

```bash
sam init
```

Choose:
- Template: AWS Quick Start Templates
- Runtime: python3.11 (or your preferred language)
- Name: my-sam-app

This creates:

```
my-sam-app/
├── template.yaml          # Infrastructure definition
├── samconfig.toml         # Deploy configuration
├── hello_world/
│   ├── __init__.py
│   ├── app.py             # Lambda function code
│   └── requirements.txt   # Python dependencies
├── events/
│   └── event.json         # Sample test event
└── tests/
    └── unit/
        └── test_handler.py
```

**The `template.yaml` file:**

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Transform: AWS::Serverless-2016-10-31
Description: My SAM Application

Globals:
  Function:
    Timeout: 10
    Runtime: python3.11
    MemorySize: 128

Resources:
  HelloWorldFunction:
    Type: AWS::Serverless::Function
    Properties:
      CodeUri: hello_world/
      Handler: app.lambda_handler
      Architectures:
        - x86_64
      Events:
        HelloWorld:
          Type: Api
          Properties:
            Path: /hello
            Method: get

  OrdersTable:
    Type: AWS::DynamoDB::Table
    Properties:
      TableName: Orders
      AttributeDefinitions:
        - AttributeName: orderId
          AttributeType: S
      KeySchema:
        - AttributeName: orderId
          KeyType: HASH
      BillingMode: PAY_PER_REQUEST

Outputs:
  HelloWorldApi:
    Description: "API Gateway endpoint URL"
    Value: !Sub "https://${ServerlessRestApi}.execute-api.${AWS::Region}.amazonaws.com/Prod/hello/"
  HelloWorldFunction:
    Description: "Lambda Function ARN"
    Value: !GetAtt HelloWorldFunction.Arn
```

Key sections:
- **Globals** — Default settings applied to all functions
- **Resources** — Your Lambda functions, APIs, tables, etc.
- **Events** — What triggers each function (API, SQS, S3, Schedule, etc.)
- **Outputs** — Values exported after deployment

## sam local invoke — Run a Single Function

Run a Lambda function once with a test event:

```bash
# Use the default event
sam local invoke HelloWorldFunction --event events/event.json
```

Create a custom event:

```bash
# Generate a sample API Gateway event
sam local generate-event apigateway aws-proxy > events/api-event.json

# Invoke with it
sam local invoke HelloWorldFunction --event events/api-event.json
```

Pass environment variables:

```bash
sam local invoke HelloWorldFunction \
  --event events/event.json \
  --env-vars env.json
```

Where `env.json` contains:

```json
{
  "HelloWorldFunction": {
    "TABLE_NAME": "Orders",
    "STAGE": "local"
  }
}
```

The first invocation is slow (Docker pulls the Lambda runtime image). Subsequent invocations are fast.

## sam local start-api — Local API Gateway

Start a persistent local API Gateway that reloads on file changes:

```bash
sam local start-api
```

Output:

```
Mounting HelloWorldFunction at http://127.0.0.1:3000/hello [GET]
You can now browse to the above endpoints to invoke your functions.
```

Test it:

```bash
curl http://localhost:3000/hello
# {"message": "hello world"}

curl http://localhost:3000/hello?name=AWS
# {"message": "hello AWS"}
```

This is a real HTTP server. You can point your frontend application at `http://localhost:3000` during development.

**Hot reloading:** Edit your function code, save the file, and the next request automatically uses the new code. No restart needed.

## sam local generate-event — Mock Event Payloads

SAM can generate realistic event payloads for various triggers:

```bash
# API Gateway event
sam local generate-event apigateway aws-proxy

# S3 put event
sam local generate-event s3 put --bucket my-bucket --key photo.jpg

# SQS message event
sam local generate-event sqs receive-message

# DynamoDB stream event
sam local generate-event dynamodb update

# Scheduled event (EventBridge)
sam local generate-event cloudwatch scheduled-event

# SNS notification
sam local generate-event sns notification
```

Save them for reuse:

```bash
sam local generate-event s3 put --bucket uploads --key file.pdf > events/s3-upload.json
sam local invoke ProcessFileFunction --event events/s3-upload.json
```

## sam local start-lambda — SDK Testing

If your code invokes Lambda functions using the AWS SDK (e.g., one Lambda calling another), use `start-lambda`:

```bash
sam local start-lambda
```

This starts a Lambda endpoint at `http://127.0.0.1:3001`. Configure your SDK to use it:

```python
import boto3

# Point the SDK at the local Lambda endpoint
lambda_client = boto3.client(
    'lambda',
    endpoint_url='http://127.0.0.1:3001',
    region_name='us-east-1',
    aws_access_key_id='test',
    aws_secret_access_key='test'
)

response = lambda_client.invoke(
    FunctionName='HelloWorldFunction',
    Payload=b'{"name": "SDK test"}'
)

print(response['Payload'].read().decode())
```

## Debugging Locally with VS Code

Add a launch configuration to `.vscode/launch.json`:

```json
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "SAM: Debug Lambda",
      "type": "python",
      "request": "attach",
      "port": 5678,
      "host": "localhost",
      "pathMappings": [
        {
          "localRoot": "${workspaceFolder}/hello_world",
          "remoteRoot": "/var/task"
        }
      ]
    }
  ]
}
```

Invoke with debug mode:

```bash
sam local invoke HelloWorldFunction \
  --event events/event.json \
  --debug-port 5678
```

SAM pauses execution until your debugger attaches. Then you can set breakpoints, inspect variables, and step through code.

## Building and Deploying

When you're ready to deploy to real AWS:

```bash
# Build (installs dependencies, compiles if needed)
sam build

# Deploy (first time — interactive guided setup)
sam deploy --guided
```

The guided deploy asks for:
- Stack name
- AWS region
- Whether to allow IAM role creation
- Whether to save settings to `samconfig.toml`

Subsequent deploys:

```bash
sam build && sam deploy
```

SAM creates a CloudFormation stack, uploads your code to S3, and provisions all resources defined in `template.yaml`.

## Hands-On: Build and Test a SAM App Locally

Complete workflow from scratch:

```bash
# 1. Create a new project
sam init --runtime python3.11 --name todo-api --app-template hello-world

cd todo-api

# 2. Edit the function (hello_world/app.py)
cat > hello_world/app.py << 'EOF'
import json

todos = []

def lambda_handler(event, context):
    method = event.get("httpMethod", "GET")
    
    if method == "GET":
        return {
            "statusCode": 200,
            "body": json.dumps({"todos": todos})
        }
    elif method == "POST":
        body = json.loads(event.get("body", "{}"))
        todos.append(body)
        return {
            "statusCode": 201,
            "body": json.dumps({"message": "Todo added", "todo": body})
        }
    
    return {"statusCode": 405, "body": "Method not allowed"}
EOF

# 3. Start local API
sam local start-api

# 4. In another terminal, test it
curl http://localhost:3000/hello
curl -X POST http://localhost:3000/hello \
  -H "Content-Type: application/json" \
  -d '{"task": "Learn SAM", "done": false}'
```

## SAM CLI vs. LocalStack — When to Use Which

| Scenario | Use SAM CLI | Use LocalStack |
|----------|-------------|----------------|
| Lambda + API Gateway only | Yes | Also works |
| Need SQS, SNS, S3 together | No (use LocalStack) | Yes |
| Want hot reloading | Yes (built-in) | Limited |
| Plan to deploy with SAM | Yes | No |
| Need DynamoDB with Lambda | Pair with DynamoDB Local | Yes (built-in) |
| CI/CD integration tests | Either | Either |
| Infrastructure as Code | template.yaml | CloudFormation or Terraform |

**They work well together:** Use SAM for your Lambda functions and pair it with LocalStack for other services (SQS, SNS, S3) that your functions interact with.

## Key Takeaways

- SAM CLI lets you run Lambda functions and API Gateway locally using Docker
- `sam local invoke` runs a function once; `sam local start-api` starts a persistent HTTP server
- `sam local generate-event` creates realistic test payloads for any trigger type
- Hot reloading means you edit code and immediately see results without restarting
- `sam build && sam deploy` takes your tested local code straight to production
- SAM is ideal for serverless development; combine with LocalStack for non-serverless services
