# Why Emulate Locally (Save Money, Faster Iteration, Offline Dev)

Understand the benefits of running AWS services on your own machine — faster feedback loops, zero cloud costs during development, and the ability to work offline.

## What You'll Learn

- Why developing directly against AWS costs money and slows you down
- Which AWS services can be emulated locally and which cannot
- The local-first development workflow used by professional teams
- How to choose between LocalStack, SAM CLI, and DynamoDB Local

## The Cost Problem with Cloud-First Development

When you're learning AWS, it's tempting to do everything in the real cloud. The problem is that costs sneak up on you:

- Every Lambda invocation, S3 request, and DynamoDB read/write has a price
- Forgetting to tear down resources leads to surprise bills
- NAT Gateways alone cost ~$32/month if left running
- Even "free tier" resources expire after 12 months

Here's a typical beginner mistake:

```
Month 1: "I'll just spin up a small RDS instance to learn"
Month 2: Forgot about it
Month 3: $25 charge on credit card
```

Local emulation costs exactly **$0**. Your laptop is the cloud.

## Speed — Local Feedback in Seconds

Deploying to AWS takes time:

| Action | Cloud Deploy Time | Local Time |
|--------|------------------|------------|
| Lambda code change | 30-60 seconds (zip + upload + propagate) | 1-2 seconds |
| API Gateway test | Deploy + wait for endpoint | Instant (localhost) |
| DynamoDB table tweak | ~10 seconds | Instant |
| Full stack redeploy | 3-5 minutes (CloudFormation) | 10-15 seconds |

When you're iterating on code — fixing bugs, testing edge cases, tweaking responses — waiting 60 seconds per change kills your flow. Locally, you save a file and re-run immediately.

## Offline Development and Portability

Local emulation means you can code anywhere:

- On an airplane with no WiFi
- In a coffee shop with unreliable internet
- On a train during your commute
- When AWS has an outage (it happens)

Your development environment lives entirely on your machine. No internet dependency, no credentials needed for day-to-day coding.

## What Can Be Emulated Locally

These services work well with local emulators:

| Service | Emulator | Fidelity |
|---------|----------|----------|
| S3 | LocalStack, Minio | High |
| Lambda | SAM CLI, LocalStack | High |
| DynamoDB | DynamoDB Local, LocalStack | High |
| SQS | LocalStack | High |
| SNS | LocalStack | Medium-High |
| API Gateway | SAM CLI, LocalStack | Medium-High |
| Step Functions | LocalStack | Medium |
| CloudWatch Logs | LocalStack | Medium |
| IAM (basic) | LocalStack | Low-Medium |
| Kinesis | LocalStack | Medium |

## When Local Emulation Falls Short

Some services are too complex or too tightly integrated to emulate well:

**Cannot emulate meaningfully:**
- **Cognito** — Full OAuth flows, hosted UI, user pools with all triggers
- **CloudFront** — CDN behavior, edge caching, geographic distribution
- **Route 53** — Real DNS resolution and health checks
- **ACM** — Real SSL/TLS certificate provisioning
- **VPC Networking** — Actual network isolation, peering, Transit Gateway
- **IAM (full)** — Real permission boundaries, SCPs, cross-account roles

**Partially emulable but with significant gaps:**
- **RDS** — You can run PostgreSQL/MySQL in Docker, but it won't have RDS-specific features like automated backups, read replicas, or IAM auth
- **ElastiCache** — Run Redis in Docker, but no cluster mode or failover behavior
- **ECS/EKS** — Run containers in Docker, but no task definitions or service mesh

## Overview of Emulation Tools

| Tool | Best For | Cost |
|------|----------|------|
| **LocalStack** | Multi-service emulation (S3, SQS, Lambda, DynamoDB) | Free (Hobby) / Paid (Pro) |
| **SAM CLI** | Lambda + API Gateway development | Free |
| **DynamoDB Local** | DynamoDB-heavy applications | Free |
| **Minio** | S3-compatible object storage | Free |
| **ElasticMQ** | SQS alternative | Free |

## Choosing the Right Tool for the Job

**Use LocalStack when:**
- You need multiple services working together (Lambda reads from SQS, writes to DynamoDB)
- You want a single `docker-compose up` to start your entire backend
- You're building event-driven architectures

**Use SAM CLI when:**
- You're building serverless apps with `template.yaml`
- You want hot-reloading for Lambda functions
- You plan to deploy with `sam deploy` later

**Use DynamoDB Local when:**
- Your app is DynamoDB-heavy and you want the highest fidelity emulation
- You need to run integration tests against DynamoDB in CI/CD
- You're using NoSQL Workbench for table design

## The Local-First Workflow

Professional teams follow this pattern:

```
1. Write code on your laptop
2. Test against local emulators (LocalStack, SAM, DynamoDB Local)
3. Run integration tests locally
4. Only deploy to AWS for:
   - Final integration testing
   - Staging/production releases
   - Services that can't be emulated
5. Tear down cloud resources after testing
```

This approach means 90% of your development happens locally at zero cost, and you only touch AWS when absolutely necessary.

## Key Takeaways

- Local emulation saves money, speeds up development, and works offline
- Core services (S3, Lambda, DynamoDB, SQS, SNS) emulate well
- Complex managed services (Cognito, CloudFront, full IAM) require real AWS
- Use the right tool: LocalStack for multi-service, SAM for serverless, DynamoDB Local for database work
- The goal is to minimize cloud usage during development, not eliminate it entirely
