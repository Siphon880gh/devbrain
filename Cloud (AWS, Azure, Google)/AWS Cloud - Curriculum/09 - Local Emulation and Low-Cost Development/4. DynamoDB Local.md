# DynamoDB Local — Official Amazon DynamoDB Emulator

Run Amazon's official DynamoDB emulator on your machine via Docker or a downloadable JAR to develop and test table designs without incurring any AWS charges.

## What You'll Learn

- What DynamoDB Local is and why Amazon provides it
- How to run it via Docker
- How to connect using the AWS CLI and SDKs (Python and Node.js)
- Key differences between DynamoDB Local and the real service
- How to use NoSQL Workbench for visual table design
- How to integrate DynamoDB Local with SAM CLI

## What Is DynamoDB Local?

DynamoDB Local is an official emulator provided by Amazon that runs a DynamoDB-compatible database on your machine. It's a Java application packaged as a JAR file or Docker image.

Why use it:
- **Free** — No AWS charges for reads, writes, or storage
- **Fast** — No network latency; responses in milliseconds
- **Offline** — Works without internet
- **Safe** — Experiment with table designs without affecting production data
- **Testable** — Run integration tests in CI/CD pipelines

It supports the full DynamoDB API: CreateTable, PutItem, GetItem, Query, Scan, BatchWriteItem, Transactions, Streams, and more.

## Installation Options — Docker vs. JAR

**Option 1: Docker (recommended)**

```bash
docker pull amazon/dynamodb-local

docker run -d \
  --name dynamodb-local \
  -p 8000:8000 \
  amazon/dynamodb-local
```

**Option 2: Docker Compose**

```yaml
version: "3.8"
services:
  dynamodb-local:
    image: amazon/dynamodb-local
    container_name: dynamodb-local
    ports:
      - "8000:8000"
    command: "-jar DynamoDBLocal.jar -sharedDb -dbPath /data"
    volumes:
      - dynamodb-data:/data

volumes:
  dynamodb-data:
```

The `-sharedDb` flag makes all clients share a single database file (simpler for development). The `-dbPath` with a volume persists data between container restarts.

**Option 3: Download JAR directly**

```bash
wget https://d1ni2b6xgvw0s0.cloudfront.net/v2.x/dynamodb_local_latest.tar.gz
tar -xzf dynamodb_local_latest.tar.gz
java -jar DynamoDBLocal.jar -sharedDb -port 8000
```

Requires Java 11+ installed.

## Starting DynamoDB Local

Start with Docker:

```bash
docker run -d --name dynamodb-local -p 8000:8000 amazon/dynamodb-local
```

Verify it's running:

```bash
aws dynamodb list-tables --endpoint-url http://localhost:8000
```

Expected output:

```json
{
    "TableNames": []
}
```

Stop and remove:

```bash
docker stop dynamodb-local
docker rm dynamodb-local
```

## Connecting with the AWS CLI and SDKs

**AWS CLI:**

Every command needs `--endpoint-url http://localhost:8000`:

```bash
# Create a table
aws dynamodb create-table \
  --endpoint-url http://localhost:8000 \
  --table-name Users \
  --attribute-definitions \
    AttributeName=userId,AttributeType=S \
  --key-schema \
    AttributeName=userId,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST

# Put an item
aws dynamodb put-item \
  --endpoint-url http://localhost:8000 \
  --table-name Users \
  --item '{
    "userId": {"S": "user-001"},
    "name": {"S": "Alice"},
    "email": {"S": "alice@example.com"},
    "age": {"N": "30"}
  }'

# Get the item
aws dynamodb get-item \
  --endpoint-url http://localhost:8000 \
  --table-name Users \
  --key '{"userId": {"S": "user-001"}}'

# Query (requires a sort key or GSI for range queries)
aws dynamodb scan \
  --endpoint-url http://localhost:8000 \
  --table-name Users
```

**Python (boto3):**

```python
import boto3

dynamodb = boto3.resource(
    'dynamodb',
    endpoint_url='http://localhost:8000',
    region_name='us-east-1',
    aws_access_key_id='anything',
    aws_secret_access_key='anything'
)

# Create table
table = dynamodb.create_table(
    TableName='Products',
    KeySchema=[
        {'AttributeName': 'productId', 'KeyType': 'HASH'},
        {'AttributeName': 'category', 'KeyType': 'RANGE'}
    ],
    AttributeDefinitions=[
        {'AttributeName': 'productId', 'AttributeType': 'S'},
        {'AttributeName': 'category', 'AttributeType': 'S'}
    ],
    BillingMode='PAY_PER_REQUEST'
)

table.meta.client.get_waiter('table_exists').wait(TableName='Products')

# Put items
table = dynamodb.Table('Products')
table.put_item(Item={
    'productId': 'prod-001',
    'category': 'electronics',
    'name': 'Wireless Mouse',
    'price': 29.99
})

# Get item
response = table.get_item(Key={
    'productId': 'prod-001',
    'category': 'electronics'
})
print(response['Item'])

# Query by partition key
from boto3.dynamodb.conditions import Key

response = table.query(
    KeyConditionExpression=Key('productId').eq('prod-001')
)
print(response['Items'])
```

**Node.js (AWS SDK v3):**

```javascript
const { DynamoDBClient } = require("@aws-sdk/client-dynamodb");
const { DynamoDBDocumentClient, PutCommand, GetCommand } = require("@aws-sdk/lib-dynamodb");

const client = new DynamoDBClient({
  endpoint: "http://localhost:8000",
  region: "us-east-1",
  credentials: {
    accessKeyId: "anything",
    secretAccessKey: "anything",
  },
});

const docClient = DynamoDBDocumentClient.from(client);

async function main() {
  // Put item
  await docClient.send(new PutCommand({
    TableName: "Users",
    Item: {
      userId: "user-002",
      name: "Bob",
      email: "bob@example.com",
    },
  }));

  // Get item
  const result = await docClient.send(new GetCommand({
    TableName: "Users",
    Key: { userId: "user-002" },
  }));

  console.log(result.Item);
}

main();
```

## Differences from the Real DynamoDB Service

| Feature | DynamoDB Local | Real DynamoDB |
|---------|---------------|---------------|
| Capacity limits | No throttling ever | Provisioned or on-demand limits |
| IAM permissions | Not enforced | Fully enforced |
| Encryption | Not applicable | At-rest encryption |
| Global tables | Not supported | Multi-region replication |
| Backups | Not supported | Point-in-time recovery, on-demand |
| DynamoDB Streams | Supported (basic) | Full Streams with Lambda triggers |
| TTL | Supported | Supported (may behave slightly different) |
| Transactions | Supported | Supported |
| Pricing | Free | Pay per read/write/storage |
| Performance | Depends on your machine | Consistent single-digit ms |

**What this means for you:** DynamoDB Local is great for developing your data access patterns, but you should do a final test against real DynamoDB before production to verify IAM permissions and capacity behavior.

## NoSQL Workbench GUI

Amazon provides NoSQL Workbench, a free desktop application for designing and visualizing DynamoDB tables.

**Download:** [https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/workbench.html](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/workbench.html)

Features:
- **Data Modeler** — Design tables visually, define keys and indexes
- **Visualizer** — See how your data looks with sample items
- **Operation Builder** — Build and test queries without writing code
- **Connect to DynamoDB Local** — Point it at `http://localhost:8000`

To connect NoSQL Workbench to DynamoDB Local:
1. Open NoSQL Workbench
2. Go to "Operation Builder"
3. Click "Add connection"
4. Choose "DynamoDB Local"
5. Set port to 8000

## Using DynamoDB Local in Integration Tests

Create a test fixture that spins up a fresh table for each test:

```python
import pytest
import boto3

@pytest.fixture
def dynamodb_table():
    """Create a fresh table for each test, delete after."""
    dynamodb = boto3.resource(
        'dynamodb',
        endpoint_url='http://localhost:8000',
        region_name='us-east-1',
        aws_access_key_id='test',
        aws_secret_access_key='test'
    )
    
    table_name = 'TestTable'
    
    # Delete if exists from a previous failed run
    try:
        dynamodb.Table(table_name).delete()
        dynamodb.Table(table_name).meta.client.get_waiter(
            'table_not_exists'
        ).wait(TableName=table_name)
    except Exception:
        pass
    
    table = dynamodb.create_table(
        TableName=table_name,
        KeySchema=[{'AttributeName': 'pk', 'KeyType': 'HASH'}],
        AttributeDefinitions=[{'AttributeName': 'pk', 'AttributeType': 'S'}],
        BillingMode='PAY_PER_REQUEST'
    )
    table.meta.client.get_waiter('table_exists').wait(TableName=table_name)
    
    yield table
    
    table.delete()


def test_put_and_get(dynamodb_table):
    dynamodb_table.put_item(Item={'pk': '123', 'data': 'hello'})
    response = dynamodb_table.get_item(Key={'pk': '123'})
    assert response['Item']['data'] == 'hello'
```

## Integration with SAM CLI

SAM CLI can use DynamoDB Local for your Lambda functions. In your SAM template, reference the table, then pass the endpoint as an environment variable:

```yaml
Resources:
  MyFunction:
    Type: AWS::Serverless::Function
    Properties:
      CodeUri: src/
      Handler: app.handler
      Runtime: python3.11
      Environment:
        Variables:
          DYNAMODB_ENDPOINT: http://host.docker.internal:8000
          TABLE_NAME: Users
```

In your Lambda code:

```python
import os
import boto3

endpoint = os.environ.get('DYNAMODB_ENDPOINT')

if endpoint:
    dynamodb = boto3.resource('dynamodb', endpoint_url=endpoint)
else:
    dynamodb = boto3.resource('dynamodb')

def handler(event, context):
    table = dynamodb.Table(os.environ['TABLE_NAME'])
    # ... your logic
```

Note: Use `host.docker.internal` (not `localhost`) because your Lambda runs inside a Docker container and needs to reach DynamoDB Local running on the host.

## Key Takeaways

- DynamoDB Local is Amazon's official free emulator — run it in Docker on port 8000
- Add `--endpoint-url http://localhost:8000` to all CLI commands
- SDKs connect by setting the `endpoint_url` parameter with any dummy credentials
- It supports the full API (CRUD, queries, transactions, streams) but doesn't enforce IAM or capacity
- Use NoSQL Workbench for visual table design connected to your local instance
- Perfect for integration tests — spin up tables per test, tear them down after
- When using with SAM CLI, use `host.docker.internal:8000` as the endpoint inside Lambda containers
