# Cheap-as-Possible Cloud Setup When Emulators Are Not Enough

Configure real AWS services with minimal cost exposure for the cases where local emulation cannot fully replace the cloud.

## What You'll Learn

- When you must use real AWS instead of emulators
- How to stay within the AWS Free Tier
- Tricks to minimize costs when free tier isn't enough
- How to avoid the most expensive beginner mistakes
- Setting up budget alarms to catch cost spikes
- A cost-saving checklist to follow for every project

## When You Must Use Real AWS

Local emulators can't replicate everything. You need real AWS for:

- **Cognito** — Testing full OAuth/OIDC flows, hosted sign-in UI, social login
- **CloudFront** — CDN caching behavior, custom domains, SSL certificates
- **Route 53** — Real DNS resolution, domain registration
- **IAM** — Validating actual permission policies before production
- **VPC networking** — Testing security groups, NACLs, private subnets
- **Cross-service integrations** — S3 event triggers firing Lambda, CloudWatch alarms triggering SNS
- **Performance testing** — Real-world latency, cold starts under load
- **Final pre-production validation** — Before launching anything users will touch

The goal: use real AWS as little as possible and tear down resources immediately after testing.

## Free Tier Services and Limits

AWS offers three types of free tier:

**Always Free (no expiration):**
| Service | Free Amount |
|---------|------------|
| Lambda | 1M requests + 400,000 GB-seconds/month |
| DynamoDB | 25 GB storage + 25 RCU + 25 WCU |
| S3 | Not always-free (see 12-month) |
| SNS | 1M publishes/month |
| SQS | 1M requests/month |
| CloudWatch | 10 custom metrics, 10 alarms |
| API Gateway | 1M REST API calls/month (first 12 months) |

**12-Month Free (from account creation):**
| Service | Free Amount |
|---------|------------|
| EC2 | 750 hours/month t2.micro or t3.micro |
| S3 | 5 GB storage + 20,000 GET + 2,000 PUT |
| RDS | 750 hours/month db.t2.micro or db.t3.micro |
| ElastiCache | 750 hours/month cache.t2.micro |
| CloudFront | 1 TB data transfer/month |

**Short-Term Trials:**
| Service | Trial |
|---------|-------|
| Redshift | 2 months |
| Lightsail | 3 months |
| Inspector | 15-day trial |

Check your free tier usage: [https://console.aws.amazon.com/billing/home#/freetier](https://console.aws.amazon.com/billing/home#/freetier)

## Staying Under the Free Tier Radar

Rules to live by:

**1. Use the smallest instance sizes**

```bash
# EC2 — always t3.micro for learning
aws ec2 run-instances \
  --instance-type t3.micro \
  --image-id ami-0abcdef1234567890 \
  --key-name my-key

# RDS — always db.t3.micro, single-AZ
aws rds create-db-instance \
  --db-instance-identifier my-dev-db \
  --db-instance-class db.t3.micro \
  --engine postgres \
  --master-username admin \
  --master-user-password mysecretpass \
  --allocated-storage 20 \
  --no-multi-az
```

**2. Avoid NAT Gateways**

NAT Gateways cost ~$32/month just for existing, plus data transfer charges. For learning:

- Use public subnets instead (instances get public IPs directly)
- Use VPC Endpoints for AWS service access from private subnets (free for gateway endpoints to S3 and DynamoDB)
- If you must use private subnets, use a NAT instance on t3.micro instead

```bash
# Gateway VPC Endpoint for S3 (free)
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-12345 \
  --service-name com.amazonaws.us-east-1.s3 \
  --route-table-ids rtb-12345
```

**3. Avoid public IPv4 addresses when possible**

As of 2024, AWS charges $0.005/hour ($3.60/month) for every public IPv4 address. For learning:

- Use AWS CloudShell or Session Manager to connect to instances (no public IP needed)
- Use IPv6-only where possible
- Release Elastic IPs immediately when not in use

**4. Pick single-AZ for everything**

Multi-AZ doubles cost for RDS, ElastiCache, etc. For learning, single-AZ is fine:

```bash
# Single-AZ RDS (half the cost of multi-AZ)
--no-multi-az

# Single-node ElastiCache
--num-cache-nodes 1
```

**5. Use on-demand DynamoDB billing**

```bash
# PAY_PER_REQUEST means you pay nothing when idle
--billing-mode PAY_PER_REQUEST
```

## Budget Alarms and Billing Alerts

**Set up billing alerts immediately after creating your AWS account.**

**Step 1: Enable Cost Explorer**

Go to Billing → Cost Explorer → Enable.

**Step 2: Create a budget**

```bash
# Create a $5/month budget with email alert
aws budgets create-budget \
  --account-id $(aws sts get-caller-identity --query Account --output text) \
  --budget '{
    "BudgetName": "Monthly-5-Dollar-Limit",
    "BudgetLimit": {
      "Amount": "5",
      "Unit": "USD"
    },
    "TimeUnit": "MONTHLY",
    "BudgetType": "COST"
  }' \
  --notifications-with-subscribers '[
    {
      "Notification": {
        "NotificationType": "ACTUAL",
        "ComparisonOperator": "GREATER_THAN",
        "Threshold": 80,
        "ThresholdType": "PERCENTAGE"
      },
      "Subscribers": [
        {
          "SubscriptionType": "EMAIL",
          "Address": "your@email.com"
        }
      ]
    },
    {
      "Notification": {
        "NotificationType": "ACTUAL",
        "ComparisonOperator": "GREATER_THAN",
        "Threshold": 100,
        "ThresholdType": "PERCENTAGE"
      },
      "Subscribers": [
        {
          "SubscriptionType": "EMAIL",
          "Address": "your@email.com"
        }
      ]
    }
  ]'
```

This sends an email when you hit 80% ($4) and 100% ($5) of your budget.

**Step 3: Set up a zero-spend alarm for unexpected charges**

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name "billing-alarm-1-dollar" \
  --alarm-description "Alert when charges exceed $1" \
  --metric-name EstimatedCharges \
  --namespace AWS/Billing \
  --statistic Maximum \
  --period 21600 \
  --threshold 1 \
  --comparison-operator GreaterThanThreshold \
  --evaluation-periods 1 \
  --alarm-actions arn:aws:sns:us-east-1:ACCOUNT_ID:billing-alerts \
  --dimensions Name=Currency,Value=USD
```

## Using Spot Instances for Non-Critical Workloads

Spot instances cost 60-90% less than on-demand. Use them for:

- Running tests
- Batch processing
- Learning/experimenting (you don't care if it's interrupted)

```bash
# Launch a Spot instance (up to 70% cheaper)
aws ec2 run-instances \
  --instance-type t3.micro \
  --image-id ami-0abcdef1234567890 \
  --instance-market-options '{"MarketType": "spot", "SpotOptions": {"SpotInstanceType": "one-time"}}'
```

Spot instances can be terminated with 2 minutes notice, so don't use them for databases or long-running stateful services.

## Schedule Resources (Stop When Not in Use)

If you need EC2 or RDS for more than a quick test, schedule them to run only during your work hours:

**Stop EC2 at night using EventBridge + Lambda:**

```python
import boto3

def lambda_handler(event, context):
    ec2 = boto3.client('ec2')
    
    # Find instances tagged "Environment=dev"
    response = ec2.describe_instances(
        Filters=[
            {'Name': 'tag:Environment', 'Values': ['dev']},
            {'Name': 'instance-state-name', 'Values': ['running']}
        ]
    )
    
    instance_ids = []
    for reservation in response['Reservations']:
        for instance in reservation['Instances']:
            instance_ids.append(instance['InstanceId'])
    
    if instance_ids:
        ec2.stop_instances(InstanceIds=instance_ids)
        print(f"Stopped: {instance_ids}")
```

**Simpler approach — just stop manually when done:**

```bash
# Stop all dev-tagged instances
aws ec2 stop-instances --instance-ids \
  $(aws ec2 describe-instances \
    --filters "Name=tag:Environment,Values=dev" "Name=instance-state-name,Values=running" \
    --query "Reservations[].Instances[].InstanceId" --output text)

# Stop RDS instance
aws rds stop-db-instance --db-instance-identifier my-dev-db
```

Stopped EC2 instances cost $0 for compute (you still pay for EBS storage). Stopped RDS instances auto-start after 7 days, so set a reminder.

## Teardown-After-Use Patterns

The safest approach: create resources, test, and destroy in one session.

```bash
#!/bin/bash
# create-test-destroy.sh

# Create
aws dynamodb create-table \
  --table-name TestTable \
  --attribute-definitions AttributeName=id,AttributeType=S \
  --key-schema AttributeName=id,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST

# Wait for it
aws dynamodb wait table-exists --table-name TestTable

# Run your tests
python run_tests.py

# Destroy immediately
aws dynamodb delete-table --table-name TestTable

echo "Cleaned up. Cost: $0.00"
```

## Tag Everything

Tags let you find and destroy resources later:

```bash
# Always tag resources when creating them
aws ec2 run-instances \
  --instance-type t3.micro \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Project,Value=learning},{Key=Environment,Value=dev},{Key=DeleteAfter,Value=2026-05-10}]' \
  ...

# Find all resources with a tag
aws resourcegroupstaggingapi get-resources \
  --tag-filters Key=Project,Values=learning
```

## Cost-Saving Checklist for Learners

Before creating any resource, ask:

- [ ] Can I test this locally with LocalStack, SAM CLI, or Docker?
- [ ] Am I within Free Tier limits for this service?
- [ ] Am I using the smallest possible instance size?
- [ ] Is Multi-AZ disabled?
- [ ] Have I avoided NAT Gateways?
- [ ] Did I tag the resource for easy cleanup?
- [ ] Do I have a budget alarm set?
- [ ] Will I remember to tear this down when I'm done?
- [ ] Can I use a Spot instance instead of on-demand?
- [ ] Have I released any Elastic IPs I'm not using?

## Key Takeaways

- Set a $5-$10 budget alarm before doing anything in AWS
- Use the smallest instance sizes: t3.micro for EC2, db.t3.micro for RDS
- Avoid NAT Gateways, public IPv4 addresses, and Multi-AZ for learning
- Tag everything with `Project`, `Environment`, and `DeleteAfter` tags
- Stop or terminate resources immediately after testing
- Use Spot instances for non-critical workloads (60-90% savings)
- Serverless services (Lambda, DynamoDB on-demand, SQS) are effectively free at learning scale
- Check your billing dashboard weekly — surprises compound fast
