# LocalStack — Free Local Emulator for S3, Lambda, DynamoDB, SQS, SNS, and More

Get started with LocalStack's free Hobby tier to emulate core AWS services on your laptop using Docker.

## What You'll Learn

- What LocalStack is and how it works
- How to install and start LocalStack with Docker
- How to point your AWS CLI at the local endpoint
- How to create S3 buckets, Lambda functions, and DynamoDB tables locally
- Which services are available on the free Hobby tier

## What Is LocalStack?

LocalStack is a cloud service emulator that runs entirely on your machine inside a Docker container. It provides a single endpoint (`http://localhost:4566`) that mimics the behavior of dozens of AWS services. When your code sends a request to "S3" or "DynamoDB," LocalStack intercepts it and responds as AWS would — but everything stays on your laptop.

As of 2026, LocalStack uses a tiered model:
- **Hobby** (free) — Requires a LocalStack account and auth token, covers core services
- **Starter / Pro / Team** (paid) — Adds advanced services like Cognito, RDS, ECS

Even the free Hobby tier is powerful enough for most learning and development.

## Installing LocalStack and Docker

**Prerequisites:**
- Docker Desktop installed and running
- Python 3.8+ (for the `localstack` CLI)
- An AWS CLI installation (for interacting with services)

**Step 1: Install the LocalStack CLI**

```bash
pip install localstack
```

**Step 2: Sign up for a LocalStack account**

Go to [https://app.localstack.cloud](https://app.localstack.cloud) and create a free account. Even the Hobby tier requires authentication now.

**Step 3: Set your auth token**

After signing up, find your auth token in the LocalStack dashboard and export it:

```bash
export LOCALSTACK_AUTH_TOKEN="your-token-here"
```

Add this to your `~/.bashrc` or `~/.zshrc` so it persists:

```bash
echo 'export LOCALSTACK_AUTH_TOKEN="your-token-here"' >> ~/.zshrc
```

**Step 4: Pull the Docker image**

```bash
docker pull localstack/localstack
```

## Starting the LocalStack Container

The simplest way to start LocalStack:

```bash
localstack start
```

This pulls the Docker image (if not already present) and starts all services on port 4566.

You can also start it directly with Docker:

```bash
docker run --rm -it \
  -p 4566:4566 \
  -p 4510-4559:4510-4559 \
  -e LOCALSTACK_AUTH_TOKEN="${LOCALSTACK_AUTH_TOKEN}" \
  -v /var/run/docker.sock:/var/run/docker.sock \
  localstack/localstack
```

Or with `docker-compose.yml`:

```yaml
version: "3.8"
services:
  localstack:
    image: localstack/localstack
    ports:
      - "4566:4566"
      - "4510-4559:4510-4559"
    environment:
      - LOCALSTACK_AUTH_TOKEN=${LOCALSTACK_AUTH_TOKEN}
    volumes:
      - "/var/run/docker.sock:/var/run/docker.sock"
```

Start it:

```bash
docker compose up -d
```

Verify it's running:

```bash
curl http://localhost:4566/_localstack/health
```

You should see a JSON response listing available services.

## Using the AWS CLI with LocalStack

You have two options to point your CLI at LocalStack:

**Option A: Use `--endpoint-url` on every command**

```bash
aws --endpoint-url=http://localhost:4566 s3 ls
```

**Option B: Set the `AWS_ENDPOINT_URL` environment variable (recommended)**

```bash
export AWS_ENDPOINT_URL=http://localhost:4566
```

Now every `aws` command automatically targets LocalStack:

```bash
aws s3 ls          # hits localhost:4566, not real AWS
aws dynamodb list-tables   # same
```

You also need dummy credentials (LocalStack doesn't validate them, but the CLI requires them):

```bash
export AWS_ACCESS_KEY_ID=test
export AWS_SECRET_ACCESS_KEY=test
export AWS_DEFAULT_REGION=us-east-1
```

**Tip:** Create a shell alias or a named AWS CLI profile to avoid mixing up local and real AWS:

```bash
# In ~/.aws/config
[profile localstack]
region = us-east-1
endpoint_url = http://localhost:4566

# In ~/.aws/credentials
[localstack]
aws_access_key_id = test
aws_secret_access_key = test
```

Then use: `aws --profile localstack s3 ls`

## Hands-On: Create an S3 Bucket and Upload a File Locally

```bash
# Create a bucket
aws --endpoint-url=http://localhost:4566 s3 mb s3://my-test-bucket

# Verify it exists
aws --endpoint-url=http://localhost:4566 s3 ls

# Create a test file
echo "Hello from LocalStack!" > hello.txt

# Upload it
aws --endpoint-url=http://localhost:4566 s3 cp hello.txt s3://my-test-bucket/

# List bucket contents
aws --endpoint-url=http://localhost:4566 s3 ls s3://my-test-bucket/

# Download it back
aws --endpoint-url=http://localhost:4566 s3 cp s3://my-test-bucket/hello.txt downloaded.txt
cat downloaded.txt
```

Everything happens on your machine. No internet, no AWS bill.

## Hands-On: Deploy a Lambda Function to LocalStack

Create a simple Lambda function:

```bash
# Create the function code
cat > lambda_function.py << 'EOF'
import json

def handler(event, context):
    name = event.get("name", "World")
    return {
        "statusCode": 200,
        "body": json.dumps({"message": f"Hello, {name}!"})
    }
EOF

# Zip it
zip function.zip lambda_function.py

# Create the function in LocalStack
aws --endpoint-url=http://localhost:4566 lambda create-function \
  --function-name hello-function \
  --runtime python3.11 \
  --handler lambda_function.handler \
  --zip-file fileb://function.zip \
  --role arn:aws:iam::000000000000:role/lambda-role

# Invoke it
aws --endpoint-url=http://localhost:4566 lambda invoke \
  --function-name hello-function \
  --payload '{"name": "LocalStack"}' \
  output.json

cat output.json
```

Expected output:

```json
{"statusCode": 200, "body": "{\"message\": \"Hello, LocalStack!\"}"}
```

## Complete Workflow Example

Here's a realistic workflow combining S3, Lambda, and SQS:

```bash
# Start LocalStack
localstack start -d

# Create an SQS queue
aws --endpoint-url=http://localhost:4566 sqs create-queue \
  --queue-name order-queue

# Create a DynamoDB table
aws --endpoint-url=http://localhost:4566 dynamodb create-table \
  --table-name Orders \
  --attribute-definitions AttributeName=orderId,AttributeType=S \
  --key-schema AttributeName=orderId,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST

# Send a message to the queue
aws --endpoint-url=http://localhost:4566 sqs send-message \
  --queue-url http://localhost:4566/000000000000/order-queue \
  --message-body '{"orderId": "123", "item": "Widget", "qty": 5}'

# Receive the message
aws --endpoint-url=http://localhost:4566 sqs receive-message \
  --queue-url http://localhost:4566/000000000000/order-queue
```

## Tips for Switching Between Local and Real AWS

1. **Use environment variables** — Toggle `AWS_ENDPOINT_URL` to switch targets
2. **Use AWS profiles** — `--profile localstack` vs `--profile default`
3. **Use wrapper scripts** — Create a `local-aws` alias that sets the endpoint
4. **Keep credentials separate** — Never use real credentials with LocalStack
5. **Data doesn't persist by default** — When you stop LocalStack, everything is gone (use volumes to persist if needed)

To persist data between restarts:

```yaml
# docker-compose.yml
volumes:
  - "./localstack-data:/var/lib/localstack"
```

## Key Takeaways

- LocalStack emulates AWS services locally inside Docker on port 4566
- The free Hobby tier (2026) requires signup and an auth token but covers S3, Lambda, DynamoDB, SQS, SNS, and more
- Point your AWS CLI at `http://localhost:4566` using `--endpoint-url` or `AWS_ENDPOINT_URL`
- Your existing AWS CLI commands work unchanged — just redirect the endpoint
- Data is ephemeral by default; stop the container and it's gone
- Perfect for learning, prototyping, and running integration tests at zero cost
