# Load Balancers — ALB, NLB, and When to Use Each

Distribute traffic across targets with Application Load Balancers and Network Load Balancers.

## What You'll Learn

- Why load balancing is essential for reliable applications
- How ALBs work at Layer 7 (HTTP/HTTPS) with advanced routing
- How NLBs work at Layer 4 (TCP/UDP) for extreme performance
- How to choose between ALB and NLB
- How target groups and health checks keep traffic flowing to healthy instances
- How to set up an ALB with SSL termination in front of EC2 instances
- Pricing and cost optimization strategies

## Why Load Balancing Matters

Without a load balancer, your application has a **single point of failure**. If that one server dies, your entire app goes down. Even if you have multiple servers, how does a user know which one to connect to?

A load balancer solves both problems:

```
Users → Load Balancer → Server 1 (healthy ✓)
                      → Server 2 (healthy ✓)
                      → Server 3 (unhealthy ✗ — skipped)
```

Benefits:
- **High availability** — Traffic automatically shifts away from failed instances
- **Scalability** — Add or remove instances behind the load balancer transparently
- **SSL termination** — Handle HTTPS at the load balancer, simplifying your servers
- **Single DNS entry** — One stable endpoint regardless of how many servers are behind it

## Application Load Balancer (ALB)

The ALB operates at **Layer 7** (HTTP/HTTPS), meaning it can inspect the content of requests and make routing decisions based on URLs, headers, cookies, and more.

### Path-Based Routing

Route different URL paths to different target groups:

```
myapp.com/api/*     → API servers (target group A)
myapp.com/images/*  → Image processing servers (target group B)
myapp.com/*         → Frontend servers (target group C)
```

### Host-Based Routing

Route different subdomains to different services on the same ALB:

```
api.myapp.com    → API target group
admin.myapp.com  → Admin dashboard target group
www.myapp.com    → Main website target group
```

### Key ALB Features

- **HTTP/HTTPS** — Understands web protocols, can inspect headers and cookies
- **WebSocket support** — Maintains persistent connections
- **Redirect rules** — HTTP → HTTPS redirect at the load balancer level
- **Fixed response** — Return a static response without hitting any server
- **Authentication** — Integrate with Cognito or OIDC providers directly
- **Sticky sessions** — Route a user to the same target using cookies

### Listener Rules

An ALB listener checks for connection requests on a specific port and protocol. Rules determine how the listener routes requests:

```
Listener: HTTPS:443
  Rule 1: IF path is /api/* → Forward to api-target-group
  Rule 2: IF host is admin.myapp.com → Forward to admin-target-group
  Rule 3: IF path is /health → Return fixed 200 "OK"
  Default: Forward to web-target-group
```

Rules are evaluated in priority order (lowest number wins).

## Network Load Balancer (NLB)

The NLB operates at **Layer 4** (TCP/UDP), making routing decisions based only on IP addresses and ports. It doesn't inspect packet contents.

### Key NLB Features

- **Ultra-low latency** — Millions of requests per second with single-digit millisecond latency
- **Static IP** — Each NLB gets one static IP per AZ (or you can assign Elastic IPs)
- **TCP/UDP/TLS** — Handles any TCP or UDP protocol, not just HTTP
- **Preserves source IP** — Target sees the client's real IP address
- **No security group on NLB** — Traffic filtering happens at the target level

### Use Cases for NLB

- Gaming servers (UDP traffic)
- IoT message brokers (MQTT over TCP)
- Financial trading platforms (microsecond latency matters)
- gRPC services
- Any non-HTTP protocol

## ALB vs NLB — When to Use Each

| Feature | ALB | NLB |
|---------|-----|-----|
| OSI Layer | Layer 7 (HTTP) | Layer 4 (TCP/UDP) |
| Protocols | HTTP, HTTPS, WebSocket | TCP, UDP, TLS |
| Routing | Path, host, header, query string | Port and IP only |
| Latency | ~ms | ~µs (microseconds) |
| Static IP | No (use DNS name) | Yes |
| Cost | ~$0.0225/hr + LCU | ~$0.0225/hr + NLCU |
| Best for | Web apps, APIs, microservices | Gaming, IoT, extreme performance |

**Rule of thumb**: If your traffic is HTTP/HTTPS and you need smart routing, use ALB. If you need raw TCP/UDP performance or a static IP, use NLB.

## Target Groups and Health Checks

A **target group** is a collection of resources that receive traffic from the load balancer. Targets can be:

- EC2 instances (by instance ID)
- IP addresses (for containers, on-premises servers)
- Lambda functions (ALB only)
- Another ALB (NLB forwarding to ALB)

### Health Checks

The load balancer periodically checks each target's health. Unhealthy targets stop receiving traffic until they recover.

ALB health check settings:
- **Protocol**: HTTP or HTTPS
- **Path**: e.g., `/health` or `/api/status`
- **Healthy threshold**: Consecutive successes needed (default: 5)
- **Unhealthy threshold**: Consecutive failures needed (default: 2)
- **Interval**: Time between checks (default: 30s)
- **Timeout**: How long to wait for a response (default: 5s)

A properly designed health check endpoint should verify your app is truly functional — not just that the process is running, but that it can connect to its database and dependencies.

### Deregistration Delay

When an instance is removed from a target group (scaling down or failing health checks), the load balancer gives it a **deregistration delay** (default: 300 seconds) to finish serving in-flight requests before cutting it off completely.

## Hands-On: Set Up an ALB with Multiple Targets

```bash
# Create the ALB (must specify at least 2 subnets in different AZs)
aws elbv2 create-load-balancer \
  --name my-web-alb \
  --subnets subnet-0abc123 subnet-0def456 \
  --security-groups sg-0abc123 \
  --scheme internet-facing \
  --type application

# Save the ALB ARN from output
ALB_ARN="arn:aws:elasticloadbalancing:us-east-1:123456789:loadbalancer/app/my-web-alb/abc123"

# Create a target group
aws elbv2 create-target-group \
  --name web-targets \
  --protocol HTTP \
  --port 80 \
  --vpc-id vpc-0abc123 \
  --health-check-path /health \
  --health-check-interval-seconds 30 \
  --healthy-threshold-count 3 \
  --unhealthy-threshold-count 2

TG_ARN="arn:aws:elasticloadbalancing:us-east-1:123456789:targetgroup/web-targets/abc123"

# Register EC2 instances as targets
aws elbv2 register-targets \
  --target-group-arn $TG_ARN \
  --targets Id=i-0abc123 Id=i-0def456

# Create an HTTPS listener (requires an ACM certificate)
aws elbv2 create-listener \
  --load-balancer-arn $ALB_ARN \
  --protocol HTTPS \
  --port 443 \
  --certificates CertificateArn=arn:aws:acm:us-east-1:123456789:certificate/abc-123 \
  --default-actions Type=forward,TargetGroupArn=$TG_ARN

# Create HTTP listener that redirects to HTTPS
aws elbv2 create-listener \
  --load-balancer-arn $ALB_ARN \
  --protocol HTTP \
  --port 80 \
  --default-actions '[{
    "Type": "redirect",
    "RedirectConfig": {
      "Protocol": "HTTPS",
      "Port": "443",
      "StatusCode": "HTTP_301"
    }
  }]'
```

### SSL/TLS Termination with ACM

AWS Certificate Manager (ACM) provides free SSL/TLS certificates for use with ALBs and other AWS services.

```bash
# Request a certificate (free)
aws acm request-certificate \
  --domain-name myapp.com \
  --subject-alternative-names "*.myapp.com" \
  --validation-method DNS

# ACM gives you a CNAME record to add to Route 53 for validation
# Once validated, attach it to your ALB listener (shown above)
```

SSL termination at the ALB means:
- The ALB handles the encryption/decryption (CPU-intensive work)
- Traffic between ALB and your EC2 instances can be HTTP (port 80)
- You manage certificates in one place instead of on every server

### Adding Path-Based Routing Rules

```bash
LISTENER_ARN="arn:aws:elasticloadbalancing:us-east-1:123456789:listener/app/my-web-alb/abc/def"

# Create a rule: /api/* goes to API target group
aws elbv2 create-rule \
  --listener-arn $LISTENER_ARN \
  --priority 10 \
  --conditions '[{"Field": "path-pattern", "Values": ["/api/*"]}]' \
  --actions '[{"Type": "forward", "TargetGroupArn": "'$API_TG_ARN'"}]'
```

## Key Takeaways

- ALB is your go-to for web applications — it does path routing, host routing, redirects, and auth
- NLB is for non-HTTP protocols or when you need static IPs and extreme performance
- Target groups define where traffic goes; health checks ensure it only goes to healthy targets
- Always redirect HTTP to HTTPS at the ALB level using a listener rule
- Use ACM for free SSL certificates — they auto-renew and integrate natively with ALB
- Pricing is ~$16/month base cost per ALB plus usage charges (LCU) based on new connections, active connections, bandwidth, and rule evaluations
- An ALB can host multiple services using listener rules — you don't need one ALB per service
