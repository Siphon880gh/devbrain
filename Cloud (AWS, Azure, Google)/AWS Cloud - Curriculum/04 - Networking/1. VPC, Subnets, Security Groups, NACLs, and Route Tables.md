# VPC, Subnets, Security Groups, NACLs, and Route Tables

Build and secure your own virtual network in AWS — the foundation of all cloud architectures.

## What You'll Learn

- What a VPC is and why every AWS resource lives inside one
- How CIDR notation works for defining IP ranges
- The difference between public and private subnets
- How Security Groups and NACLs protect your resources at different levels
- How route tables, Internet Gateways, and NAT Gateways control traffic flow
- How to build a production-style VPC from scratch using the CLI

## VPC Fundamentals

A **Virtual Private Cloud (VPC)** is your own isolated network inside AWS. Think of it as renting a floor in a building — you control the rooms, doors, and hallways, but the building (AWS) provides the structure.

Every AWS account comes with a **default VPC** in each region. The default VPC has public subnets, an Internet Gateway, and permissive settings so things "just work" when you're getting started. But for production workloads, you create **custom VPCs** with tighter controls.

### CIDR Notation — Defining Your IP Range

When you create a VPC, you assign it a block of private IP addresses using CIDR notation:

```
10.0.0.0/16 → 65,536 IP addresses (10.0.0.0 – 10.0.255.255)
10.0.0.0/24 → 256 IP addresses (10.0.0.0 – 10.0.0.255)
10.0.1.0/24 → 256 IP addresses (10.0.1.0 – 10.0.1.255)
```

The number after the `/` is how many bits are fixed. A `/16` gives you a large block (common for VPCs), and a `/24` gives you a smaller block (common for subnets).

Common private IP ranges for VPCs:
- `10.0.0.0/16` — Most popular choice
- `172.16.0.0/16` — Alternative range
- `192.168.0.0/16` — Familiar from home networks

### Default VPC vs Custom VPC

| Feature | Default VPC | Custom VPC |
|---------|-------------|------------|
| Created automatically | Yes, one per region | No, you create it |
| Subnets | Public in each AZ | You design the layout |
| Internet Gateway | Attached | You attach one if needed |
| Use case | Quick demos, learning | Production workloads |

## Subnets — Public vs Private

Subnets divide your VPC into smaller segments. Each subnet lives in one Availability Zone.

**Public subnet** — Has a route to an Internet Gateway. Resources here can be reached from the internet (if they have a public IP). Use for: load balancers, bastion hosts, NAT Gateways.

**Private subnet** — No direct route to the internet. Resources here are hidden from the outside world. Use for: application servers, databases, internal services.

A typical setup uses at least two AZs for high availability:

```
VPC: 10.0.0.0/16
├── Public Subnet A:  10.0.1.0/24  (us-east-1a)
├── Public Subnet B:  10.0.2.0/24  (us-east-1b)
├── Private Subnet A: 10.0.10.0/24 (us-east-1a)
└── Private Subnet B: 10.0.20.0/24 (us-east-1b)
```

### Internet Gateway and NAT Gateway

An **Internet Gateway (IGW)** connects your VPC to the internet. It's free and one per VPC. Public subnets route `0.0.0.0/0` traffic to the IGW.

A **NAT Gateway** lets private subnet resources access the internet (for updates, API calls) without being reachable FROM the internet. It sits in a public subnet and costs ~$0.045/hr + data processing fees.

```
Internet ←→ IGW ←→ Public Subnet (NAT Gateway lives here)
                          ↕
                    Private Subnet (routes outbound traffic to NAT)
```

## Security Groups vs NACLs

AWS gives you two layers of network security that work differently:

### Security Groups (Instance-Level, Stateful)

Security Groups act as firewalls attached to individual resources (EC2 instances, RDS databases, Lambda in VPC).

- **Stateful**: If you allow inbound traffic, the response is automatically allowed out
- **Allow rules only**: You specify what to permit; everything else is denied by default
- **Instance-level**: Attached directly to ENIs (network interfaces)

Example Security Group for a web server:

| Direction | Protocol | Port | Source | Purpose |
|-----------|----------|------|--------|---------|
| Inbound | TCP | 80 | 0.0.0.0/0 | HTTP from anywhere |
| Inbound | TCP | 443 | 0.0.0.0/0 | HTTPS from anywhere |
| Inbound | TCP | 22 | 203.0.113.5/32 | SSH from your IP only |
| Outbound | All | All | 0.0.0.0/0 | Allow all outbound |

### NACLs (Subnet-Level, Stateless)

Network Access Control Lists operate at the subnet boundary.

- **Stateless**: You must explicitly allow both inbound AND outbound traffic
- **Allow and deny rules**: You can block specific IPs
- **Subnet-level**: Applies to all resources in the subnet
- **Numbered rules**: Evaluated in order, lowest number first

Example NACL for a public subnet:

| Rule # | Direction | Protocol | Port | Source | Action |
|--------|-----------|----------|------|--------|--------|
| 100 | Inbound | TCP | 80 | 0.0.0.0/0 | ALLOW |
| 110 | Inbound | TCP | 443 | 0.0.0.0/0 | ALLOW |
| 120 | Inbound | TCP | 1024-65535 | 0.0.0.0/0 | ALLOW |
| * | Inbound | All | All | 0.0.0.0/0 | DENY |
| 100 | Outbound | TCP | 1024-65535 | 0.0.0.0/0 | ALLOW |
| * | Outbound | All | All | 0.0.0.0/0 | DENY |

Rule 120 inbound and 100 outbound handle **ephemeral ports** — the random high ports used for return traffic. Without these, responses never reach clients.

### When to Use Each

- **Security Groups**: Your primary firewall. Use for every resource.
- **NACLs**: Extra defense layer. Use to block known bad IPs or restrict entire subnets.

## Route Tables and Internet Gateways

A **route table** contains rules (routes) that determine where network traffic goes. Every subnet is associated with exactly one route table.

Default route table for a public subnet:

| Destination | Target | Purpose |
|-------------|--------|---------|
| 10.0.0.0/16 | local | Traffic within the VPC stays local |
| 0.0.0.0/0 | igw-abc123 | Everything else goes to the internet |

Route table for a private subnet:

| Destination | Target | Purpose |
|-------------|--------|---------|
| 10.0.0.0/16 | local | Traffic within the VPC |
| 0.0.0.0/0 | nat-xyz789 | Outbound internet via NAT Gateway |

The `local` route is always present and cannot be removed — it allows communication between all subnets within your VPC.

## Hands-On: Build a Custom VPC

Create a production-style VPC with public and private subnets:

```bash
# Create the VPC
aws ec2 create-vpc \
  --cidr-block 10.0.0.0/16 \
  --tag-specifications 'ResourceType=vpc,Tags=[{Key=Name,Value=my-vpc}]'

# Save the VPC ID (from the output)
VPC_ID=vpc-0abc123def456

# Create an Internet Gateway and attach it
aws ec2 create-internet-gateway \
  --tag-specifications 'ResourceType=internet-gateway,Tags=[{Key=Name,Value=my-igw}]'
IGW_ID=igw-0abc123def456

aws ec2 attach-internet-gateway --vpc-id $VPC_ID --internet-gateway-id $IGW_ID

# Create public subnet
aws ec2 create-subnet \
  --vpc-id $VPC_ID \
  --cidr-block 10.0.1.0/24 \
  --availability-zone us-east-1a \
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=public-subnet-a}]'
PUBLIC_SUBNET_ID=subnet-0abc123

# Create private subnet
aws ec2 create-subnet \
  --vpc-id $VPC_ID \
  --cidr-block 10.0.10.0/24 \
  --availability-zone us-east-1a \
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=private-subnet-a}]'
PRIVATE_SUBNET_ID=subnet-0def456

# Create a route table for the public subnet
aws ec2 create-route-table --vpc-id $VPC_ID \
  --tag-specifications 'ResourceType=route-table,Tags=[{Key=Name,Value=public-rt}]'
PUBLIC_RT_ID=rtb-0abc123

# Add internet route to public route table
aws ec2 create-route \
  --route-table-id $PUBLIC_RT_ID \
  --destination-cidr-block 0.0.0.0/0 \
  --gateway-id $IGW_ID

# Associate public subnet with the route table
aws ec2 associate-route-table \
  --subnet-id $PUBLIC_SUBNET_ID \
  --route-table-id $PUBLIC_RT_ID

# Enable auto-assign public IPs for the public subnet
aws ec2 modify-subnet-attribute \
  --subnet-id $PUBLIC_SUBNET_ID \
  --map-public-ip-on-launch
```

To add a NAT Gateway for the private subnet:

```bash
# Allocate an Elastic IP for the NAT Gateway
aws ec2 allocate-address --domain vpc
EIP_ALLOC_ID=eipalloc-0abc123

# Create NAT Gateway in the public subnet
aws ec2 create-nat-gateway \
  --subnet-id $PUBLIC_SUBNET_ID \
  --allocation-id $EIP_ALLOC_ID \
  --tag-specifications 'ResourceType=natgateway,Tags=[{Key=Name,Value=my-nat}]'
NAT_GW_ID=nat-0abc123

# Create route table for private subnet
aws ec2 create-route-table --vpc-id $VPC_ID \
  --tag-specifications 'ResourceType=route-table,Tags=[{Key=Name,Value=private-rt}]'
PRIVATE_RT_ID=rtb-0def456

# Route outbound traffic through NAT
aws ec2 create-route \
  --route-table-id $PRIVATE_RT_ID \
  --destination-cidr-block 0.0.0.0/0 \
  --nat-gateway-id $NAT_GW_ID

# Associate private subnet
aws ec2 associate-route-table \
  --subnet-id $PRIVATE_SUBNET_ID \
  --route-table-id $PRIVATE_RT_ID
```

Create a Security Group for a web server:

```bash
aws ec2 create-security-group \
  --group-name web-sg \
  --description "Allow HTTP and HTTPS" \
  --vpc-id $VPC_ID

SG_ID=sg-0abc123

aws ec2 authorize-security-group-ingress \
  --group-id $SG_ID \
  --protocol tcp --port 80 --cidr 0.0.0.0/0

aws ec2 authorize-security-group-ingress \
  --group-id $SG_ID \
  --protocol tcp --port 443 --cidr 0.0.0.0/0
```

## Key Takeaways

- A VPC is your isolated network in AWS — define it with a CIDR block like `10.0.0.0/16`
- Public subnets route to an Internet Gateway; private subnets route through a NAT Gateway
- Security Groups are stateful and attached to resources — your primary firewall
- NACLs are stateless and attached to subnets — use for broad allow/deny rules
- Route tables determine where traffic flows — the `local` route enables intra-VPC communication
- Always design for multiple AZs from the start for high availability
- NAT Gateways cost money (~$32/month minimum) — use VPC endpoints for AWS services instead when possible
