# API Gateway — HTTP and REST APIs in Front of Lambda or EC2

Expose your backend services as managed APIs with authentication, throttling, and routing.

## What You'll Learn

- What API Gateway does and why you'd use a managed API layer
- The differences between HTTP API and REST API (and when to pick each)
- How to create an HTTP API integrated with Lambda
- How stages, routes, and CORS work
- How to add authentication with JWT authorizers or Cognito
- How throttling and usage plans protect your backend
- A complete walkthrough: building a serverless API endpoint from scratch

## What is API Gateway?

**API Gateway** is a fully managed service that sits between your clients (browsers, mobile apps, other services) and your backend (Lambda functions, EC2 instances, any HTTP endpoint). It handles all the cross-cutting concerns you'd otherwise have to build yourself:

```
Client → API Gateway → Lambda Function
                     → EC2 / ECS Service
                     → Any HTTP endpoint
```

What it handles for you:
- **Request routing** — Direct different paths/methods to different backends
- **Authentication** — Verify JWTs or Cognito tokens before requests reach your code
- **Throttling** — Protect backends from being overwhelmed
- **CORS** — Handle cross-origin headers without cluttering your code
- **Request/response transformation** — Modify payloads in transit
- **API keys and usage plans** — Meter and limit third-party access
- **Logging and monitoring** — Automatic CloudWatch integration

Without API Gateway, you'd need a reverse proxy (Nginx), auth middleware, rate limiting library, and custom logging — all maintained by you.

## HTTP API vs REST API

AWS offers two types of API Gateway. They solve the same problem but differ in features and cost:

| Feature | HTTP API | REST API |
|---------|----------|----------|
| Cost | $1.00 per million requests | $3.50 per million requests |
| Latency | Lower (~10ms added) | Higher (~30ms added) |
| JWT authorizers | Yes (native) | No (use Lambda authorizer) |
| Cognito authorizer | Yes | Yes |
| API keys / usage plans | No | Yes |
| Request validation | No | Yes |
| Caching | No | Yes (built-in) |
| WebSocket | No | No (use WebSocket API) |
| WAF integration | No | Yes |
| Resource policies | No | Yes |

**Choose HTTP API when**: You want a simple, fast, cheap API with JWT auth. This covers 80% of use cases.

**Choose REST API when**: You need API keys, request validation, caching, WAF, or fine-grained access control.

For most new projects, start with HTTP API and switch to REST API only if you need the extra features.

## Integrating with Lambda and EC2

### Lambda Integration (Most Common)

API Gateway invokes your Lambda function for each request and returns the response to the client. This is the serverless pattern — no servers to manage.

```
POST /users → Lambda: createUser
GET /users/{id} → Lambda: getUser
PUT /users/{id} → Lambda: updateUser
DELETE /users/{id} → Lambda: deleteUser
```

Each route can point to a different Lambda function, or a single function can handle all routes (monolithic Lambda pattern).

### HTTP Integration (EC2, ECS, External)

API Gateway can proxy requests to any HTTP endpoint:

```
GET /legacy/* → http://my-ec2-instance.us-east-1.compute.amazonaws.com:3000
```

This is useful when migrating from servers to serverless incrementally — new routes go to Lambda, old routes forward to your existing servers.

## Stages, Routes, and CORS

### Routes

A route is a combination of HTTP method and path that maps to a backend:

```
GET    /items         → getItems function
POST   /items         → createItem function
GET    /items/{id}    → getItem function (path parameter)
DELETE /items/{id}    → deleteItem function
```

Path parameters (like `{id}`) are passed to your Lambda function in the event object.

### Stages

Stages let you deploy multiple versions of your API:

```
https://abc123.execute-api.us-east-1.amazonaws.com/prod/items
https://abc123.execute-api.us-east-1.amazonaws.com/dev/items
```

HTTP APIs have a default `$default` stage that doesn't add a path prefix:
```
https://abc123.execute-api.us-east-1.amazonaws.com/items
```

### CORS (Cross-Origin Resource Sharing)

If your frontend (`myapp.com`) calls your API (`api.myapp.com`), browsers enforce CORS. API Gateway can handle this for you:

```bash
# CORS is configured at the API level for HTTP APIs
aws apigatewayv2 update-api \
  --api-id abc123 \
  --cors-configuration '{
    "AllowOrigins": ["https://myapp.com"],
    "AllowMethods": ["GET", "POST", "PUT", "DELETE"],
    "AllowHeaders": ["Content-Type", "Authorization"],
    "MaxAge": 86400
  }'
```

With this configured, API Gateway automatically handles preflight OPTIONS requests and adds the correct headers to responses.

## Authentication and Authorization

### JWT Authorizer (HTTP API)

The simplest way to secure an HTTP API. API Gateway validates JWT tokens (from Cognito, Auth0, Firebase, or any OIDC provider) before the request reaches your Lambda.

```bash
# Create a JWT authorizer
aws apigatewayv2 create-authorizer \
  --api-id abc123 \
  --authorizer-type JWT \
  --name cognito-jwt \
  --identity-source '$request.header.Authorization' \
  --jwt-configuration '{
    "Audience": ["your-app-client-id"],
    "Issuer": "https://cognito-idp.us-east-1.amazonaws.com/us-east-1_ABC123"
  }'
```

Attach it to routes:
```bash
aws apigatewayv2 update-route \
  --api-id abc123 \
  --route-id routeId123 \
  --authorization-type JWT \
  --authorizer-id authId456
```

Now any request without a valid JWT in the `Authorization` header gets a `401 Unauthorized` before it ever reaches your Lambda function.

### Cognito Authorizer

Works with Amazon Cognito User Pools directly. Users sign in through Cognito and send the ID token or access token with each request.

### Lambda Authorizer (REST API)

A custom Lambda function that runs before your main function. Use when you need complex auth logic (checking a database, validating custom tokens, IP whitelisting).

## Throttling and Usage Plans

### Default Throttling

API Gateway applies throttling at multiple levels to protect your backend:

- **Account-level**: 10,000 requests/second across all APIs in a region
- **Per-route**: Configurable limits per route

```bash
# Set route-level throttling
aws apigatewayv2 update-stage \
  --api-id abc123 \
  --stage-name '$default' \
  --route-settings '{
    "POST /items": {
      "ThrottlingBurstLimit": 100,
      "ThrottlingRateLimit": 50
    }
  }'
```

This means the `POST /items` route allows:
- **Rate limit**: 50 requests/second sustained
- **Burst limit**: 100 requests in a burst

Requests exceeding these limits receive a `429 Too Many Requests` response.

### Usage Plans and API Keys (REST API Only)

For APIs consumed by third parties, usage plans let you create tiered access:

```
Free tier:   1,000 requests/day, 10/second
Pro tier:    100,000 requests/day, 100/second
Enterprise:  Unlimited, 1,000/second
```

Each consumer gets an API key that maps to a usage plan. API Gateway tracks usage and enforces limits automatically.

## Hands-On: Create an HTTP API Backed by Lambda

Build a complete serverless API from scratch — a simple items CRUD.

### Step 1: Create the Lambda Function

```bash
# Create a simple handler (items-handler.js)
cat << 'EOF' > index.mjs
export const handler = async (event) => {
  const method = event.requestContext.http.method;
  const path = event.requestContext.http.path;
  const pathParams = event.pathParameters || {};
  const body = event.body ? JSON.parse(event.body) : null;

  // In-memory store (use DynamoDB in production)
  // This resets on each cold start — just for demonstration

  switch (`${method} ${path.replace(/\/items\/.*/, '/items/{id}')}`) {
    case 'GET /items':
      return {
        statusCode: 200,
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ items: [], message: 'List items' })
      };

    case 'POST /items':
      return {
        statusCode: 201,
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ item: body, message: 'Item created' })
      };

    case 'GET /items/{id}':
      return {
        statusCode: 200,
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ id: pathParams.id, message: 'Get item' })
      };

    case 'DELETE /items/{id}':
      return {
        statusCode: 200,
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ id: pathParams.id, message: 'Item deleted' })
      };

    default:
      return {
        statusCode: 404,
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ message: 'Not found' })
      };
  }
};
EOF

# Package and create the function
zip function.zip index.mjs

aws lambda create-function \
  --function-name items-api \
  --runtime nodejs20.x \
  --handler index.handler \
  --zip-file fileb://function.zip \
  --role arn:aws:iam::123456789:role/lambda-basic-role
```

### Step 2: Create the HTTP API

```bash
# Create the API
aws apigatewayv2 create-api \
  --name items-api \
  --protocol-type HTTP \
  --cors-configuration '{
    "AllowOrigins": ["*"],
    "AllowMethods": ["GET", "POST", "PUT", "DELETE"],
    "AllowHeaders": ["Content-Type"]
  }'

API_ID="abc123def"

# Create the Lambda integration
aws apigatewayv2 create-integration \
  --api-id $API_ID \
  --integration-type AWS_PROXY \
  --integration-uri arn:aws:lambda:us-east-1:123456789:function:items-api \
  --payload-format-version "2.0"

INTEGRATION_ID="integ123"

# Create routes
aws apigatewayv2 create-route \
  --api-id $API_ID \
  --route-key "GET /items" \
  --target "integrations/$INTEGRATION_ID"

aws apigatewayv2 create-route \
  --api-id $API_ID \
  --route-key "POST /items" \
  --target "integrations/$INTEGRATION_ID"

aws apigatewayv2 create-route \
  --api-id $API_ID \
  --route-key "GET /items/{id}" \
  --target "integrations/$INTEGRATION_ID"

aws apigatewayv2 create-route \
  --api-id $API_ID \
  --route-key "DELETE /items/{id}" \
  --target "integrations/$INTEGRATION_ID"

# Create a stage with auto-deploy
aws apigatewayv2 create-stage \
  --api-id $API_ID \
  --stage-name '$default' \
  --auto-deploy
```

### Step 3: Grant API Gateway Permission to Invoke Lambda

```bash
aws lambda add-permission \
  --function-name items-api \
  --statement-id apigateway-invoke \
  --action lambda:InvokeFunction \
  --principal apigateway.amazonaws.com \
  --source-arn "arn:aws:execute-api:us-east-1:123456789:$API_ID/*/*"
```

### Step 4: Test Your API

```bash
# Get your API endpoint
aws apigatewayv2 get-api --api-id $API_ID --query 'ApiEndpoint'
# Output: "https://abc123def.execute-api.us-east-1.amazonaws.com"

# Test it
curl https://abc123def.execute-api.us-east-1.amazonaws.com/items

curl -X POST https://abc123def.execute-api.us-east-1.amazonaws.com/items \
  -H "Content-Type: application/json" \
  -d '{"name": "Widget", "price": 9.99}'

curl https://abc123def.execute-api.us-east-1.amazonaws.com/items/abc-123
```

### Custom Domain (Optional)

Instead of the auto-generated URL, use your own domain:

```bash
# Create a custom domain name
aws apigatewayv2 create-domain-name \
  --domain-name api.myapp.com \
  --domain-name-configurations '[{
    "CertificateArn": "arn:aws:acm:us-east-1:123456789:certificate/abc-123",
    "EndpointType": "REGIONAL"
  }]'

# Map it to your API
aws apigatewayv2 create-api-mapping \
  --domain-name api.myapp.com \
  --api-id $API_ID \
  --stage '$default'

# Create a Route 53 record for api.myapp.com pointing to the API Gateway domain
```

## Key Takeaways

- API Gateway is a managed reverse proxy — it handles routing, auth, throttling, and CORS so you don't have to
- HTTP API is 70% cheaper and faster than REST API — use it unless you need specific REST API features (API keys, caching, WAF)
- JWT authorizers validate tokens at the gateway level, so unauthorized requests never reach your Lambda
- Always configure CORS at the API Gateway level rather than in your Lambda function
- Throttling protects your backend and downstream services from traffic spikes
- The API endpoint URL is auto-generated; use a custom domain for production APIs
- Pricing: HTTP API costs $1.00/million requests; first 1 million requests/month are free for 12 months
- A single Lambda can handle all routes (simpler deployment) or each route can have its own function (better isolation)
