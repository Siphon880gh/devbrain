# Route 53 ‚Äî DNS, Hosted Zones, and Domain Registration

Manage DNS records, register domains, and route traffic intelligently with AWS Route 53.

## What You'll Learn

- How DNS translates domain names to IP addresses
- What Route 53 is and why the name "53" (DNS uses port 53)
- How hosted zones organize your DNS records
- The different record types and when to use each
- How routing policies enable advanced traffic management
- How to register a domain and configure DNS records
- How health checks enable automatic failover

## How DNS Works

When someone types `myapp.com` into their browser, their computer needs to find the IP address of your server. This translation is handled by the **Domain Name System (DNS)**.

The lookup process:

```
1. Browser asks: "What's the IP for myapp.com?"
2. Local DNS resolver checks its cache
3. If not cached ‚Üí asks Root nameserver ‚Üí ".com" TLD nameserver ‚Üí your authoritative nameserver
4. Your nameserver responds: "myapp.com ‚Üí 54.210.33.100"
5. Browser connects to that IP address
```

This entire process takes milliseconds but involves multiple servers cooperating to resolve the name.

**Route 53** is AWS's DNS service. It serves as both:
- A **DNS service** (hosting your DNS records and answering queries)
- A **domain registrar** (purchasing and managing domain names)

The name comes from TCP/UDP port 53, which is the standard port for DNS traffic.

## Hosted Zones and Record Types

### Hosted Zones

A **hosted zone** is a container for DNS records belonging to a domain. When you create a hosted zone for `myapp.com`, you manage all records under that domain there.

**Public hosted zone** ‚Äî Answers DNS queries from the internet. Use this for any publicly accessible service.

**Private hosted zone** ‚Äî Only answers queries from within associated VPCs. Use for internal service discovery (e.g., `database.internal.myapp.com` resolving to a private IP).

```bash
# Create a public hosted zone
aws route53 create-hosted-zone \
  --name myapp.com \
  --caller-reference "2024-01-15-initial"

# Create a private hosted zone for internal use
aws route53 create-hosted-zone \
  --name internal.myapp.com \
  --caller-reference "2024-01-15-internal" \
  --vpc VPCRegion=us-east-1,VPCId=vpc-0abc123def456
```

A hosted zone costs $0.50/month regardless of how many records it contains.

### Record Types

| Record Type | Purpose | Example |
|-------------|---------|---------|
| **A** | Maps domain to IPv4 address | `myapp.com ‚Üí 54.210.33.100` |
| **AAAA** | Maps domain to IPv6 address | `myapp.com ‚Üí 2001:db8::1` |
| **CNAME** | Maps domain to another domain | `www.myapp.com ‚Üí myapp.com` |
| **Alias** | AWS-specific, maps to AWS resources | `myapp.com ‚Üí d1234.cloudfront.net` |
| **MX** | Mail server routing | `myapp.com ‚Üí 10 mail.myapp.com` |
| **TXT** | Text data (verification, SPF) | `myapp.com ‚Üí "v=spf1 include:_spf.google.com"` |
| **NS** | Nameserver delegation | Auto-created with hosted zone |
| **SOA** | Start of authority | Auto-created with hosted zone |

### Alias Records ‚Äî The AWS Superpower

Alias records are unique to Route 53 and solve a key DNS limitation. Standard DNS does not allow a CNAME at the zone apex (the bare domain like `myapp.com`). Alias records can:

- Point `myapp.com` (zone apex) to a CloudFront distribution, ALB, or S3 bucket
- Respond with the actual IP address (faster resolution)
- Cost nothing for queries to AWS resources (CNAME queries cost $0.40/million)

```bash
# Create an Alias record pointing to a CloudFront distribution
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890 \
  --change-batch '{
    "Changes": [{
      "Action": "CREATE",
      "ResourceRecordSet": {
        "Name": "myapp.com",
        "Type": "A",
        "AliasTarget": {
          "HostedZoneId": "Z2FDTNDATAQYW2",
          "DNSName": "d1234abcd.cloudfront.net",
          "EvaluateTargetHealth": false
        }
      }
    }]
  }'
```

## Routing Policies (Simple, Weighted, Latency, Failover)

Route 53 doesn't just resolve names ‚Äî it can route traffic intelligently.

### Simple Routing

One record, one or more values. If multiple IPs are listed, Route 53 returns them in random order.

```bash
# Simple A record
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890 \
  --change-batch '{
    "Changes": [{
      "Action": "CREATE",
      "ResourceRecordSet": {
        "Name": "myapp.com",
        "Type": "A",
        "TTL": 300,
        "ResourceRecords": [
          {"Value": "54.210.33.100"},
          {"Value": "54.210.33.101"}
        ]
      }
    }]
  }'
```

### Weighted Routing

Split traffic by percentage. Great for gradual deployments or A/B testing.

```bash
# Send 90% of traffic to the primary server
# Record 1: weight 90
"ResourceRecordSet": {
  "Name": "myapp.com",
  "Type": "A",
  "SetIdentifier": "primary",
  "Weight": 90,
  "TTL": 300,
  "ResourceRecords": [{"Value": "54.210.33.100"}]
}

# Record 2: weight 10 (sends 10% to the new version)
"ResourceRecordSet": {
  "Name": "myapp.com",
  "Type": "A",
  "SetIdentifier": "canary",
  "Weight": 10,
  "TTL": 300,
  "ResourceRecords": [{"Value": "54.210.33.200"}]
}
```

### Latency-Based Routing

Sends users to the region with the lowest latency from their location.

```
User in Tokyo ‚Üí routed to ap-northeast-1 (50ms latency)
User in London ‚Üí routed to eu-west-1 (20ms latency)
User in New York ‚Üí routed to us-east-1 (10ms latency)
```

### Failover Routing

Automatically switches to a backup when the primary fails a health check.

```
Primary: production ALB (active ‚Äî health check passes)
Secondary: S3 static "maintenance page" (passive ‚Äî used when primary fails)
```

### Geolocation Routing

Route based on the user's geographic location (country or continent). Useful for serving region-specific content or complying with data residency requirements.

## Domain Registration

You can purchase domains directly through Route 53. Prices vary by TLD:
- `.com` ‚Äî ~$13/year
- `.io` ‚Äî ~$39/year
- `.dev` ‚Äî ~$14/year

```bash
# Check domain availability
aws route53domains check-domain-availability \
  --domain-name myapp.com

# Register a domain (creates the hosted zone automatically)
aws route53domains register-domain \
  --domain-name myapp.com \
  --duration-in-years 1 \
  --admin-contact FirstName=Jane,LastName=Doe,ContactType=PERSON,\
Email=jane@example.com,PhoneNumber=+1.5551234567,\
AddressLine1="123 Main St",City=Seattle,State=WA,\
CountryCode=US,ZipCode=98101 \
  --registrant-contact FirstName=Jane,LastName=Doe,ContactType=PERSON,\
Email=jane@example.com,PhoneNumber=+1.5551234567,\
AddressLine1="123 Main St",City=Seattle,State=WA,\
CountryCode=US,ZipCode=98101 \
  --tech-contact FirstName=Jane,LastName=Doe,ContactType=PERSON,\
Email=jane@example.com,PhoneNumber=+1.5551234567,\
AddressLine1="123 Main St",City=Seattle,State=WA,\
CountryCode=US,ZipCode=98101
```

If you already own a domain registered elsewhere (GoDaddy, Namecheap), you can either:
1. **Transfer** the domain to Route 53
2. **Keep** it there but point the NS records to Route 53's nameservers

## Hands-On: Register a Domain and Configure DNS

Practical workflow for pointing a domain at your infrastructure:

```bash
# Step 1: Create the hosted zone (if domain registered elsewhere)
aws route53 create-hosted-zone \
  --name myapp.com \
  --caller-reference "$(date +%s)"

# Step 2: Note the 4 NS records from the output ‚Äî update these
# at your registrar if domain is registered elsewhere

# Step 3: Create an A record pointing to your EC2 instance
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890 \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "myapp.com",
        "Type": "A",
        "TTL": 300,
        "ResourceRecords": [{"Value": "54.210.33.100"}]
      }
    }]
  }'

# Step 4: Create a CNAME for www
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890 \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "www.myapp.com",
        "Type": "CNAME",
        "TTL": 300,
        "ResourceRecords": [{"Value": "myapp.com"}]
      }
    }]
  }'

# Step 5: Create a health check for failover
aws route53 create-health-check \
  --caller-reference "primary-health-$(date +%s)" \
  --health-check-config '{
    "IPAddress": "54.210.33.100",
    "Port": 443,
    "Type": "HTTPS",
    "ResourcePath": "/health",
    "RequestInterval": 30,
    "FailureThreshold": 3
  }'
```

### Health Checks

Route 53 health checks monitor your endpoints every 10 or 30 seconds from multiple locations worldwide. If an endpoint fails the threshold (default: 3 consecutive failures), Route 53 stops routing traffic to it.

Health check types:
- **Endpoint** ‚Äî Checks a specific IP/domain + port + path
- **Calculated** ‚Äî Combines multiple health checks (e.g., "healthy if 2 of 3 pass")
- **CloudWatch alarm** ‚Äî Healthy/unhealthy based on a CloudWatch metric

Health checks cost ~$0.50-$2.00/month each depending on configuration.

## Key Takeaways

- Route 53 is both a DNS hosting service and a domain registrar
- Use Alias records (free) instead of CNAMEs when pointing to AWS resources
- Hosted zones cost $0.50/month; DNS queries cost $0.40 per million
- Routing policies enable blue/green deployments, regional routing, and automatic failover
- Health checks monitor endpoints and trigger failover automatically
- TTL (Time To Live) controls how long resolvers cache your records ‚Äî lower TTL means faster propagation but more queries
- Always set up both the apex domain (`myapp.com`) and `www` subdomain
