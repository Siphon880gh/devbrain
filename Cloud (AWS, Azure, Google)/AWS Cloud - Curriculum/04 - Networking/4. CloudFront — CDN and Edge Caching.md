# CloudFront — CDN and Edge Caching

Deliver content globally with low latency using AWS CloudFront's content delivery network.

## What You'll Learn

- What a CDN is and why it makes websites faster worldwide
- How CloudFront distributions, edge locations, and origins work together
- How to configure cache behaviors and TTLs
- How to serve a static website from S3 through CloudFront with HTTPS
- How Origin Access Control (OAC) secures your S3 bucket
- How to invalidate cached content when you deploy updates
- CloudFront pricing and free tier

## What is a CDN?

A **Content Delivery Network (CDN)** caches copies of your content at servers around the world. Instead of every user fetching files from your origin server (which might be in a single region), they get the content from the nearest edge location.

Without CDN:
```
User in Sydney → Request travels to us-east-1 (200ms round trip)
User in London → Request travels to us-east-1 (80ms round trip)
User in New York → Request travels to us-east-1 (10ms round trip)
```

With CloudFront:
```
User in Sydney → Nearest edge location in Sydney (10ms)
User in London → Nearest edge location in London (5ms)
User in New York → Nearest edge location in Virginia (5ms)
```

CloudFront has **450+ edge locations** across 90+ cities worldwide. The first request for a file goes to your origin; subsequent requests are served from the edge cache.

## CloudFront Distributions and Origins

A **distribution** is a CloudFront instance — it's what you create and configure. Each distribution gets a domain name like `d1234abcd.cloudfront.net`.

An **origin** is where CloudFront fetches the original content from. Supported origins:

- **S3 bucket** — Static files, websites, media
- **Application Load Balancer** — Dynamic web applications
- **EC2 instance** — Direct server access (less common)
- **Custom origin** — Any HTTP server with a public URL
- **MediaStore / MediaPackage** — Video streaming

You can have multiple origins in one distribution and route requests to different origins based on URL path patterns:

```
/api/*        → ALB origin (dynamic content)
/images/*     → S3 bucket origin (static assets)
/videos/*     → MediaPackage origin (streaming)
/*            → S3 bucket origin (default — website files)
```

### Edge Locations vs Regional Edge Caches

```
User → Edge Location (450+ worldwide, smallest cache)
         ↓ cache miss
     Regional Edge Cache (13 locations, larger cache)
         ↓ cache miss
     Origin (your S3 bucket or ALB)
```

Regional Edge Caches sit between edge locations and your origin. They have larger caches and reduce the load on your origin server.

## Caching Behavior and TTLs

**Cache behaviors** define how CloudFront handles requests matching specific URL patterns. Each behavior specifies:

- Which origin to forward requests to
- Whether to cache responses (and for how long)
- Which HTTP methods to allow (GET, POST, PUT, etc.)
- Whether to forward query strings, cookies, or headers

### TTL (Time To Live)

TTL controls how long CloudFront keeps a cached copy before checking the origin again:

| Setting | Value | Use Case |
|---------|-------|----------|
| Minimum TTL | 0 seconds | Allow origin to control caching |
| Default TTL | 86400 (24 hours) | Standard static assets |
| Maximum TTL | 31536000 (1 year) | Versioned assets (style.abc123.css) |

Best practice for static sites: Use long TTLs with **file versioning**. Instead of `style.css`, use `style.a3f2b1.css`. When you update the file, the filename changes, so you never serve stale content.

### Cache Keys

CloudFront decides whether a request is a "cache hit" based on the **cache key**. By default, the cache key includes:
- The URL path
- Query strings (if configured to forward them)

If you forward cookies or headers as part of the cache key, you reduce cache hit rates because more variations mean more cache entries.

## HTTPS and Custom Domains

### Using Your Own Domain

Instead of `d1234abcd.cloudfront.net`, you can use `cdn.myapp.com` or even `myapp.com`:

1. Request an SSL certificate in ACM (**must be in us-east-1** for CloudFront)
2. Add your domain as a CNAME in the CloudFront distribution settings
3. Create a Route 53 Alias record pointing your domain to the CloudFront distribution

```bash
# Request certificate in us-east-1 (required for CloudFront)
aws acm request-certificate \
  --region us-east-1 \
  --domain-name myapp.com \
  --subject-alternative-names "*.myapp.com" \
  --validation-method DNS
```

### Viewer Protocol Policy

Control how users connect to CloudFront:
- **HTTP and HTTPS** — Allow both (not recommended)
- **Redirect HTTP to HTTPS** — Best for most sites
- **HTTPS Only** — Reject HTTP connections entirely

## Invalidations

When you update files at your origin, CloudFront might still serve the old cached version until the TTL expires. **Invalidations** force CloudFront to remove specific files from all edge caches.

```bash
# Invalidate specific files
aws cloudfront create-invalidation \
  --distribution-id E1234567890 \
  --paths "/index.html" "/css/style.css"

# Invalidate everything (use sparingly)
aws cloudfront create-invalidation \
  --distribution-id E1234567890 \
  --paths "/*"
```

Pricing: The first 1,000 invalidation paths per month are free. After that, $0.005 per path. A wildcard (`/*`) counts as one path but invalidates everything.

**Better alternative**: Use versioned filenames (`app.v2.js`) so you never need invalidations for assets. Only invalidate `index.html` which must keep the same name.

## Hands-On: Serve an S3 Static Site via CloudFront

This walkthrough sets up a complete static website with CloudFront, HTTPS, and proper S3 security.

### Step 1: Create the S3 Bucket

```bash
# Create bucket (doesn't need to be configured as website hosting)
aws s3 mb s3://myapp-website-assets

# Upload your site files
aws s3 sync ./build/ s3://myapp-website-assets/
```

Unlike direct S3 website hosting, you do NOT enable "Static website hosting" on the bucket when using CloudFront with OAC. CloudFront handles everything.

### Step 2: Create Origin Access Control (OAC)

OAC ensures only CloudFront can read from your S3 bucket — users cannot bypass CloudFront and access S3 directly.

```bash
# Create an OAC
aws cloudfront create-origin-access-control \
  --origin-access-control-config '{
    "Name": "myapp-oac",
    "Description": "OAC for myapp S3 bucket",
    "SigningProtocol": "sigv4",
    "SigningBehavior": "always",
    "OriginAccessControlOriginType": "s3"
  }'
```

### Step 3: Create the CloudFront Distribution

```bash
aws cloudfront create-distribution \
  --distribution-config '{
    "CallerReference": "myapp-2024",
    "Comment": "My App Static Site",
    "DefaultRootObject": "index.html",
    "Enabled": true,
    "Origins": {
      "Quantity": 1,
      "Items": [{
        "Id": "myapp-s3-origin",
        "DomainName": "myapp-website-assets.s3.us-east-1.amazonaws.com",
        "S3OriginConfig": {
          "OriginAccessIdentity": ""
        },
        "OriginAccessControlId": "E1234OACID"
      }]
    },
    "DefaultCacheBehavior": {
      "TargetOriginId": "myapp-s3-origin",
      "ViewerProtocolPolicy": "redirect-to-https",
      "AllowedMethods": {
        "Quantity": 2,
        "Items": ["GET", "HEAD"]
      },
      "CachePolicyId": "658327ea-f89d-4fab-a63d-7e88639e58f6",
      "Compress": true
    },
    "ViewerCertificate": {
      "ACMCertificateArn": "arn:aws:acm:us-east-1:123456789:certificate/abc-123",
      "SSLSupportMethod": "sni-only",
      "MinimumProtocolVersion": "TLSv1.2_2021"
    },
    "Aliases": {
      "Quantity": 1,
      "Items": ["myapp.com"]
    },
    "CustomErrorResponses": {
      "Quantity": 1,
      "Items": [{
        "ErrorCode": 403,
        "ResponsePagePath": "/index.html",
        "ResponseCode": "200",
        "ErrorCachingMinTTL": 10
      }]
    }
  }'
```

The `CustomErrorResponses` section is critical for single-page applications (React, Vue, Angular). When CloudFront gets a 403 from S3 (file not found), it serves `index.html` instead, letting your client-side router handle the URL.

### Step 4: Update S3 Bucket Policy

Allow CloudFront to read from the bucket:

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "AllowCloudFrontServicePrincipal",
    "Effect": "Allow",
    "Principal": {
      "Service": "cloudfront.amazonaws.com"
    },
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::myapp-website-assets/*",
    "Condition": {
      "StringEquals": {
        "AWS:SourceArn": "arn:aws:cloudfront::123456789:distribution/E1234567890"
      }
    }
  }]
}
```

```bash
aws s3api put-bucket-policy \
  --bucket myapp-website-assets \
  --policy file://bucket-policy.json
```

### Step 5: Create Route 53 Alias Record

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890 \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "myapp.com",
        "Type": "A",
        "AliasTarget": {
          "HostedZoneId": "Z2FDTNDATAQYW2",
          "DNSName": "d1234abcd.cloudfront.net",
          "EvaluateTargetHealth": false
        }
      }
    }]
  }'
```

The `HostedZoneId` value `Z2FDTNDATAQYW2` is always the same for all CloudFront distributions — it's a fixed constant.

## Key Takeaways

- CloudFront caches content at 450+ edge locations worldwide, drastically reducing latency
- Use OAC (not the older OAI) to restrict S3 access to CloudFront only
- ACM certificates for CloudFront **must** be created in `us-east-1`
- Use versioned filenames for assets and only invalidate `index.html` on deploys
- Enable compression (`Compress: true`) — CloudFront will gzip/brotli assets automatically
- For SPAs, configure custom error responses to redirect 403/404 to `index.html`
- Free tier: 1 TB data transfer out + 10,000,000 HTTP requests per month for the first 12 months
- After free tier: ~$0.085/GB for the first 10 TB from North America/Europe
