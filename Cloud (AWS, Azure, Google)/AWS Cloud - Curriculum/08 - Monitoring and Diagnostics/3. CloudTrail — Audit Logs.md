# CloudTrail ‚Äî Audit Logs for Every API Call

Discover how AWS CloudTrail records every API call made in your account, giving you an audit trail for security, compliance, and troubleshooting.

## What You'll Learn

- What CloudTrail records and why it matters

## What Is CloudTrail?

CloudTrail is AWS's audit log. It records **every API call** made in your AWS account ‚Äî whether that call came from the console, CLI, SDK, or another AWS service.

Every time someone:
- Launches an EC2 instance
- Creates an S3 bucket
- Changes a security group rule
- Deletes a Lambda function
- Logs into the console

...CloudTrail records who did it, when, from where, and what the request/response looked like.

Think of CloudTrail as a security camera for your AWS account. You might not watch the footage every day, but when something goes wrong, it's invaluable.

**Key difference from CloudWatch:**
- **CloudWatch** = performance monitoring (how is my system doing?)
- **CloudTrail** = audit logging (who did what in my account?)

## Management Events vs. Data Events

CloudTrail categorizes API calls into two types:

### Management Events (Control Plane)

These are operations that manage your AWS resources:
- `CreateBucket`, `DeleteBucket`
- `RunInstances`, `TerminateInstances`
- `CreateFunction`, `UpdateFunctionCode`
- `AttachRolePolicy`, `CreateUser`
- Console sign-in events

Management events are logged **by default** and are free for the last 90 days of history.

### Data Events (Data Plane)

These are operations on the data *within* your resources:
- `GetObject`, `PutObject` (S3)
- `Invoke` (Lambda)
- `GetItem`, `PutItem` (DynamoDB)

Data events are **not logged by default** because they're high-volume (imagine logging every S3 download). You must explicitly enable them, and they cost $0.10 per 100,000 events.

| | Management Events | Data Events |
|---|---|---|
| Example | CreateBucket | PutObject |
| Default | Logged automatically | Must be enabled |
| Volume | Low (dozens/day) | High (millions/day) |
| Cost | Free (90-day history) | $0.10/100K events |

## Trails, Event History, and S3 Storage

### Event History (No Setup Required)

Without creating a trail, you automatically get **90 days** of management event history. You can search it in the console or via CLI:

```bash
# Look up events from the last 7 days
aws cloudtrail lookup-events \
  --start-time "2024-01-08T00:00:00Z" \
  --end-time "2024-01-15T00:00:00Z" \
  --max-results 20
```

### Creating a Trail

A trail delivers events as JSON log files to an S3 bucket for long-term storage (beyond 90 days):

```bash
# Create an S3 bucket for CloudTrail logs
aws s3 mb s3://my-cloudtrail-logs-123456

# Apply the required bucket policy (CloudTrail needs write access)
aws s3api put-bucket-policy \
  --bucket my-cloudtrail-logs-123456 \
  --policy '{
    "Version": "2012-10-17",
    "Statement": [{
      "Sid": "AWSCloudTrailWrite",
      "Effect": "Allow",
      "Principal": {"Service": "cloudtrail.amazonaws.com"},
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-cloudtrail-logs-123456/AWSLogs/123456789012/*",
      "Condition": {
        "StringEquals": {"s3:x-amz-acl": "bucket-owner-full-control"}
      }
    },
    {
      "Sid": "AWSCloudTrailAclCheck",
      "Effect": "Allow",
      "Principal": {"Service": "cloudtrail.amazonaws.com"},
      "Action": "s3:GetBucketAcl",
      "Resource": "arn:aws:s3:::my-cloudtrail-logs-123456"
    }]
  }'

# Create the trail
aws cloudtrail create-trail \
  --name my-audit-trail \
  --s3-bucket-name my-cloudtrail-logs-123456 \
  --is-multi-region-trail

# Start logging
aws cloudtrail start-logging --name my-audit-trail
```

The `--is-multi-region-trail` flag records events from all AWS regions, not just the region where you created the trail. Always use this ‚Äî resources can be created in any region.

## Reading a CloudTrail Event

Each event is a JSON object. Here's what a real event looks like when someone deletes an S3 bucket:

```json
{
  "eventVersion": "1.08",
  "userIdentity": {
    "type": "IAMUser",
    "principalId": "AIDAEXAMPLE123",
    "arn": "arn:aws:iam::123456789012:user/alice",
    "accountId": "123456789012",
    "userName": "alice"
  },
  "eventTime": "2024-01-15T14:23:45Z",
  "eventSource": "s3.amazonaws.com",
  "eventName": "DeleteBucket",
  "awsRegion": "us-east-1",
  "sourceIPAddress": "203.0.113.25",
  "userAgent": "aws-cli/2.15.0",
  "requestParameters": {
    "bucketName": "my-important-data"
  },
  "responseElements": null,
  "errorCode": null,
  "errorMessage": null
}
```

**Key fields to look at:**

| Field | What It Tells You |
|-------|-------------------|
| `userIdentity` | WHO did it (IAM user, role, or root) |
| `eventTime` | WHEN it happened |
| `eventName` | WHAT they did |
| `sourceIPAddress` | WHERE they were (IP address) |
| `userAgent` | HOW they did it (console, CLI, SDK) |
| `requestParameters` | The details of the request |
| `errorCode` | If the action was denied or failed |

## Multi-Region and Organization Trails

### Multi-Region Trail

A multi-region trail captures events in all AWS regions. If someone creates resources in `ap-southeast-1` while your team works in `us-east-1`, you'll still see it:

```bash
aws cloudtrail update-trail \
  --name my-audit-trail \
  --is-multi-region-trail
```

### Organization Trail

If you use AWS Organizations, you can create a single trail that captures events from all member accounts:

```bash
aws cloudtrail create-trail \
  --name org-trail \
  --s3-bucket-name org-cloudtrail-logs \
  --is-multi-region-trail \
  --is-organization-trail
```

This is essential for enterprises ‚Äî one place to audit activity across all accounts.

## Integrating CloudTrail with CloudWatch Logs

You can stream CloudTrail events to CloudWatch Logs for real-time alerting:

```bash
aws cloudtrail update-trail \
  --name my-audit-trail \
  --cloud-watch-logs-log-group-arn "arn:aws:logs:us-east-1:123456789012:log-group:CloudTrail/Events:*" \
  --cloud-watch-logs-role-arn "arn:aws:iam::123456789012:role/CloudTrail-CloudWatch-Role"
```

Once connected, you can create **metric filters** that trigger alarms:

```bash
# Create a metric filter for security group changes
aws logs put-metric-filter \
  --log-group-name "CloudTrail/Events" \
  --filter-name "SecurityGroupChanges" \
  --filter-pattern '{ ($.eventName = "AuthorizeSecurityGroupIngress") || ($.eventName = "RevokeSecurityGroupIngress") }' \
  --metric-transformations \
    metricName=SecurityGroupChangeCount,metricNamespace=CloudTrailMetrics,metricValue=1
```

Then create an alarm on that metric to get notified whenever someone modifies a security group.

### CloudTrail Lake

CloudTrail Lake lets you run SQL queries directly on your event data without needing to set up an S3 bucket or Athena:

```sql
-- Find who deleted resources in the last 7 days
SELECT
  eventTime,
  userIdentity.userName,
  eventName,
  requestParameters
FROM events
WHERE eventName LIKE 'Delete%'
  AND eventTime > '2024-01-08 00:00:00'
ORDER BY eventTime DESC

-- Find all actions by a specific user
SELECT eventTime, eventName, eventSource, sourceIPAddress
FROM events
WHERE userIdentity.userName = 'alice'
  AND eventTime > '2024-01-01 00:00:00'
ORDER BY eventTime DESC
```

CloudTrail Lake retains data for up to 7 years and charges based on data ingested ($2.50/GB).

## Hands-On: Find Who Did What in Your Account

**Scenario 1: "Who deleted my S3 bucket?"**

```bash
# Search by event name
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=DeleteBucket \
  --max-results 10
```

**Scenario 2: "What did user alice do today?"**

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=Username,AttributeValue=alice \
  --start-time "2024-01-15T00:00:00Z" \
  --end-time "2024-01-15T23:59:59Z"
```

**Scenario 3: "Who changed this security group?"**

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=ResourceName,AttributeValue=sg-0abc123def456 \
  --max-results 20
```

**Scenario 4: "Show me all console logins"**

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=ConsoleLogin \
  --start-time "2024-01-01T00:00:00Z"
```

**Available lookup attributes:**
- `EventId` ‚Äî specific event
- `EventName` ‚Äî the API action
- `ReadOnly` ‚Äî true/false
- `Username` ‚Äî IAM user name
- `ResourceType` ‚Äî e.g., `AWS::S3::Bucket`
- `ResourceName` ‚Äî specific resource ID/name
- `EventSource` ‚Äî e.g., `s3.amazonaws.com`
- `AccessKeyId` ‚Äî which access key was used

## Security and Compliance Use Cases

**1. Detecting unauthorized access:**
Monitor for `ConsoleLogin` events with `errorCode: "Failed"` ‚Äî multiple failed logins may indicate a brute force attempt.

**2. Tracking privilege escalation:**
Alert on `AttachUserPolicy`, `AttachRolePolicy`, or `PutUserPolicy` events ‚Äî someone may be granting themselves excessive permissions.

**3. Compliance auditing:**
Prove that only authorized users accessed sensitive resources. CloudTrail provides the evidence that auditors need.

**4. Incident investigation:**
When a resource is misconfigured or deleted, trace back through CloudTrail to find the exact moment and person responsible.

**5. Protecting the trail itself:**
Enable log file validation to detect if someone tampers with your audit logs:

```bash
aws cloudtrail update-trail \
  --name my-audit-trail \
  --enable-log-file-validation
```

This creates a hash digest file every hour, so you can verify logs haven't been modified.

## Key Takeaways

- CloudTrail records every API call in your account ‚Äî it's your security audit log
- Management events (create/delete/modify resources) are logged free for 90 days
- Data events (S3 object access, Lambda invocations) must be explicitly enabled
- Always create a multi-region trail so you don't miss activity in other regions
- Store trail logs in S3 for long-term retention beyond 90 days
- Integrate with CloudWatch Logs to create real-time alerts on suspicious activity
- Use CloudTrail Lake for SQL queries when investigating incidents
- **Free tier:** one copy of management events per region, 90-day event history
