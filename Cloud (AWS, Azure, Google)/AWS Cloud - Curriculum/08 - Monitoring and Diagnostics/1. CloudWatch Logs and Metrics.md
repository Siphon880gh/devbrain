# CloudWatch Logs and Metrics ÔÇö Observability Basics

Learn how AWS CloudWatch collects, stores, and surfaces logs and metrics so you can see what your applications and infrastructure are doing in real time.

## What You'll Learn

- What CloudWatch is and why observability matters
- How metrics work: namespaces, dimensions, statistics, and periods
- Built-in metrics for EC2, Lambda, and other services
- How to publish your own custom metrics
- How log groups and log streams organize your application logs
- How to send logs from Lambda (automatic) and EC2 (CloudWatch Agent)
- Retention policies and cost considerations

## How CloudWatch Fits into AWS

CloudWatch is the central monitoring service for everything running in AWS. Think of it as two things combined:

1. **A metrics database** ÔÇö stores time-series numbers (CPU usage, request count, error rate)
2. **A log aggregation system** ÔÇö collects and stores text logs from your applications and services

Almost every AWS service automatically sends metrics to CloudWatch at no extra cost. When your EC2 instance is running, AWS is already recording its CPU utilization, network traffic, and disk I/O. When your Lambda function executes, AWS records its duration, errors, and invocation count.

You don't need to install anything or opt in ÔÇö basic metrics just appear. This is what makes CloudWatch the natural starting point for monitoring on AWS.

## Metrics ÔÇö Numbers Over Time

A **metric** is a time-ordered set of data points. Each data point has a timestamp and a numeric value.

**Key concepts:**

| Concept | What It Means |
|---------|---------------|
| **Namespace** | A container that groups related metrics. Example: `AWS/EC2`, `AWS/Lambda`, `AWS/RDS` |
| **Metric name** | The specific measurement. Example: `CPUUtilization`, `Duration`, `Errors` |
| **Dimension** | A key-value pair that identifies which resource the metric belongs to. Example: `InstanceId=i-0abc123` |
| **Period** | The time granularity for aggregation (60 seconds, 5 minutes, etc.) |
| **Statistic** | How data points in a period are aggregated: `Average`, `Sum`, `Minimum`, `Maximum`, `SampleCount` |

**Example:** The metric `AWS/EC2 Ôćĺ CPUUtilization` with dimension `InstanceId=i-0abc123` and a period of 300 seconds tells you the average CPU usage of that specific instance in 5-minute intervals.

### Common Built-In Metrics

**EC2:**
- `CPUUtilization` ÔÇö percentage of allocated CPU in use
- `NetworkIn` / `NetworkOut` ÔÇö bytes transferred
- `StatusCheckFailed` ÔÇö indicates hardware or software problems

**Lambda:**
- `Invocations` ÔÇö number of times the function ran
- `Duration` ÔÇö execution time in milliseconds
- `Errors` ÔÇö invocations that resulted in an error
- `Throttles` ÔÇö invocations that were rejected due to concurrency limits
- `ConcurrentExecutions` ÔÇö functions running simultaneously

**S3:**
- `NumberOfObjects` ÔÇö total objects in a bucket
- `BucketSizeBytes` ÔÇö total storage used

## Namespaces and Dimensions

Namespaces prevent metric name collisions. AWS services use the format `AWS/ServiceName`. When you publish custom metrics, you pick your own namespace (e.g., `MyApp/Orders`).

Dimensions let you slice metrics by specific resources:

```
Namespace: AWS/EC2
Metric: CPUUtilization
Dimensions:
  InstanceId: i-0abc123def456
```

You can have multiple dimensions on a single metric. For example, you might publish a custom metric with both `Environment=production` and `Service=payment-api` dimensions, letting you filter by either.

## Logs ÔÇö Streams, Groups, and Retention

CloudWatch Logs organizes log data into two levels:

- **Log group** ÔÇö a collection of log streams that share the same retention and permissions settings. Typically one log group per application or service. Example: `/aws/lambda/my-function`
- **Log stream** ÔÇö a sequence of log events from a single source. Example: one stream per Lambda execution environment, or one per EC2 instance.

### Log Events

Each log event has:
- A **timestamp** (when the event occurred)
- A **message** (the actual log text)

### Sending Logs from Lambda

Lambda sends logs to CloudWatch automatically. Every `console.log()` (Node.js), `print()` (Python), or stdout output goes to a log group named `/aws/lambda/<function-name>`. Each execution environment creates its own log stream.

```python
# This Python Lambda code automatically logs to CloudWatch
def handler(event, context):
    print("Received event:", event)
    # This line appears in CloudWatch Logs automatically
    return {"statusCode": 200, "body": "OK"}
```

No configuration required ÔÇö Lambda handles log delivery as long as your function's execution role has the `logs:CreateLogGroup`, `logs:CreateLogStream`, and `logs:PutLogEvents` permissions (included in `AWSLambdaBasicExecutionRole`).

### Sending Logs from EC2

EC2 instances do NOT send logs automatically. You must install the **CloudWatch Agent**:

```bash
# Install the CloudWatch Agent on Amazon Linux 2
sudo yum install -y amazon-cloudwatch-agent

# Create a basic configuration
sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-config-wizard

# Start the agent
sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \
  -a fetch-config \
  -m ec2 \
  -s \
  -c file:/opt/aws/amazon-cloudwatch-agent/bin/config.json
```

A minimal agent config to ship `/var/log/myapp.log`:

```json
{
  "logs": {
    "logs_collected": {
      "files": {
        "collect_list": [
          {
            "file_path": "/var/log/myapp.log",
            "log_group_name": "/ec2/myapp",
            "log_stream_name": "{instance_id}"
          }
        ]
      }
    }
  }
}
```

The EC2 instance also needs an IAM role with `CloudWatchAgentServerPolicy` attached.

### Retention Settings

By default, log groups retain logs **forever** (and you pay for storage). Always set a retention policy:

```bash
# Set retention to 30 days
aws logs put-retention-policy \
  --log-group-name "/aws/lambda/my-function" \
  --retention-in-days 30
```

Available retention periods: 1, 3, 5, 7, 14, 30, 60, 90, 120, 150, 180, 365, 400, 545, 731, 1096, 1827, 2192, 2557, 2922, 3288, 3653 days ÔÇö or never expire.

### Structured Logging (JSON)

Plain text logs are hard to search and filter. Use JSON-formatted (structured) logs instead:

```python
import json
import logging

logger = logging.getLogger()
logger.setLevel(logging.INFO)

def handler(event, context):
    # Structured log entry
    logger.info(json.dumps({
        "action": "process_order",
        "order_id": "ORD-12345",
        "customer_id": "CUST-789",
        "amount": 49.99,
        "status": "success"
    }))
    return {"statusCode": 200}
```

With structured logs, you can use CloudWatch Logs Insights to query specific fields:

```
fields @timestamp, action, order_id, amount
| filter status = "success"
| sort @timestamp desc
```

## Viewing Metrics in the Console

1. Open the **CloudWatch** console
2. Click **Metrics** Ôćĺ **All metrics** in the left sidebar
3. Select a namespace (e.g., `AWS/Lambda`)
4. Choose a dimension (e.g., `FunctionName`)
5. Check the metric you want to graph (e.g., `Duration`)

You can overlay multiple metrics, change the time range, and switch between statistics (Average, Sum, Max).

## Pushing Custom Metrics

When built-in metrics aren't enough, publish your own with `PutMetricData`:

```bash
# Publish a custom metric via CLI
aws cloudwatch put-metric-data \
  --namespace "MyApp/Orders" \
  --metric-name "OrdersProcessed" \
  --value 1 \
  --unit Count \
  --dimensions Environment=production,Service=checkout
```

From a Lambda function (Python):

```python
import boto3
from datetime import datetime

cloudwatch = boto3.client("cloudwatch")

def handler(event, context):
    # Your business logic here
    orders_processed = process_orders(event)

    # Publish custom metric
    cloudwatch.put_metric_data(
        Namespace="MyApp/Orders",
        MetricData=[{
            "MetricName": "OrdersProcessed",
            "Value": orders_processed,
            "Unit": "Count",
            "Timestamp": datetime.utcnow(),
            "Dimensions": [
                {"Name": "Environment", "Value": "production"}
            ]
        }]
    )
```

Custom metrics cost $0.30 per metric per month (first 10,000 metrics). The free tier includes **10 custom metrics**.

## Hands-On: Explore EC2 and Lambda Metrics

**Exercise 1 ÔÇö View Lambda metrics:**

```bash
# List available metrics for a Lambda function
aws cloudwatch list-metrics \
  --namespace "AWS/Lambda" \
  --dimensions Name=FunctionName,Value=my-function

# Get average duration over the last hour (5-minute periods)
aws cloudwatch get-metric-statistics \
  --namespace "AWS/Lambda" \
  --metric-name "Duration" \
  --dimensions Name=FunctionName,Value=my-function \
  --start-time $(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%S) \
  --end-time $(date -u +%Y-%m-%dT%H:%M:%S) \
  --period 300 \
  --statistics Average Maximum
```

**Exercise 2 ÔÇö View recent logs:**

```bash
# List log groups
aws logs describe-log-groups --log-group-name-prefix "/aws/lambda/"

# Get the most recent log events
aws logs get-log-events \
  --log-group-name "/aws/lambda/my-function" \
  --log-stream-name "2024/01/15/[$LATEST]abc123" \
  --limit 20
```

**Exercise 3 ÔÇö Tail logs in real time:**

```bash
# Follow logs live (great for debugging)
aws logs tail "/aws/lambda/my-function" --follow
```

## Key Takeaways

- CloudWatch is the built-in monitoring service ÔÇö metrics and logs in one place
- Metrics are organized by namespace, named by metric name, and filtered by dimensions
- Most AWS services emit metrics automatically; you publish custom metrics with `PutMetricData`
- Lambda sends logs automatically; EC2 requires the CloudWatch Agent
- Always set a retention policy on log groups to control costs
- Use structured (JSON) logging so you can query logs effectively later
- **Free tier:** 10 custom metrics, 5 GB log data ingestion, 5 GB log data archived, 3 dashboards with up to 50 metrics each
