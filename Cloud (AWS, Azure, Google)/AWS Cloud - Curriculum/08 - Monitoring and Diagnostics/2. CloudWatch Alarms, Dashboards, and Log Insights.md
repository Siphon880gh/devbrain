# CloudWatch Alarms, Dashboards, and Log Insights

Set up alarms that notify you when things go wrong, build dashboards for at-a-glance health checks, and query your logs with CloudWatch Logs Insights.

## What You'll Learn

- How CloudWatch Alarms monitor metrics and trigger actions
- The three alarm states: OK, ALARM, and INSUFFICIENT_DATA
- How to create alarms via CLI and connect them to SNS notifications
- How to build dashboards with different widget types
- How to write Log Insights queries to analyze your log data
- Practical query patterns for debugging and monitoring

## Alarms — Thresholds, States, and Actions

An alarm watches a single metric over time and takes action when that metric crosses a threshold you define.

**How it works:**

1. You pick a metric (e.g., `AWS/Lambda` → `Errors`)
2. You set a threshold (e.g., "greater than 5")
3. You set an evaluation period (e.g., "for 3 consecutive 1-minute periods")
4. You define what happens when the threshold is breached (e.g., send an email)

### Alarm States

Every alarm is always in one of three states:

| State | Meaning |
|-------|---------|
| **OK** | The metric is within the defined threshold |
| **ALARM** | The metric has breached the threshold |
| **INSUFFICIENT_DATA** | Not enough data points to determine state (common right after creation) |

An alarm transitions between states. You can trigger actions on any transition — not just when entering ALARM, but also when returning to OK (useful for "all clear" notifications).

### Alarm Actions

When an alarm changes state, it can:

- **Send an SNS notification** — email, SMS, or trigger a Lambda function
- **Auto Scale** — add or remove EC2 instances
- **EC2 actions** — stop, terminate, reboot, or recover an instance

## Creating a Metric Alarm

### Via the CLI

Create an alarm that fires when a Lambda function has more than 3 errors in a 5-minute period:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name "lambda-high-errors" \
  --alarm-description "Fires when my-function has more than 3 errors" \
  --namespace "AWS/Lambda" \
  --metric-name "Errors" \
  --dimensions Name=FunctionName,Value=my-function \
  --statistic Sum \
  --period 300 \
  --threshold 3 \
  --comparison-operator GreaterThanThreshold \
  --evaluation-periods 1 \
  --alarm-actions "arn:aws:sns:us-east-1:123456789012:my-alerts" \
  --ok-actions "arn:aws:sns:us-east-1:123456789012:my-alerts"
```

Breaking down the key parameters:

- `--period 300` — aggregate data in 5-minute windows
- `--statistic Sum` — add up all errors in the period
- `--threshold 3` — fire if the sum exceeds 3
- `--evaluation-periods 1` — fire after just one period breaches (use 3 for less sensitivity)
- `--alarm-actions` — what to do when entering ALARM state
- `--ok-actions` — what to do when returning to OK state

### Create an alarm for EC2 CPU usage:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name "high-cpu-web-server" \
  --namespace "AWS/EC2" \
  --metric-name "CPUUtilization" \
  --dimensions Name=InstanceId,Value=i-0abc123def456 \
  --statistic Average \
  --period 300 \
  --threshold 80 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --evaluation-periods 3 \
  --alarm-actions "arn:aws:sns:us-east-1:123456789012:ops-team"
```

This alarm only fires if CPU stays above 80% for three consecutive 5-minute periods (15 minutes total) — reducing false alarms from temporary spikes.

### Check alarm status:

```bash
# List all alarms
aws cloudwatch describe-alarms --state-value ALARM

# Check a specific alarm
aws cloudwatch describe-alarms --alarm-names "lambda-high-errors"
```

## SNS Integration for Notifications

Alarms on their own just change state. To actually get notified, you need an **SNS topic** connected to the alarm:

```bash
# Create an SNS topic
aws sns create-topic --name my-alerts
# Returns: arn:aws:sns:us-east-1:123456789012:my-alerts

# Subscribe your email to the topic
aws sns subscribe \
  --topic-arn "arn:aws:sns:us-east-1:123456789012:my-alerts" \
  --protocol email \
  --notification-endpoint "you@example.com"
```

After subscribing, you'll receive a confirmation email — you must click the link to activate the subscription.

**Other subscription types:**
- `sms` — text messages (note: costs per message)
- `lambda` — invoke a Lambda function (great for Slack/Teams integration)
- `https` — POST to a webhook URL

### Auto Scaling Actions

You can also trigger Auto Scaling policies from alarms:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name "scale-out-alarm" \
  --namespace "AWS/EC2" \
  --metric-name "CPUUtilization" \
  --dimensions Name=AutoScalingGroupName,Value=my-asg \
  --statistic Average \
  --period 300 \
  --threshold 70 \
  --comparison-operator GreaterThanThreshold \
  --evaluation-periods 2 \
  --alarm-actions "arn:aws:autoscaling:us-east-1:123456789012:scalingPolicy:abc-123:autoScalingGroupName/my-asg:policyName/scale-out"
```

## Dashboards — Building a Single Pane of Glass

A CloudWatch dashboard is a customizable page of widgets that display metrics, logs, and text. Use them to get a real-time view of your system's health without clicking through individual services.

### Widget Types

| Widget | Purpose |
|--------|---------|
| **Line** | Time-series graph showing metric trends |
| **Stacked area** | Like line but filled — good for seeing composition |
| **Number** | Shows the current value of a metric (big and bold) |
| **Gauge** | Circular meter with color-coded ranges |
| **Bar** | Horizontal bars comparing values |
| **Text** | Markdown content for notes, links, or runbook references |
| **Log** | Displays results of a Logs Insights query |
| **Alarm status** | Shows alarm states (green/red/gray) |

### Creating a Dashboard via CLI

```bash
# Create a dashboard with a CPU utilization graph
aws cloudwatch put-dashboard \
  --dashboard-name "Production-Overview" \
  --dashboard-body '{
    "widgets": [
      {
        "type": "metric",
        "x": 0, "y": 0, "width": 12, "height": 6,
        "properties": {
          "title": "Lambda Errors",
          "metrics": [
            ["AWS/Lambda", "Errors", "FunctionName", "my-function", {"stat": "Sum"}]
          ],
          "period": 300,
          "region": "us-east-1"
        }
      },
      {
        "type": "metric",
        "x": 12, "y": 0, "width": 12, "height": 6,
        "properties": {
          "title": "Lambda Duration (avg)",
          "metrics": [
            ["AWS/Lambda", "Duration", "FunctionName", "my-function", {"stat": "Average"}]
          ],
          "period": 300,
          "region": "us-east-1",
          "view": "singleValue"
        }
      },
      {
        "type": "text",
        "x": 0, "y": 6, "width": 24, "height": 2,
        "properties": {
          "markdown": "## Runbook\n- If errors spike, check [CloudWatch Logs](/cloudwatch/home#logsV2:log-groups)\n- Escalation: #ops-team in Slack"
        }
      }
    ]
  }'
```

**Practical tip:** Create one dashboard per environment (production, staging) or per team. Place alarm status widgets at the top so you immediately see red/green.

## CloudWatch Logs Insights — Query Language Basics

Logs Insights lets you run SQL-like queries against your log data. It's the fastest way to search, filter, and analyze logs without exporting them.

### Query Syntax

A basic query follows this structure:

```
fields @timestamp, @message
| filter <condition>
| stats <aggregation> by <grouping>
| sort <field> desc
| limit <number>
```

Key built-in fields:
- `@timestamp` — when the event was logged
- `@message` — the full log line
- `@logStream` — which stream it came from
- `@log` — the log group

### Basic Filtering

```
# Find all error messages
fields @timestamp, @message
| filter @message like /ERROR/
| sort @timestamp desc
| limit 50
```

```
# Filter by structured JSON field
fields @timestamp, action, order_id, status
| filter status = "failed"
| sort @timestamp desc
```

### Aggregation (Stats)

```
# Count errors per hour
fields @timestamp, @message
| filter @message like /ERROR/
| stats count(*) as error_count by bin(1h)
```

```
# Average Lambda duration by function
fields @timestamp, @duration
| stats avg(@duration) as avg_ms, max(@duration) as max_ms by @logStream
```

## Writing Useful Queries

Here are practical queries you'll use repeatedly:

### Count errors over time

```
fields @timestamp, @message
| filter @message like /ERROR/
| stats count(*) as errors by bin(5m)
| sort @timestamp desc
```

### Find the top 10 error messages

```
fields @timestamp, @message
| filter @message like /ERROR/
| stats count(*) as occurrences by @message
| sort occurrences desc
| limit 10
```

### Latency percentiles for an API

```
fields @timestamp, @duration
| stats
    avg(@duration) as avg_ms,
    pct(@duration, 50) as p50,
    pct(@duration, 90) as p90,
    pct(@duration, 99) as p99,
    max(@duration) as max_ms
  by bin(5m)
```

### Cold starts in Lambda

```
fields @timestamp, @message, @duration, @initDuration
| filter ispresent(@initDuration)
| stats count(*) as cold_starts, avg(@initDuration) as avg_init_ms by bin(1h)
```

### Find specific request IDs

```
fields @timestamp, @message, @requestId
| filter @requestId = "abc123-def456-ghi789"
| sort @timestamp asc
```

### Memory usage near limit

```
fields @timestamp, @maxMemoryUsed, @memorySize
| filter @maxMemoryUsed / @memorySize > 0.8
| sort @timestamp desc
| limit 20
```

## Hands-On: Alarm + Dashboard for a Lambda Function

**Step 1 — Create an SNS topic and subscribe:**

```bash
aws sns create-topic --name lambda-alerts
aws sns subscribe \
  --topic-arn "arn:aws:sns:us-east-1:123456789012:lambda-alerts" \
  --protocol email \
  --notification-endpoint "you@example.com"
```

**Step 2 — Create an error alarm:**

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name "my-function-errors" \
  --namespace "AWS/Lambda" \
  --metric-name "Errors" \
  --dimensions Name=FunctionName,Value=my-function \
  --statistic Sum \
  --period 60 \
  --threshold 1 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --evaluation-periods 1 \
  --alarm-actions "arn:aws:sns:us-east-1:123456789012:lambda-alerts"
```

**Step 3 — Create a duration alarm (slow responses):**

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name "my-function-slow" \
  --namespace "AWS/Lambda" \
  --metric-name "Duration" \
  --dimensions Name=FunctionName,Value=my-function \
  --statistic Average \
  --period 300 \
  --threshold 5000 \
  --comparison-operator GreaterThanThreshold \
  --evaluation-periods 2 \
  --alarm-actions "arn:aws:sns:us-east-1:123456789012:lambda-alerts"
```

**Step 4 — Verify your alarms:**

```bash
aws cloudwatch describe-alarms \
  --alarm-name-prefix "my-function"
```

Then build a dashboard in the console: CloudWatch → Dashboards → Create → add Line widgets for Duration and Errors, add a Number widget for Invocations, and an Alarm Status widget showing both alarms.

## Key Takeaways

- Alarms watch a metric and change state (OK → ALARM → INSUFFICIENT_DATA) when thresholds are breached
- Use `evaluation-periods` > 1 to avoid false alarms from momentary spikes
- Connect alarms to SNS topics to get email/SMS/webhook notifications
- Dashboards give you a single page view of your system health — use them daily
- Logs Insights queries are powerful and fast; learn `filter`, `stats`, `pct()`, and `bin()`
- Use structured JSON logs to make Logs Insights queries much more effective
- **Free tier:** 10 alarms, 3 dashboards (50 metrics each)
