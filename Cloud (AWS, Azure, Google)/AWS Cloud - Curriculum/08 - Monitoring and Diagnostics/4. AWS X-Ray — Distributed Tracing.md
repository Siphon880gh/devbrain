# AWS X-Ray — Distributed Tracing for Serverless and Microservices

Trace requests as they flow through Lambda functions, API Gateway, DynamoDB, and other services to pinpoint latency bottlenecks and errors.

## What You'll Learn

- What distributed tracing is and why you need it
- X-Ray core concepts: traces, segments, subsegments, annotations, and metadata
- How to enable tracing on Lambda and API Gateway (one-click)
- How to instrument Node.js and Python code with the X-Ray SDK
- How to read the service map and identify bottlenecks
- How to analyze latency, errors, and throttling
- X-Ray Groups and Insights for proactive alerting

## Why Distributed Tracing Matters

In a monolithic application, when a request is slow, you check one log file. In a distributed system — where a single user request might hit API Gateway, then Lambda, then DynamoDB, then another Lambda, then SNS — a slow request could be caused by any one of those services.

**The problem:** CloudWatch tells you *that* something is slow. But when your system has 5+ services, you need to know *where* in the chain the slowness happens.

**Distributed tracing** solves this by assigning a unique trace ID to each request and tracking that ID as it flows through every service. You get a timeline showing exactly how long each service took.

```
User Request (total: 850ms)
├── API Gateway (23ms)
├── Lambda: auth-function (120ms)
│   └── DynamoDB: GetItem (45ms)
├── Lambda: process-order (680ms)
│   ├── DynamoDB: PutItem (30ms)
│   └── SNS: Publish (25ms)
└── API Gateway response (27ms)
```

Now you can see that `process-order` is the bottleneck, and most of its time isn't spent in DynamoDB or SNS — it's in the function's own code.

## X-Ray Concepts — Traces, Segments, and Subsegments

### Trace

A **trace** represents a single request from start to finish. It collects all the segments and subsegments generated as the request moves through your system. Each trace has a unique **Trace ID** like:

```
1-65a5f123-abcdef1234567890abcdef12
```

The format is: `1-{unix epoch in hex}-{96-bit random hex}`

### Segment

A **segment** represents the work done by a single service. When API Gateway handles a request, it creates a segment. When Lambda executes your function, it creates a segment.

Each segment records:
- Service name
- Start time and end time
- Whether it succeeded or errored
- HTTP request/response details

### Subsegment

A **subsegment** is a more granular unit within a segment. When your Lambda function calls DynamoDB, that DynamoDB call is recorded as a subsegment within the Lambda segment.

You can also create custom subsegments to measure specific parts of your code:

```python
from aws_xray_sdk.core import xray_recorder

@xray_recorder.capture("validate_input")
def validate_input(data):
    # This function's execution time is tracked as a subsegment
    ...
```

### Annotations

Key-value pairs that are **indexed** and searchable. Use them to filter traces:

```python
from aws_xray_sdk.core import xray_recorder

segment = xray_recorder.current_segment()
segment.put_annotation("customer_tier", "premium")
segment.put_annotation("order_id", "ORD-12345")
```

You can then search: "Show me all traces where `customer_tier = premium` and response was slow."

### Metadata

Key-value pairs that are **not indexed**. Use for detailed debugging data you don't need to search by:

```python
segment.put_metadata("request_body", event["body"])
segment.put_metadata("item_count", len(items))
```

## Enabling X-Ray on Lambda and API Gateway

### Lambda — One Setting

Enabling X-Ray on Lambda requires no code changes:

```bash
# Enable active tracing on an existing function
aws lambda update-function-configuration \
  --function-name my-function \
  --tracing-config Mode=Active
```

Or in your SAM/CloudFormation template:

```yaml
MyFunction:
  Type: AWS::Serverless::Function
  Properties:
    FunctionName: my-function
    Runtime: python3.12
    Handler: app.handler
    Tracing: Active
```

With Active tracing enabled, Lambda automatically:
- Creates a segment for each invocation
- Records duration, cold start, errors
- Propagates the trace ID to downstream AWS SDK calls

### API Gateway

Enable tracing on your API stage:

```bash
aws apigateway update-stage \
  --rest-api-id abc123 \
  --stage-name prod \
  --patch-operations op=replace,path=/tracingEnabled,value=true
```

API Gateway will create its own segment and pass the trace ID to your Lambda integration.

## The X-Ray Service Map

The service map is a visual diagram that X-Ray generates automatically. It shows:

- Every service involved in your traces (as nodes)
- The connections between services (as edges)
- Average latency for each service
- Error rates (nodes turn yellow for 4xx, red for 5xx)
- Request volume (node size scales with traffic)

To view it: **X-Ray console** → **Service map**

The service map answers questions like:
- Which service is the bottleneck?
- Where are errors occurring?
- What's the dependency chain?
- Is a downstream service affecting my latency?

## Analyzing Traces in the Console

### Trace List

The trace list shows individual requests with their total duration and status. You can filter by:
- Time range
- Status code (2xx, 4xx, 5xx)
- Duration (e.g., > 3 seconds)
- Annotation values
- URL path

### Trace Detail

Click a trace to see its waterfall timeline:

```
[====== API Gateway (850ms) ======]
  [=== Lambda: auth (120ms) ===]
    [= DynamoDB GetItem (45ms) =]
  [======= Lambda: process (680ms) =======]
    [= DynamoDB PutItem (30ms) =]
    [= SNS Publish (25ms) =]
```

Each bar shows:
- Green = successful
- Yellow = client error (4xx)
- Red = server error (5xx) or fault
- Purple = throttled (429)

## Instrumenting Node.js with the X-Ray SDK

Install the SDK:

```bash
npm install aws-xray-sdk
```

Wrap the AWS SDK to automatically trace all AWS service calls:

```javascript
const AWSXRay = require("aws-xray-sdk");
const AWS = AWSXRay.captureAWS(require("aws-sdk"));

const dynamodb = new AWS.DynamoDB.DocumentClient();

exports.handler = async (event) => {
  // This DynamoDB call is automatically traced as a subsegment
  const result = await dynamodb.get({
    TableName: "Orders",
    Key: { orderId: event.pathParameters.id }
  }).promise();

  return {
    statusCode: 200,
    body: JSON.stringify(result.Item)
  };
};
```

### Custom Subsegments

Measure specific operations:

```javascript
const AWSXRay = require("aws-xray-sdk");

exports.handler = async (event) => {
  // Create a custom subsegment
  const subsegment = AWSXRay.getSegment().addNewSubsegment("process_payment");
  
  try {
    const result = await processPayment(event.body);
    subsegment.addAnnotation("payment_method", "credit_card");
    subsegment.addMetadata("response", result);
    subsegment.close();
    return { statusCode: 200, body: JSON.stringify(result) };
  } catch (err) {
    subsegment.addError(err);
    subsegment.close();
    throw err;
  }
};
```

## Instrumenting Python with the X-Ray SDK

Install the SDK:

```bash
pip install aws-xray-sdk
```

Patch the AWS SDK (boto3) for automatic tracing:

```python
from aws_xray_sdk.core import xray_recorder, patch_all

# Patch all supported libraries (boto3, requests, sqlite3, etc.)
patch_all()

import boto3

dynamodb = boto3.resource("dynamodb")
table = dynamodb.Table("Orders")

def handler(event, context):
    # This DynamoDB call is automatically traced
    response = table.get_item(Key={"orderId": event["pathParameters"]["id"]})
    
    return {
        "statusCode": 200,
        "body": str(response["Item"])
    }
```

### Custom Subsegments in Python

```python
from aws_xray_sdk.core import xray_recorder

def handler(event, context):
    # Decorator approach
    result = validate_and_process(event)
    return {"statusCode": 200, "body": result}

@xray_recorder.capture("validate_and_process")
def validate_and_process(event):
    # Manual subsegment within
    with xray_recorder.in_subsegment("validation") as subseg:
        validated = validate(event["body"])
        subseg.put_annotation("is_valid", True)

    with xray_recorder.in_subsegment("processing") as subseg:
        result = process(validated)
        subseg.put_annotation("items_processed", len(result))

    return result
```

### Tracing HTTP Calls

The SDK can also trace outbound HTTP requests:

```python
from aws_xray_sdk.core import xray_recorder, patch
patch(["requests"])

import requests

def handler(event, context):
    # This external HTTP call is traced as a subsegment
    response = requests.get("https://api.example.com/data")
    return {"statusCode": 200, "body": response.text}
```

## X-Ray with API Gateway + Lambda + DynamoDB

Here's a complete traced serverless architecture:

```
Client → API Gateway → Lambda → DynamoDB
                                    ↓
                                   SNS
```

When tracing is enabled on all components, a single request generates:

1. **API Gateway segment** — total request time, HTTP method, status
2. **Lambda segment** — function duration, cold start, memory used
3. **DynamoDB subsegment** — table name, operation, items returned, latency
4. **SNS subsegment** — topic, publish time

All linked by the same Trace ID, giving you end-to-end visibility.

### SAM Template with Full Tracing

```yaml
Globals:
  Function:
    Tracing: Active
  Api:
    TracingEnabled: true

Resources:
  OrderApi:
    Type: AWS::Serverless::Api
    Properties:
      StageName: prod

  GetOrderFunction:
    Type: AWS::Serverless::Function
    Properties:
      Runtime: python3.12
      Handler: get_order.handler
      Policies:
        - DynamoDBReadPolicy:
            TableName: !Ref OrdersTable
        - Version: "2012-10-17"
          Statement:
            - Effect: Allow
              Action: xray:PutTraceSegments
              Resource: "*"
      Events:
        GetOrder:
          Type: Api
          Properties:
            RestApiId: !Ref OrderApi
            Path: /orders/{id}
            Method: GET

  OrdersTable:
    Type: AWS::DynamoDB::Table
    Properties:
      TableName: Orders
      AttributeDefinitions:
        - AttributeName: orderId
          AttributeType: S
      KeySchema:
        - AttributeName: orderId
          KeyType: HASH
      BillingMode: PAY_PER_REQUEST
```

## X-Ray Groups and Insights

### Groups

Groups let you define a filter expression to segment your traces. For example, create a group for slow requests only:

```bash
aws xray create-group \
  --group-name "SlowRequests" \
  --filter-expression "responsetime > 3"
```

Other useful filter expressions:

```
# Only errors
fault = true

# Specific service with high latency
service("payment-api") { responsetime > 2 }

# Specific annotation value
annotation.customer_tier = "premium"

# Errors on a specific URL
http.url CONTAINS "/checkout" AND fault = true
```

### Insights

X-Ray Insights uses machine learning to automatically detect anomalies in your traces:
- Sudden latency spikes
- Elevated error rates
- A service that starts timing out

Enable Insights on a group:

```bash
aws xray update-group \
  --group-name "SlowRequests" \
  --insights-configuration InsightsEnabled=true,NotificationsEnabled=true
```

When an anomaly is detected, X-Ray can send a notification via EventBridge, which you can route to SNS, Slack, or PagerDuty.

## X-Ray SDK vs. AWS Distro for OpenTelemetry

AWS now also supports **OpenTelemetry** (an open-source standard) as an alternative to the X-Ray SDK:

| | X-Ray SDK | AWS Distro for OpenTelemetry (ADOT) |
|---|---|---|
| Vendor lock-in | AWS only | Open standard, works with multiple backends |
| Setup | Simpler for AWS-only | More configuration, but more flexible |
| When to use | Pure AWS, quick start | Multi-cloud, already using OpenTelemetry |

For beginners and AWS-focused projects, the X-Ray SDK is simpler. If your organization uses multiple cloud providers or wants portability, consider ADOT.

## Hands-On: Trace a Serverless API End-to-End

**Step 1 — Enable tracing on your Lambda:**

```bash
aws lambda update-function-configuration \
  --function-name my-api-handler \
  --tracing-config Mode=Active
```

**Step 2 — Add the X-Ray SDK to your function (Python):**

```python
from aws_xray_sdk.core import patch_all
patch_all()

import boto3

dynamodb = boto3.resource("dynamodb")
table = dynamodb.Table("MyTable")

def handler(event, context):
    item_id = event["pathParameters"]["id"]
    response = table.get_item(Key={"id": item_id})
    
    if "Item" not in response:
        return {"statusCode": 404, "body": "Not found"}
    
    return {"statusCode": 200, "body": str(response["Item"])}
```

**Step 3 — Invoke the function a few times:**

```bash
# Invoke directly
aws lambda invoke \
  --function-name my-api-handler \
  --payload '{"pathParameters": {"id": "123"}}' \
  output.json

# Or hit the API Gateway endpoint
curl https://abc123.execute-api.us-east-1.amazonaws.com/prod/items/123
```

**Step 4 — View traces in the console:**

1. Open the **X-Ray** console
2. Click **Traces** in the left sidebar
3. Set the time range to the last 5 minutes
4. Click on any trace to see the waterfall timeline
5. Click **Service map** to see the visual dependency graph

**Step 5 — Filter traces:**

In the filter bar, try:
- `responsetime > 1` — find slow requests
- `fault = true` — find errors
- `service("MyTable")` — find traces that hit DynamoDB

## Key Takeaways

- Distributed tracing tracks a request across multiple services using a single Trace ID
- X-Ray uses traces (full request), segments (per service), and subsegments (per operation)
- Enable Active Tracing on Lambda with one CLI command — no code changes needed
- Use the X-Ray SDK to automatically trace AWS SDK calls and add custom subsegments
- Annotations are indexed and searchable; metadata is for debugging details
- The service map gives you an instant visual of your architecture's health
- Use Groups to filter traces and Insights for anomaly detection
- **Free tier:** 100,000 traces recorded and 1,000,000 traces scanned per month
