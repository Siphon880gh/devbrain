# CloudFormation

CloudFormation is AWS's native Infrastructure as Code (IaC) service. Instead of manually clicking through the Console to create resources, you describe your entire infrastructure in a template file (YAML or JSON), and CloudFormation builds it all for you — consistently, repeatably, and with the ability to tear it all down in one command.

## What You'll Learn

- Why Infrastructure as Code matters
- How to read and write CloudFormation templates
- How to use Parameters, Resources, and Outputs
- How to create, update, and delete stacks with the CLI
- Intrinsic functions like `!Ref`, `!Sub`, and `!GetAtt`
- How change sets and drift detection keep your infrastructure safe

## Why Infrastructure as Code

Imagine you need to set up an environment with an S3 bucket, a Lambda function, an IAM role, and an API Gateway. Doing this manually in the Console:

- Takes 30+ minutes of clicking
- Is different every time (you'll forget a setting)
- Can't be reviewed in a pull request
- Can't be duplicated across staging and production

With Infrastructure as Code, you write a template file that describes all of those resources. Then you run one command. Five minutes later, everything exists — exactly the same every time. You can version-control it, review it, duplicate it across environments, and delete the entire thing cleanly.

CloudFormation is AWS's built-in IaC tool. You write a template, submit it to CloudFormation, and it creates a **stack** — a collection of resources that are managed as a single unit.

## Template Anatomy (Resources, Parameters, Outputs)

A CloudFormation template has several sections. Only `Resources` is required, but most real templates use all of these:

```yaml
AWSTemplateFormatVersion: "2010-09-09"
Description: "My first CloudFormation template"

Parameters:
  EnvironmentName:
    Type: String
    Default: dev
    AllowedValues:
      - dev
      - staging
      - prod
    Description: "The deployment environment"

  BucketPrefix:
    Type: String
    Default: myapp
    Description: "Prefix for the S3 bucket name"

Resources:
  AppBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: !Sub "${BucketPrefix}-${EnvironmentName}-assets"
      VersioningConfiguration:
        Status: Enabled

  ProcessorRole:
    Type: AWS::IAM::Role
    Properties:
      RoleName: !Sub "${EnvironmentName}-processor-role"
      AssumeRolePolicyDocument:
        Version: "2012-10-17"
        Statement:
          - Effect: Allow
            Principal:
              Service: lambda.amazonaws.com
            Action: sts:AssumeRole
      ManagedPolicyArns:
        - arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole

  ProcessorFunction:
    Type: AWS::Lambda::Function
    Properties:
      FunctionName: !Sub "${EnvironmentName}-processor"
      Runtime: python3.12
      Handler: index.handler
      Role: !GetAtt ProcessorRole.Arn
      Code:
        ZipFile: |
          import json
          def handler(event, context):
              print("Processing event:", json.dumps(event))
              return {"statusCode": 200, "body": "Processed"}

Outputs:
  BucketName:
    Description: "Name of the created S3 bucket"
    Value: !Ref AppBucket
    Export:
      Name: !Sub "${EnvironmentName}-bucket-name"

  FunctionArn:
    Description: "ARN of the Lambda function"
    Value: !GetAtt ProcessorFunction.Arn

  RoleArn:
    Description: "ARN of the IAM role"
    Value: !GetAtt ProcessorRole.Arn
```

### Section Breakdown

**AWSTemplateFormatVersion** — Always `"2010-09-09"`. This is the only valid version and hasn't changed since CloudFormation launched.

**Description** — A human-readable description of what this template does. Shows up in the Console.

**Parameters** — Input values that make your template reusable. When you create a stack, CloudFormation prompts you for these values (or uses the defaults). Parameters support types like `String`, `Number`, `AWS::EC2::KeyPair::KeyName`, and more.

**Resources** — The only required section. Each resource has a logical name (like `AppBucket`), a `Type` (the AWS resource type), and `Properties` specific to that resource type.

**Outputs** — Values you want to see after the stack is created. Outputs can also be exported for use by other stacks.

## Intrinsic Functions

CloudFormation provides built-in functions for dynamic values:

**!Ref** — Returns the value of a parameter or the physical ID of a resource:

```yaml
# References the EnvironmentName parameter
BucketName: !Ref EnvironmentName

# References an S3 bucket (returns the bucket name)
BucketName: !Ref AppBucket
```

**!Sub** — String substitution. Inserts variable values into a string:

```yaml
BucketName: !Sub "${BucketPrefix}-${EnvironmentName}-assets"
```

**!GetAtt** — Gets an attribute from a resource. Different resources expose different attributes:

```yaml
# Get the ARN of a Lambda function
FunctionArn: !GetAtt ProcessorFunction.Arn

# Get the domain name of an S3 bucket
BucketDomain: !GetAtt AppBucket.DomainName
```

**Fn::Join** — Joins a list of strings with a delimiter:

```yaml
BucketName: !Join
  - "-"
  - - !Ref BucketPrefix
    - !Ref EnvironmentName
    - "assets"
```

**!Select** — Picks an item from a list by index:

```yaml
Zone: !Select [0, !GetAZs ""]
```

**Fn::If** — Conditional values based on a condition defined in the `Conditions` section:

```yaml
Conditions:
  IsProd: !Equals [!Ref EnvironmentName, prod]

Resources:
  AppBucket:
    Type: AWS::S3::Bucket
    Properties:
      VersioningConfiguration:
        Status: !If [IsProd, Enabled, Suspended]
```

## Stacks and Change Sets

### Stacks

A **stack** is a running instance of a template. When you deploy a template, CloudFormation creates a stack. All the resources in that stack are managed together — if you delete the stack, all its resources are deleted.

### Creating a Stack (CLI)

```bash
aws cloudformation create-stack \
  --stack-name my-app-dev \
  --template-body file://template.yaml \
  --parameters ParameterKey=EnvironmentName,ParameterValue=dev \
  --capabilities CAPABILITY_NAMED_IAM
```

The `--capabilities CAPABILITY_NAMED_IAM` flag is required when your template creates IAM resources with custom names. This is a safety measure.

Wait for the stack to finish creating:

```bash
aws cloudformation wait stack-create-complete --stack-name my-app-dev
```

### Updating a Stack

Edit your template, then update:

```bash
aws cloudformation update-stack \
  --stack-name my-app-dev \
  --template-body file://template.yaml \
  --parameters ParameterKey=EnvironmentName,ParameterValue=dev \
  --capabilities CAPABILITY_NAMED_IAM
```

### Change Sets

A **change set** lets you preview what an update will do before it happens. This is critical in production — you don't want surprises.

```bash
# Create a change set
aws cloudformation create-change-set \
  --stack-name my-app-dev \
  --change-set-name add-logging \
  --template-body file://template-v2.yaml \
  --capabilities CAPABILITY_NAMED_IAM

# Review what will change
aws cloudformation describe-change-set \
  --stack-name my-app-dev \
  --change-set-name add-logging

# Execute if everything looks good
aws cloudformation execute-change-set \
  --stack-name my-app-dev \
  --change-set-name add-logging
```

### Checking Stack Status

```bash
aws cloudformation describe-stacks --stack-name my-app-dev
```

Common statuses:
- `CREATE_COMPLETE` — everything worked
- `CREATE_IN_PROGRESS` — still building
- `ROLLBACK_COMPLETE` — creation failed, resources were cleaned up
- `UPDATE_ROLLBACK_COMPLETE` — an update failed and was rolled back

### Viewing Stack Events

When something fails, check the events to find the error:

```bash
aws cloudformation describe-stack-events \
  --stack-name my-app-dev \
  --query "StackEvents[?ResourceStatus=='CREATE_FAILED'].[LogicalResourceId, ResourceStatusReason]" \
  --output table
```

### Deleting a Stack

```bash
aws cloudformation delete-stack --stack-name my-app-dev
```

This removes all resources that the stack created.

## Drift Detection

Over time, someone might manually change a resource that CloudFormation created (for example, editing a security group in the Console). This creates **drift** — a mismatch between what CloudFormation thinks exists and what actually exists.

Detect drift on a stack:

```bash
aws cloudformation detect-stack-drift --stack-name my-app-dev
```

Check the results:

```bash
aws cloudformation describe-stack-drift-detection-status \
  --stack-drift-detection-id <detection-id>
```

View which resources drifted:

```bash
aws cloudformation describe-stack-resource-drifts \
  --stack-name my-app-dev \
  --stack-resource-drift-status-filters MODIFIED DELETED
```

## Nested Stacks

As your infrastructure grows, a single template becomes hard to manage. Nested stacks let you break your infrastructure into smaller, reusable templates.

A parent template references child templates stored in S3:

```yaml
Resources:
  NetworkStack:
    Type: AWS::CloudFormation::Stack
    Properties:
      TemplateURL: https://s3.amazonaws.com/my-templates/network.yaml
      Parameters:
        EnvironmentName: !Ref EnvironmentName

  AppStack:
    Type: AWS::CloudFormation::Stack
    Properties:
      TemplateURL: https://s3.amazonaws.com/my-templates/app.yaml
      Parameters:
        VpcId: !GetAtt NetworkStack.Outputs.VpcId
        EnvironmentName: !Ref EnvironmentName
```

Each child template is a self-contained CloudFormation template with its own Resources and Outputs. The parent passes parameters down and reads outputs back up.

## Key Takeaways

- CloudFormation lets you define AWS infrastructure in YAML/JSON templates
- A **stack** is a deployed instance of a template — create, update, or delete everything at once
- Use **Parameters** to make templates reusable across environments
- Use **Outputs** to expose important values like ARNs and endpoint URLs
- **Change sets** let you preview updates before applying them
- **Drift detection** catches manual changes that don't match your template
- The `--capabilities CAPABILITY_NAMED_IAM` flag is required for templates that create IAM resources
- **Nested stacks** break large templates into manageable, reusable pieces
