# CodePipeline, CodeBuild, and CodeDeploy

AWS provides a full CI/CD (Continuous Integration / Continuous Deployment) suite built into the platform. CodePipeline orchestrates the workflow, CodeBuild compiles and tests your code, and CodeDeploy handles pushing updates to your servers or services. Together, they let you go from a code commit to a production deployment automatically.

## What You'll Learn

- The CI/CD pipeline concept and why it matters
- How CodePipeline orchestrates multi-stage workflows
- How CodeBuild compiles, tests, and packages your code
- How CodeDeploy pushes updates with zero-downtime strategies
- How to wire up a complete pipeline from GitHub to production

## The CI/CD Pipeline Concept

Without CI/CD, deploying looks like this:
1. Developer finishes code on their laptop
2. They manually zip it up and upload it somewhere
3. They SSH into a server and pull the new code
4. They restart the service and hope nothing breaks

With CI/CD:
1. Developer pushes code to GitHub
2. A pipeline automatically runs tests, builds the project, and deploys it
3. If anything fails, the pipeline stops and notifies the team

CI/CD gives you speed, consistency, and safety. Every deployment follows the same steps, every time.

## CodePipeline: Orchestrating Stages

CodePipeline is the orchestrator. It defines a series of **stages**, and each stage contains one or more **actions**. A pipeline might look like:

```
Source → Build → Test → Deploy
```

### Pipeline Stages

**Source Stage** — Where the code comes from. CodePipeline connects to:
- GitHub (via GitHub App connection or OAuth)
- CodeCommit (AWS's Git service)
- S3 (trigger on file upload)
- Bitbucket

**Build Stage** — Compiles the code, runs unit tests, produces artifacts. Usually runs CodeBuild.

**Deploy Stage** — Pushes the built artifacts to their destination. Can use:
- CodeDeploy (to EC2, ECS, Lambda)
- CloudFormation (deploy/update a stack)
- S3 (deploy a static website)
- Elastic Beanstalk
- ECS (direct container deployment)

### Creating a Pipeline with the CLI

```bash
aws codepipeline create-pipeline --cli-input-json file://pipeline.json
```

The `pipeline.json` defines the stages:

```json
{
  "pipeline": {
    "name": "my-app-pipeline",
    "roleArn": "arn:aws:iam::123456789012:role/codepipeline-role",
    "stages": [
      {
        "name": "Source",
        "actions": [
          {
            "name": "GitHub-Source",
            "actionTypeId": {
              "category": "Source",
              "owner": "AWS",
              "provider": "CodeStarSourceConnection",
              "version": "1"
            },
            "configuration": {
              "ConnectionArn": "arn:aws:codestar-connections:us-east-1:123456789012:connection/abc-123",
              "FullRepositoryId": "myuser/myrepo",
              "BranchName": "main"
            },
            "outputArtifacts": [{ "name": "SourceOutput" }]
          }
        ]
      },
      {
        "name": "Build",
        "actions": [
          {
            "name": "CodeBuild",
            "actionTypeId": {
              "category": "Build",
              "owner": "AWS",
              "provider": "CodeBuild",
              "version": "1"
            },
            "configuration": {
              "ProjectName": "my-app-build"
            },
            "inputArtifacts": [{ "name": "SourceOutput" }],
            "outputArtifacts": [{ "name": "BuildOutput" }]
          }
        ]
      },
      {
        "name": "Deploy",
        "actions": [
          {
            "name": "Deploy-to-S3",
            "actionTypeId": {
              "category": "Deploy",
              "owner": "AWS",
              "provider": "S3",
              "version": "1"
            },
            "configuration": {
              "BucketName": "my-app-website",
              "Extract": "true"
            },
            "inputArtifacts": [{ "name": "BuildOutput" }]
          }
        ]
      }
    ],
    "artifactStore": {
      "type": "S3",
      "location": "my-pipeline-artifacts-bucket"
    }
  }
}
```

### Useful Pipeline Commands

```bash
# List pipelines
aws codepipeline list-pipelines

# Get pipeline status
aws codepipeline get-pipeline-state --name my-app-pipeline

# Manually trigger a pipeline
aws codepipeline start-pipeline-execution --name my-app-pipeline

# View execution history
aws codepipeline list-pipeline-executions --pipeline-name my-app-pipeline
```

## CodeBuild: Building and Testing

CodeBuild is a fully managed build service. You don't need to maintain Jenkins servers or build agents — CodeBuild spins up a fresh container for each build, runs your commands, and tears it down.

### The buildspec.yml File

Every CodeBuild project needs a `buildspec.yml` in the root of your repository. This file defines what happens during the build:

```yaml
version: 0.2

env:
  variables:
    NODE_ENV: "production"
  parameter-store:
    DB_PASSWORD: "/myapp/db-password"

phases:
  install:
    runtime-versions:
      nodejs: 20
    commands:
      - echo "Installing dependencies..."
      - npm ci

  pre_build:
    commands:
      - echo "Running tests..."
      - npm test
      - echo "Running linter..."
      - npm run lint

  build:
    commands:
      - echo "Building application..."
      - npm run build

  post_build:
    commands:
      - echo "Build completed on $(date)"

artifacts:
  files:
    - "dist/**/*"
    - "package.json"
    - "appspec.yml"
  discard-paths: no

cache:
  paths:
    - "node_modules/**/*"
```

### Build Phases Explained

**install** — Install dependencies and runtime versions. CodeBuild supports Node.js, Python, Java, Go, Ruby, .NET, and more.

**pre_build** — Steps that run before the main build. Usually tests, linting, or pulling credentials.

**build** — The main build step. Compile, bundle, package — whatever your project needs.

**post_build** — Cleanup, notifications, or final packaging. Runs even if the build phase fails.

### Build Environments

CodeBuild provides managed images with pre-installed tools:

- `aws/codebuild/amazonlinux2-x86_64-standard:5.0` — Linux with most runtimes
- `aws/codebuild/amazonlinux2-aarch64-standard:3.0` — ARM-based builds

You can also use custom Docker images for specialized build environments.

### Creating a CodeBuild Project

```bash
aws codebuild create-project \
  --name my-app-build \
  --source type=GITHUB,location=https://github.com/myuser/myrepo \
  --artifacts type=S3,location=my-build-artifacts \
  --environment type=LINUX_CONTAINER,computeType=BUILD_GENERAL1_SMALL,image=aws/codebuild/amazonlinux2-x86_64-standard:5.0 \
  --service-role arn:aws:iam::123456789012:role/codebuild-role
```

### Trigger a Build Manually

```bash
aws codebuild start-build --project-name my-app-build
```

## CodeDeploy: Deployment Strategies

CodeDeploy automates deployments to EC2 instances, ECS services, or Lambda functions. It supports multiple strategies to minimize risk.

### Deployment Types

**In-place deployment** (EC2 only) — Stops the application on each instance, installs the new version, then restarts. Instances are updated one at a time or in batches.

**Blue/Green deployment** — Creates a new set of instances with the new version, shifts traffic over, then terminates the old instances. Zero downtime.

**Lambda deployments** — Gradually shifts traffic from the old function version to the new one using aliases. Options include:
- `AllAtOnce` — Instant switch
- `Linear10PercentEvery1Minute` — Gradual shift
- `Canary10Percent5Minutes` — Send 10% first, wait 5 minutes, then shift the rest

### The appspec.yml File

CodeDeploy uses `appspec.yml` to define what happens during deployment.

For EC2 deployments:

```yaml
version: 0.0
os: linux

files:
  - source: /
    destination: /var/www/myapp

hooks:
  BeforeInstall:
    - location: scripts/stop-server.sh
      timeout: 60

  AfterInstall:
    - location: scripts/install-deps.sh
      timeout: 120

  ApplicationStart:
    - location: scripts/start-server.sh
      timeout: 60

  ValidateService:
    - location: scripts/health-check.sh
      timeout: 60
```

For Lambda deployments:

```yaml
version: 0.0

Resources:
  - MyFunction:
      Type: AWS::Lambda::Function
      Properties:
        Name: "my-function"
        Alias: "live"
        CurrentVersion: "1"
        TargetVersion: "2"
```

### Lifecycle Hooks (EC2)

CodeDeploy runs scripts at specific points during deployment:

1. **BeforeInstall** — Run before files are copied (stop old service, clean up)
2. **AfterInstall** — Run after files are copied (install dependencies, set permissions)
3. **ApplicationStart** — Start the new version
4. **ValidateService** — Verify the deployment worked (health checks)

Example hook script (`scripts/start-server.sh`):

```bash
#!/bin/bash
cd /var/www/myapp
npm install --production
pm2 restart myapp || pm2 start server.js --name myapp
```

## Hands-On: Complete Pipeline (GitHub → Build → Deploy to S3)

Here's how to set up a pipeline that builds a React app from GitHub and deploys it to S3 as a static website.

### Step 1: Create the S3 Website Bucket

```bash
aws s3 mb s3://my-react-app-website
aws s3 website s3://my-react-app-website --index-document index.html --error-document index.html
```

### Step 2: Create the buildspec.yml

In your React project root:

```yaml
version: 0.2

phases:
  install:
    runtime-versions:
      nodejs: 20
    commands:
      - npm ci

  pre_build:
    commands:
      - npm test -- --watchAll=false

  build:
    commands:
      - npm run build

artifacts:
  files:
    - "**/*"
  base-directory: build
```

### Step 3: Connect GitHub

In the AWS Console, go to **Developer Tools > Settings > Connections** and create a connection to your GitHub account. This gives AWS permission to read your repository.

### Step 4: Create the Pipeline

Use the Console or CLI to create a pipeline with three stages:
1. **Source**: GitHub (via the connection you created)
2. **Build**: CodeBuild (using the `buildspec.yml` in your repo)
3. **Deploy**: S3 (to your website bucket, with Extract enabled)

### Step 5: Push and Watch

Every time you push to `main`, the pipeline triggers automatically:

```bash
git add . && git commit -m "Update homepage" && git push
```

Monitor the pipeline:

```bash
aws codepipeline get-pipeline-state --name my-react-pipeline \
  --query "stageStates[].{Stage:stageName,Status:latestExecution.status}" \
  --output table
```

### Adding Manual Approval

For production deployments, add a manual approval stage between Build and Deploy:

```json
{
  "name": "Approval",
  "actions": [
    {
      "name": "ManualApproval",
      "actionTypeId": {
        "category": "Approval",
        "owner": "AWS",
        "provider": "Manual",
        "version": "1"
      },
      "configuration": {
        "NotificationArn": "arn:aws:sns:us-east-1:123456789012:deployment-approvals",
        "CustomData": "Please review the staging deployment before approving."
      }
    }
  ]
}
```

## Key Takeaways

- **CodePipeline** orchestrates the flow: Source → Build → Test → Deploy
- **CodeBuild** compiles and tests code in managed containers — no servers to maintain
- **CodeDeploy** pushes updates to EC2, ECS, or Lambda with rollback support
- The `buildspec.yml` defines your build steps; `appspec.yml` defines your deployment steps
- **Blue/Green deployments** give you zero-downtime updates
- Pipelines trigger automatically on code push — every commit goes through the same process
- Add **Manual Approval** stages for production deployments that need human review
