# AWS CDK

The AWS Cloud Development Kit (CDK) lets you define cloud infrastructure using real programming languages — TypeScript, Python, Java, C#, or Go — instead of YAML or JSON. You write code that describes your infrastructure, and CDK converts it into a CloudFormation template and deploys it. This means you get loops, conditionals, type checking, autocomplete, and everything else your language offers.

## What You'll Learn

- What CDK is and why it exists
- How CDK compares to CloudFormation
- Core concepts: App, Stack, and Construct
- The difference between L1, L2, and L3 constructs
- How to set up CDK and deploy your first stack
- Essential CDK CLI commands

## CDK vs CloudFormation

CDK generates CloudFormation under the hood. Think of it as a higher-level layer on top of CloudFormation:

| Feature | CloudFormation | CDK |
|---------|---------------|-----|
| Language | YAML / JSON | TypeScript, Python, Java, C#, Go |
| Abstraction level | Low — you specify every property | High — smart defaults fill in the gaps |
| Loops / conditionals | Limited (Conditions, Mappings) | Full language features |
| IDE support | Basic YAML validation | Full autocomplete, type checking |
| Learning curve | Learn AWS resource syntax | Learn CDK constructs + some AWS knowledge |
| Output | Deployed CloudFormation stack | Deployed CloudFormation stack |

CDK doesn't replace CloudFormation — it **compiles down to it**. When you run `cdk deploy`, CDK first generates a CloudFormation template (`cdk synth`), then submits that template to CloudFormation. You can always inspect the generated template.

## Getting Started with CDK

### Prerequisites

You need Node.js installed (even if you're writing CDK in Python, because the CDK CLI runs on Node):

```bash
node --version  # v18 or higher recommended
```

### Install the CDK CLI

```bash
npm install -g aws-cdk
cdk --version
```

### Bootstrap Your AWS Account

Before deploying your first CDK app, you need to "bootstrap" your AWS account. This creates an S3 bucket and IAM roles that CDK uses to deploy:

```bash
cdk bootstrap aws://123456789012/us-east-1
```

Replace `123456789012` with your actual AWS account ID. You only need to do this once per account/region combination.

### Create a New CDK Project

```bash
mkdir my-cdk-app && cd my-cdk-app
cdk init app --language typescript
```

This creates a project structure:

```
my-cdk-app/
├── bin/
│   └── my-cdk-app.ts        # Entry point — creates the App
├── lib/
│   └── my-cdk-app-stack.ts  # Your stack definition
├── test/
│   └── my-cdk-app.test.ts   # Unit tests
├── cdk.json                  # CDK configuration
├── package.json
└── tsconfig.json
```

## Constructs, Stacks, and Apps

CDK has three levels of organization:

**App** — The top-level container. A CDK app can contain one or more stacks. This is defined in `bin/my-cdk-app.ts`:

```typescript
import * as cdk from 'aws-cdk-lib';
import { MyCdkAppStack } from '../lib/my-cdk-app-stack';

const app = new cdk.App();
new MyCdkAppStack(app, 'MyCdkAppStack', {
  env: { account: '123456789012', region: 'us-east-1' },
});
```

**Stack** — Maps directly to a CloudFormation stack. A stack contains constructs (resources). This is defined in `lib/my-cdk-app-stack.ts`:

```typescript
import * as cdk from 'aws-cdk-lib';
import { Construct } from 'constructs';

export class MyCdkAppStack extends cdk.Stack {
  constructor(scope: Construct, id: string, props?: cdk.StackProps) {
    super(scope, id, props);
    // Your resources go here
  }
}
```

**Construct** — A cloud component. It can represent a single resource (like an S3 bucket) or a higher-level pattern (like an API Gateway backed by a Lambda function). Constructs are the building blocks of CDK.

## L1, L2, and L3 Constructs

### L1 Constructs (CFN Resources)

These are direct 1:1 mappings to CloudFormation resource types. They start with `Cfn` and require you to set every property manually — just like writing CloudFormation in YAML:

```typescript
import * as s3 from 'aws-cdk-lib/aws-s3';

new s3.CfnBucket(this, 'MyBucket', {
  bucketName: 'my-raw-bucket',
  versioningConfiguration: {
    status: 'Enabled',
  },
});
```

### L2 Constructs (Curated)

These are the sweet spot. L2 constructs provide sensible defaults, convenience methods, and grant helpers. They handle the boilerplate so you can focus on what matters:

```typescript
import * as s3 from 'aws-cdk-lib/aws-s3';

const bucket = new s3.Bucket(this, 'MyBucket', {
  versioned: true,
  removalPolicy: cdk.RemovalPolicy.DESTROY,
  autoDeleteObjects: true,
});
```

Notice how `versioned: true` is much cleaner than the L1 version. L2 constructs also have methods like `bucket.grantRead(myLambda)` that automatically create the right IAM permissions.

### L3 Constructs (Patterns)

These are high-level patterns that combine multiple resources. For example, `LambdaRestApi` creates an API Gateway, connects it to a Lambda function, and configures everything:

```typescript
import * as apigateway from 'aws-cdk-lib/aws-apigateway';

new apigateway.LambdaRestApi(this, 'MyApi', {
  handler: myFunction,
});
```

This single construct creates an API Gateway REST API, a Lambda integration, an IAM role, and a deployment — all wired together.

## Hands-On: S3 Bucket + Lambda with CDK

Here's a complete stack that creates an S3 bucket and a Lambda function with permission to read from it:

```typescript
import * as cdk from 'aws-cdk-lib';
import * as s3 from 'aws-cdk-lib/aws-s3';
import * as lambda from 'aws-cdk-lib/aws-lambda';
import { Construct } from 'constructs';

export class MyCdkAppStack extends cdk.Stack {
  constructor(scope: Construct, id: string, props?: cdk.StackProps) {
    super(scope, id, props);

    const bucket = new s3.Bucket(this, 'DataBucket', {
      versioned: true,
      removalPolicy: cdk.RemovalPolicy.DESTROY,
      autoDeleteObjects: true,
    });

    const processor = new lambda.Function(this, 'Processor', {
      runtime: lambda.Runtime.PYTHON_3_12,
      handler: 'index.handler',
      code: lambda.Code.fromInline(`
import json
import os

def handler(event, context):
    bucket_name = os.environ['BUCKET_NAME']
    print(f"Processing data from {bucket_name}")
    return {"statusCode": 200, "body": json.dumps("Done")}
      `),
      environment: {
        BUCKET_NAME: bucket.bucketName,
      },
    });

    bucket.grantRead(processor);

    new cdk.CfnOutput(this, 'BucketNameOutput', {
      value: bucket.bucketName,
      description: 'The name of the S3 bucket',
    });

    new cdk.CfnOutput(this, 'FunctionArnOutput', {
      value: processor.functionArn,
      description: 'The ARN of the Lambda function',
    });
  }
}
```

The key line is `bucket.grantRead(processor)`. This single call creates an IAM policy that grants the Lambda function read access to the bucket, and attaches it to the function's role. In CloudFormation, this would be 15-20 lines of YAML.

## CDK CLI Commands

### Synthesize (Generate CloudFormation)

```bash
cdk synth
```

This outputs the generated CloudFormation template to `cdk.out/` and prints it to the terminal. Always run this before deploying to inspect what will be created.

### Compare Changes

```bash
cdk diff
```

Shows what will change compared to the currently deployed stack. Similar to CloudFormation change sets but faster to use.

### Deploy

```bash
cdk deploy
```

Synthesizes the template and deploys it via CloudFormation. If the stack doesn't exist, it creates one. If it does exist, it updates it.

Deploy a specific stack (if your app has multiple):

```bash
cdk deploy MyCdkAppStack
```

Deploy all stacks:

```bash
cdk deploy --all
```

### Destroy

```bash
cdk destroy
```

Deletes the CloudFormation stack and all its resources.

### List Stacks

```bash
cdk list
```

Shows all stacks defined in your CDK app.

### Watch Mode (Hot Deploy)

```bash
cdk watch
```

Monitors your source files and automatically deploys changes. Useful during development — it uses CloudFormation's hotswap feature for faster Lambda updates.

## Project Organization Tips

For larger projects, organize constructs into separate files:

```
lib/
├── my-cdk-app-stack.ts       # Main stack
├── database-construct.ts     # DynamoDB tables
├── api-construct.ts          # API Gateway + Lambda
└── storage-construct.ts      # S3 buckets
```

Each file exports a Construct class:

```typescript
import { Construct } from 'constructs';
import * as dynamodb from 'aws-cdk-lib/aws-dynamodb';

export class DatabaseConstruct extends Construct {
  public readonly table: dynamodb.Table;

  constructor(scope: Construct, id: string) {
    super(scope, id);

    this.table = new dynamodb.Table(this, 'MainTable', {
      partitionKey: { name: 'pk', type: dynamodb.AttributeType.STRING },
      sortKey: { name: 'sk', type: dynamodb.AttributeType.STRING },
      billingMode: dynamodb.BillingMode.PAY_PER_REQUEST,
    });
  }
}
```

Then compose them in your stack:

```typescript
const db = new DatabaseConstruct(this, 'Database');
const api = new ApiConstruct(this, 'Api', { table: db.table });
```

## Key Takeaways

- CDK lets you write infrastructure in TypeScript, Python, Java, C#, or Go
- It compiles down to CloudFormation — you're still using CloudFormation under the hood
- **L1** constructs are raw CloudFormation resources, **L2** constructs add smart defaults and helper methods, **L3** constructs are complete patterns
- `bucket.grantRead(lambda)` is the CDK magic — one line replaces 20 lines of IAM YAML
- Run `cdk synth` to inspect the generated template, `cdk diff` to preview changes, `cdk deploy` to ship
- Bootstrap your account once with `cdk bootstrap` before your first deployment
