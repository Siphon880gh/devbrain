# Terraform with AWS

Terraform is an open-source Infrastructure as Code tool by HashiCorp. Unlike CloudFormation (which only works with AWS), Terraform works across multiple cloud providers — AWS, Azure, GCP, and hundreds of others — using the same language and workflow. You describe infrastructure in HCL (HashiCorp Configuration Language), and Terraform figures out how to create it.

## What You'll Learn

- What Terraform is and how it differs from CloudFormation and CDK
- HCL syntax basics for AWS resources
- How to configure the AWS provider
- How Terraform tracks infrastructure with state files
- Essential commands: init, plan, apply, destroy
- Variables, outputs, and modules for reusable infrastructure
- State management with S3 and DynamoDB locking

## Terraform vs CloudFormation vs CDK

| Feature | Terraform | CloudFormation | CDK |
|---------|-----------|---------------|-----|
| Provider | Multi-cloud (AWS, Azure, GCP, etc.) | AWS only | AWS only |
| Language | HCL | YAML / JSON | TypeScript, Python, Java, C#, Go |
| State management | You manage (local file or remote backend) | AWS manages it | AWS manages (via CloudFormation) |
| Drift detection | `terraform plan` | Drift detection API | Via CloudFormation |
| Community modules | Terraform Registry (huge) | Limited | Construct Hub |
| Rollback on failure | No automatic rollback | Automatic rollback | Via CloudFormation |
| Cost | Free (open source) | Free | Free |

**When to choose Terraform:** You use multiple cloud providers, you want a single tool for everything, or your team already knows HCL.

**When to choose CloudFormation:** You're all-in on AWS and want AWS-managed state with automatic rollback.

**When to choose CDK:** You want the power of a real programming language with AWS-managed state.

## Getting Started

### Install Terraform

On macOS:

```bash
brew tap hashicorp/tap
brew install hashicorp/tap/terraform
terraform --version
```

On Linux:

```bash
sudo apt-get update && sudo apt-get install -y gnupg software-properties-common
wget -O- https://apt.releases.hashicorp.com/gpg | gpg --dearmor | sudo tee /usr/share/keyrings/hashicorp-archive-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] https://apt.releases.hashicorp.com $(lsb_release -cs) main" | sudo tee /etc/apt/sources.list.d/hashicorp.list
sudo apt update && sudo apt install terraform
```

### Project Structure

A typical Terraform project:

```
my-terraform-project/
├── main.tf          # Resource definitions
├── variables.tf     # Input variables
├── outputs.tf       # Output values
├── providers.tf     # Provider configuration
└── terraform.tfvars # Variable values (don't commit secrets)
```

## Providers, Resources, and State

### AWS Provider Configuration

Every Terraform project starts with a provider. The AWS provider tells Terraform how to authenticate and which region to use.

`providers.tf`:

```hcl
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
  required_version = ">= 1.5.0"
}

provider "aws" {
  region = var.aws_region
}
```

Terraform uses your AWS CLI credentials by default (`~/.aws/credentials`). You can also set `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` environment variables.

### HCL Syntax Basics

HCL uses a block-based syntax. Here's how you define resources:

```hcl
resource "aws_s3_bucket" "app_assets" {
  bucket = "my-app-assets-12345"

  tags = {
    Environment = "dev"
    Project     = "my-app"
  }
}

resource "aws_s3_bucket_versioning" "app_assets_versioning" {
  bucket = aws_s3_bucket.app_assets.id

  versioning_configuration {
    status = "Enabled"
  }
}
```

The format is `resource "TYPE" "LOCAL_NAME" { ... }`. The type comes from the provider (e.g., `aws_s3_bucket`), and the local name is how you reference it within your Terraform code.

### Variables

`variables.tf`:

```hcl
variable "aws_region" {
  description = "AWS region to deploy into"
  type        = string
  default     = "us-east-1"
}

variable "environment" {
  description = "Deployment environment"
  type        = string
  default     = "dev"

  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Environment must be dev, staging, or prod."
  }
}

variable "lambda_memory" {
  description = "Memory allocation for Lambda function (MB)"
  type        = number
  default     = 128
}
```

Set variable values in `terraform.tfvars`:

```hcl
aws_region  = "us-east-1"
environment = "staging"
```

Or pass them on the command line:

```bash
terraform apply -var="environment=prod"
```

### Outputs

`outputs.tf`:

```hcl
output "bucket_name" {
  description = "Name of the S3 bucket"
  value       = aws_s3_bucket.app_assets.bucket
}

output "function_arn" {
  description = "ARN of the Lambda function"
  value       = aws_lambda_function.processor.arn
}
```

## Hands-On: S3 Bucket + Lambda with Terraform

Here's a complete example. Create `main.tf`:

```hcl
resource "aws_s3_bucket" "data_bucket" {
  bucket = "${var.environment}-data-bucket-${random_id.suffix.hex}"

  tags = {
    Environment = var.environment
  }
}

resource "aws_s3_bucket_versioning" "data_bucket" {
  bucket = aws_s3_bucket.data_bucket.id

  versioning_configuration {
    status = "Enabled"
  }
}

resource "random_id" "suffix" {
  byte_length = 4
}

resource "aws_iam_role" "lambda_role" {
  name = "${var.environment}-processor-role"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Action = "sts:AssumeRole"
        Effect = "Allow"
        Principal = {
          Service = "lambda.amazonaws.com"
        }
      }
    ]
  })
}

resource "aws_iam_role_policy_attachment" "lambda_basic" {
  role       = aws_iam_role.lambda_role.name
  policy_arn = "arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole"
}

resource "aws_iam_role_policy" "lambda_s3_read" {
  name = "s3-read-access"
  role = aws_iam_role.lambda_role.id

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect = "Allow"
        Action = [
          "s3:GetObject",
          "s3:ListBucket"
        ]
        Resource = [
          aws_s3_bucket.data_bucket.arn,
          "${aws_s3_bucket.data_bucket.arn}/*"
        ]
      }
    ]
  })
}

data "archive_file" "lambda_zip" {
  type        = "zip"
  source_file = "${path.module}/src/handler.py"
  output_path = "${path.module}/build/handler.zip"
}

resource "aws_lambda_function" "processor" {
  filename         = data.archive_file.lambda_zip.output_path
  function_name    = "${var.environment}-processor"
  role             = aws_iam_role.lambda_role.arn
  handler          = "handler.lambda_handler"
  runtime          = "python3.12"
  source_code_hash = data.archive_file.lambda_zip.output_base64sha256
  memory_size      = var.lambda_memory

  environment {
    variables = {
      BUCKET_NAME = aws_s3_bucket.data_bucket.bucket
      ENVIRONMENT = var.environment
    }
  }
}
```

Create the Lambda handler at `src/handler.py`:

```python
import json
import os

def lambda_handler(event, context):
    bucket_name = os.environ['BUCKET_NAME']
    environment = os.environ['ENVIRONMENT']
    print(f"Processing in {environment} from bucket {bucket_name}")
    return {"statusCode": 200, "body": json.dumps("Done")}
```

## State File Concept

Terraform keeps track of what it created in a **state file** (`terraform.tfstate`). This file maps your HCL resources to real AWS resources. Without it, Terraform wouldn't know what already exists.

### Local State (Default)

By default, state is stored locally. This works for learning but not for teams, because:
- If you lose the file, Terraform loses track of your resources
- Two people can't work on the same infrastructure simultaneously

### Remote State with S3 Backend

For real projects, store state in S3 with DynamoDB for locking:

```hcl
terraform {
  backend "s3" {
    bucket         = "my-terraform-state-bucket"
    key            = "my-app/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-lock"
    encrypt        = true
  }
}
```

The DynamoDB table prevents two people from running `terraform apply` at the same time (state locking). Create the lock table:

```bash
aws dynamodb create-table \
  --table-name terraform-lock \
  --attribute-definitions AttributeName=LockID,AttributeType=S \
  --key-schema AttributeName=LockID,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST
```

## Terraform Commands

### Initialize

```bash
terraform init
```

Downloads provider plugins, initializes the backend, and sets up the working directory. Run this first in any new project or after adding a new provider.

### Plan

```bash
terraform plan
```

Shows what Terraform will do without making any changes. Think of it as a dry run. The output shows resources to be created (`+`), changed (`~`), or destroyed (`-`).

Save a plan for later execution:

```bash
terraform plan -out=tfplan
terraform apply tfplan
```

### Apply

```bash
terraform apply
```

Creates or updates infrastructure. Terraform shows you the plan and asks for confirmation. Use `-auto-approve` to skip confirmation (useful in CI/CD, dangerous in production).

### Destroy

```bash
terraform destroy
```

Deletes all resources managed by the current state. Asks for confirmation.

### Other Useful Commands

```bash
terraform fmt          # Format HCL files consistently
terraform validate     # Check syntax and configuration
terraform show         # Display current state
terraform state list   # List resources in state
terraform import       # Import existing AWS resource into Terraform
```

## Modules

Modules let you package and reuse infrastructure. A module is just a directory with Terraform files.

Create a module at `modules/lambda-api/main.tf`:

```hcl
variable "function_name" {
  type = string
}

variable "handler" {
  type    = string
  default = "handler.lambda_handler"
}

variable "runtime" {
  type    = string
  default = "python3.12"
}

variable "source_path" {
  type = string
}

resource "aws_lambda_function" "this" {
  filename      = var.source_path
  function_name = var.function_name
  role          = aws_iam_role.this.arn
  handler       = var.handler
  runtime       = var.runtime
}

resource "aws_iam_role" "this" {
  name = "${var.function_name}-role"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Action    = "sts:AssumeRole"
      Effect    = "Allow"
      Principal = { Service = "lambda.amazonaws.com" }
    }]
  })
}

output "function_arn" {
  value = aws_lambda_function.this.arn
}
```

Use the module in your root configuration:

```hcl
module "api_handler" {
  source        = "./modules/lambda-api"
  function_name = "api-handler"
  source_path   = "./build/api.zip"
}

module "worker" {
  source        = "./modules/lambda-api"
  function_name = "background-worker"
  source_path   = "./build/worker.zip"
  runtime       = "nodejs20.x"
  handler       = "index.handler"
}
```

You can also use published modules from the Terraform Registry:

```hcl
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.1.0"

  name = "my-vpc"
  cidr = "10.0.0.0/16"
  azs  = ["us-east-1a", "us-east-1b"]
}
```

## Key Takeaways

- Terraform is cloud-agnostic IaC — one tool for AWS, Azure, GCP, and more
- HCL syntax is readable and declarative: you say what you want, Terraform figures out how to build it
- The **state file** is critical — it maps your code to real resources; store it in S3 with DynamoDB locking for teams
- `terraform plan` before `terraform apply` — always review before making changes
- **Modules** let you package and reuse infrastructure patterns
- Terraform doesn't auto-rollback on failure (unlike CloudFormation), so always plan first
- Use `terraform.tfvars` for environment-specific values, but never commit secrets to version control
