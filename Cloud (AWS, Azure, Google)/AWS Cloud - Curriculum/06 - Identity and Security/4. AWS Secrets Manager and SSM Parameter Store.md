# AWS Secrets Manager and SSM Parameter Store

Every application needs configuration values — database passwords, API keys, connection strings, feature flags. Hardcoding these in your source code is one of the most common security mistakes in cloud development. AWS provides two services for storing these values safely: **Secrets Manager** and **SSM Parameter Store**.

## What You'll Learn

- Why hardcoding secrets is dangerous and what to do instead
- How AWS Secrets Manager stores and automatically rotates secrets
- How SSM Parameter Store stores configuration values and secrets
- The differences between the two services (features, pricing, use cases)
- CLI commands for storing and retrieving values from both services
- How to access secrets from Lambda and EC2 using the AWS SDK

## Why You Should Never Hardcode Secrets

Consider this code:

```python
import psycopg2

conn = psycopg2.connect(
    host="mydb.abc123.us-east-1.rds.amazonaws.com",
    database="myapp",
    user="admin",
    password="SuperSecret123!"   # DON'T DO THIS
)
```

This password will end up in:
- Your Git repository (and every clone, fork, and backup of it)
- Your CI/CD pipeline logs
- Your deployment artifacts
- Every developer's laptop who has access to the repo

If the password needs to change, you have to redeploy every application that uses it. If it gets leaked, you have to rotate it manually across all systems.

The correct approach: store secrets in a dedicated secrets service and retrieve them at runtime.

## AWS Secrets Manager

### What It Does

Secrets Manager is purpose-built for storing sensitive values like:

- Database credentials (username/password)
- API keys for third-party services
- OAuth tokens
- SSH keys
- TLS certificates

Each secret is stored encrypted using AWS KMS. You can store plain text or JSON (useful for key-value pairs like a database username and password together).

### Storing a Secret

```bash
aws secretsmanager create-secret \
  --name prod/myapp/database \
  --description "Production database credentials" \
  --secret-string '{"username":"admin","password":"SuperSecret123!","host":"mydb.abc123.us-east-1.rds.amazonaws.com","port":"5432","dbname":"myapp"}'
```

The name uses a path-like structure (`prod/myapp/database`) — this is a convention that helps organize secrets and set up IAM policies by path.

### Retrieving a Secret

```bash
aws secretsmanager get-secret-value \
  --secret-id prod/myapp/database
```

This returns JSON with a `SecretString` field containing your credentials.

### Retrieving from Lambda (Python)

```python
import json
import boto3

def get_db_credentials():
    client = boto3.client("secretsmanager")
    response = client.get_secret_value(SecretId="prod/myapp/database")
    return json.loads(response["SecretString"])

def handler(event, context):
    creds = get_db_credentials()
    # creds["username"], creds["password"], etc.
```

The Lambda function's execution role needs this permission:

```json
{
  "Effect": "Allow",
  "Action": "secretsmanager:GetSecretValue",
  "Resource": "arn:aws:secretsmanager:us-east-1:123456789012:secret:prod/myapp/database-*"
}
```

Note the wildcard at the end — Secrets Manager appends a random suffix to the ARN.

### Automatic Rotation

The killer feature of Secrets Manager is **automatic rotation**. You can configure it to rotate database passwords on a schedule (e.g., every 30 days) using a Lambda function:

```bash
aws secretsmanager rotate-secret \
  --secret-id prod/myapp/database \
  --rotation-lambda-arn arn:aws:lambda:us-east-1:123456789012:function:RotateDBPassword \
  --rotation-rules '{"AutomaticallyAfterDays": 30}'
```

For RDS databases, AWS provides pre-built rotation Lambda functions. The rotation process:

1. The rotation Lambda creates a new password
2. It updates the password in the database
3. It stores the new password in Secrets Manager
4. Your application automatically picks up the new password next time it retrieves the secret

No downtime. No manual intervention. No stale passwords sitting around for years.

### Versioning

Secrets Manager keeps previous versions of your secrets. Each version has labels:

- `AWSCURRENT` — the current active version
- `AWSPREVIOUS` — the last version before the current one
- `AWSPENDING` — used during rotation (the new version before it becomes current)

This means if a rotation fails, you can roll back.

## SSM Parameter Store

### What It Does

SSM Parameter Store (part of AWS Systems Manager) is a more general-purpose configuration store. It handles:

- Application configuration values (feature flags, URLs, timeouts)
- Secrets (encrypted with KMS)
- Any key-value pair your application needs

### Parameter Types

| Type | Description | Example |
|------|-------------|---------|
| **String** | Plain text value | `https://api.example.com` |
| **StringList** | Comma-separated values | `us-east-1,us-west-2,eu-west-1` |
| **SecureString** | Encrypted with KMS | Database passwords, API keys |

### Tiers

| Feature | Standard | Advanced |
|---------|----------|----------|
| **Price** | Free (up to 10,000 params) | $0.05 per parameter/month |
| **Max size** | 4 KB | 8 KB |
| **Parameter policies** | No | Yes (expiration, notifications) |
| **Throughput** | 40 TPS default (can request up to 1,000) | 1,000 TPS |

For most applications, the **Standard tier is free and sufficient**.

### Storing Parameters

Plain text config value:

```bash
aws ssm put-parameter \
  --name "/myapp/prod/api-url" \
  --value "https://api.example.com/v2" \
  --type String
```

Encrypted secret:

```bash
aws ssm put-parameter \
  --name "/myapp/prod/api-key" \
  --value "sk-abc123def456" \
  --type SecureString
```

By default, SecureString parameters are encrypted with the AWS managed KMS key (`aws/ssm`). You can specify a custom KMS key with `--key-id`.

### Retrieving Parameters

Get a single parameter:

```bash
aws ssm get-parameter \
  --name "/myapp/prod/api-url"
```

Get a decrypted SecureString:

```bash
aws ssm get-parameter \
  --name "/myapp/prod/api-key" \
  --with-decryption
```

Get all parameters under a path (useful for loading all config at once):

```bash
aws ssm get-parameters-by-path \
  --path "/myapp/prod" \
  --with-decryption \
  --recursive
```

### Retrieving from Lambda (Python)

```python
import boto3

ssm = boto3.client("ssm")

def get_config():
    response = ssm.get_parameters_by_path(
        Path="/myapp/prod",
        WithDecryption=True,
        Recursive=True
    )
    config = {}
    for param in response["Parameters"]:
        key = param["Name"].split("/")[-1]
        config[key] = param["Value"]
    return config

def handler(event, context):
    config = get_config()
    api_url = config["api-url"]
    api_key = config["api-key"]
```

Required IAM permissions:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "ssm:GetParameter",
        "ssm:GetParametersByPath"
      ],
      "Resource": "arn:aws:ssm:us-east-1:123456789012:parameter/myapp/prod/*"
    },
    {
      "Effect": "Allow",
      "Action": "kms:Decrypt",
      "Resource": "arn:aws:kms:us-east-1:123456789012:key/your-kms-key-id"
    }
  ]
}
```

The `kms:Decrypt` permission is needed only if you use SecureString parameters.

## Secrets Manager vs SSM Parameter Store

| Feature | Secrets Manager | SSM Parameter Store (Standard) |
|---------|----------------|-------------------------------|
| **Price** | $0.40/secret/month + $0.05 per 10K API calls | Free (up to 10,000 parameters) |
| **Auto rotation** | Built-in with Lambda | Not built-in (you'd build it yourself) |
| **Max size** | 64 KB | 4 KB |
| **Cross-account access** | Native support via resource policies | Requires more setup |
| **Versioning** | Automatic with labels | Version numbers only |
| **Best for** | Database passwords, API keys that need rotation | App config, feature flags, non-rotating secrets |

### When to Use Which

**Use Secrets Manager when:**
- You need automatic credential rotation (especially for RDS, Redshift, DocumentDB)
- You need cross-account secret sharing
- The secret value is large (up to 64 KB)
- You're willing to pay for the convenience

**Use SSM Parameter Store when:**
- You're storing application configuration (URLs, feature flags, timeouts)
- You want to keep costs at zero (Standard tier)
- You don't need automatic rotation
- You want hierarchical organization with path-based access control

**Many teams use both:** SSM Parameter Store for general config and Secrets Manager for database credentials that need rotation.

## Hands-On: Storing a Database Password

Let's store and retrieve a database password using both services.

### With Secrets Manager

Store it:

```bash
aws secretsmanager create-secret \
  --name demo/database \
  --secret-string '{"username":"admin","password":"DemoPass123!"}'
```

Retrieve it:

```bash
aws secretsmanager get-secret-value \
  --secret-id demo/database \
  --query SecretString \
  --output text
```

Update it:

```bash
aws secretsmanager update-secret \
  --secret-id demo/database \
  --secret-string '{"username":"admin","password":"NewPass456!"}'
```

Delete it (with a 7-day recovery window):

```bash
aws secretsmanager delete-secret \
  --secret-id demo/database \
  --recovery-window-in-days 7
```

### With SSM Parameter Store

Store it:

```bash
aws ssm put-parameter \
  --name "/demo/database/password" \
  --value "DemoPass123!" \
  --type SecureString
```

Retrieve it:

```bash
aws ssm get-parameter \
  --name "/demo/database/password" \
  --with-decryption \
  --query Parameter.Value \
  --output text
```

Update it (add `--overwrite` to replace the existing value):

```bash
aws ssm put-parameter \
  --name "/demo/database/password" \
  --value "NewPass456!" \
  --type SecureString \
  --overwrite
```

Delete it:

```bash
aws ssm delete-parameter \
  --name "/demo/database/password"
```

## Key Takeaways

- **Never hardcode secrets** in source code, environment variables in Dockerfiles, or config files committed to Git
- **Secrets Manager** is ideal for credentials that need automatic rotation — especially database passwords
- **SSM Parameter Store** (Standard) is free and great for general application configuration
- Use **path-based naming** (`/app/env/key`) to organize parameters and scope IAM policies
- Both services encrypt secrets using **AWS KMS**
- The AWS SDK retrieves secrets at runtime — your code never stores credentials on disk
- **Use both together**: Parameter Store for config, Secrets Manager for rotating credentials
