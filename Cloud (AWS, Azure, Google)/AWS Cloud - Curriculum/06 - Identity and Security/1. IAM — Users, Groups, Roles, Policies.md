# IAM — Users, Groups, Roles, Policies

IAM (Identity and Access Management) controls **who** can do **what** in your AWS account. Every API call to AWS is checked against IAM. If you don't have permission, the call is denied. This lesson covers the four building blocks of IAM and how to use them safely.

## What You'll Learn

- The difference between the root user and IAM users
- How users, groups, roles, and policies work together
- How to read and write IAM policy JSON documents
- The principle of least privilege and why it matters
- How to create users, groups, and attach policies via the CLI

## The Root User vs IAM Users

When you first create an AWS account, you get a **root user**. This user has unrestricted access to everything — billing, account closure, every service. You should treat the root user like a fire alarm: break glass only in emergencies.

**Root user rules:**
- Enable MFA on the root user immediately
- Never create access keys for the root user
- Never use the root user for day-to-day work
- Only use it for tasks that require it (like changing account settings or closing the account)

Instead, create **IAM users** for yourself and your team. An IAM user is an identity within your account that represents a person or application. Each IAM user has their own credentials (password for console, access keys for CLI/API) and their own set of permissions.

## Core IAM Components

### Users

A **user** represents a single person or application. Each user can have:

- A **console password** — for logging into the AWS web console
- **Access keys** (access key ID + secret access key) — for CLI and SDK access
- **MFA device** — an extra layer of security on sign-in

A new user has **zero permissions** by default. They can log in, but they can't do anything until you attach a policy.

### Groups

A **group** is a collection of IAM users. Instead of attaching policies to each user individually, you attach policies to a group and then add users to that group.

For example:
- A **Developers** group with permissions to deploy Lambda functions and access S3
- An **Admins** group with broader permissions
- A **ReadOnly** group for auditors who only need to view resources

Groups make permission management scalable. When a new developer joins, add them to the Developers group. When they leave, remove them. No need to touch individual policies.

> **Note:** Groups cannot be nested. A group cannot contain another group. A user can belong to multiple groups.

### Roles

A **role** is an identity with permissions, but unlike a user, it has no permanent credentials. Instead, a role provides **temporary security credentials** when someone (or something) assumes it.

Roles are used for:
- **AWS services** — An EC2 instance assuming a role to access S3
- **Cross-account access** — A user in Account A assuming a role in Account B
- **Federated users** — Users from an external identity provider (like Google or Active Directory)

We'll cover service roles in detail in the next lesson.

### Policies

A **policy** is a JSON document that defines permissions. It answers: "Who can do what on which resources, and under what conditions?"

There are three types:

| Type | Description | Use Case |
|------|-------------|----------|
| **AWS Managed** | Created and maintained by AWS | Quick start, common use cases like `ReadOnlyAccess` |
| **Customer Managed** | Created by you in your account | Custom permissions tailored to your app |
| **Inline** | Embedded directly on a user, group, or role | One-off exceptions (generally avoid these) |

AWS managed policies are convenient but often grant more permissions than you need. Customer managed policies let you be precise.

## Policies and Permissions

### Policy Structure

Every IAM policy is a JSON document with this structure:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3ReadAccess",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-app-bucket",
        "arn:aws:s3:::my-app-bucket/*"
      ]
    }
  ]
}
```

Breaking down each field:

- **Version** — Always `"2012-10-17"`. This is the policy language version, not a date you choose.
- **Statement** — An array of permission rules. A policy can have multiple statements.
- **Sid** (optional) — A human-readable identifier for the statement.
- **Effect** — Either `"Allow"` or `"Deny"`. Deny always wins if there's a conflict.
- **Action** — The AWS API actions being allowed or denied. Format is `service:Action`.
- **Resource** — The specific AWS resources (identified by ARN) the actions apply to.
- **Condition** (optional) — Extra constraints like IP address, time, or MFA requirement.

### How AWS Evaluates Permissions

When you make an API call, AWS checks your permissions in this order:

1. By default, everything is **denied** (implicit deny)
2. AWS looks for any **explicit Allow** that matches the action and resource
3. If an **explicit Deny** exists anywhere, it overrides any Allow
4. If no Allow is found, the request is denied

The golden rule: **Deny always beats Allow.**

### Condition Example

You can add conditions to make policies more precise. Here's a policy that only allows S3 access from a specific IP range:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:*",
      "Resource": "*",
      "Condition": {
        "IpAddress": {
          "aws:SourceIp": "203.0.113.0/24"
        }
      }
    }
  ]
}
```

## The Principle of Least Privilege

Least privilege means giving each user or service **only the permissions they need** to do their job — nothing more.

In practice:
- Don't give a developer `AdministratorAccess` when they only need to deploy Lambda functions
- Don't give a Lambda function `s3:*` when it only reads from one bucket
- Don't use `Resource: "*"` when you can scope to specific ARNs
- Start with zero permissions and add only what breaks

AWS provides a tool called **IAM Access Analyzer** that reviews your policies and flags overly permissive access. You can also check **IAM Access Advisor** to see which services a user or role has actually accessed — if they haven't used a permission in 90 days, consider removing it.

## Hands-On: Creating Users and Attaching Policies

### Create a Group

```bash
aws iam create-group --group-name Developers
```

### Create a Customer Managed Policy

Save the following as `dev-s3-policy.json`:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-app-bucket",
        "arn:aws:s3:::my-app-bucket/*"
      ]
    }
  ]
}
```

Create the policy:

```bash
aws iam create-policy \
  --policy-name DevS3Access \
  --policy-document file://dev-s3-policy.json
```

This returns a policy ARN — copy it for the next step.

### Attach the Policy to the Group

```bash
aws iam attach-group-policy \
  --group-name Developers \
  --policy-arn arn:aws:iam::123456789012:policy/DevS3Access
```

### Create a User and Add Them to the Group

```bash
aws iam create-user --user-name alice

aws iam add-user-to-group \
  --user-name alice \
  --group-name Developers
```

### Give the User Console Access

```bash
aws iam create-login-profile \
  --user-name alice \
  --password "TempPassword123!" \
  --password-reset-required
```

The `--password-reset-required` flag forces the user to set their own password on first login.

### Give the User CLI Access

```bash
aws iam create-access-key --user-name alice
```

This outputs an `AccessKeyId` and `SecretAccessKey`. The secret key is shown only once — save it immediately.

### Verify Permissions

List what policies are attached to the user (directly and via groups):

```bash
aws iam list-attached-group-policies --group-name Developers
aws iam list-groups-for-user --user-name alice
```

## Key Takeaways

- **Root user** has unlimited power — lock it down with MFA and never use it daily
- **Users** represent people or apps; they start with zero permissions
- **Groups** let you manage permissions for multiple users at once
- **Roles** provide temporary credentials for services, cross-account access, and federation
- **Policies** are JSON documents with Effect, Action, Resource, and optional Condition
- **Deny always beats Allow** — an explicit deny overrides everything
- Always follow the **principle of least privilege** — grant only what's needed
- Prefer **customer managed policies** over inline policies for reusability and auditability
