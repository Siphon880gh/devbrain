# Cognito — User Authentication

Building user sign-up and sign-in from scratch is painful — you have to handle password hashing, email verification, token generation, MFA, and a dozen security edge cases. **Amazon Cognito** handles all of this for you, giving your web and mobile apps a fully managed authentication system.

## What You'll Learn

- What Cognito does and when to use it
- The difference between User Pools and Identity Pools
- How the sign-up, sign-in, and token flow works
- What JWT tokens (ID, access, refresh) contain and how they're used
- How to integrate Cognito with API Gateway as an authorizer
- How to create a User Pool via the CLI

## What is Amazon Cognito?

Cognito is a managed authentication and authorization service. It lets your applications:

- **Sign up users** with email/password, phone number, or username
- **Sign in users** and issue secure tokens
- **Add social login** (Google, Facebook, Apple, Amazon)
- **Add enterprise login** via SAML or OpenID Connect (like Okta or Active Directory)
- **Enable MFA** (SMS or TOTP authenticator apps)
- **Provide a hosted sign-in UI** so you don't have to build login pages

Cognito scales automatically. Whether you have 100 users or 10 million, the infrastructure is managed for you.

## User Pools vs Identity Pools

Cognito has two main components, and the names are confusing. Here's the clear distinction:

### User Pools — "Who are you?"

A **User Pool** is a user directory. It handles:

- User registration (sign-up)
- User authentication (sign-in)
- Password policies and recovery
- Email and phone verification
- MFA
- Token issuance (JWT tokens)

Think of a User Pool as your application's user database with a built-in authentication API. When a user signs in, the User Pool returns **JWT tokens** that your app uses to identify the user and authorize API calls.

**Use a User Pool when:** You need to authenticate users for your web or mobile app.

### Identity Pools — "What AWS resources can you access?"

An **Identity Pool** (also called Federated Identities) maps authenticated users to **temporary AWS credentials**. It takes a token from a User Pool (or a social provider like Google) and exchanges it for IAM credentials that can access AWS services directly.

**Use an Identity Pool when:** Your frontend app needs to access AWS services directly — for example, uploading files to S3 from a mobile app or querying DynamoDB from a browser.

### How They Work Together

A common pattern:

1. User signs in via a **User Pool** → gets JWT tokens
2. The JWT token is sent to an **Identity Pool** → gets temporary AWS credentials (access key, secret key, session token)
3. The frontend uses those credentials to call AWS services directly (e.g., upload to S3)

For most web apps with a backend API, you only need a **User Pool**. The backend validates the JWT token and handles AWS service calls with its own IAM role. You add an Identity Pool only if the client needs direct AWS access.

## Sign-Up, Sign-In, and Token Flow

### The Sign-Up Flow

1. User submits email and password to your app
2. Your app calls Cognito's `SignUp` API
3. Cognito stores the user and sends a verification code via email (or SMS)
4. User enters the verification code
5. Your app calls `ConfirmSignUp` — the user is now verified and can sign in

### The Sign-In Flow

1. User submits email and password
2. Your app calls Cognito's `InitiateAuth` API
3. If MFA is enabled, Cognito returns a challenge — user provides the MFA code
4. Cognito validates credentials and returns three JWT tokens

### The Three Tokens

After successful sign-in, Cognito returns three tokens:

| Token | Purpose | Default Expiry |
|-------|---------|----------------|
| **ID Token** | Contains user identity claims (email, name, custom attributes). Used to identify who the user is. | 1 hour |
| **Access Token** | Contains user's group memberships and scopes. Used to authorize API calls. | 1 hour |
| **Refresh Token** | Used to get new ID and access tokens without re-authenticating. | 30 days |

The **ID token** is what your backend inspects to know who's calling. It contains claims like:

```json
{
  "sub": "a1b2c3d4-5678-90ab-cdef-example11111",
  "email": "alice@example.com",
  "email_verified": true,
  "cognito:username": "alice",
  "cognito:groups": ["admin"],
  "iat": 1625000000,
  "exp": 1625003600,
  "iss": "https://cognito-idp.us-east-1.amazonaws.com/us-east-1_abc123"
}
```

The **access token** is sent with API requests. API Gateway or your backend validates it.

The **refresh token** is stored securely on the client. When the access token expires, the client sends the refresh token to get a new pair of ID and access tokens — without asking the user to sign in again.

## Integrating Cognito with API Gateway

One of the most powerful Cognito integrations is using it as an **authorizer** for API Gateway. This means API Gateway validates the JWT token before your backend code even runs.

### How It Works

1. Client signs in via Cognito → receives an access token
2. Client sends API request with `Authorization: Bearer <access_token>` header
3. API Gateway's Cognito authorizer validates the token (checks signature, expiry, issuer)
4. If valid, the request passes through to your Lambda function (or other backend)
5. If invalid or expired, API Gateway returns `401 Unauthorized` immediately

This offloads authentication from your backend entirely. Your Lambda function only handles business logic.

### Setting Up the Authorizer via CLI

First, create the authorizer on your API:

```bash
aws apigatewayv2 create-authorizer \
  --api-id your-api-id \
  --authorizer-type JWT \
  --identity-source '$request.header.Authorization' \
  --name CognitoAuth \
  --jwt-configuration '{
    "Audience": ["your-app-client-id"],
    "Issuer": "https://cognito-idp.us-east-1.amazonaws.com/us-east-1_abc123"
  }'
```

Then attach the authorizer to your route:

```bash
aws apigatewayv2 update-route \
  --api-id your-api-id \
  --route-id your-route-id \
  --authorization-type JWT \
  --authorizer-id your-authorizer-id
```

Now any request to that route without a valid Cognito token gets rejected automatically.

## Hands-On: Setting Up a User Pool

### Create a User Pool

```bash
aws cognito-idp create-user-pool \
  --pool-name MyAppUsers \
  --auto-verified-attributes email \
  --username-attributes email \
  --policies '{
    "PasswordPolicy": {
      "MinimumLength": 8,
      "RequireUppercase": true,
      "RequireLowercase": true,
      "RequireNumbers": true,
      "RequireSymbols": false
    }
  }' \
  --mfa-configuration OPTIONAL \
  --schema '[
    {
      "Name": "email",
      "Required": true,
      "Mutable": true
    },
    {
      "Name": "name",
      "Required": true,
      "Mutable": true
    }
  ]'
```

This creates a User Pool where:
- Users sign in with their email address
- Emails are automatically sent for verification
- Passwords must be at least 8 characters with mixed case and numbers
- MFA is available but not required

Copy the `UserPool.Id` from the output (it looks like `us-east-1_abc123`).

### Create an App Client

Your web or mobile app needs an **app client** to interact with the User Pool:

```bash
aws cognito-idp create-user-pool-client \
  --user-pool-id us-east-1_abc123 \
  --client-name MyWebApp \
  --explicit-auth-flows ALLOW_USER_PASSWORD_AUTH ALLOW_REFRESH_TOKEN_AUTH \
  --generate-secret false
```

The `--generate-secret false` flag is important for public clients (browser-based apps, mobile apps) that can't securely store a client secret.

### Configure the Hosted UI (Optional)

Cognito provides a pre-built sign-in page so you don't have to build one:

```bash
aws cognito-idp create-user-pool-domain \
  --user-pool-id us-east-1_abc123 \
  --domain my-app-auth
```

This creates a hosted UI at:
`https://my-app-auth.auth.us-east-1.amazoncognito.com`

Users can sign up, sign in, reset their password, and complete MFA — all without you writing a single line of frontend auth code.

### Test the Flow

Sign up a user:

```bash
aws cognito-idp sign-up \
  --client-id your-app-client-id \
  --username alice@example.com \
  --password MyPassword123 \
  --user-attributes Name=name,Value=Alice
```

Confirm the user (simulating email verification):

```bash
aws cognito-idp admin-confirm-sign-up \
  --user-pool-id us-east-1_abc123 \
  --username alice@example.com
```

Sign in and get tokens:

```bash
aws cognito-idp initiate-auth \
  --client-id your-app-client-id \
  --auth-flow USER_PASSWORD_AUTH \
  --auth-parameters USERNAME=alice@example.com,PASSWORD=MyPassword123
```

This returns `IdToken`, `AccessToken`, and `RefreshToken`. You can decode the ID token at [jwt.io](https://jwt.io) to inspect the claims.

## Key Takeaways

- **User Pools** handle authentication — sign-up, sign-in, MFA, and token issuance
- **Identity Pools** exchange tokens for temporary AWS credentials (for direct AWS access from clients)
- Most web apps with a backend API only need a **User Pool**
- Cognito returns three JWT tokens: **ID** (who you are), **Access** (what you can do), **Refresh** (get new tokens)
- Use Cognito as an **API Gateway authorizer** to validate tokens before requests reach your backend
- The **hosted UI** gives you a complete sign-in experience without building login pages
- Cognito supports social login (Google, Facebook, Apple) and enterprise federation (SAML, OIDC)
