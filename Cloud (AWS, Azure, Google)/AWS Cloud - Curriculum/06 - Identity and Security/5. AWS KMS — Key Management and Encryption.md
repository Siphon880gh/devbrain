# AWS KMS — Key Management and Encryption

Encryption protects your data from unauthorized access. Even if someone gains access to a disk or intercepts network traffic, encrypted data is unreadable without the key. **AWS Key Management Service (KMS)** is the central service for creating and managing encryption keys across your AWS account.

## What You'll Learn

- The difference between encryption at rest and encryption in transit
- How KMS keys work (AWS managed vs customer managed)
- What envelope encryption is and why it's necessary
- How KMS integrates with S3, EBS, RDS, and DynamoDB
- How to create keys, encrypt data, and decrypt data via the CLI
- Key policies, rotation, and pricing

## Encryption at Rest vs In Transit

### Encryption at Rest

Data "at rest" is data stored on a physical medium — an S3 bucket, an EBS volume, a DynamoDB table, an RDS database. Encryption at rest means the data is encrypted when it's written to disk and decrypted when it's read back.

If someone steals the physical hard drive or gains unauthorized access to the storage layer, they see scrambled bytes instead of your data.

Most AWS services support encryption at rest, and many enable it by default (like EBS volumes and S3 buckets created through the console).

### Encryption in Transit

Data "in transit" is data moving between two points — from your browser to a server, from one AWS service to another, from your app to a database. Encryption in transit uses **TLS (Transport Layer Security)** to protect data on the network.

AWS services use TLS for their API endpoints by default. When you call `s3.getObject()`, that request travels over HTTPS. For database connections, you can enforce TLS by configuring your RDS instance to require SSL.

KMS handles encryption **at rest**. TLS handles encryption **in transit**. Both are essential.

## KMS Key Types

### AWS Managed Keys

These are keys that AWS creates and manages on your behalf. They exist in your account but you can't control their rotation schedule, key policy, or deletion. They have aliases like `aws/s3`, `aws/ebs`, and `aws/rds`.

- **Free** — No monthly charge
- Automatically rotated every year
- You can't use them outside their specific service (the `aws/s3` key only works with S3)
- Good for: Getting started, basic compliance requirements

### Customer Managed Keys (CMKs)

These are keys you create and control. You set the key policy, decide who can use them, enable/disable them, and schedule deletion.

- **$1/month** per key
- **$0.03 per 10,000 API calls** (encrypt, decrypt, generate data key)
- You control the key policy (who can administer it, who can use it)
- You can enable automatic rotation (once per year)
- Can be used across services and even across accounts
- Good for: Production workloads, audit requirements, cross-account encryption

### Customer Managed Keys with Imported Key Material

You can import your own key material into KMS if your organization requires that keys are generated by a specific hardware security module (HSM). This is rare and adds complexity — for most use cases, letting KMS generate the key material is simpler and equally secure.

### Symmetric vs Asymmetric Keys

| Type | Description | Use Case |
|------|-------------|----------|
| **Symmetric** (AES-256) | Single key for encrypt and decrypt. The key never leaves KMS. | Default for most AWS service integrations. |
| **Asymmetric** (RSA or ECC) | Public key + private key pair. Public key can be downloaded. | Encrypt outside AWS, sign data, verify signatures. |

For encrypting data within AWS, you almost always use **symmetric keys**. Asymmetric keys are for scenarios where someone outside your AWS account needs to encrypt data that only you can decrypt.

## Envelope Encryption

KMS has a 4 KB limit on the data you can encrypt directly with a KMS key. For anything larger — files, database columns, application data — KMS uses **envelope encryption**.

Here's how it works:

1. You call `GenerateDataKey` on your KMS key
2. KMS returns two things:
   - A **plaintext data key** (use this to encrypt your data)
   - An **encrypted copy of the data key** (encrypted by the KMS key)
3. You encrypt your data with the plaintext data key
4. You store the **encrypted data** alongside the **encrypted data key**
5. You **discard the plaintext data key** from memory

To decrypt:
1. You send the **encrypted data key** to KMS
2. KMS decrypts it using the KMS key and returns the **plaintext data key**
3. You use the plaintext data key to decrypt your data

This way, the KMS key never leaves KMS, and you can encrypt data of any size. AWS services that integrate with KMS (S3, EBS, RDS) handle envelope encryption automatically — you just enable encryption and pick a key.

```
┌──────────────────────────────────────────────────────┐
│                     KMS Service                       │
│                                                       │
│  KMS Key (never leaves KMS)                          │
│       │                                               │
│       ├── GenerateDataKey → plaintext data key        │
│       │                   → encrypted data key        │
│       │                                               │
│       └── Decrypt(encrypted data key)                 │
│                   → plaintext data key                │
└──────────────────────────────────────────────────────┘

Your Application:
  1. Get data key from KMS
  2. Encrypt data with plaintext data key
  3. Store encrypted data + encrypted data key together
  4. Throw away plaintext data key
```

## KMS Integration with AWS Services

### S3

S3 supports three types of server-side encryption:

- **SSE-S3** — S3 manages the key entirely (default). Free, no KMS involvement.
- **SSE-KMS** — S3 uses a KMS key. You get audit trails in CloudTrail and can control the key.
- **SSE-C** — You provide the encryption key with each request. S3 never stores the key.

Enable SSE-KMS on a bucket:

```bash
aws s3api put-bucket-encryption \
  --bucket my-secure-bucket \
  --server-side-encryption-configuration '{
    "Rules": [
      {
        "ApplyServerSideEncryptionByDefault": {
          "SSEAlgorithm": "aws:kms",
          "KMSMasterKeyID": "arn:aws:kms:us-east-1:123456789012:key/your-key-id"
        },
        "BucketKeyEnabled": true
      }
    ]
  }'
```

The `BucketKeyEnabled: true` setting reduces KMS API calls (and costs) by generating a bucket-level key that's reused for objects in the bucket.

### EBS

EBS volumes support encryption by default. When you create an encrypted volume, EBS uses KMS to generate a data key and encrypts the entire volume.

```bash
aws ec2 create-volume \
  --availability-zone us-east-1a \
  --size 100 \
  --volume-type gp3 \
  --encrypted \
  --kms-key-id arn:aws:kms:us-east-1:123456789012:key/your-key-id
```

You can set encryption as the default for all new EBS volumes in a region:

```bash
aws ec2 enable-ebs-encryption-by-default
```

### RDS

RDS encryption protects the underlying storage, automated backups, read replicas, and snapshots:

```bash
aws rds create-db-instance \
  --db-instance-identifier my-database \
  --db-instance-class db.t3.medium \
  --engine postgres \
  --master-username admin \
  --master-user-password MyPassword123 \
  --allocated-storage 20 \
  --storage-encrypted \
  --kms-key-id arn:aws:kms:us-east-1:123456789012:key/your-key-id
```

> **Important:** You can't enable encryption on an existing unencrypted RDS instance. You have to create an encrypted snapshot and restore from it.

### DynamoDB

DynamoDB encrypts all tables at rest by default using an AWS owned key (free). You can switch to a customer managed KMS key for more control:

```bash
aws dynamodb update-table \
  --table-name MyTable \
  --sse-specification Enabled=true,SSEType=KMS,KMSMasterKeyId=your-key-id
```

## Hands-On: Creating a KMS Key and Encrypting Data

### Create a Customer Managed Key

```bash
aws kms create-key \
  --description "My application encryption key" \
  --key-usage ENCRYPT_DECRYPT \
  --key-spec SYMMETRIC_DEFAULT
```

Copy the `KeyId` from the output. Then create a human-friendly alias:

```bash
aws kms create-alias \
  --alias-name alias/my-app-key \
  --target-key-id your-key-id
```

Now you can reference the key as `alias/my-app-key` instead of the UUID.

### Encrypt Small Data Directly

For data under 4 KB (like a short secret):

```bash
aws kms encrypt \
  --key-id alias/my-app-key \
  --plaintext "my-secret-value" \
  --output text \
  --query CiphertextBlob > encrypted.txt
```

### Decrypt It

```bash
aws kms decrypt \
  --ciphertext-blob fileb://encrypted.txt \
  --output text \
  --query Plaintext | base64 --decode
```

### Generate a Data Key (for Envelope Encryption)

```bash
aws kms generate-data-key \
  --key-id alias/my-app-key \
  --key-spec AES_256
```

This returns both the `Plaintext` (base64-encoded data key) and `CiphertextBlob` (encrypted data key). Your application uses the plaintext key to encrypt data locally, then discards it and stores only the ciphertext blob alongside the encrypted data.

## Key Policies

Every KMS key has a **key policy** — a resource-based policy that defines who can manage and use the key. The default key policy gives the account root full access, which means IAM policies can then grant access to specific users and roles.

Here's a key policy that allows one role to use the key for encryption and another to administer it:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "EnableRootAccountAccess",
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::123456789012:root" },
      "Action": "kms:*",
      "Resource": "*"
    },
    {
      "Sid": "AllowKeyAdministration",
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::123456789012:role/KeyAdmin" },
      "Action": [
        "kms:Create*",
        "kms:Describe*",
        "kms:Enable*",
        "kms:List*",
        "kms:Put*",
        "kms:Update*",
        "kms:Revoke*",
        "kms:Disable*",
        "kms:Get*",
        "kms:Delete*",
        "kms:ScheduleKeyDeletion",
        "kms:CancelKeyDeletion"
      ],
      "Resource": "*"
    },
    {
      "Sid": "AllowKeyUsage",
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::123456789012:role/MyAppRole" },
      "Action": [
        "kms:Encrypt",
        "kms:Decrypt",
        "kms:GenerateDataKey"
      ],
      "Resource": "*"
    }
  ]
}
```

This separates administration (managing the key) from usage (encrypting/decrypting data) — a security best practice.

## Key Rotation

Customer managed keys can be set to rotate automatically every year:

```bash
aws kms enable-key-rotation --key-id your-key-id
```

When a key rotates:
- KMS creates new key material
- The key ID and ARN stay the same
- Old key material is preserved (so data encrypted with old material can still be decrypted)
- New encryption operations use the new material

This is seamless — you don't need to re-encrypt existing data.

## Pricing

| Component | Cost |
|-----------|------|
| Customer managed key | $1/month per key |
| API requests (encrypt, decrypt, etc.) | $0.03 per 10,000 requests |
| AWS managed keys | Free |
| Automatic key rotation | Free (for customer managed keys) |

For a typical application with one or two customer managed keys and moderate API usage, KMS costs a few dollars per month.

## Key Takeaways

- **Encryption at rest** protects stored data; **encryption in transit** (TLS) protects data on the network
- **AWS managed keys** are free and automatic; **customer managed keys** give you control over policies and rotation ($1/month)
- **Envelope encryption** lets you encrypt data of any size — KMS encrypts a data key, you encrypt your data with that key
- Most AWS services (S3, EBS, RDS, DynamoDB) integrate directly with KMS — just enable encryption and pick a key
- **Key policies** control who can administer and use a key — separate these roles for security
- Enable **automatic key rotation** on customer managed keys for compliance
- Use **S3 bucket keys** to reduce KMS API call costs when encrypting many objects
