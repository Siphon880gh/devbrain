# IAM Roles for Services

When an AWS service needs to access another service on your behalf — like an EC2 instance reading from S3, or a Lambda function writing to DynamoDB — it needs credentials. The right way to provide those credentials is through **IAM roles**, not hardcoded access keys.

## What You'll Learn

- Why hardcoding access keys is dangerous and unnecessary
- How IAM roles provide temporary credentials to AWS services
- What trust policies are and how AssumeRole works
- How to set up roles for EC2 (instance profiles), Lambda (execution roles), and ECS (task roles)
- How to create and attach a service role using the CLI

## Why Roles Instead of Access Keys

Imagine you have an EC2 instance that needs to read files from an S3 bucket. You could create an IAM user, generate access keys, and paste them into a config file on the instance. This works, but it's a terrible idea:

- **Keys don't expire** — If someone steals them, they have access until you manually revoke them
- **Keys get leaked** — Developers accidentally commit them to Git, paste them in Slack, or leave them in logs
- **Keys are hard to rotate** — You have to update every application that uses them
- **Keys give permanent access** — There's no time limit on what they can do

IAM roles solve all of these problems. When a service assumes a role, it receives **temporary credentials** that:

- Automatically expire (default 1 hour, configurable up to 12 hours)
- Are automatically rotated by AWS — your code never sees or stores them
- Are delivered through the instance metadata service (for EC2) or the execution environment (for Lambda)
- Require no manual management

Your code doesn't change at all. The AWS SDK automatically picks up role credentials from the environment. You write `s3.getObject(...)` and it just works — no access keys anywhere in your code.

## How Service Roles Work

Every IAM role has two components:

### 1. Trust Policy (Who Can Assume This Role)

The trust policy is a JSON document that specifies **which entities** are allowed to assume the role. For a service role, you're saying "I trust this AWS service to use this role."

Here's a trust policy that allows the Lambda service to assume a role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

The `Principal` field identifies who can assume the role. For services, it's the service's identifier (like `lambda.amazonaws.com`, `ec2.amazonaws.com`, or `ecs-tasks.amazonaws.com`).

### 2. Permission Policy (What the Role Can Do)

This is a standard IAM policy that defines what actions the role is allowed to perform. It's the same policy format you learned in the previous lesson.

When a service assumes the role, it gets both pieces: the trust policy says "yes, you're allowed to become this role" and the permission policy says "here's what you can do while wearing this hat."

## Trust Policies and Assume Role

Under the hood, all role assumption works through the **AWS Security Token Service (STS)** and the `AssumeRole` API. Here's what happens:

1. A Lambda function starts executing
2. The Lambda service calls `sts:AssumeRole` on the function's execution role
3. STS checks the trust policy — does it allow `lambda.amazonaws.com`? Yes.
4. STS returns **temporary credentials**: an access key ID, secret access key, and session token
5. These credentials are injected into the Lambda execution environment
6. The AWS SDK in your code automatically uses them
7. After the credentials expire, new ones are fetched automatically

You never interact with STS directly for service roles — AWS handles it. But understanding the flow helps when debugging permission errors.

## Roles for Different Services

### EC2 Instance Profiles

EC2 instances can't directly assume a role. Instead, you attach a role through an **instance profile** — a container that holds the role and makes it available to the instance.

When you create a role for EC2 in the console, AWS creates the instance profile automatically. Via CLI, you need to create both:

```bash
aws iam create-role \
  --role-name EC2-S3-ReadOnly \
  --assume-role-policy-document file://ec2-trust-policy.json

aws iam create-instance-profile \
  --instance-profile-name EC2-S3-ReadOnly

aws iam add-role-to-instance-profile \
  --instance-profile-name EC2-S3-ReadOnly \
  --role-name EC2-S3-ReadOnly
```

Where `ec2-trust-policy.json` contains:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Then attach to a running instance:

```bash
aws ec2 associate-iam-instance-profile \
  --instance-id i-0abcd1234efgh5678 \
  --iam-instance-profile Name=EC2-S3-ReadOnly
```

Code running on that EC2 instance can now access S3 without any credentials in the code. The SDK automatically retrieves temporary credentials from the instance metadata service at `http://169.254.169.254`.

### Lambda Execution Roles

Every Lambda function must have an **execution role**. This role defines what AWS services the function can access. Lambda also needs permission to write logs to CloudWatch, so most execution roles include that.

Here's a complete example — creating a role for a Lambda function that reads and writes to a specific DynamoDB table:

**Step 1:** Create the trust policy file (`lambda-trust-policy.json`):

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

**Step 2:** Create the role:

```bash
aws iam create-role \
  --role-name LambdaDynamoDBRole \
  --assume-role-policy-document file://lambda-trust-policy.json
```

**Step 3:** Attach the basic Lambda execution policy (for CloudWatch logs):

```bash
aws iam attach-role-policy \
  --role-name LambdaDynamoDBRole \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole
```

**Step 4:** Create and attach a custom policy for DynamoDB access. Save as `dynamo-policy.json`:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "dynamodb:GetItem",
        "dynamodb:PutItem",
        "dynamodb:UpdateItem",
        "dynamodb:Query"
      ],
      "Resource": "arn:aws:dynamodb:us-east-1:123456789012:table/Orders"
    }
  ]
}
```

```bash
aws iam put-role-policy \
  --role-name LambdaDynamoDBRole \
  --policy-name DynamoDBAccess \
  --policy-document file://dynamo-policy.json
```

**Step 5:** Assign the role when creating the Lambda function:

```bash
aws lambda create-function \
  --function-name ProcessOrders \
  --runtime nodejs20.x \
  --role arn:aws:iam::123456789012:role/LambdaDynamoDBRole \
  --handler index.handler \
  --zip-file fileb://function.zip
```

Inside your Lambda code, the DynamoDB call requires no credentials:

```javascript
const { DynamoDBClient, GetItemCommand } = require("@aws-sdk/client-dynamodb");
const client = new DynamoDBClient({});

exports.handler = async (event) => {
  const result = await client.send(new GetItemCommand({
    TableName: "Orders",
    Key: { orderId: { S: event.orderId } }
  }));
  return result.Item;
};
```

The SDK automatically uses the execution role's temporary credentials.

### ECS Task Roles

ECS containers use **task roles** to access AWS services. This is configured in the task definition:

```json
{
  "family": "my-api",
  "taskRoleArn": "arn:aws:iam::123456789012:role/ECSTaskRole",
  "executionRoleArn": "arn:aws:iam::123456789012:role/ecsTaskExecutionRole",
  "containerDefinitions": [...]
}
```

There are two roles in ECS:
- **Task role** (`taskRoleArn`) — Permissions for your application code (e.g., accessing S3 or DynamoDB)
- **Execution role** (`executionRoleArn`) — Permissions for the ECS agent to pull container images from ECR and write logs to CloudWatch

The trust policy for ECS task roles uses `ecs-tasks.amazonaws.com` as the principal.

## Hands-On: Attaching a Role to an EC2 Instance

Let's walk through a practical scenario: you have an EC2 instance running a web application that needs to read objects from an S3 bucket called `my-app-assets`.

**1. Create the trust policy** (`ec2-trust.json`):

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Service": "ec2.amazonaws.com" },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

**2. Create the role and instance profile:**

```bash
aws iam create-role \
  --role-name WebAppS3Reader \
  --assume-role-policy-document file://ec2-trust.json

aws iam create-instance-profile \
  --instance-profile-name WebAppS3Reader

aws iam add-role-to-instance-profile \
  --instance-profile-name WebAppS3Reader \
  --role-name WebAppS3Reader
```

**3. Attach an S3 read policy:**

```bash
aws iam attach-role-policy \
  --role-name WebAppS3Reader \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
```

**4. Attach the instance profile to your EC2 instance:**

```bash
aws ec2 associate-iam-instance-profile \
  --instance-id i-0abcd1234efgh5678 \
  --iam-instance-profile Name=WebAppS3Reader
```

**5. SSH into the instance and test** — no credentials needed:

```bash
aws s3 ls s3://my-app-assets/
```

If it returns the bucket contents, the role is working. The CLI on the instance automatically picks up the role credentials.

## Key Takeaways

- **Never hardcode access keys** — Use IAM roles for any service-to-service communication
- Roles provide **temporary credentials** that are automatically rotated by AWS
- Every role has a **trust policy** (who can assume it) and a **permission policy** (what it can do)
- **EC2** uses instance profiles to deliver role credentials via the metadata service
- **Lambda** requires an execution role — always include `AWSLambdaBasicExecutionRole` for logging
- **ECS** has separate task roles (for your app) and execution roles (for the ECS agent)
- The **AWS SDK automatically picks up role credentials** — your code doesn't need to handle credentials at all
