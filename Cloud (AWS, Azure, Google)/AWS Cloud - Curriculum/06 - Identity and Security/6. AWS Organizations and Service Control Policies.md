# AWS Organizations and Service Control Policies

As your company grows, running everything in a single AWS account becomes a problem. One developer accidentally deletes a production database. A test workload racks up a $5,000 bill that gets mixed with production costs. A compromised access key in the dev environment exposes production resources. The solution: **multiple AWS accounts** managed through **AWS Organizations**.

## What You'll Learn

- Why companies use multiple AWS accounts instead of one
- How AWS Organizations structures accounts with Organizational Units (OUs)
- How Service Control Policies (SCPs) act as guardrails across accounts
- Practical SCP examples (region restrictions, service restrictions)
- How consolidated billing works
- What AWS Control Tower does

## Why Multi-Account Architecture

A single AWS account for everything seems simple, but it creates real problems:

**Security risk** — If one IAM user is compromised, they could potentially access everything. Separate accounts create hard security boundaries. An admin in the dev account has zero access to production by default.

**Billing chaos** — When dev, staging, and production share an account, you can't easily see what costs belong to what. Separate accounts give you clean cost attribution per environment or team.

**Service limits** — AWS applies service quotas per account. If your dev team runs a load test that hits the Lambda concurrency limit, it could take down production in the same account.

**Compliance** — Regulations (HIPAA, PCI-DSS, SOC 2) often require isolation between workloads handling sensitive data and everything else.

A common multi-account structure looks like this:

```
Management Account (billing, organization admin — no workloads)
├── Security OU
│   ├── Log Archive Account (CloudTrail, Config logs)
│   └── Audit Account (security tools, GuardDuty)
├── Production OU
│   ├── Prod-App Account
│   └── Prod-Data Account
├── Development OU
│   ├── Dev Account
│   └── Staging Account
└── Sandbox OU
    └── Experiments Account (developers can try anything)
```

## How AWS Organizations Works

### The Management Account

When you create an organization, the account you create it from becomes the **management account** (formerly called the "master account"). This account:

- Pays the bills for all member accounts (consolidated billing)
- Creates and manages member accounts
- Applies Service Control Policies
- Cannot be restricted by SCPs (it's always the admin)

> **Best practice:** Don't run any workloads in the management account. It should only be used for organization administration and billing.

### Creating an Organization

```bash
aws organizations create-organization --feature-set ALL
```

The `ALL` feature set enables SCPs and all organization features. The alternative, `CONSOLIDATED_BILLING`, only enables shared billing without policy features.

### Creating Member Accounts

```bash
aws organizations create-account \
  --account-name "Production" \
  --email "aws-prod@yourcompany.com"
```

Each account needs a unique email address. Many companies use email aliases (like `aws-prod@yourcompany.com`, `aws-dev@yourcompany.com`) to manage this.

You can also **invite** existing AWS accounts to join your organization:

```bash
aws organizations invite-account-to-organization \
  --target '{"Id":"987654321098","Type":"ACCOUNT"}'
```

## Organizational Units and Hierarchy

**Organizational Units (OUs)** are containers for grouping accounts. They form a hierarchy, and you attach SCPs at the OU level to apply policies to all accounts within.

### Create OUs

```bash
aws organizations create-organizational-unit \
  --parent-id r-abc1 \
  --name "Production"
```

The `--parent-id` is the root ID of your organization (starts with `r-`). You can nest OUs up to five levels deep.

```bash
aws organizations create-organizational-unit \
  --parent-id r-abc1 \
  --name "Development"

aws organizations create-organizational-unit \
  --parent-id r-abc1 \
  --name "Sandbox"
```

### Move Accounts into OUs

```bash
aws organizations move-account \
  --account-id 111111111111 \
  --source-parent-id r-abc1 \
  --destination-parent-id ou-abc1-prod1234
```

### List Your Organization Structure

```bash
aws organizations list-roots
aws organizations list-organizational-units-for-parent --parent-id r-abc1
aws organizations list-accounts-for-parent --parent-id ou-abc1-prod1234
```

## Service Control Policies (SCPs)

SCPs are the most powerful feature of AWS Organizations. They set **permission guardrails** that restrict what actions accounts within an OU (or individual accounts) can perform — even if an IAM user or role in that account has the permission.

### How SCPs Work

- SCPs don't **grant** permissions — they **restrict** them
- They set the maximum allowed permissions for an account
- Even an account's root user is restricted by SCPs
- The management account is never affected by SCPs
- SCPs use the same JSON syntax as IAM policies

Think of SCPs as a fence around the account. IAM policies inside the account determine what users can do, but they can never exceed what the SCP allows.

```
SCP says: "You can use EC2, S3, Lambda, DynamoDB"
IAM policy says: "This user has AdministratorAccess"
Result: The user can only administer EC2, S3, Lambda, DynamoDB
```

### SCP Strategy: Deny List vs Allow List

**Deny list approach (recommended):** Start with the default `FullAWSAccess` SCP and add explicit deny statements for things you want to block. This is easier to manage because new AWS services are allowed by default.

**Allow list approach:** Remove the `FullAWSAccess` SCP and create an SCP that explicitly allows only the services you want. This is more restrictive but requires updating the SCP every time you need a new service.

### Example: Deny All Regions Except Approved Ones

This SCP prevents accounts from launching resources in any region other than `us-east-1` and `us-west-2`:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyUnapprovedRegions",
      "Effect": "Deny",
      "Action": "*",
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "us-east-1",
            "us-west-2"
          ]
        },
        "ArnNotLike": {
          "aws:PrincipalARN": "arn:aws:iam::*:role/OrganizationAdmin"
        }
      }
    }
  ]
}
```

The `ArnNotLike` condition exempts a specific admin role so you're not locked out completely.

### Example: Prevent Leaving the Organization

Stop member accounts from removing themselves:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyLeaveOrganization",
      "Effect": "Deny",
      "Action": "organizations:LeaveOrganization",
      "Resource": "*"
    }
  ]
}
```

### Example: Deny Expensive Services

Prevent developers in sandbox accounts from launching expensive services:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyExpensiveServices",
      "Effect": "Deny",
      "Action": [
        "redshift:*",
        "emr:*",
        "sagemaker:*",
        "es:*",
        "ec2:RunInstances"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "ec2:InstanceType": [
            "t3.micro",
            "t3.small",
            "t3.medium"
          ]
        }
      }
    }
  ]
}
```

### Example: Require Encryption on S3 Buckets

Force all S3 buckets to have encryption enabled:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyUnencryptedS3",
      "Effect": "Deny",
      "Action": "s3:PutObject",
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "s3:x-amz-server-side-encryption": "aws:kms"
        }
      }
    }
  ]
}
```

## Hands-On: Creating an Organization and Applying an SCP

### Step 1: Enable SCPs

If you haven't already, enable SCPs for your organization:

```bash
aws organizations enable-policy-type \
  --root-id r-abc1 \
  --policy-type SERVICE_CONTROL_POLICY
```

### Step 2: Create the SCP

Save the region restriction policy as `deny-regions.json`, then create it:

```bash
aws organizations create-policy \
  --name "RestrictRegions" \
  --description "Only allow us-east-1 and us-west-2" \
  --type SERVICE_CONTROL_POLICY \
  --content file://deny-regions.json
```

Copy the `PolicyId` from the output.

### Step 3: Attach the SCP to an OU

```bash
aws organizations attach-policy \
  --policy-id p-abc123 \
  --target-id ou-abc1-prod1234
```

Now every account in the Production OU is restricted to only `us-east-1` and `us-west-2`. Any API call targeting another region is denied.

### Step 4: Verify

List policies attached to the OU:

```bash
aws organizations list-policies-for-target \
  --target-id ou-abc1-prod1234 \
  --filter SERVICE_CONTROL_POLICY
```

### Detach an SCP

If you need to remove a policy:

```bash
aws organizations detach-policy \
  --policy-id p-abc123 \
  --target-id ou-abc1-prod1234
```

## Consolidated Billing

All accounts in an organization share a single bill paid by the management account. This provides:

**Cost visibility** — You can see spending broken down by account, making it easy to track costs per team or environment.

**Volume discounts** — AWS pricing tiers (like S3 storage tiers or EC2 savings plans) are calculated across all accounts combined. If Account A uses 50 TB of S3 and Account B uses 50 TB, you get the volume pricing for 100 TB.

**Reserved Instance sharing** — Reserved Instances and Savings Plans purchased in one account can apply to matching usage in any account in the organization (unless you disable this).

Use **AWS Cost Explorer** to view costs by account:

```bash
aws ce get-cost-and-usage \
  --time-period Start=2025-01-01,End=2025-02-01 \
  --granularity MONTHLY \
  --metrics "BlendedCost" \
  --group-by Type=DIMENSION,Key=LINKED_ACCOUNT
```

## AWS Control Tower

Setting up a multi-account environment with the right guardrails, logging, and account provisioning is complex. **AWS Control Tower** automates this setup.

Control Tower provides:

- **Landing zone** — A pre-configured multi-account environment with best-practice structure
- **Account Factory** — A self-service portal for creating new accounts that automatically conform to your standards
- **Guardrails** — Pre-built SCPs and AWS Config rules (categorized as mandatory, strongly recommended, and elective)
- **Dashboard** — A single view of compliance across all accounts

Control Tower creates several accounts automatically:
- A **Log Archive** account for centralized CloudTrail and Config logs
- An **Audit** account for security tools and cross-account audit access

For teams starting fresh with AWS Organizations, Control Tower is the fastest way to get a well-architected multi-account setup. You can enable it from the AWS Console and it provisions everything in about 30 minutes.

> **Note:** Control Tower is primarily managed through the Console, not the CLI. It builds on top of Organizations, SCPs, AWS Config, and CloudTrail.

## Key Takeaways

- **Multiple accounts** provide hard security boundaries, clean billing, and isolated service limits
- **AWS Organizations** groups accounts into a hierarchy with the management account at the top
- **Organizational Units (OUs)** let you group accounts by purpose (production, development, sandbox)
- **SCPs restrict permissions** at the account level — even admin users can't exceed SCP boundaries
- The **deny list approach** (default FullAWSAccess + explicit denies) is easier to manage than allow lists
- Common SCPs: restrict regions, block expensive services, prevent leaving the organization, enforce encryption
- **Consolidated billing** combines usage across accounts for volume discounts and cost visibility
- **AWS Control Tower** automates multi-account setup with pre-built guardrails and account provisioning
