# Step Functions — Orchestrating Workflows with State Machines

Coordinate multi-step processes visually using AWS Step Functions and state machines.

## What You'll Learn

- Why orchestration matters and what problems it solves
- What state machines are and how Step Functions implements them
- The different state types: Task, Choice, Wait, Parallel, and Map
- Standard vs Express workflow types and when to use each
- How to handle errors and retries
- How to build a multi-step order processing workflow
- Amazon States Language (ASL) syntax

## Why Orchestration?

In the previous lessons, you learned about SQS, SNS, and EventBridge — services that let you pass messages and events between components. These work great for simple, one-step reactions: "When X happens, do Y."

But many real-world processes have multiple steps with branching logic:

1. Customer places an order
2. Validate the payment information
3. If payment is valid, charge the credit card; if not, notify the customer
4. If the charge succeeds, update inventory and ship the order
5. If any step fails, roll back previous steps and send an error notification

You could wire this together with a chain of Lambda functions triggering each other through SQS, but that approach gets messy fast. You lose visibility into where each order is in the process, error handling becomes scattered across multiple functions, and debugging failed orders means digging through logs from five different Lambda functions.

**AWS Step Functions** solves this by letting you define the entire workflow as a **state machine** — a visual, manageable sequence of steps with built-in error handling, retries, and branching logic. You see the whole flow in one place, and the Step Functions console shows you exactly where each execution is in real time.

## State Machine Concepts

A **state machine** is a model with a finite set of **states** and **transitions** between them. At any moment, the machine is in exactly one state. Based on the current state and the input it receives, it transitions to the next state.

In Step Functions, you define your state machine using **Amazon States Language (ASL)** — a JSON-based language that describes states, transitions, inputs, outputs, and error handling.

Here's the simplest possible state machine — one state that returns a result:

```json
{
  "Comment": "Hello World state machine",
  "StartAt": "SayHello",
  "States": {
    "SayHello": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:hello",
      "End": true
    }
  }
}
```

Key ASL concepts:

- **StartAt** — the name of the first state to execute
- **States** — an object containing all state definitions
- **End: true** — marks this state as a terminal state (the workflow stops here)
- **Next** — specifies which state to transition to (used instead of End)

## Task, Choice, Wait, and Parallel States

Step Functions provides several state types, each serving a different purpose.

### Task State

Performs work by calling an AWS service. This is the most common state type.

```json
{
  "ChargeCard": {
    "Type": "Task",
    "Resource": "arn:aws:lambda:us-east-1:123456789012:function:charge-card",
    "Next": "UpdateInventory"
  }
}
```

Task states can call Lambda functions, but they also integrate directly with over 200 AWS services — DynamoDB, SQS, SNS, ECS, Glue, and more — without needing a Lambda in between:

```json
{
  "SaveToDatabase": {
    "Type": "Task",
    "Resource": "arn:aws:states:::dynamodb:putItem",
    "Parameters": {
      "TableName": "Orders",
      "Item": {
        "orderId": {"S.$": "$.orderId"},
        "status": {"S": "confirmed"}
      }
    },
    "Next": "NotifyCustomer"
  }
}
```

### Choice State

Adds branching logic — like an if/else statement. Evaluates conditions against the input and transitions to different states accordingly.

```json
{
  "CheckOrderTotal": {
    "Type": "Choice",
    "Choices": [
      {
        "Variable": "$.total",
        "NumericGreaterThan": 1000,
        "Next": "RequireManagerApproval"
      },
      {
        "Variable": "$.isPrime",
        "BooleanEquals": true,
        "Next": "ApplyPrimeDiscount"
      }
    ],
    "Default": "ProcessNormally"
  }
}
```

### Wait State

Pauses the workflow for a set duration or until a specific time.

```json
{
  "WaitForShipment": {
    "Type": "Wait",
    "Seconds": 3600,
    "Next": "CheckShipmentStatus"
  }
}
```

Or wait until a specific timestamp:

```json
{
  "WaitUntilDelivery": {
    "Type": "Wait",
    "TimestampPath": "$.estimatedDelivery",
    "Next": "SendDeliveryNotification"
  }
}
```

### Parallel State

Runs multiple branches simultaneously and waits for all of them to complete before continuing.

```json
{
  "ProcessInParallel": {
    "Type": "Parallel",
    "Branches": [
      {
        "StartAt": "SendEmail",
        "States": {
          "SendEmail": {
            "Type": "Task",
            "Resource": "arn:aws:lambda:us-east-1:123456789012:function:send-email",
            "End": true
          }
        }
      },
      {
        "StartAt": "UpdateAnalytics",
        "States": {
          "UpdateAnalytics": {
            "Type": "Task",
            "Resource": "arn:aws:lambda:us-east-1:123456789012:function:update-analytics",
            "End": true
          }
        }
      }
    ],
    "Next": "Done"
  }
}
```

### Map State

Iterates over an array in the input and runs the same steps for each item — like a for-each loop.

```json
{
  "ProcessEachItem": {
    "Type": "Map",
    "ItemsPath": "$.orderItems",
    "ItemProcessor": {
      "ProcessorConfig": {
        "Mode": "INLINE"
      },
      "StartAt": "CheckStock",
      "States": {
        "CheckStock": {
          "Type": "Task",
          "Resource": "arn:aws:lambda:us-east-1:123456789012:function:check-stock",
          "End": true
        }
      }
    },
    "Next": "ConfirmOrder"
  }
}
```

## Standard vs Express Workflows

Step Functions offers two workflow types:

| Feature | Standard | Express |
|---------|----------|---------|
| **Max duration** | 1 year | 5 minutes |
| **Execution model** | Exactly-once | At-least-once (async) or at-most-once (sync) |
| **Execution history** | Full history in console | CloudWatch Logs only |
| **Pricing** | Per state transition | Per execution + duration |
| **Use case** | Long-running, auditable workflows | High-volume, short-duration processing |

**Standard workflows** are the default. Use them for business-critical processes where you need a complete audit trail and exactly-once execution — order processing, account onboarding, ETL pipelines.

**Express workflows** are for high-volume, event-processing workloads — IoT data ingestion, streaming transformations, real-time file processing. They can handle over 100,000 executions per second.

## Error Handling and Retries

Real-world workflows fail. Network timeouts, throttling, transient errors — Step Functions has built-in mechanisms to handle all of them.

### Retry

Add a `Retry` block to any Task state to automatically retry on specific errors:

```json
{
  "ChargeCard": {
    "Type": "Task",
    "Resource": "arn:aws:lambda:us-east-1:123456789012:function:charge-card",
    "Retry": [
      {
        "ErrorEquals": ["States.TaskFailed", "States.Timeout"],
        "IntervalSeconds": 2,
        "MaxAttempts": 3,
        "BackoffRate": 2.0
      }
    ],
    "Next": "UpdateInventory"
  }
}
```

This retries up to 3 times, starting with a 2-second wait and doubling each time (2s, 4s, 8s) — exponential backoff.

### Catch

If retries are exhausted, `Catch` routes the execution to a fallback state:

```json
{
  "ChargeCard": {
    "Type": "Task",
    "Resource": "arn:aws:lambda:us-east-1:123456789012:function:charge-card",
    "Retry": [
      {
        "ErrorEquals": ["States.TaskFailed"],
        "MaxAttempts": 3
      }
    ],
    "Catch": [
      {
        "ErrorEquals": ["States.ALL"],
        "Next": "HandlePaymentFailure"
      }
    ],
    "Next": "UpdateInventory"
  }
}
```

Common error types:

- `States.ALL` — matches any error
- `States.TaskFailed` — the task returned an error
- `States.Timeout` — the task exceeded its timeout
- Custom error strings thrown by your Lambda functions

## Hands-On: Build a Multi-Step Order Processing Workflow

Here's a complete state machine definition for an order processing pipeline:

```
Validate Order → Charge Card → (success?) → Update Inventory → Ship Order → Notify Customer
                                (failure?) → Notify Payment Failed
```

```json
{
  "Comment": "Order processing workflow",
  "StartAt": "ValidateOrder",
  "States": {
    "ValidateOrder": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:validate-order",
      "Next": "ChargeCard",
      "Catch": [{
        "ErrorEquals": ["ValidationError"],
        "Next": "NotifyInvalidOrder"
      }]
    },
    "ChargeCard": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:charge-card",
      "Retry": [{
        "ErrorEquals": ["States.TaskFailed"],
        "IntervalSeconds": 2,
        "MaxAttempts": 3,
        "BackoffRate": 2.0
      }],
      "Next": "CheckPaymentResult",
      "Catch": [{
        "ErrorEquals": ["States.ALL"],
        "Next": "NotifyPaymentFailed"
      }]
    },
    "CheckPaymentResult": {
      "Type": "Choice",
      "Choices": [{
        "Variable": "$.paymentStatus",
        "StringEquals": "approved",
        "Next": "FulfillOrder"
      }],
      "Default": "NotifyPaymentFailed"
    },
    "FulfillOrder": {
      "Type": "Parallel",
      "Branches": [
        {
          "StartAt": "UpdateInventory",
          "States": {
            "UpdateInventory": {
              "Type": "Task",
              "Resource": "arn:aws:states:::dynamodb:updateItem",
              "Parameters": {
                "TableName": "Inventory",
                "Key": {"itemId": {"S.$": "$.itemId"}},
                "UpdateExpression": "SET stock = stock - :qty",
                "ExpressionAttributeValues": {":qty": {"N.$": "$.quantity"}}
              },
              "End": true
            }
          }
        },
        {
          "StartAt": "ShipOrder",
          "States": {
            "ShipOrder": {
              "Type": "Task",
              "Resource": "arn:aws:lambda:us-east-1:123456789012:function:ship-order",
              "End": true
            }
          }
        }
      ],
      "Next": "NotifyCustomer"
    },
    "NotifyCustomer": {
      "Type": "Task",
      "Resource": "arn:aws:states:::sns:publish",
      "Parameters": {
        "TopicArn": "arn:aws:sns:us-east-1:123456789012:order-notifications",
        "Message.$": "States.Format('Order {} has been confirmed and shipped!', $.orderId)"
      },
      "End": true
    },
    "NotifyPaymentFailed": {
      "Type": "Task",
      "Resource": "arn:aws:states:::sns:publish",
      "Parameters": {
        "TopicArn": "arn:aws:sns:us-east-1:123456789012:order-notifications",
        "Message": "Payment failed. Please update your payment method."
      },
      "End": true
    },
    "NotifyInvalidOrder": {
      "Type": "Task",
      "Resource": "arn:aws:states:::sns:publish",
      "Parameters": {
        "TopicArn": "arn:aws:sns:us-east-1:123456789012:order-notifications",
        "Message": "Your order could not be validated. Please check your details."
      },
      "End": true
    }
  }
}
```

### Create the State Machine via CLI

```bash
aws stepfunctions create-state-machine \
  --name "order-processing" \
  --definition file://order-workflow.json \
  --role-arn arn:aws:iam::123456789012:role/StepFunctionsExecutionRole
```

### Start an Execution

```bash
aws stepfunctions start-execution \
  --state-machine-arn arn:aws:states:us-east-1:123456789012:stateMachine:order-processing \
  --input '{"orderId": "ORD-123", "itemId": "WIDGET-A", "quantity": 2, "total": 59.99}'
```

### Check Execution Status

```bash
aws stepfunctions describe-execution \
  --execution-arn arn:aws:states:us-east-1:123456789012:execution:order-processing:abc-123
```

Open the Step Functions console to see a visual diagram of the workflow with each state highlighted green (success), red (failed), or blue (in progress). This visual debugging is one of the biggest advantages of Step Functions over chaining Lambda functions manually.

## Key Takeaways

- **Step Functions orchestrates multi-step workflows** as state machines with visual monitoring
- **State types** cover all common patterns: Task (do work), Choice (branch), Wait (pause), Parallel (concurrent), Map (iterate)
- **Standard workflows** for long-running, auditable processes; **Express workflows** for high-volume, short-duration tasks
- **Built-in error handling** with Retry (exponential backoff) and Catch (fallback states) eliminates manual retry logic
- **Direct service integrations** let you call DynamoDB, SQS, SNS, and 200+ AWS services without a Lambda function in between
- **ASL (Amazon States Language)** is the JSON format for defining state machines — save it as a file and create the state machine via CLI or CloudFormation
- **Free tier**: 4,000 state transitions per month for Standard workflows
