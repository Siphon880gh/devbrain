# When to Use SQS vs SNS vs EventBridge vs Step Functions

A decision guide for choosing the right messaging or orchestration service for your architecture.

## What You'll Learn

- When to use each service and their core strengths
- How the four services compare across key dimensions
- Common architectural patterns that combine multiple services
- A decision flowchart to pick the right service for your use case

## Quick Comparison Table

| Feature | SQS | SNS | EventBridge | Step Functions |
|---------|-----|-----|-------------|----------------|
| **Pattern** | Point-to-point queue | Pub/sub broadcast | Event routing | Workflow orchestration |
| **Delivery** | Pull (consumers poll) | Push to subscribers | Push to targets | Managed execution |
| **Consumers** | 1 consumer per message | Many subscribers | Many targets via rules | Sequential/parallel steps |
| **Message retention** | Up to 14 days | No retention (immediate) | Archive up to indefinitely | Execution history stored |
| **Ordering** | FIFO available | FIFO available | Ordered per rule | Defined by state machine |
| **Max message size** | 256 KB | 256 KB | 256 KB | 256 KB input/output |
| **Filtering** | None (consumer-side) | Attribute-based | Rich content-based | Choice state logic |
| **Error handling** | DLQ, visibility timeout | DLQ, retry policies | DLQ, retry policies | Retry, Catch, fallback states |
| **Throughput** | Nearly unlimited | Nearly unlimited | Thousands/sec per bus | Thousands/sec (Express) |
| **Free tier** | 1M requests/month | 1M publishes/month | — | 4,000 transitions/month |

## SQS — Best For

**Point-to-point messaging and workload buffering.**

Use SQS when you need to decouple a producer from a single consumer, especially when the consumer might be slower than the producer or might go offline temporarily.

**Choose SQS when:**

- One service produces work, one service consumes it
- You need to buffer requests during traffic spikes (the queue absorbs the burst)
- You need messages to persist if the consumer is temporarily unavailable
- You want consumers to process at their own pace
- You need exactly-once processing with FIFO queues

**Real-world examples:**

- A web API places image-resize jobs in a queue; worker EC2 instances process them at their own speed
- An order service drops payment requests into a queue; the payment processor handles them one at a time
- A data pipeline buffers records before batch-inserting into a database

**Key mental model**: SQS is a **buffer** between two services. Think of it as a to-do list — items go in, one worker pulls them off and completes them.

## SNS — Best For

**Broadcasting a message to multiple subscribers simultaneously.**

Use SNS when a single event needs to reach multiple destinations, and you want push-based delivery without the complexity of event routing rules.

**Choose SNS when:**

- One event needs to notify multiple services at the same time
- You need to send notifications to humans (email, SMS)
- You want simple attribute-based filtering per subscriber
- You're building the fan-out pattern (SNS → multiple SQS queues)

**Real-world examples:**

- A new user signs up → send a welcome email, notify the analytics service, and trigger a setup Lambda
- A deployment completes → notify the team via Slack webhook, email, and SMS
- An alarm fires in CloudWatch → SNS delivers the alert to an on-call engineer's phone and triggers an auto-remediation Lambda

**Key mental model**: SNS is a **megaphone**. You shout once, and everyone who's listening hears it.

## EventBridge — Best For

**Routing events to targets based on content, with rich filtering and multiple source types.**

Use EventBridge when you need intelligent event routing — matching events by their content and sending them to different targets based on what's in the event. It's also the right choice when you need to react to AWS service events or integrate with SaaS providers.

**Choose EventBridge when:**

- You need to route events based on their content (not just attributes)
- You want to react to AWS service events (S3 uploads, EC2 state changes, IAM logins)
- You need scheduled tasks (cron jobs, periodic triggers)
- You want archive and replay for debugging or reprocessing
- You're building an event-driven architecture with many producers and consumers
- You need cross-account or cross-region event routing

**Real-world examples:**

- S3 file upload → EventBridge matches the file extension → routes `.jpg` to an image processor and `.csv` to a data importer
- EC2 instance terminates → EventBridge triggers a cleanup Lambda and notifies the ops team
- Every hour on weekdays → EventBridge runs a report-generation Step Function

**Key mental model**: EventBridge is a **smart router**. Events flow in, rules inspect them, and each event gets directed to exactly the right destination.

## Step Functions — Best For

**Multi-step workflows that need coordination, branching, error handling, and visibility.**

Use Step Functions when your process has multiple sequential or parallel steps, conditional logic, wait states, or complex error handling that would be painful to implement with just queues and events.

**Choose Step Functions when:**

- Your process has multiple steps that must execute in a specific order
- You need branching logic (if this, then that)
- You need to wait for a period of time or for a human approval
- You need built-in retry with exponential backoff
- You want visual monitoring of where each execution is in the process
- You need to coordinate parallel tasks and wait for all to complete

**Real-world examples:**

- Order processing: validate → charge → fulfill → ship → notify (with error handling at each step)
- User onboarding: create account → verify email → wait for verification → provision resources → send welcome kit
- ETL pipeline: extract from S3 → transform with Lambda → load into Redshift → notify completion
- Machine learning: preprocess data → train model → evaluate → if accuracy > threshold → deploy model

**Key mental model**: Step Functions is a **flowchart executor**. You draw the process, and AWS runs it.

## Common Patterns and Combinations

These services are frequently combined. Here are the patterns you'll see most often in production:

### Pattern 1: SNS + SQS Fan-Out

**When**: One event needs to reach multiple independent consumers, each needing their own buffer.

```
Order Placed → [SNS Topic] → [SQS: email-queue]     → Email Lambda
                            → [SQS: inventory-queue] → Inventory Lambda
                            → [SQS: analytics-queue] → Analytics Lambda
```

Each consumer has its own queue, so if one consumer is slow or fails, the others are unaffected. This is the single most common integration pattern on AWS.

### Pattern 2: EventBridge + Lambda Event Processing

**When**: AWS service events or custom app events need to trigger different actions based on content.

```
S3 Upload    → [EventBridge] → Rule: *.jpg → Image Processor Lambda
EC2 Stopped  → [EventBridge] → Rule: prod instances → Restart Lambda
Custom Event → [EventBridge] → Rule: high priority → PagerDuty via HTTP
```

EventBridge acts as the central nervous system — every event flows through it, and rules determine what happens.

### Pattern 3: Step Functions for Complex Workflows

**When**: A process has multiple steps, branching, error handling, and needs visibility.

```
API Gateway → Step Functions → Validate (Lambda)
                             → Charge (Lambda)
                             → Choice: approved?
                               → Yes: Ship (Lambda) → Notify (SNS)
                               → No: Refund (Lambda) → Notify (SNS)
```

### Pattern 4: EventBridge + Step Functions

**When**: An event should trigger a multi-step workflow.

```
S3 Upload → [EventBridge Rule] → Step Functions → Extract metadata
                                                 → Generate thumbnails
                                                 → Update database
                                                 → Notify user
```

### Pattern 5: Step Functions + SQS for Batch Processing

**When**: A workflow needs to process items from a queue with orchestrated steps.

```
SQS Queue → Lambda (trigger) → Step Functions → Process item
                                              → Validate result
                                              → Store in DynamoDB
                                              → Send confirmation
```

## Decision Flowchart

Use this text-based flowchart to pick the right service:

```
START: What does your system need to do?
│
├─ "Send a message from one service to one other service"
│   └─→ Does the consumer need to process at its own pace?
│       ├─ Yes → SQS
│       └─ No  → Consider direct invocation (Lambda, API call)
│
├─ "Broadcast a message to multiple services"
│   └─→ Do you need content-based routing (different targets for different events)?
│       ├─ Yes → EventBridge
│       └─ No  → Do you need simple fan-out to known subscribers?
│           ├─ Yes → SNS (+ SQS queues for buffered fan-out)
│           └─ No  → EventBridge (more flexible long-term)
│
├─ "React to AWS service events (S3, EC2, etc.)"
│   └─→ EventBridge (AWS events go to the default bus automatically)
│
├─ "Run something on a schedule (cron job)"
│   └─→ EventBridge Scheduler or EventBridge scheduled rules
│
├─ "Coordinate multiple steps in a specific order"
│   └─→ Does the process have branching, retries, or wait states?
│       ├─ Yes → Step Functions
│       └─ No  → Simple chain: SQS or EventBridge → Lambda → next step
│
└─ "Process a high volume of events with routing"
    └─→ EventBridge (or Kinesis if you need real-time stream processing)
```

### Quick Rules of Thumb

1. **One producer, one consumer** → SQS
2. **One event, many consumers** → SNS
3. **Route events by content** → EventBridge
4. **Multi-step process with logic** → Step Functions
5. **Need a cron job** → EventBridge Scheduler
6. **Fan-out with buffering** → SNS + SQS
7. **React to AWS events** → EventBridge
8. **Not sure?** Start with EventBridge — it's the most flexible and you can always add SQS queues as targets if you need buffering.

## Key Takeaways

- **SQS** is a buffer for point-to-point messaging — one producer, one consumer, with message persistence and retry via DLQ
- **SNS** is a broadcast system — one message pushed to many subscribers simultaneously, with simple attribute-based filtering
- **EventBridge** is a smart event router — matches events by their content using powerful patterns, integrates with 100+ AWS services and SaaS providers, supports scheduling
- **Step Functions** is a workflow engine — coordinates multi-step processes with branching, parallel execution, error handling, and visual monitoring
- These services are **complementary, not competing** — most production architectures combine two or more of them
- **Start simple**: if you just need to decouple two services, use SQS. Add complexity (SNS, EventBridge, Step Functions) only when your requirements demand it
