# SNS — Simple Notification Service

Push notifications to multiple subscribers using pub/sub messaging and the fan-out pattern.

## Pub/Sub Messaging Model

In the previous lesson, SQS gave us **point-to-point** messaging: one producer sends a message, one consumer receives it. But what if a single event needs to reach multiple services?

For example, when a customer places an order, you might need to:
- Send a confirmation email
- Update the inventory database
- Notify the warehouse to start packing
- Log the event for analytics

You could have the order service call each of these services directly, but that creates tight coupling — the order service needs to know about every downstream service and handle failures for each one.

The **pub/sub** (publish/subscribe) pattern solves this. A **publisher** sends a message to a **topic** — a named channel. **Subscribers** register interest in that topic and receive a copy of every message published to it. The publisher doesn't know or care who the subscribers are.

```
                           ┌─→ Email Service
                           │
Order Service → [Topic] ───┼─→ Inventory Service
                           │
                           ├─→ Warehouse Service
                           │
                           └─→ Analytics Service
```

**Amazon SNS** is AWS's fully managed pub/sub service. It handles message delivery, retries, and fan-out to millions of subscribers. You don't run any servers.

## Topics and Subscriptions

### Topics

A **topic** is a communication channel. When you publish a message to a topic, SNS delivers a copy to every active subscription on that topic.

Topics have an Amazon Resource Name (ARN) like `arn:aws:sns:us-east-1:123456789012:order-events`.

### Subscription Types

SNS supports several subscription protocols — this is where it gets powerful:

| Protocol | Description | Use Case |
|----------|-------------|----------|
| **Email** | Sends an email to the subscriber | Alerts to humans |
| **SMS** | Sends a text message | Urgent alerts, 2FA |
| **Lambda** | Invokes a Lambda function | Serverless processing |
| **SQS** | Delivers to an SQS queue | Buffered processing |
| **HTTP/HTTPS** | POSTs to a URL endpoint | Webhooks, external services |
| **Firehose** | Delivers to a Kinesis Data Firehose stream | Data archival, analytics |

A single topic can have subscriptions of different types simultaneously. You might have one topic that sends email to an admin, triggers a Lambda, and delivers to an SQS queue — all from one published message.

### Creating a Topic and Subscribing via CLI

Create a topic:

```bash
aws sns create-topic --name order-events
```

Output:

```json
{
    "TopicArn": "arn:aws:sns:us-east-1:123456789012:order-events"
}
```

Subscribe an email address (the subscriber must confirm via a link in the email):

```bash
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --protocol email \
  --notification-endpoint admin@example.com
```

Subscribe an SQS queue:

```bash
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --protocol sqs \
  --notification-endpoint arn:aws:sqs:us-east-1:123456789012:order-processing-queue
```

Subscribe a Lambda function:

```bash
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --protocol lambda \
  --notification-endpoint arn:aws:lambda:us-east-1:123456789012:function:process-order
```

Publish a message:

```bash
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --subject "New Order" \
  --message '{"orderId": "12345", "amount": 59.99, "status": "placed"}'
```

Every subscriber receives a copy of that message within seconds.

## Fan-Out Pattern (SNS + SQS)

The **fan-out pattern** is one of the most common architectures on AWS. It combines SNS with multiple SQS queues so that a single event gets delivered to multiple independent consumers, each with their own queue.

Why not just subscribe Lambda functions directly? Because SQS gives you **buffering and retry**. If a consumer goes down, messages wait safely in its queue. Each consumer processes at its own pace.

```
                                ┌─→ [SQS: email-queue]     → Email Lambda
                                │
Order Service → [SNS Topic] ────┼─→ [SQS: inventory-queue] → Inventory Lambda
                                │
                                └─→ [SQS: analytics-queue] → Analytics Lambda
```

Each SQS queue operates independently. If the analytics Lambda is slow, it doesn't affect email delivery or inventory updates. If the email Lambda crashes, its messages sit in the email queue until it recovers.

### Setting Up Fan-Out

```bash
# 1. Create the SNS topic
aws sns create-topic --name order-events

# 2. Create the SQS queues
aws sqs create-queue --queue-name email-queue
aws sqs create-queue --queue-name inventory-queue
aws sqs create-queue --queue-name analytics-queue

# 3. Subscribe each queue to the topic
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --protocol sqs \
  --notification-endpoint arn:aws:sqs:us-east-1:123456789012:email-queue

aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --protocol sqs \
  --notification-endpoint arn:aws:sqs:us-east-1:123456789012:inventory-queue

aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --protocol sqs \
  --notification-endpoint arn:aws:sqs:us-east-1:123456789012:analytics-queue
```

**Important**: The SQS queues need a resource policy that allows the SNS topic to send messages to them. You set this with `aws sqs set-queue-attributes` using the `Policy` attribute.

## Message Filtering

Without filtering, every subscriber gets every message. But what if the email service only cares about high-value orders, and the inventory service only cares about orders with physical goods?

**Message filtering** lets each subscription define a **filter policy** — a JSON document that specifies which messages it wants to receive based on message attributes. Messages that don't match the filter are not delivered to that subscriber.

### Publishing with Attributes

```bash
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --message '{"orderId": "12345", "amount": 250.00}' \
  --message-attributes '{
    "orderType": {"DataType": "String", "StringValue": "physical"},
    "orderValue": {"DataType": "Number", "StringValue": "250"}
  }'
```

### Setting a Filter Policy on a Subscription

Only receive messages where `orderType` is "physical":

```bash
aws sns set-subscription-attributes \
  --subscription-arn arn:aws:sns:us-east-1:123456789012:order-events:abc-123 \
  --attribute-name FilterPolicy \
  --attribute-value '{"orderType": ["physical"]}'
```

Only receive messages where `orderValue` is greater than 100:

```bash
aws sns set-subscription-attributes \
  --subscription-arn arn:aws:sns:us-east-1:123456789012:order-events:abc-123 \
  --attribute-name FilterPolicy \
  --attribute-value '{"orderValue": [{"numeric": [">", 100]}]}'
```

Filter policies support exact match, prefix match, numeric comparisons, and "anything-but" logic. Filtering happens on the SNS side before delivery, so your consumers don't waste compute processing irrelevant messages.

## Hands-On: Create a Topic and Subscribe an SQS Queue

Here's a complete walkthrough using Python (boto3) that publishes an event and has it delivered to an SQS queue via SNS:

```python
import boto3
import json
import time

sns = boto3.client('sns', region_name='us-east-1')
sqs = boto3.client('sqs', region_name='us-east-1')

# Create the SNS topic
topic = sns.create_topic(Name='demo-notifications')
topic_arn = topic['TopicArn']
print(f"Topic ARN: {topic_arn}")

# Create the SQS queue
queue = sqs.create_queue(QueueName='demo-subscriber')
queue_url = queue['QueueUrl']

# Get the queue ARN for the subscription
attrs = sqs.get_queue_attributes(QueueUrl=queue_url, AttributeNames=['QueueArn'])
queue_arn = attrs['Attributes']['QueueArn']

# Allow SNS to send messages to this queue (resource policy)
policy = {
    "Version": "2012-10-17",
    "Statement": [{
        "Effect": "Allow",
        "Principal": {"Service": "sns.amazonaws.com"},
        "Action": "sqs:SendMessage",
        "Resource": queue_arn,
        "Condition": {"ArnEquals": {"aws:SourceArn": topic_arn}}
    }]
}
sqs.set_queue_attributes(
    QueueUrl=queue_url,
    Attributes={'Policy': json.dumps(policy)}
)

# Subscribe the SQS queue to the SNS topic
sns.subscribe(
    TopicArn=topic_arn,
    Protocol='sqs',
    Endpoint=queue_arn
)
print("SQS queue subscribed to SNS topic.")

# Publish a message
sns.publish(
    TopicArn=topic_arn,
    Subject='Test Event',
    Message=json.dumps({'event': 'user_signup', 'userId': 'u-9876'})
)
print("Message published!")

# Wait a moment for delivery, then read from the queue
time.sleep(2)
response = sqs.receive_message(QueueUrl=queue_url, WaitTimeSeconds=5)
if 'Messages' in response:
    for msg in response['Messages']:
        sns_message = json.loads(msg['Body'])
        payload = json.loads(sns_message['Message'])
        print(f"Received event: {payload}")
        sqs.delete_message(QueueUrl=queue_url, ReceiptHandle=msg['ReceiptHandle'])
```

Notice that when SNS delivers to SQS, it wraps the original message in an SNS envelope. The original message is in the `Message` field of the outer JSON.

## Key Takeaways

- **SNS is pub/sub** — one message, many subscribers, push-based delivery
- **Topics** are named channels; **subscriptions** define who gets the messages and how (email, SQS, Lambda, HTTP, SMS)
- **Fan-out with SNS + SQS** is a foundational AWS pattern — it gives each consumer an independent, buffered queue
- **Message filtering** lets subscribers receive only messages matching specific attributes, reducing unnecessary processing
- **SNS delivers immediately** — unlike SQS, messages aren't stored. If a subscriber is down, the message may be lost (unless delivered to SQS, which buffers it)
- **SNS vs SQS**: SNS pushes to many subscribers; SQS holds messages for one consumer. They're often used together.
- **Free tier**: 1 million SNS publishes per month, 100,000 HTTP notifications, and 1,000 email notifications are free
