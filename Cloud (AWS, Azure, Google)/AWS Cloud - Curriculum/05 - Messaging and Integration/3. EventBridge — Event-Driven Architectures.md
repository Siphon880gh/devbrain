# EventBridge — Event-Driven Architectures and Rules

Build loosely-coupled, event-driven systems with EventBridge's serverless event bus.

## What You'll Learn

- What event-driven architecture is and why it matters
- How EventBridge's event buses, rules, and targets work together
- How to write event patterns for precise filtering
- How to create scheduled rules using cron and rate expressions
- How to route an S3 event through EventBridge to a Lambda function
- How EventBridge differs from SNS
- CLI commands for creating and managing rules

## What is Event-Driven Architecture?

In a traditional architecture, services call each other directly. Service A makes an API call to Service B, waits for a response, and then continues. This works fine for simple systems, but as you add more services, the web of direct connections becomes brittle and hard to change.

In an **event-driven architecture**, services communicate by producing and consuming **events** — small JSON documents that describe something that happened. "A file was uploaded." "An order was placed." "A user signed in." Services don't call each other; they emit events, and other services react to the events they care about.

This approach gives you:

- **Loose coupling** — the producer doesn't know who consumes its events
- **Extensibility** — adding a new consumer doesn't require changing the producer
- **Auditability** — events form a natural record of everything that happened in your system

**Amazon EventBridge** is a serverless event bus that makes event-driven architecture practical. It receives events from your applications, AWS services, and third-party SaaS providers, then routes them to targets based on rules you define.

## Event Buses, Rules, and Targets

EventBridge has three core concepts:

### Event Buses

An event bus is a pipeline that receives events. There are three types:

1. **Default event bus** — every AWS account has one. AWS services (like S3, EC2, and IAM) automatically send events to the default bus.
2. **Custom event buses** — you create these for your own application events. Useful for separating your app events from AWS events.
3. **Partner event buses** — receive events from third-party SaaS providers like Shopify, Datadog, or Zendesk through pre-built integrations.

### Rules

A **rule** matches incoming events and routes them to targets. Each rule has:

- An **event pattern** (which events to match) or a **schedule** (run on a timer)
- One or more **targets** (where to send matched events)

A single event bus can have hundreds of rules, and each rule can have up to 5 targets. A single event can match multiple rules simultaneously.

### Targets

A **target** is the AWS service that processes the matched event. EventBridge supports over 20 target types, including:

| Target | Use Case |
|--------|----------|
| Lambda | Run serverless code in response to events |
| SQS | Buffer events for asynchronous processing |
| SNS | Fan out to multiple subscribers |
| Step Functions | Start a workflow |
| Kinesis Data Streams | Stream events for real-time analytics |
| CloudWatch Logs | Log events for debugging |
| Another event bus | Forward events cross-account or cross-region |
| API Gateway | Trigger an API endpoint |

```
                         ┌─ Rule A (pattern: S3 events)  ──→ Lambda Function
                         │
[Event Bus] ─── events ──┼─ Rule B (pattern: order events) ─→ SQS Queue
                         │
                         └─ Rule C (schedule: every hour) ──→ Step Functions
```

## Event Patterns and Filtering

Event patterns are JSON objects that define which events a rule should match. EventBridge compares the pattern against incoming events and only routes events that match.

### Anatomy of an Event

Every EventBridge event has a standard structure:

```json
{
  "version": "0",
  "id": "abc123-def456",
  "source": "aws.s3",
  "detail-type": "Object Created",
  "account": "123456789012",
  "time": "2025-01-15T10:30:00Z",
  "region": "us-east-1",
  "detail": {
    "bucket": {"name": "my-uploads"},
    "object": {"key": "photos/sunset.jpg", "size": 1024000}
  }
}
```

The `source` identifies which service or application produced the event. The `detail-type` describes what happened. The `detail` contains the event-specific payload.

### Writing Event Patterns

**Match all S3 events:**

```json
{
  "source": ["aws.s3"]
}
```

**Match only "Object Created" events from S3:**

```json
{
  "source": ["aws.s3"],
  "detail-type": ["Object Created"]
}
```

**Match S3 uploads to a specific bucket:**

```json
{
  "source": ["aws.s3"],
  "detail-type": ["Object Created"],
  "detail": {
    "bucket": {
      "name": ["my-uploads"]
    }
  }
}
```

**Match custom application events with prefix matching:**

```json
{
  "source": ["myapp.orders"],
  "detail": {
    "status": ["placed", "confirmed"],
    "region": [{"prefix": "us-"}]
  }
}
```

EventBridge supports these matching operators:

- **Exact match**: `["value"]`
- **Prefix**: `[{"prefix": "us-"}]`
- **Numeric range**: `[{"numeric": [">", 100, "<=", 500]}]`
- **Anything-but**: `[{"anything-but": ["cancelled"]}]`
- **Exists**: `[{"exists": true}]` or `[{"exists": false}]`

This filtering is far more powerful than what SNS offers. You can match on deeply nested fields, combine multiple conditions, and use range operators — all evaluated by EventBridge before the event reaches your target.

## Scheduling with EventBridge

EventBridge can also trigger targets on a schedule, replacing the old CloudWatch Events cron jobs. Two schedule types are available:

### Rate Expressions

Run at fixed intervals:

```
rate(5 minutes)
rate(1 hour)
rate(7 days)
```

### Cron Expressions

Run at specific times (all times are UTC):

```
cron(0 9 * * ? *)       # Every day at 9:00 AM UTC
cron(0 */2 * * ? *)     # Every 2 hours
cron(0 9 ? * MON-FRI *) # Weekdays at 9:00 AM UTC
cron(0 0 1 * ? *)       # First day of every month at midnight
```

The cron format is: `cron(minutes hours day-of-month month day-of-week year)`

**Create a scheduled rule via CLI:**

```bash
# Run a Lambda function every 5 minutes
aws events put-rule \
  --name "cleanup-every-5-min" \
  --schedule-expression "rate(5 minutes)" \
  --state ENABLED

# Add the Lambda function as the target
aws events put-targets \
  --rule "cleanup-every-5-min" \
  --targets "Id"="1","Arn"="arn:aws:lambda:us-east-1:123456789012:function:cleanup"
```

**Tip**: For production scheduling workloads, consider **EventBridge Scheduler** — a newer, dedicated scheduling service that supports one-time schedules, time zones, and higher throughput.

## Hands-On: Route an S3 Event to a Lambda Function

This walkthrough creates a rule that triggers a Lambda function whenever a file is uploaded to a specific S3 bucket.

### Step 1: Enable S3 Event Notifications to EventBridge

By default, S3 doesn't send events to EventBridge. You need to enable it on the bucket:

```bash
aws s3api put-bucket-notification-configuration \
  --bucket my-uploads \
  --notification-configuration '{"EventBridgeConfiguration": {}}'
```

### Step 2: Create the EventBridge Rule

```bash
aws events put-rule \
  --name "s3-upload-to-lambda" \
  --event-pattern '{
    "source": ["aws.s3"],
    "detail-type": ["Object Created"],
    "detail": {
      "bucket": {
        "name": ["my-uploads"]
      }
    }
  }' \
  --state ENABLED
```

### Step 3: Add the Lambda Function as a Target

```bash
aws events put-targets \
  --rule "s3-upload-to-lambda" \
  --targets "Id"="1","Arn"="arn:aws:lambda:us-east-1:123456789012:function:process-upload"
```

### Step 4: Grant EventBridge Permission to Invoke the Lambda

```bash
aws lambda add-permission \
  --function-name process-upload \
  --statement-id eventbridge-invoke \
  --action lambda:InvokeFunction \
  --principal events.amazonaws.com \
  --source-arn arn:aws:events:us-east-1:123456789012:rule/s3-upload-to-lambda
```

### The Lambda Handler

```python
def lambda_handler(event, context):
    bucket = event['detail']['bucket']['name']
    key = event['detail']['object']['key']
    size = event['detail']['object']['size']
    print(f"New file uploaded: s3://{bucket}/{key} ({size} bytes)")
    # Process the file — resize image, extract text, etc.
    return {'statusCode': 200}
```

### How This Differs from S3 Event Notifications

S3 has its own event notification system that can trigger Lambda directly. So why use EventBridge?

- **Richer filtering** — EventBridge patterns can match on object key prefix, suffix, size, and more
- **Multiple targets** — one S3 event can trigger Lambda, SQS, and Step Functions simultaneously
- **Cross-account routing** — forward events to other AWS accounts
- **Schema discovery** — EventBridge can automatically detect the structure of your events
- **Archive and replay** — store events and replay them later for debugging or reprocessing

## Key Takeaways

- **EventBridge is a serverless event bus** that routes events from AWS services, your apps, and SaaS partners to targets based on rules
- **Event patterns** give you powerful filtering on any field in the event, including nested fields, numeric ranges, and prefix matching
- **Scheduled rules** replace cron jobs — use rate expressions for intervals or cron expressions for specific times
- **Over 20 target types** including Lambda, SQS, SNS, Step Functions, and other event buses
- **EventBridge vs SNS**: EventBridge has richer filtering, more target types, schema registry, and archive/replay. SNS is simpler and better for high-throughput fan-out with raw pub/sub.
- **Every AWS account gets a default event bus** that receives events from AWS services automatically — you just need to create rules to react to them
