## Why Message Queues?

Imagine you have a web application that processes orders. When a customer places an order, your app needs to charge their credit card, update inventory, and send a confirmation email. If any of those steps is slow or fails, the customer gets an error — even though the order itself was fine.

A **message queue** sits between services and holds messages until the receiving service is ready to process them. Instead of calling the payment service directly, your app drops a message into a queue saying "process this order." The payment service picks it up when it's ready.

This pattern is called **decoupling**, and it gives you three major benefits:

1. **Resilience** — If the payment service goes down for a few minutes, messages wait in the queue. Nothing is lost.
2. **Scalability** — If orders spike on Black Friday, messages pile up in the queue and workers process them at their own pace.
3. **Simplicity** — The web app doesn't need to know anything about the payment service. It just sends a message and moves on.

**Amazon SQS** is AWS's fully managed message queue. You don't manage any servers — AWS handles availability, scaling, and durability behind the scenes. Messages are stored redundantly across multiple Availability Zones.

## Standard vs FIFO Queues

SQS offers two queue types, and the choice matters depending on your use case.

### Standard Queues

- **Nearly unlimited throughput** — handles as many messages per second as you need
- **At-least-once delivery** — a message might occasionally be delivered more than once
- **Best-effort ordering** — messages generally arrive in order, but it's not guaranteed

Standard queues are the default. Use them when your application can handle occasional duplicate messages and doesn't need strict ordering. Examples: log processing, image thumbnailing, sending emails.

### FIFO Queues

- **Exactly-once processing** — each message is delivered once and remains available until a consumer processes and deletes it
- **Strict ordering** — messages are processed in the exact order they were sent
- **Limited throughput** — 300 messages/second without batching, 3,000 with batching

FIFO queue names must end with `.fifo`. Use them when order matters — like financial transactions, ticket booking, or inventory updates where processing a "cancel" before the "create" would cause problems.

```
Standard Queue:  Message A → Message C → Message B → Message A (duplicate)
FIFO Queue:      Message A → Message B → Message C (always in order, no duplicates)
```

## Visibility Timeout and Long Polling

### Key Configuration Settings

**Message size**: Each message can be up to **256 KB**. If you need to send larger payloads, store the data in S3 and put the S3 object key in the message. The SQS Extended Client Library for Java and Python handles this pattern automatically.

**Retention period**: Messages stay in the queue for **1 minute to 14 days** (default: 4 days). If no consumer picks up the message in that time, it's deleted automatically.

### Visibility Timeout

When a consumer receives a message from the queue, the message isn't deleted — it becomes **invisible** to other consumers for a period called the **visibility timeout** (default: 30 seconds).

Here's why this matters:

1. Consumer A receives a message — the message becomes invisible
2. Consumer A processes the message and deletes it — done
3. But if Consumer A crashes before deleting it, the visibility timeout expires and the message reappears in the queue for another consumer to pick up

If your processing takes longer than 30 seconds, increase the visibility timeout. If it's too short, another consumer might pick up the same message while the first one is still working on it.

### Long Polling vs Short Polling

**Short polling** (default) returns immediately, even if the queue is empty. This costs you money because each empty response counts as a request.

**Long polling** waits up to 20 seconds for a message to arrive before returning. This reduces the number of empty responses and saves money.

Always enable long polling unless you have a specific reason not to:

```bash
aws sqs set-queue-attributes \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/my-queue \
  --attributes ReceiveMessageWaitTimeSeconds=20
```

## Dead-Letter Queues

Sometimes a message can't be processed — maybe the payload is malformed, or it triggers a bug in your code. Without a safety net, this "poison message" gets received, fails, becomes visible again, fails again, and loops forever.

A **Dead-Letter Queue (DLQ)** catches these problem messages. You configure a **maxReceiveCount** on your source queue — after a message has been received that many times without being deleted, SQS automatically moves it to the DLQ.

```bash
# Create the dead-letter queue
aws sqs create-queue --queue-name my-queue-dlq

# Get the DLQ ARN
aws sqs get-queue-attributes \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/my-queue-dlq \
  --attribute-names QueueArn

# Configure the source queue to use the DLQ (move after 3 failed attempts)
aws sqs set-queue-attributes \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/my-queue \
  --attributes '{
    "RedrivePolicy": "{\"deadLetterTargetArn\":\"arn:aws:sqs:us-east-1:123456789012:my-queue-dlq\",\"maxReceiveCount\":\"3\"}"
  }'
```

**Tip**: Set up a CloudWatch alarm on the DLQ's `ApproximateNumberOfMessagesVisible` metric so you get notified when messages land there. Dead-letter queues should be monitored, not ignored.

## Hands-On: Send and Receive Messages with SQS

### Using the AWS CLI

Create a queue:

```bash
aws sqs create-queue --queue-name my-first-queue
```

The output gives you the queue URL — you'll need it for all subsequent commands:

```json
{
    "QueueUrl": "https://sqs.us-east-1.amazonaws.com/123456789012/my-first-queue"
}
```

Send a message:

```bash
aws sqs send-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/my-first-queue \
  --message-body '{"orderId": "12345", "item": "widget", "quantity": 2}'
```

Receive a message (with long polling):

```bash
aws sqs receive-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/my-first-queue \
  --wait-time-seconds 20 \
  --max-number-of-messages 1
```

Delete the message after processing (use the ReceiptHandle from the receive response):

```bash
aws sqs delete-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/my-first-queue \
  --receipt-handle "AQEBzL...your-receipt-handle..."
```

### Using Python (boto3)

```python
import boto3
import json

sqs = boto3.client('sqs', region_name='us-east-1')
queue_url = 'https://sqs.us-east-1.amazonaws.com/123456789012/my-first-queue'

# Send a message
sqs.send_message(
    QueueUrl=queue_url,
    MessageBody=json.dumps({
        'orderId': '12345',
        'item': 'widget',
        'quantity': 2
    })
)
print("Message sent!")

# Receive and process messages
response = sqs.receive_message(
    QueueUrl=queue_url,
    MaxNumberOfMessages=1,
    WaitTimeSeconds=20
)

if 'Messages' in response:
    for message in response['Messages']:
        body = json.loads(message['Body'])
        print(f"Processing order: {body['orderId']}")

        # Delete the message after successful processing
        sqs.delete_message(
            QueueUrl=queue_url,
            ReceiptHandle=message['ReceiptHandle']
        )
        print("Message deleted.")
else:
    print("No messages available.")
```

### Triggering Lambda from SQS

One of the most common patterns is having a Lambda function automatically process messages from an SQS queue. AWS handles the polling for you — when messages arrive, Lambda is invoked with a batch of messages.

```bash
# Create an event source mapping (your Lambda needs permission to read from SQS)
aws lambda create-event-source-mapping \
  --function-name my-processor-function \
  --event-source-arn arn:aws:sqs:us-east-1:123456789012:my-first-queue \
  --batch-size 10
```

Your Lambda handler receives messages like this:

```python
def lambda_handler(event, context):
    for record in event['Records']:
        body = json.loads(record['body'])
        print(f"Processing order: {body['orderId']}")
        # Your processing logic here
    return {'statusCode': 200}
```

If the Lambda function throws an error, the message becomes visible again after the visibility timeout — and eventually lands in the DLQ if you've configured one.

## Key Takeaways

- **SQS decouples producers from consumers** — the sender doesn't wait for the receiver
- **Standard queues** give you maximum throughput with at-least-once delivery; **FIFO queues** give you strict ordering with exactly-once processing
- **Visibility timeout** prevents multiple consumers from processing the same message simultaneously — set it higher than your processing time
- **Dead-Letter Queues** catch messages that repeatedly fail processing — always configure one
- **Long polling** reduces costs by waiting for messages instead of returning empty responses
- **Lambda + SQS** is a powerful serverless pattern — AWS handles the polling and scaling automatically
- **Free tier**: 1 million SQS requests per month are free (shared across Standard and FIFO)
