# S3 Access — Public, Private, Pre-Signed URLs, Bucket Policies, and ACLs

Controlling who can access your S3 objects and how.

## What You'll Learn

- Why S3 objects are private by default and what that means
- How bucket policies control access with JSON rules
- Why ACLs exist but should generally be avoided
- How Block Public Access acts as a safety net
- How to generate pre-signed URLs for temporary access

## Public vs Private Access

By default, **everything in S3 is private**. When you upload an object, only you (the AWS account owner) and IAM users/roles with explicit permissions can access it. Nobody on the internet can see it.

This is a deliberate security decision. Making something public requires you to take explicit steps — and AWS has added multiple layers of protection to prevent accidental public exposure.

There are four mechanisms that control S3 access, evaluated together:

1. **IAM Policies** — Attached to users, groups, or roles in your account
2. **Bucket Policies** — Attached to the bucket itself (resource-based)
3. **ACLs** — Legacy per-object permissions (not recommended)
4. **Block Public Access** — Account or bucket-level override that blocks public access regardless of policies

For a request to succeed, it must be **allowed** by at least one policy and **not denied** by any policy or Block Public Access setting.

## Bucket Policies

A **bucket policy** is a JSON document attached to a bucket that defines who can do what with the bucket and its objects. This is the primary way to grant access to S3 resources.

### Basic Structure

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DescriptiveName",
      "Effect": "Allow",
      "Principal": "who",
      "Action": "what they can do",
      "Resource": "which bucket/objects"
    }
  ]
}
```

### Example: Allow Public Read Access to All Objects

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PublicReadGetObject",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-website-bucket/*"
    }
  ]
}
```

- `"Principal": "*"` means anyone (the public internet)
- `"Action": "s3:GetObject"` means they can only download objects (not list, delete, or upload)
- `"Resource"` ends with `/*` to mean all objects in the bucket

### Example: Allow a Specific AWS Account to Upload

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowPartnerUpload",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:root"
      },
      "Action": [
        "s3:PutObject",
        "s3:PutObjectAcl"
      ],
      "Resource": "arn:aws:s3:::my-shared-bucket/uploads/*"
    }
  ]
}
```

### Example: Deny Unencrypted Uploads

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyUnencryptedUploads",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-secure-bucket/*",
      "Condition": {
        "StringNotEquals": {
          "s3:x-amz-server-side-encryption": "aws:kms"
        }
      }
    }
  ]
}
```

Apply a bucket policy via CLI:

```bash
aws s3api put-bucket-policy \
  --bucket my-bucket \
  --policy file://policy.json
```

### IAM Policies vs Bucket Policies

| Feature           | IAM Policy                        | Bucket Policy                     |
| ----------------- | --------------------------------- | --------------------------------- |
| Attached to       | Users, groups, or roles           | The bucket itself                 |
| Controls          | What that identity can access     | Who can access this bucket        |
| Cross-account     | Requires role assumption          | Can directly grant cross-account access |
| Use when          | Managing internal team permissions | Granting public or cross-account access |

In practice, use **IAM policies** for your own team members and **bucket policies** when you need to grant access to external accounts or the public.

## Access Control Lists (ACLs)

ACLs are the **original** access control mechanism in S3 — they predate bucket policies. They allow you to grant basic read/write permissions to specific AWS accounts or predefined groups.

**AWS strongly recommends disabling ACLs** for new buckets. Bucket policies and IAM policies are more flexible and easier to audit.

When you create a new bucket, AWS now defaults to **"Bucket owner enforced"** — which disables ACLs entirely.

```bash
# Confirm ACLs are disabled (recommended setting)
aws s3api get-bucket-ownership-controls --bucket my-bucket
```

Expected output for a properly configured bucket:

```json
{
  "OwnershipControls": {
    "Rules": [
      {
        "ObjectOwnership": "BucketOwnerEnforced"
      }
    ]
  }
}
```

If you encounter legacy buckets that still use ACLs, the most common grants are:

- `private` — Owner has full control, no one else has access (default)
- `public-read` — Everyone can read, only owner can write
- `authenticated-read` — Any authenticated AWS user can read

> **Recommendation:** Always use `BucketOwnerEnforced` for new buckets and migrate old buckets away from ACLs when possible.

## Block Public Access

**Block Public Access** is a set of four account-level and bucket-level settings that act as a **safety net** against accidental public exposure. Even if a bucket policy or ACL grants public access, these settings can override and block it.

The four settings:

| Setting                                | What It Blocks                                     |
| -------------------------------------- | -------------------------------------------------- |
| `BlockPublicAcls`                      | Prevents adding new ACLs that grant public access  |
| `IgnorePublicAcls`                     | Ignores any existing ACLs that grant public access |
| `BlockPublicPolicy`                    | Prevents adding bucket policies that grant public access |
| `RestrictPublicBuckets`                | Restricts access to buckets with public policies to authorized users only |

By default, **all four settings are ON** for new buckets and new accounts.

```bash
# Check Block Public Access settings for a bucket
aws s3api get-public-access-block --bucket my-bucket

# Enable all Block Public Access settings (recommended)
aws s3api put-public-access-block \
  --bucket my-bucket \
  --public-access-block-configuration \
    BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true
```

To host a public website, you must **explicitly disable** some of these settings for that specific bucket:

```bash
# Disable Block Public Access for a website bucket ONLY
aws s3api put-public-access-block \
  --bucket my-public-website \
  --public-access-block-configuration \
    BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=false,RestrictPublicBuckets=false
```

> **Tip:** Set Block Public Access at the **account level** first, then selectively disable it only on buckets that truly need public access. This way, new buckets are safe by default.

## Pre-Signed URLs

A **pre-signed URL** gives someone temporary access to a private S3 object without making it public. The URL contains authentication information embedded in query parameters and expires after a set time.

Use cases:
- Share a file with a client without making it public
- Allow a user to upload a file to a specific key without giving them AWS credentials
- Generate download links in a web application

### Generate a Pre-Signed URL via CLI

```bash
# Generate a URL that expires in 1 hour (3600 seconds)
aws s3 presign s3://my-bucket/reports/quarterly-report.pdf --expires-in 3600
```

Output:

```
https://my-bucket.s3.us-east-1.amazonaws.com/reports/quarterly-report.pdf?X-Amz-Algorithm=AWS4-HMAC-SHA256&X-Amz-Credential=AKIA...&X-Amz-Date=20240115T120000Z&X-Amz-Expires=3600&X-Amz-SignedHeaders=host&X-Amz-Signature=abc123...
```

Anyone with this URL can download the file for the next hour — no AWS account needed. After the expiration, the URL stops working.

### Generate a Pre-Signed URL in Python (boto3)

```python
import boto3

s3 = boto3.client('s3')

url = s3.generate_presigned_url(
    'get_object',
    Params={
        'Bucket': 'my-bucket',
        'Key': 'reports/quarterly-report.pdf'
    },
    ExpiresIn=3600
)

print(url)
```

### Pre-Signed URL for Uploads

You can also create pre-signed URLs that allow someone to **upload** a file:

```python
url = s3.generate_presigned_url(
    'put_object',
    Params={
        'Bucket': 'my-bucket',
        'Key': 'uploads/user-file.pdf',
        'ContentType': 'application/pdf'
    },
    ExpiresIn=600
)
```

The recipient uses an HTTP PUT request to upload to this URL. This is how many web apps handle file uploads without proxying through a server.

## Hands-On: Generate a Pre-Signed URL

**Step 1 — Upload a private file:**

```bash
echo "This is a secret report." > report.txt
aws s3 cp report.txt s3://your-bucket/private/report.txt
```

**Step 2 — Try to access it publicly (this should fail):**

Open `https://your-bucket.s3.amazonaws.com/private/report.txt` in your browser. You should get an "Access Denied" XML response.

**Step 3 — Generate a pre-signed URL:**

```bash
aws s3 presign s3://your-bucket/private/report.txt --expires-in 300
```

**Step 4 — Open the pre-signed URL in your browser.** The file downloads.

**Step 5 — Wait 5 minutes and try again.** The link now returns an error because it expired.

**Step 6 — Clean up:**

```bash
aws s3 rm s3://your-bucket/private/report.txt
```

## Key Takeaways

- S3 objects are **private by default** — public access requires explicit configuration
- **Bucket policies** (JSON) are the primary tool for granting access — use them over ACLs
- **ACLs are legacy** — disable them with `BucketOwnerEnforced` on new buckets
- **Block Public Access** is a safety net that overrides policies and ACLs — keep it on unless you have a specific reason
- **Pre-signed URLs** grant temporary access to private objects without changing permissions
- When granting access, use **IAM policies** for your own team and **bucket policies** for external access
