# Securing S3

IAM Policies, Bucket Policies, Block Public Access, and Encryption.

- How IAM policies and bucket policies work together to control S3 access

## IAM Policies for S3

IAM policies are attached to **users, groups, or roles** in your AWS account. They define what that identity is allowed to do across all AWS services, including S3.

### Example: Allow a User to Read and Write to a Specific Bucket

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "s3:DeleteObject"
      ],
      "Resource": "arn:aws:s3:::my-app-bucket/*"
    },
    {
      "Effect": "Allow",
      "Action": "s3:ListBucket",
      "Resource": "arn:aws:s3:::my-app-bucket"
    }
  ]
}
```

Notice the two statements. `s3:GetObject`, `s3:PutObject`, and `s3:DeleteObject` are **object-level** actions — they need the `/*` suffix on the ARN. `s3:ListBucket` is a **bucket-level** action — it uses the bucket ARN without `/*`. Mixing these up is a common source of "Access Denied" errors.

### Example: Read-Only Access to a Prefix

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::data-lake/public-datasets/*"
    }
  ]
}
```

### Common S3 IAM Actions

| Action                    | What It Allows                    |
| ------------------------- | --------------------------------- |
| `s3:GetObject`            | Download objects                  |
| `s3:PutObject`            | Upload objects                    |
| `s3:DeleteObject`         | Delete objects                    |
| `s3:ListBucket`           | List objects in a bucket          |
| `s3:GetBucketLocation`    | Check which region a bucket is in |
| `s3:GetBucketPolicy`      | Read the bucket policy            |
| `s3:PutBucketPolicy`      | Modify the bucket policy          |
| `s3:GetEncryptionConfiguration` | Read encryption settings    |
| `s3:PutEncryptionConfiguration` | Set encryption settings     |

## Bucket Policies vs IAM Policies

Both control access, but they approach it from different directions:

| Aspect            | IAM Policy                              | Bucket Policy                          |
| ----------------- | --------------------------------------- | -------------------------------------- |
| **Attached to**   | A user, group, or role                  | The bucket                             |
| **Question asked**| "What can this identity access?"        | "Who can access this bucket?"          |
| **Cross-account** | Requires assuming a role                | Can grant access directly              |
| **Public access** | Cannot grant public access              | Can grant access to `"Principal": "*"` |
| **Size limit**    | 6,144 characters                        | 20 KB                                  |

**When to use which:**

- **IAM policies** for managing your own team's permissions ("Developers can read the staging bucket")
- **Bucket policies** for granting access to other AWS accounts, services, or the public ("CloudFront can read from this bucket," "Account 123456789012 can write here")

When both an IAM policy and a bucket policy apply to the same request, AWS evaluates them together. An explicit **Deny** in either one always wins.

## Block Public Access Settings

Block Public Access is a **safety net** — four independent settings that override bucket policies and ACLs to prevent accidental public exposure.

```bash
# View current settings
aws s3api get-public-access-block --bucket my-bucket
```

```json
{
  "PublicAccessBlockConfiguration": {
    "BlockPublicAcls": true,
    "IgnorePublicAcls": true,
    "BlockPublicPolicy": true,
    "RestrictPublicBuckets": true
  }
}
```

**What each setting does:**

- **BlockPublicAcls** — Rejects any PUT request that includes a public ACL
- **IgnorePublicAcls** — Ignores all public ACLs (existing objects with public ACLs become private)
- **BlockPublicPolicy** — Rejects bucket policies that grant public access
- **RestrictPublicBuckets** — Even if a bucket has a public policy, restricts access to authorized users only

All four are **enabled by default** on new buckets and accounts.

### Account-Level vs Bucket-Level

You can set Block Public Access at two levels:

```bash
# Account level (applies to ALL buckets)
aws s3control put-public-access-block \
  --account-id 123456789012 \
  --public-access-block-configuration \
    BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true

# Bucket level (overrides for a specific bucket)
aws s3api put-public-access-block \
  --bucket my-public-website \
  --public-access-block-configuration \
    BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=false,RestrictPublicBuckets=false
```

> **Best practice:** Enable all four settings at the account level. Only disable specific settings on buckets that genuinely need public access (like a static website bucket).

## Server-Side Encryption (SSE-S3, SSE-KMS, SSE-C)

S3 encrypts all new objects by default using SSE-S3. But you have three encryption options depending on your key management needs.

### SSE-S3 (Amazon S3-Managed Keys)

- AWS manages the encryption keys entirely
- Uses AES-256 encryption
- No extra cost, no extra configuration
- You cannot audit key usage or control key rotation independently

```bash
# This is now the default — no flag needed
aws s3 cp file.txt s3://my-bucket/file.txt

# Explicitly request SSE-S3
aws s3 cp file.txt s3://my-bucket/file.txt \
  --sse AES256
```

### SSE-KMS (AWS Key Management Service Keys)

- Encryption keys are managed in AWS KMS
- You can create your own KMS key (Customer Managed Key) or use the AWS-managed key
- Every encrypt/decrypt operation is logged in **CloudTrail** — full audit trail
- You can control who has access to the key via KMS key policies
- KMS charges per API call (encrypt, decrypt, generate-data-key)

```bash
# Encrypt with the default AWS-managed KMS key
aws s3 cp file.txt s3://my-bucket/file.txt --sse aws:kms

# Encrypt with a specific Customer Managed Key
aws s3 cp file.txt s3://my-bucket/file.txt \
  --sse aws:kms \
  --sse-kms-key-id arn:aws:kms:us-east-1:123456789012:key/abcd-1234-efgh-5678
```

### SSE-C (Customer-Provided Keys)

- **You** provide the encryption key with every request
- AWS uses your key to encrypt/decrypt but never stores it
- If you lose the key, your data is unrecoverable
- Requires HTTPS — the CLI enforces this automatically

```bash
# Generate a 256-bit key (base64 encoded)
openssl rand -out sse-c.key 32
SSE_KEY=$(base64 < sse-c.key)
SSE_KEY_MD5=$(openssl dgst -md5 -binary sse-c.key | base64)

# Upload with your key
aws s3api put-object \
  --bucket my-bucket \
  --key sensitive-file.txt \
  --body sensitive-file.txt \
  --sse-customer-algorithm AES256 \
  --sse-customer-key "$SSE_KEY" \
  --sse-customer-key-md5 "$SSE_KEY_MD5"
```

### Set Default Encryption on a Bucket

```bash
# Default to SSE-KMS for all objects
aws s3api put-bucket-encryption \
  --bucket my-bucket \
  --server-side-encryption-configuration '{
    "Rules": [{
      "ApplyServerSideEncryptionByDefault": {
        "SSEAlgorithm": "aws:kms",
        "KMSMasterKeyID": "arn:aws:kms:us-east-1:123456789012:key/abcd-1234"
      },
      "BucketKeyEnabled": true
    }]
  }'
```

> **Tip:** Enable `BucketKeyEnabled` when using SSE-KMS. It reduces KMS API calls (and costs) by using a bucket-level key to create data keys, rather than calling KMS for every single object.

### Choosing an Encryption Option

| Feature          | SSE-S3          | SSE-KMS                | SSE-C              |
| ---------------- | --------------- | ---------------------- | ------------------- |
| Key management   | AWS handles it  | KMS (you control keys) | You handle it       |
| Audit trail      | No              | Yes (CloudTrail)       | No                  |
| Extra cost       | No              | KMS API charges        | No                  |
| Compliance       | Basic           | Strong (HIPAA, PCI)    | You manage compliance |
| Complexity       | None            | Low                    | High                |

For most workloads, **SSE-S3** is fine. Use **SSE-KMS** when you need audit trails, key rotation control, or compliance requirements. Use **SSE-C** only when you must manage your own keys (rare).

## S3 Access Logging

Access logging records every request made to your bucket. Each log entry includes the requester, bucket name, request time, action, response status, and error code.

```bash
# Enable access logging (logs go to a separate bucket)
aws s3api put-bucket-logging \
  --bucket my-app-bucket \
  --bucket-logging-status '{
    "LoggingEnabled": {
      "TargetBucket": "my-access-logs-bucket",
      "TargetPrefix": "s3-logs/my-app-bucket/"
    }
  }'
```

> **Important:** Never log a bucket to itself — this creates an infinite loop of log files.

Log files are delivered on a best-effort basis and may take a few hours to appear. Each log entry looks like:

```
79a59df900b949e55d96a1e698fbacedfd6e09d98eacf8f8d5218e7cd47ef2be
my-app-bucket [06/Feb/2024:00:00:38 +0000] 192.0.2.3 arn:aws:iam::123456789012:user/alice
REST.GET.OBJECT reports/q4.pdf "GET /reports/q4.pdf HTTP/1.1" 200 - 3462 3462 ...
```

## S3 Object Lock

**Object Lock** prevents objects from being deleted or overwritten for a specified retention period. This is how you implement **WORM** (Write Once Read Many) compliance — required in industries like finance and healthcare.

Two retention modes:

| Mode            | Can Root Delete? | Can Anyone Overwrite? | Use Case                 |
| --------------- | ---------------- | --------------------- | ------------------------ |
| **Governance**  | Yes (with special permission) | No           | Internal policies        |
| **Compliance**  | No — not even root | No                  | Regulatory requirements  |

```bash
# Enable Object Lock when creating a bucket
aws s3api create-bucket \
  --bucket my-compliance-bucket \
  --object-lock-enabled-for-bucket

# Set a default retention policy
aws s3api put-object-lock-configuration \
  --bucket my-compliance-bucket \
  --object-lock-configuration '{
    "ObjectLockEnabled": "Enabled",
    "Rule": {
      "DefaultRetention": {
        "Mode": "COMPLIANCE",
        "Years": 7
      }
    }
  }'
```

> **Warning:** Compliance mode is truly immutable. Once set, even the AWS account root user cannot delete objects until the retention period expires. Test with Governance mode first.

## VPC Endpoints for Private S3 Access

By default, when an EC2 instance accesses S3, the traffic goes over the public internet (even within the same region). A **VPC Gateway Endpoint** creates a private route from your VPC to S3 — traffic never leaves the AWS network.

Benefits:
- **Security** — No internet exposure
- **Performance** — Lower latency, more consistent
- **Cost** — No NAT Gateway data processing charges for S3 traffic

```bash
# Create a VPC Gateway Endpoint for S3
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --service-name com.amazonaws.us-east-1.s3 \
  --route-table-ids rtb-0123456789abcdef0
```

You can restrict your bucket to **only accept traffic from your VPC** by combining a VPC endpoint with a bucket policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyAccessUnlessFromVPC",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::my-private-bucket",
        "arn:aws:s3:::my-private-bucket/*"
      ],
      "Condition": {
        "StringNotEquals": {
          "aws:sourceVpce": "vpce-0123456789abcdef0"
        }
      }
    }
  ]
}
```

## Common S3 Misconfigurations

These are the most frequent S3 security mistakes and how to prevent them:

### 1. Accidentally Public Buckets

**Problem:** A bucket policy or ACL exposes data to the internet.
**Fix:** Enable Block Public Access at the account level. Use AWS Config rule `s3-bucket-public-read-prohibited` to detect violations.

### 2. Overly Broad IAM Policies

**Problem:** Policies using `"Resource": "*"` or `"Action": "s3:*"` grant access to all buckets.
**Fix:** Always scope to specific bucket ARNs and specific actions.

```json
// BAD — gives full S3 access to everything
{ "Action": "s3:*", "Resource": "*" }

// GOOD — scoped to one bucket, specific actions
{ "Action": ["s3:GetObject", "s3:PutObject"], "Resource": "arn:aws:s3:::my-bucket/*" }
```

### 3. No Encryption Enforcement

**Problem:** Objects are uploaded without encryption.
**Fix:** Set default encryption on the bucket and add a deny policy for unencrypted uploads.

### 4. Logging Not Enabled

**Problem:** No record of who accessed what.
**Fix:** Enable S3 access logging or use CloudTrail data events for S3.

### 5. No Versioning on Critical Buckets

**Problem:** Accidental deletes permanently lose data.
**Fix:** Enable versioning and set lifecycle rules to manage old versions.

### 6. Stale Pre-Signed URLs

**Problem:** Pre-signed URLs with long expiration times remain usable longer than intended.
**Fix:** Use short expiration times (minutes, not days). Rotate the signing credentials if you need to invalidate URLs immediately.

## Hands-On: Lock Down a Bucket

**Step 1 — Create a bucket with versioning and encryption:**

```bash
aws s3 mb s3://your-secure-bucket-demo --region us-east-1

aws s3api put-bucket-versioning \
  --bucket your-secure-bucket-demo \
  --versioning-configuration Status=Enabled

aws s3api put-bucket-encryption \
  --bucket your-secure-bucket-demo \
  --server-side-encryption-configuration '{
    "Rules": [{"ApplyServerSideEncryptionByDefault": {"SSEAlgorithm": "AES256"}}]
  }'
```

**Step 2 — Enable Block Public Access:**

```bash
aws s3api put-public-access-block \
  --bucket your-secure-bucket-demo \
  --public-access-block-configuration \
    BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true
```

**Step 3 — Add a bucket policy restricting to your account only:**

Save as `secure-policy.json`:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyExternalAccess",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::your-secure-bucket-demo",
        "arn:aws:s3:::your-secure-bucket-demo/*"
      ],
      "Condition": {
        "StringNotEquals": {
          "aws:PrincipalAccount": "YOUR_ACCOUNT_ID"
        }
      }
    }
  ]
}
```

```bash
aws s3api put-bucket-policy \
  --bucket your-secure-bucket-demo \
  --policy file://secure-policy.json
```

**Step 4 — Enable access logging:**

```bash
aws s3 mb s3://your-secure-bucket-demo-logs

aws s3api put-bucket-logging \
  --bucket your-secure-bucket-demo \
  --bucket-logging-status '{
    "LoggingEnabled": {
      "TargetBucket": "your-secure-bucket-demo-logs",
      "TargetPrefix": "access-logs/"
    }
  }'
```

**Step 5 — Verify the configuration:**

```bash
aws s3api get-bucket-encryption --bucket your-secure-bucket-demo
aws s3api get-public-access-block --bucket your-secure-bucket-demo
aws s3api get-bucket-versioning --bucket your-secure-bucket-demo
aws s3api get-bucket-logging --bucket your-secure-bucket-demo
```

**Step 6 — Clean up when done:**

```bash
aws s3 rb s3://your-secure-bucket-demo --force
aws s3 rb s3://your-secure-bucket-demo-logs --force
```

## Key Takeaways

- Use **IAM policies** for internal team access; use **bucket policies** for cross-account and public access
- **Block Public Access** is your safety net — enable all four settings at the account level
- **SSE-S3** is the default encryption; use **SSE-KMS** when you need audit trails and key control
- **Access logging** records who accessed your bucket — enable it for security-sensitive buckets
- **Object Lock** provides WORM compliance — Compliance mode is truly immutable
- **VPC endpoints** let EC2 instances access S3 privately without internet traffic
- Always scope IAM and bucket policies to **specific ARNs and actions** — never use wildcards for production
