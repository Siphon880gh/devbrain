# S3 Storage Classes and Lifecycle Rules

Standard, IA, Glacier Instant, Glacier Deep Archive — and automating transitions.

## What You'll Learn

- The seven S3 storage classes and when to use each
- How costs vary between storage classes (storage price vs retrieval cost)
- What Intelligent-Tiering does and when it makes sense
- How to create lifecycle rules to automatically move or delete objects
- How to configure lifecycle rules via CLI and console

## Storage Classes Overview

Not all data is accessed the same way. A profile photo might be accessed thousands of times per day, while a tax return from three years ago might never be accessed again. S3 offers **seven storage classes** so you can match your storage cost to your access pattern.

The core trade-off: **cheaper storage = more expensive or slower retrieval**.

| Storage Class                  | Use Case                          | Storage Cost | Retrieval Cost | Retrieval Time     | Min Duration |
| ------------------------------ | --------------------------------- | ------------ | -------------- | ------------------ | ------------ |
| **Standard**                   | Frequently accessed data          | $$$$         | Free           | Milliseconds       | None         |
| **Intelligent-Tiering**        | Unknown or changing access        | $$$$*        | Free**         | Milliseconds       | None         |
| **Standard-IA**                | Infrequent but needs fast access  | $$$          | Per-GB fee     | Milliseconds       | 30 days      |
| **One Zone-IA**                | Infrequent, recreatable data      | $$           | Per-GB fee     | Milliseconds       | 30 days      |
| **Glacier Instant Retrieval**  | Archive with instant access       | $$           | Per-GB fee     | Milliseconds       | 90 days      |
| **Glacier Flexible Retrieval** | Archive with hours-level access   | $            | Per-GB fee     | 1–12 hours         | 90 days      |
| **Glacier Deep Archive**       | Long-term archive                 | ¢            | Per-GB fee     | 12–48 hours        | 180 days     |

*\* Intelligent-Tiering matches the price of whatever tier your data is currently in*
*\*\* No retrieval fee, but a small monthly monitoring fee per object*

> **Min Duration** means you're charged for at least that many days even if you delete the object sooner. Delete a Standard-IA object after 5 days and you still pay for 30 days of storage.

## Standard vs Infrequent Access

### S3 Standard

This is the default. Use it for data that is accessed frequently — website assets, application data, active datasets.

- No retrieval fee
- No minimum storage duration
- Highest storage cost per GB

### S3 Standard-IA (Infrequent Access)

Same durability and availability as Standard, but at a lower storage price. The catch: you pay a per-GB retrieval fee every time you download data.

Good for:
- Backups you rarely restore
- Disaster recovery copies
- Older log files you might need occasionally

```bash
# Upload directly to Standard-IA
aws s3 cp backup.tar.gz s3://my-bucket/backups/backup.tar.gz \
  --storage-class STANDARD_IA
```

### S3 One Zone-IA

Like Standard-IA, but data is stored in **only one Availability Zone** instead of three. If that AZ has an outage or hardware failure, your data could be lost.

Good for:
- Data you can regenerate (thumbnails, transcoded video)
- Secondary backup copies when the primary is elsewhere

**Not good for:** Your only copy of anything important.

```bash
aws s3 cp thumbnails/ s3://my-bucket/thumbnails/ \
  --recursive --storage-class ONEZONE_IA
```

## Intelligent-Tiering

**S3 Intelligent-Tiering** monitors access patterns and automatically moves objects between tiers:

- **Frequent Access tier** — Accessed regularly (same price as Standard)
- **Infrequent Access tier** — Not accessed for 30 days (same price as Standard-IA)
- **Archive Instant Access tier** — Not accessed for 90 days
- **Archive Access tier** — Not accessed for 90+ days (optional, you enable this)
- **Deep Archive Access tier** — Not accessed for 180+ days (optional)

There is no retrieval fee when objects move between tiers. AWS charges a small monitoring and automation fee (~$0.0025 per 1,000 objects per month).

**When to use Intelligent-Tiering:**
- You don't know the access pattern of your data
- Access patterns change over time
- You have many objects and don't want to manage lifecycle rules manually

```bash
# Upload with Intelligent-Tiering
aws s3 cp data.csv s3://my-bucket/data.csv --storage-class INTELLIGENT_TIERING
```

> **Tip:** Intelligent-Tiering is not cost-effective for very small objects. The per-object monitoring fee can exceed the storage savings for objects under 128 KB.

## Glacier Instant Retrieval

**Glacier Instant Retrieval** is for data you rarely access (once a quarter or less) but need back in milliseconds when you do.

Think of medical images, news archives, or user-generated content from years ago — accessed rarely, but when a user requests it, they expect it immediately.

- Storage cost ~68% lower than Standard
- Millisecond retrieval (same as Standard)
- Per-GB retrieval fee
- 90-day minimum storage duration

## Glacier Flexible Retrieval

Formerly known as just "S3 Glacier." This is for archive data where you can tolerate a wait to get it back.

Three retrieval options:

| Retrieval Mode | Time          | Cost        |
| -------------- | ------------- | ----------- |
| Expedited      | 1–5 minutes   | $$$         |
| Standard       | 3–5 hours     | $$          |
| Bulk           | 5–12 hours    | $           |

Good for: Compliance archives, old backups, data you need to keep but rarely access.

```bash
# Restore a Glacier object (initiate retrieval)
aws s3api restore-object \
  --bucket my-bucket \
  --key archive/old-data.tar.gz \
  --restore-request '{"Days":7,"GlacierJobParameters":{"Tier":"Standard"}}'
```

The `Days` parameter is how long the restored copy stays accessible before S3 removes it and the object goes back to Glacier.

## Glacier Deep Archive

The cheapest storage in AWS. Designed for data you must retain for 7-10 years but may never access again — regulatory compliance, legal hold, long-term backups.

- Storage cost ~95% lower than Standard
- Retrieval takes 12-48 hours
- 180-day minimum storage duration

Two retrieval options:

| Retrieval Mode | Time        |
| -------------- | ----------- |
| Standard       | 12 hours    |
| Bulk           | 48 hours    |

> **Reality check:** If you need data back in less than 12 hours, Deep Archive is not the right class. Use Glacier Flexible Retrieval or Glacier Instant Retrieval instead.

## Lifecycle Rules

**Lifecycle rules** automate the movement of objects between storage classes — and can automatically delete old objects. You define rules based on object age, prefix, or tags.

### Common Lifecycle Patterns

**Pattern 1: Transition and Archive**

```
Day 0:     Upload to Standard
Day 30:    Move to Standard-IA
Day 90:    Move to Glacier Instant Retrieval
Day 365:   Move to Glacier Deep Archive
Day 2555:  Delete (7 years)
```

**Pattern 2: Clean Up Temp Data**

```
Day 0:     Upload to Standard
Day 7:     Delete
```

**Pattern 3: Abort Incomplete Multipart Uploads**

```
After 3 days: Abort any incomplete multipart upload
```

This last one is important — incomplete multipart uploads silently consume storage and cost money.

### Create a Lifecycle Rule via CLI

Create a JSON file named `lifecycle.json`:

```json
{
  "Rules": [
    {
      "ID": "ArchiveOldData",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "logs/"
      },
      "Transitions": [
        {
          "Days": 30,
          "StorageClass": "STANDARD_IA"
        },
        {
          "Days": 90,
          "StorageClass": "GLACIER"
        }
      ],
      "Expiration": {
        "Days": 365
      }
    },
    {
      "ID": "AbortIncompleteMultipartUploads",
      "Status": "Enabled",
      "Filter": {
        "Prefix": ""
      },
      "AbortIncompleteMultipartUpload": {
        "DaysAfterInitiation": 3
      }
    }
  ]
}
```

Apply it:

```bash
aws s3api put-bucket-lifecycle-configuration \
  --bucket my-bucket \
  --lifecycle-configuration file://lifecycle.json
```

Check existing rules:

```bash
aws s3api get-bucket-lifecycle-configuration --bucket my-bucket
```

### Create a Lifecycle Rule via Console

1. Open your bucket in the S3 Console
2. Go to the **Management** tab
3. Under **Lifecycle rules**, click **Create lifecycle rule**
4. Name the rule and choose a scope (entire bucket or a prefix/tag filter)
5. Select actions: transition, expiration, or both
6. Set the day thresholds for each transition
7. Review and create

## Hands-On: Create a Lifecycle Policy

**Step 1 — Create a lifecycle rule that moves logs to IA after 30 days:**

Save this as `my-lifecycle.json`:

```json
{
  "Rules": [
    {
      "ID": "TransitionLogs",
      "Status": "Enabled",
      "Filter": { "Prefix": "logs/" },
      "Transitions": [
        { "Days": 30, "StorageClass": "STANDARD_IA" }
      ]
    }
  ]
}
```

```bash
aws s3api put-bucket-lifecycle-configuration \
  --bucket your-bucket \
  --lifecycle-configuration file://my-lifecycle.json
```

**Step 2 — Verify:**

```bash
aws s3api get-bucket-lifecycle-configuration --bucket your-bucket
```

**Step 3 — Clean up (remove lifecycle rules):**

```bash
aws s3api delete-bucket-lifecycle --bucket your-bucket
```

## Key Takeaways

- **S3 Standard** is for frequently accessed data; **Standard-IA** saves money for infrequent access
- **One Zone-IA** is cheap but risky — only one Availability Zone
- **Glacier** tiers are for archival — Instant (millisecond access), Flexible (hours), Deep Archive (half-day or more)
- **Intelligent-Tiering** automates class selection based on access patterns — great when patterns are unpredictable
- **Lifecycle rules** automate transitions and deletions based on object age
- Always create a lifecycle rule to **abort incomplete multipart uploads** — they silently waste money
- Minimum storage durations mean deleting early does not save costs
