# Other Storage Services — EBS, EFS, and S3 Glacier

When to use block storage, file storage, or archival storage.

## What You'll Learn

- The three categories of AWS storage: object, block, and file
- What EBS is, its volume types, and when to use it
- What EFS is, how it works across Availability Zones, and when to choose it
- How S3 Glacier fits into long-term archival strategies
- A decision framework for choosing the right storage service

## The Three Types of Cloud Storage

AWS offers storage in three fundamental categories. Each works differently and solves different problems:

| Type            | Service | How It Works                                 | Analogy              |
| --------------- | ------- | -------------------------------------------- | -------------------- |
| **Object**      | S3      | Flat key-value store; access via HTTP/API    | Google Drive / Dropbox |
| **Block**       | EBS     | Raw disk volumes; attach to one EC2 instance | A hard drive plugged into a computer |
| **File**        | EFS     | Shared file system; mount on many instances  | A network shared folder (NAS) |

You don't choose one for everything. Different parts of your application use different storage types.

## EBS (Elastic Block Store)

**EBS** provides persistent block-level storage volumes for EC2 instances. Think of an EBS volume as a virtual hard drive that you attach to a virtual machine.

### Key Characteristics

- **Attached to one EC2 instance at a time** (with one exception: `io2` supports multi-attach)
- **Persists independently** from the EC2 instance — you can stop/terminate the instance and the volume survives (if configured to)
- **Lives in a single Availability Zone** — you cannot attach an EBS volume in `us-east-1a` to an instance in `us-east-1b`
- **Supports snapshots** — point-in-time backups stored in S3 (you can copy snapshots across regions)

### EBS Volume Types

| Volume Type     | Name                      | Use Case                              | Max IOPS   | Max Throughput |
| --------------- | ------------------------- | ------------------------------------- | ---------- | -------------- |
| **gp3**         | General Purpose SSD       | Most workloads, boot volumes          | 16,000     | 1,000 MB/s     |
| **gp2**         | General Purpose SSD (old) | Legacy — use gp3 instead              | 16,000     | 250 MB/s       |
| **io2 Block Express** | Provisioned IOPS SSD | Databases needing consistent IOPS     | 256,000    | 4,000 MB/s     |
| **io2**         | Provisioned IOPS SSD      | Databases, latency-sensitive apps     | 64,000     | 1,000 MB/s     |
| **st1**         | Throughput Optimized HDD  | Big data, data warehouses, log processing | 500     | 500 MB/s       |
| **sc1**         | Cold HDD                  | Infrequent access, lowest cost        | 250        | 250 MB/s       |

**gp3 is the default choice** for most workloads. It's cheaper than gp2 and lets you independently configure IOPS and throughput. Use io2 only when your database demands guaranteed high IOPS (like a production MySQL or PostgreSQL).

### Working with EBS Volumes

```bash
# Create a 50 GB gp3 volume in a specific AZ
aws ec2 create-volume \
  --volume-type gp3 \
  --size 50 \
  --availability-zone us-east-1a

# Attach a volume to an EC2 instance
aws ec2 attach-volume \
  --volume-id vol-0123456789abcdef0 \
  --instance-id i-0123456789abcdef0 \
  --device /dev/xvdf

# Create a snapshot (backup)
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Daily backup"
```

After attaching, you still need to format and mount the volume from inside the EC2 instance:

```bash
# (Run these INSIDE the EC2 instance)

# Format the volume (only first time — this erases data)
sudo mkfs -t ext4 /dev/xvdf

# Create a mount point and mount
sudo mkdir /data
sudo mount /dev/xvdf /data

# Verify
df -h /data
```

### EBS Snapshots

Snapshots are incremental backups. The first snapshot copies the entire volume. Subsequent snapshots only copy blocks that changed since the last snapshot.

```bash
# Create a snapshot
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Before upgrade"

# List snapshots you own
aws ec2 describe-snapshots --owner-ids self

# Create a new volume from a snapshot (in a different AZ if needed)
aws ec2 create-volume \
  --snapshot-id snap-0123456789abcdef0 \
  --availability-zone us-east-1b \
  --volume-type gp3
```

## EFS (Elastic File System)

**EFS** is a managed NFS (Network File System) that multiple EC2 instances can mount simultaneously. Unlike EBS, which is tied to one instance in one AZ, EFS is accessible across all Availability Zones in a region.

### Key Characteristics

- **Shared access** — Hundreds of EC2 instances can read and write to the same file system at the same time
- **Spans Availability Zones** — Mount targets in each AZ provide access from any subnet
- **Elastic** — Grows and shrinks automatically as you add and remove files (no need to provision size)
- **NFS protocol** — Works with Linux instances (not Windows — use FSx for Windows File Server instead)
- **Two storage classes** — Standard (frequent access) and Infrequent Access (IA) with automatic lifecycle tiering

### When to Use EFS

- Shared configuration files across a fleet of web servers
- Content management systems where multiple servers need access to the same files
- Machine learning training data shared across GPU instances
- Home directories in multi-user environments
- Container storage (ECS and EKS can mount EFS volumes)

### Creating and Mounting EFS

```bash
# Create a file system
aws efs create-file-system \
  --performance-mode generalPurpose \
  --throughput-mode bursting \
  --encrypted \
  --tags Key=Name,Value=my-shared-fs

# The command returns a FileSystemId like fs-0123456789abcdef0

# Create a mount target in your subnet
aws efs create-mount-target \
  --file-system-id fs-0123456789abcdef0 \
  --subnet-id subnet-0123456789abcdef0 \
  --security-groups sg-0123456789abcdef0
```

Mount from an EC2 instance (install `amazon-efs-utils` first):

```bash
# Install the EFS helper (Amazon Linux 2)
sudo yum install -y amazon-efs-utils

# Mount using the EFS helper
sudo mkdir /efs
sudo mount -t efs fs-0123456789abcdef0:/ /efs

# Or mount with encryption in transit
sudo mount -t efs -o tls fs-0123456789abcdef0:/ /efs

# Verify
df -h /efs
```

### EFS Performance Modes

| Mode                | Best For                              |
| ------------------- | ------------------------------------- |
| **General Purpose** | Web serving, CMS, home directories — most workloads |
| **Max I/O**         | Big data, media processing — high parallelism (higher latency) |

### EFS Throughput Modes

| Mode          | How It Works                                          |
| ------------- | ----------------------------------------------------- |
| **Bursting**  | Throughput scales with file system size; earns burst credits |
| **Elastic**   | Automatically scales throughput up/down based on workload (recommended for unpredictable access) |
| **Provisioned** | You set a fixed throughput level regardless of size  |

## S3 Glacier for Archival

S3 Glacier is not a separate service — it is a set of S3 storage classes designed for long-term archival at very low cost. We covered the details in Lesson 4, but here is where Glacier fits in the bigger storage picture.

### When to Use Glacier

- **Compliance archives** — Legal, financial, or healthcare records you must keep for years
- **Backup retention** — Old backups you want to keep "just in case" but never expect to use
- **Media archives** — Raw footage, original photographs, research data
- **Data you're legally required to keep** — Audit trails, tax records

### Glacier Retrieval Workflow

Objects in Glacier are not directly downloadable. You must initiate a **restore** request, wait for retrieval, then download during the restore window:

```bash
# 1. Initiate restore (takes hours depending on tier)
aws s3api restore-object \
  --bucket my-archive-bucket \
  --key compliance/2019/audit-trail.tar.gz \
  --restore-request '{"Days":7,"GlacierJobParameters":{"Tier":"Standard"}}'

# 2. Check restore status
aws s3api head-object \
  --bucket my-archive-bucket \
  --key compliance/2019/audit-trail.tar.gz
# Look for: "Restore": "ongoing-request=\"false\", expiry-date=\"...\""

# 3. Download once restore is complete
aws s3 cp s3://my-archive-bucket/compliance/2019/audit-trail.tar.gz ./
```

## Choosing the Right Storage Service

Use this decision framework:

**"Do I need to mount this as a drive on an EC2 instance?"**
- Yes, one instance → **EBS**
- Yes, multiple instances → **EFS**
- No, I access via API/HTTP → **S3**

**"Is this a database or OS disk?"**
- Yes → **EBS** (block storage that EC2 can boot from)

**"Do multiple servers need to share files?"**
- Yes, Linux → **EFS**
- Yes, Windows → **FSx for Windows File Server**

**"Is this for backups, logs, images, or static assets?"**
- Yes → **S3** (and lifecycle to Glacier for older data)

**"Must I keep this for years but rarely access it?"**
- Yes → **S3 Glacier** (Deep Archive for 7+ year retention)

## Key Concepts Comparison Table

| Feature              | S3                    | EBS                  | EFS                    |
| -------------------- | --------------------- | -------------------- | ---------------------- |
| **Type**             | Object storage        | Block storage        | File storage (NFS)     |
| **Access**           | HTTP API / CLI / SDK  | Mount to EC2         | Mount to EC2(s)        |
| **Shared access**    | Yes (via API)         | One instance*        | Many instances         |
| **Availability Zone**| Regional (multi-AZ)   | Single AZ            | Regional (multi-AZ)    |
| **Max size**         | Unlimited (5 TB/object) | 64 TB per volume   | Unlimited (elastic)    |
| **Provisioning**     | No capacity planning  | Set volume size      | Automatic scaling      |
| **Performance**      | High throughput       | Low latency, high IOPS | Moderate latency     |
| **Use case**         | Assets, backups, data lake | OS disks, databases | Shared files, CMS    |
| **Cost model**       | Pay per GB stored + requests | Pay per GB provisioned | Pay per GB stored  |

*\* io2 volumes support multi-attach to up to 16 Nitro instances*

## Key Takeaways

- **S3** is object storage accessed via API — use it for files, backups, and static assets
- **EBS** is a virtual hard drive for EC2 — use it for OS disks and databases; choose **gp3** by default
- **EFS** is a shared network file system — use it when multiple instances need the same files
- EBS is **single-AZ**; EFS and S3 are **multi-AZ**
- **Snapshots** let you back up EBS volumes incrementally
- **Glacier** classes are for archival — cheap storage, slow (or costly) retrieval
- Match the storage type to the access pattern: API-based → S3, mounted volume → EBS, shared mount → EFS
