# S3 Concepts — Buckets, Objects, Keys, Regions, and URIs

The foundational building blocks of Amazon S3.

## What You'll Learn

- What Amazon S3 is and how it differs from a traditional file system
- The meaning of buckets, objects, and keys
- How regions affect where your data lives
- How to read and write S3 URIs and ARNs
- Object metadata, ETags, and versioning basics

## What Is Amazon S3?

Amazon Simple Storage Service (S3) is an **object storage** service. Unlike a hard drive or a file system with folders and directories, S3 stores data as flat **objects** inside **buckets**. There is no folder hierarchy at the storage level — what looks like a folder in the console is just a naming convention using `/` in the object key.

Think of S3 like a giant key-value store:

- The **key** is the name you give the object (like `photos/vacation/beach.jpg`)
- The **value** is the file data itself (the actual bytes of the image)
- The **bucket** is the container that holds all your key-value pairs

S3 is designed for **durability** (99.999999999% — that's eleven 9s) and **availability**. AWS replicates your data across multiple facilities within a region automatically.

**What S3 is good for:**

- Storing images, videos, backups, and log files
- Hosting static websites (HTML, CSS, JS)
- Data lakes for analytics
- Application assets and configuration files

**What S3 is NOT:**

- A file system you mount to your OS (use EFS or EBS for that)
- A database (use DynamoDB or RDS for structured data)
- A place to run code (use Lambda or EC2 for that)

## Buckets

A **bucket** is a container for objects. Every object you store in S3 lives inside a bucket.

**Key rules about buckets:**

1. **Bucket names are globally unique.** No two AWS accounts anywhere in the world can have a bucket with the same name. If someone already has `my-cool-bucket`, you cannot create it.

2. **Bucket names must follow DNS rules:**
   - 3 to 63 characters long
   - Lowercase letters, numbers, and hyphens only
   - Must start with a letter or number
   - No underscores, no uppercase, no periods (periods cause issues with SSL)

3. **Buckets are created in a specific region.** Even though bucket names are globally unique, the actual data lives in the region you choose.

4. **There is a default limit of 100 buckets per account** (you can request an increase to 1,000).

```bash
# Create a bucket in us-east-1
aws s3 mb s3://my-company-app-assets-2024

# Create a bucket in a specific region
aws s3 mb s3://my-company-eu-data --region eu-west-1

# List all your buckets
aws s3 ls
```

## Objects and Keys

An **object** is the fundamental entity stored in S3. Each object consists of:

| Component      | Description                                         |
| -------------- | --------------------------------------------------- |
| **Key**        | The unique name for the object within the bucket    |
| **Value**      | The actual data (file content), up to 5 TB per object |
| **Metadata**   | Key-value pairs describing the object               |
| **Version ID** | Unique ID if versioning is enabled                  |
| **ETag**       | Hash of the object content (used for change detection) |

### Keys and the Illusion of Folders

The key is the full "path" to your object. S3 has no real directories — the `/` character in a key is just a character, but the console and CLI display it as if it were a folder.

```
my-bucket/
├── index.html              ← key: "index.html"
├── images/logo.png         ← key: "images/logo.png"
└── data/2024/report.csv    ← key: "data/2024/report.csv"
```

The key `images/logo.png` is not a file named `logo.png` inside a folder named `images`. It is a single object whose key happens to contain a `/`.

### Object Metadata

Every object has **system metadata** (set by AWS) and optional **user metadata** (set by you):

**System metadata examples:**
- `Content-Type` — MIME type like `image/png` or `application/json`
- `Content-Length` — Size of the object in bytes
- `Last-Modified` — Timestamp of the last modification
- `ETag` — MD5 hash for simple uploads, opaque identifier for multipart

**User metadata** is custom key-value pairs you attach when uploading:

```bash
aws s3 cp report.pdf s3://my-bucket/reports/report.pdf \
  --metadata '{"department":"finance","quarter":"Q1"}'
```

### ETags

The **ETag** (entity tag) is a hash that represents the content of the object. For objects uploaded in a single PUT request, the ETag is the MD5 hash of the content. For multipart uploads, it is a more complex hash with a `-N` suffix indicating the number of parts.

ETags are useful for:
- Detecting if a file has changed
- Conditional requests (`If-None-Match` headers)
- Cache validation

### Versioning

When **versioning** is enabled on a bucket, S3 keeps every version of every object. Deleting an object doesn't remove it — it adds a "delete marker." You can retrieve any previous version.

```bash
# Enable versioning on a bucket
aws s3api put-bucket-versioning \
  --bucket my-bucket \
  --versioning-configuration Status=Enabled

# List all versions of objects in a bucket
aws s3api list-object-versions --bucket my-bucket
```

Versioning protects against accidental deletes and overwrites, but every version counts toward your storage costs.

## Regions and Data Residency

When you create a bucket, you pick an AWS region (like `us-east-1`, `eu-west-1`, or `ap-southeast-1`). Your data **never leaves that region** unless you explicitly copy or replicate it.

This matters for:

- **Latency** — Store data near your users for faster access
- **Compliance** — Some regulations require data to stay within certain countries
- **Cost** — Data transfer prices vary by region

```bash
# Check which region a bucket is in
aws s3api get-bucket-location --bucket my-bucket
```

> **Tip:** `us-east-1` is the default region. If you don't specify a region, your bucket lands there. It's also the only region where `get-bucket-location` returns `null` instead of the region name — a quirk you'll encounter.

## S3 URIs and ARNs

There are two standard ways to reference S3 resources:

### S3 URI (used by the CLI and SDKs)

```
s3://bucket-name/key-name
```

Examples:

```
s3://my-app-assets/images/logo.png
s3://data-lake-prod/raw/2024/01/events.json
```

You use S3 URIs with the CLI:

```bash
aws s3 cp s3://my-app-assets/images/logo.png ./local-logo.png
```

### S3 ARN (used in IAM policies and bucket policies)

```
arn:aws:s3:::bucket-name
arn:aws:s3:::bucket-name/key-prefix/*
```

Examples:

```
arn:aws:s3:::my-app-assets              ← the bucket itself
arn:aws:s3:::my-app-assets/*            ← all objects in the bucket
arn:aws:s3:::my-app-assets/images/*     ← all objects under images/
```

> **Note:** S3 ARNs do not include a region or account ID. This is because bucket names are globally unique — the bucket name alone is enough to identify it.

### HTTPS URL (used in browsers and HTTP clients)

```
https://bucket-name.s3.amazonaws.com/key-name
https://bucket-name.s3.us-east-1.amazonaws.com/key-name
```

## Hands-On: Create Your First Bucket

**Step 1 — Create a bucket via CLI:**

```bash
# Replace "your-unique-name" with something globally unique
aws s3 mb s3://your-unique-name-first-bucket --region us-east-1
```

**Step 2 — Upload a test file:**

```bash
echo "Hello, S3!" > hello.txt
aws s3 cp hello.txt s3://your-unique-name-first-bucket/hello.txt
```

**Step 3 — Verify the upload:**

```bash
aws s3 ls s3://your-unique-name-first-bucket/
```

You should see `hello.txt` listed with its size and timestamp.

**Step 4 — Download the file:**

```bash
aws s3 cp s3://your-unique-name-first-bucket/hello.txt downloaded.txt
cat downloaded.txt
```

**Step 5 — Clean up:**

```bash
aws s3 rm s3://your-unique-name-first-bucket/hello.txt
aws s3 rb s3://your-unique-name-first-bucket
```

> **Note:** You cannot delete a bucket that still contains objects. Remove all objects first (or use `aws s3 rb s3://bucket-name --force` to delete everything and the bucket in one step).

## Key Takeaways

- S3 is **object storage**, not a file system — there are no real folders
- **Buckets** are globally unique containers; **objects** are the files inside them
- An object's **key** is its full path name; the **value** is the file data
- **ETags** track content changes; **versioning** keeps every version of an object
- **S3 URIs** (`s3://bucket/key`) are for the CLI; **ARNs** (`arn:aws:s3:::bucket`) are for policies
- Data stays in the **region** where you create the bucket unless you move it
