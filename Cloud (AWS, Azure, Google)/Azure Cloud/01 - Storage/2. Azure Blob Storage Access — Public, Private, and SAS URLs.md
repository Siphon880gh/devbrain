In the last lesson, we covered the structure:

```text
Azure Account
└── Subscription
    └── Resource Group
        └── Storage Account
            └── Container
                └── Blob
```

And we learned that a blob URI looks like this:

```text
https://{storage-account-name}.blob.core.windows.net/{container-name}/{blob-name}
```

Now the next important lesson is:

> A blob can have a URI, but that does **not** mean everyone can open it.

The URI tells Azure **where the file is**.

The access settings decide **who can open it**.

---

# 1. A Blob URI Is Only an Address

A blob URI is like a street address.

Example:

```text
https://mycompanymedia.blob.core.windows.net/property-videos/listing-123.mp4
```

This tells Azure:

```text
Storage account: mycompanymedia
Container: property-videos
Blob: listing-123.mp4
```

But the URI by itself does not always give permission.

If the blob is private, opening the URI in a browser may show an error like:

```text
ResourceNotFound
AuthorizationFailure
PublicAccessNotPermitted
AuthenticationFailed
```

This does not always mean the file is missing.

It may simply mean:

```text
The file exists, but you do not have permission to view it.
```

---

# 2. Public vs Private Access

Azure Blob Storage can be configured in different ways.

The most important access types are:

|Access Type|What It Means|
|---|---|
|Private|Nobody can access the blob unless they authenticate or have a valid access token|
|Public blob access|People can access individual blobs by URL, but cannot list the whole container|
|Public container access|People can access blobs and may list items inside the container|
|SAS URL access|People can access the blob only with a special temporary URL|
|Identity-based access|Access is controlled by Microsoft Entra ID / Azure permissions|

For most business use cases, you usually want:

```text
Private container + SAS URL when sharing is needed
```

---

# 3. Private Access

Private access means the blob cannot be opened by the public.

Example private blob URL:

```text
https://mycompanymedia.blob.core.windows.net/client-files/report.pdf
```

Someone opening that link may get blocked.

That is expected.

Private is best for:

```text
Client documents
Internal files
Backups
Invoices
Exports
Medical or legal documents
Anything sensitive
```

A private blob is safer because just knowing the URL is not enough.

The user still needs permission.

---

# 4. Public Blob Access

Public blob access means someone can open a specific blob if they have the full URL.

Example:

```text
https://mycompanymedia.blob.core.windows.net/public-assets/logo.png
```

This can be useful for:

```text
Website images
Public PDFs
Product images
Marketing files
Downloadable brochures
```

But public access should be used carefully.

A public URL can be shared, indexed, copied, downloaded, or scraped.

Use public access only for files that are truly meant to be public.

---

# 5. Public Container Access

Public container access is broader than public blob access.

It may allow people to list blobs inside the container.

Example:

```text
https://mycompanymedia.blob.core.windows.net/public-assets
```

Depending on settings, someone may be able to view a list of objects in that container.

This is usually not what you want unless the container is intentionally used as a public file bucket.

Safer option:

```text
Allow public read access to individual blobs only, not container listing.
```

---

# 6. SAS URLs

SAS means:

```text
Shared Access Signature
```

A SAS URL is a special URL that includes a temporary permission token.

Normal blob URL:

```text
https://mycompanymedia.blob.core.windows.net/client-files/report.pdf
```

SAS blob URL:

```text
https://mycompanymedia.blob.core.windows.net/client-files/report.pdf?sv=2024-11-04&se=2026-05-01T23%3A59%3A00Z&sr=b&sp=r&sig=abc123...
```

The part after the question mark is the SAS token:

```text
?sv=...&se=...&sr=...&sp=...&sig=...
```

That token controls access.

---

# 7. Breaking Down a SAS URL

A SAS URL may look scary, but it is mostly key-value pairs.

Example:

```text
https://mycompanymedia.blob.core.windows.net/client-files/report.pdf?sv=2024-11-04&se=2026-05-01T23%3A59%3A00Z&sr=b&sp=r&sig=abc123
```

Breakdown:

|SAS Part|Meaning|
|---|---|
|`sv`|Storage service version|
|`se`|Expiration time|
|`sr`|Signed resource|
|`sp`|Signed permissions|
|`sig`|Signature proving the URL is valid|

The most important ones for beginners are:

```text
se = when the link expires
sp = what the link allows
sig = the cryptographic signature
```

---

# 8. Common SAS Permissions

The `sp` value controls what the SAS URL can do.

Common permissions:

|Permission|Meaning|
|---|---|
|`r`|Read|
|`w`|Write|
|`d`|Delete|
|`l`|List|
|`c`|Create|
|`a`|Add|

Example:

```text
sp=r
```

Means:

```text
This SAS URL allows read-only access.
```

Example:

```text
sp=rw
```

Means:

```text
This SAS URL allows read and write.
```

For sharing a file with someone, you usually only want:

```text
sp=r
```

Read-only.

---

# 9. SAS Expiration

The `se` value controls when the SAS URL expires.

Example:

```text
se=2026-05-01T23%3A59%3A00Z
```

Decoded, this means:

```text
Expires on May 1, 2026 at 23:59 UTC
```

Once the SAS expires, the link should stop working.

This is useful because you can share access temporarily.

Examples:

```text
Expires in 15 minutes
Expires in 1 hour
Expires in 24 hours
Expires in 7 days
```

For client downloads, a short expiration is usually safer.

---

# 10. Why SAS URLs Are Useful

SAS URLs let you keep the container private while still sharing specific files.

That means this can stay private:

```text
https://mycompanymedia.blob.core.windows.net/client-files/report.pdf
```

But this can work temporarily:

```text
https://mycompanymedia.blob.core.windows.net/client-files/report.pdf?sv=...&se=...&sp=r&sig=...
```

This is useful for:

```text
Client download links
Private invoices
Temporary video previews
Generated exports
One-time file delivery
App-generated upload links
```

---

# 11. Important Security Warning

A SAS URL is like a temporary key.

Anyone who has the full SAS URL can use it until it expires, unless it is revoked or blocked by other controls.

So do not casually paste SAS URLs into:

```text
Public websites
GitHub repos
Support tickets
Screenshots
Shared docs
Chat rooms
Client notes
```

Treat SAS URLs like passwords.

Especially if they include write or delete permission.

---

# 12. Better vs Riskier SAS Examples

Better:

```text
Private container
Read-only SAS
Expires in 15 minutes
Only points to one blob
```

Riskier:

```text
Container-level SAS
Read/write/delete permissions
Expires in 1 year
Shared in public chat
```

Very risky:

```text
Account-level SAS
Full permissions
No short expiration
Saved in source code
```

---

# 13. Common Beginner Mistake

A common beginner mistake is thinking:

> “I have the Azure Blob URL, so it should open in the browser.”

Not always.

The URL may be correct, but access may be blocked.

Ask these questions:

```text
Is the container private?
Is public access disabled on the storage account?
Does the URL need a SAS token?
Is the SAS token expired?
Does the SAS token have read permission?
Was the blob name typed correctly?
```

---

# 14. Another Common Mistake: Sharing the Wrong URL

Sometimes people copy the blob URL without the SAS token.

This URL may not work:

```text
https://mycompanymedia.blob.core.windows.net/client-files/report.pdf
```

But this one may work:

```text
https://mycompanymedia.blob.core.windows.net/client-files/report.pdf?sv=...&se=...&sp=r&sig=...
```

If the blob is private, the SAS token part is required.

That means everything after the `?` matters.

---

# 15. Simple Troubleshooting Table

|Problem|Possible Cause|
|---|---|
|URL opens for you but not client|You are logged into Azure; client is not|
|URL says authorization failed|Blob/container is private or SAS is invalid|
|URL says authentication failed|SAS token may be expired, malformed, or signed incorrectly|
|URL says resource not found|Blob path may be wrong, or Azure may hide private resources|
|URL worked yesterday but not today|SAS token may have expired|
|Upload fails|SAS may not include write/create permission|
|Can download but cannot list files|SAS may have read permission but not list permission|

---

# 16. Best Practice for Most Apps

For most apps, use this pattern:

```text
Keep containers private.
Generate SAS URLs only when needed.
Make SAS URLs read-only unless upload is required.
Use short expiration times.
Do not store SAS URLs permanently.
Store the blob path, not the full SAS URL.
```

Instead of storing this forever:

```text
https://mycompanymedia.blob.core.windows.net/client-files/report.pdf?sv=...&sig=...
```

Store this:

```text
client-files/report.pdf
```

Then generate a fresh SAS URL when the user needs access.

---

# 17. Practical Example

Imagine your app stores client invoices.

Storage structure:

```text
Storage Account: mycompanymedia
Container: invoices
Blob: clients/acme/invoice-1001.pdf
```

Private blob URL:

```text
https://mycompanymedia.blob.core.windows.net/invoices/clients/acme/invoice-1001.pdf
```

This should not be public.

When the client clicks “Download Invoice,” your app generates a temporary SAS URL:

```text
https://mycompanymedia.blob.core.windows.net/invoices/clients/acme/invoice-1001.pdf?sv=...&se=...&sp=r&sig=...
```

The SAS URL may expire in 10 minutes.

That way:

```text
The file stays private.
The client can still download it.
The link stops working later.
```

# 18. Simple Mental Model

Think of it like this:

```text
Blob URI = address of the file
Private access = locked door
SAS token = temporary key
Expiration = key stops working after a certain time
Permissions = what the key can do
```

Example:

```text
https://mycompanymedia.blob.core.windows.net/client-files/report.pdf
```

Means:

```text
Here is the address.
```

Example:

```text
https://mycompanymedia.blob.core.windows.net/client-files/report.pdf?sv=...&se=...&sp=r&sig=...
```

Means:

```text
Here is the address plus a temporary read-only key.
```

---

# Quick Recap

|Concept|Meaning|
|---|---|
|Blob URI|Address of a blob|
|Private blob|Blob exists but cannot be opened publicly|
|Public blob|Blob can be opened by URL|
|Public container|Container may allow broader public access|
|SAS URL|Temporary signed URL that grants access|
|`sp=r`|Read-only permission|
|`se=`|Expiration time|
|`sig=`|Signature that validates the SAS token|

The big lesson:

> A blob URI tells Azure where the file is.  
> A SAS URL tells Azure who is allowed to access it, what they can do, and for how long.
