A Storage Account is a single resource that actually contains **four separate services**:

```text
Storage Account
├── Blob Storage    ← objects/files, the focus of lessons 1–4
├── File Shares     ← real network drives (SMB / NFS)
├── Queues          ← simple FIFO message broker
└── Tables          ← cheap NoSQL key/value store
```

Lessons 1–4 covered Blob in depth. This lesson covers the other three: when to use each, when not to, and how to create one quickly.

---

## 1. Side-by-Side Comparison

| Service       | Best for                                    | Access pattern                | Common alternative           |
|---------------|---------------------------------------------|-------------------------------|------------------------------|
| Blob          | Files, images, videos, backups, big data    | HTTPS REST, SDK               | S3, GCS                      |
| File Shares   | Drop-in replacement for a network drive     | SMB, NFS, REST                | On-prem file server, NetApp  |
| Queues        | Simple async messages between processes     | HTTPS REST, SDK               | Service Bus, RabbitMQ        |
| Tables        | Cheap key/value rows                        | HTTPS REST, SDK               | Cosmos DB Table API, DynamoDB|

A single storage account can host all four. They share the same name, region, redundancy setting, keys, and firewall — but each has its own service endpoint:

```text
https://mystorage.blob.core.windows.net
https://mystorage.file.core.windows.net
https://mystorage.queue.core.windows.net
https://mystorage.table.core.windows.net
```

## 2. Azure Files — Network Drive in the Cloud

An Azure File Share is a **fully managed SMB or NFS file share**. You can mount it on Windows, macOS, Linux, and inside containers as if it were a local drive.

### When to use Files

```text
- "Lift and shift" of an app that expects \\server\share or /mnt/share
- A shared config or assets directory across multiple VMs
- A team file drop accessible to several services
- CI/CD agents that need a shared workspace
- Replacing an on-prem file server
```

### When NOT to use Files

```text
- Storing user-uploaded objects served over HTTPS  → use Blob
- High-throughput object stores                    → use Blob
- A real database                                  → use SQL or Cosmos DB
```

### Create + mount in 3 commands

```bash
RG=rg-files-demo
ACCT=myfileslearner$RANDOM
SHARE=teamfiles
LOC=eastus

az group create -n $RG -l $LOC

az storage account create -n $ACCT -g $RG -l $LOC \
  --sku Standard_LRS --kind StorageV2

az storage share-rm create \
  --resource-group $RG \
  --storage-account $ACCT \
  --name $SHARE \
  --quota 5
```

Mount on macOS/Linux (SMB):

```bash
KEY=$(az storage account keys list -g $RG -n $ACCT --query "[0].value" -o tsv)

mkdir -p ~/azure-files-mount
sudo mount -t smbfs \
  "//${ACCT}:${KEY}@${ACCT}.file.core.windows.net/${SHARE}" \
  ~/azure-files-mount
```

Mount on Windows (PowerShell):

```powershell
$user = "Azure\$env:ACCT"
$key  = (az storage account keys list -g $env:RG -n $env:ACCT --query "[0].value" -o tsv)
$cred = New-Object System.Management.Automation.PSCredential($user, (ConvertTo-SecureString $key -AsPlainText -Force))

New-PSDrive -Name Z -PSProvider FileSystem `
  -Root "\\$env:ACCT.file.core.windows.net\$env:SHARE" `
  -Credential $cred -Persist
```

Now `~/azure-files-mount` (or `Z:`) is a real folder backed by Azure.

### File tiers

Like Blob, Files supports tiers:

```text
Premium       — SSD-backed, low latency, expensive
Transaction Optimized — for chatty workloads
Hot           — general-purpose
Cool          — cheaper storage, costlier transactions
```

## 3. Azure Queues — Simple Async Messaging

A Queue Storage queue is a **FIFO list of small messages** (up to 64 KB each, base64-encoded). Producers `enqueue`, consumers `dequeue` and `delete`.

### When to use Queues

```text
- Decoupling a producer from a slower consumer
- Buffering bursty workloads (e.g., image upload → background resize)
- Simple "to-do" list for a worker pool
- Triggering Azure Functions on each message
- You want VERY cheap, very simple, no fancy features
```

### When NOT to use Queues

```text
- Need ordering across millions of messages   → use Service Bus FIFO sessions
- Need pub/sub or topics                      → use Service Bus or Event Grid
- Need millions of events/second              → use Event Hubs
- Need messages > 64 KB                       → store payload in Blob, queue a pointer
```

### Create and use a queue

```bash
QUEUE=image-jobs
az storage queue create --account-name $ACCT --account-key $KEY --name $QUEUE

az storage message put \
  --account-name $ACCT --account-key $KEY \
  --queue-name $QUEUE \
  --content "resize:photo-001.jpg"

az storage message get \
  --account-name $ACCT --account-key $KEY \
  --queue-name $QUEUE
```

### Visibility timeout pattern

When you `get` a message, it becomes invisible for a few seconds (default 30s). The consumer must `delete` it before the timeout expires:

```text
1. Consumer calls GET → message hidden for 30s
2. Consumer processes the work (resize image)
3. Consumer calls DELETE → message gone for good
4. If consumer crashes, the 30s expires and another worker can pick it up
```

This makes queues self-healing in the face of crashes.

### Queues vs. Service Bus (the always-asked question)

```text
Queue Storage      Service Bus Queues
---------------    ----------------------------------
< $0.04 per million ops   Higher per message
64 KB max msg              256 KB / 1 MB max msg
Simple                     Sessions, transactions, dead-letter, FIFO
At-least-once              At-least-once + at-most-once options
```

For learner projects, Queue Storage is plenty. Reach for Service Bus when you need real enterprise messaging.

## 4. Azure Tables — Cheap NoSQL Key/Value Store

Table Storage is a **schemaless, partitioned key/value store**. Each row has a partition key + row key + arbitrary properties:

```text
Row example:
{
  "PartitionKey": "user-123",
  "RowKey":       "session-9f8e2",
  "loginAt":      "2026-05-01T03:14:00Z",
  "userAgent":    "Mozilla/5.0 ...",
  "country":      "US"
}
```

### When to use Tables

```text
- Cheap structured data at any scale
- Simple lookups by PartitionKey + RowKey
- High write throughput
- Telemetry, audit logs, settings, user profiles
```

### When NOT to use Tables

```text
- Complex queries / joins        → use SQL Database
- Global distribution + SLA      → use Cosmos DB Table API (compatible)
- Full-text search               → use Azure AI Search
- Strong relational integrity    → use SQL Database
```

### Create and use a table

```bash
TABLE=usersessions
az storage table create --account-name $ACCT --account-key $KEY --name $TABLE

az storage entity insert \
  --account-name $ACCT --account-key $KEY \
  --table-name $TABLE \
  --entity PartitionKey=user-123 RowKey=session-001 country=US
```

Query in Python:

```python
from azure.data.tables import TableClient

table = TableClient.from_connection_string(
    conn_str="DefaultEndpointsProtocol=https;AccountName=...;AccountKey=...;EndpointSuffix=core.windows.net",
    table_name="usersessions",
)

for entity in table.query_entities("PartitionKey eq 'user-123'"):
    print(entity)
```

### Designing PartitionKey and RowKey

This is the **only** thing that matters for Table performance.

```text
PartitionKey  → groups rows that are always queried together
RowKey        → unique identifier within the partition
```

Good PartitionKey choices:

```text
user_id   when most queries are "all sessions for user X"
date      when most queries are "all events on day Y"
deviceId  when most queries are "all telemetry from device Z"
```

Bad: `RowKey="auto-incrementing id"` with no thoughtful PartitionKey. You will end up scanning the whole table.

### Table Storage vs. Cosmos DB Table API

```text
Table Storage          Cosmos DB Table API
-------------------    --------------------------------------------
Pennies per GB         Premium pricing (per RU/s + per GB)
Single region          Multi-region replication, 99.999% SLA
Eventually consistent  Five consistency models
Simple SDK             Same SDK works against both
```

Start with Table Storage. Migrate to Cosmos DB Table API only when you need global distribution or stronger SLAs — your code barely changes.

## 5. One Mental Model for All Four

```text
Blob   → Storage for files                      (objects)
Files  → Storage as a network drive             (file system)
Queue  → Storage for messages                   (work to do)
Table  → Storage for rows                       (structured records)
```

All four share:

```text
- The same Storage Account
- The same redundancy (LRS / ZRS / GRS / RA-GRS)
- The same firewall + private endpoint configuration
- The same access keys
- Per-service endpoints under *.core.windows.net
```

---

## 6. Picking the Right Service — Quick Decision Tree

```text
Need to store...

├── files served over HTTP, large objects, backups
│       → Blob Storage
│
├── a network share that an OS can mount
│       → Azure Files
│
├── small messages between processes
│       → Queue Storage  (or Service Bus for advanced needs)
│
└── structured records you query by key
        → Table Storage  (or Cosmos DB Table API for global scale)
```

## 7. Cost Snapshot

(Approximate, Standard_LRS, Hot, East US — always confirm with the calculator.)

```text
Service        Per-GB / month       Per million transactions
--------       -----------------    --------------------------
Blob Hot       ~$0.02               ~$0.05 read, $0.05 write
Files Hot      ~$0.06               higher than Blob
Queue          ~$0.05               ~$0.04 per million ops
Table          ~$0.05               ~$0.04 per million ops
```

Queues and Tables are cheap enough that for small projects they basically round to $0.

## Quick Recap

| Service | Use it when…                                   | Pitfall                               |
|---------|------------------------------------------------|---------------------------------------|
| Blob    | You need an object/file store                  | Don't put structured rows here        |
| Files   | You need an SMB/NFS share for an OS or app     | Per-GB cost is higher than Blob       |
| Queue   | You need simple async messaging                | 64 KB cap; no advanced features       |
| Table   | You need cheap key/value rows at scale         | Partition key choice is everything    |
