When something breaks in Azure — a resource was deleted, a deploy failed, a permission was changed — the **Activity Log** tells you what happened, when, and who did it.

This lesson also covers how to diagnose failures in App Service and VMs using the tools Azure provides.

---

# 1. The Activity Log

The **Activity Log** records every control-plane operation on Azure resources:

```text
- Resource created
- Resource deleted
- Configuration changed
- Role assignment added or removed
- Policy applied
- Deployment started or failed
```

It does not record:

```text
- What your application code did
- Database queries
- Application-level errors
- Data reads (viewing a file, reading a secret)
```

For those, you need Application Insights or Log Analytics (previous two lessons).

---

# 2. View the Activity Log

### Per resource:

```text
Portal → any resource → Activity log (left sidebar)
```

### Per resource group:

```text
Portal → Resource group → Activity log
```

### Per subscription:

```text
Portal → Monitor → Activity log
```

You can filter by:

```text
Time span      → last 1 hour, 24 hours, 7 days, custom
Subscription
Resource group
Resource
Event severity → Critical, Error, Warning, Informational
Operation      → search by operation name
Caller         → who initiated it (user email or service principal)
```

# 3. Activity Log via CLI

```bash
# Last 24 hours of activity for a resource group
az monitor activity-log list \
  --resource-group $RG \
  --start-time $(date -u -d "-1 day" '+%Y-%m-%dT%H:%MZ') \
  --output table

# Filter to errors only
az monitor activity-log list \
  --resource-group $RG \
  --start-time $(date -u -d "-1 day" '+%Y-%m-%dT%H:%MZ') \
  --status Failed \
  --output table

# Find operations by a specific user
az monitor activity-log list \
  --resource-group $RG \
  --caller alice@example.com \
  --start-time $(date -u -d "-7 days" '+%Y-%m-%dT%H:%MZ') \
  --output table
```

# 4. Activity Log via KQL

If you have diagnostic settings sending the Activity Log to Log Analytics:

```kql
AzureActivity
| where TimeGenerated > ago(24h)
| where ActivityStatus == "Failed"
| project TimeGenerated, OperationName, ResourceGroup, Caller, ActivityStatusValue
| order by TimeGenerated desc
```

Find who deleted a resource:

```kql
AzureActivity
| where TimeGenerated > ago(7d)
| where OperationNameValue has "delete"
| project TimeGenerated, OperationName, ResourceGroup, Resource, Caller
| order by TimeGenerated desc
```

Find all role assignment changes:

```kql
AzureActivity
| where TimeGenerated > ago(7d)
| where OperationNameValue has "roleAssignment"
| project TimeGenerated, OperationName, Caller, Properties
| order by TimeGenerated desc
```

# 5. Diagnosing App Service Failures

When an App Service is showing errors, use these steps:

### Step 1: Check the overview

```text
Portal → App Service → Overview
  - HTTP 5xx errors (shows in the chart)
  - Requests per second
  - Average response time
```

### Step 2: Stream logs

```bash
az webapp log tail --resource-group $RG --name $APP
```

This streams stdout/stderr from your app in real-time.

Enable logging first if nothing appears:

```bash
az webapp log config \
  --resource-group $RG \
  --name $APP \
  --docker-container-logging filesystem \
  --level verbose
```

### Step 3: Check Diagnose and Solve Problems

```text
Portal → App Service → Diagnose and solve problems
```

Azure's built-in diagnostic tool analyzes your app and suggests causes for common issues:

```text
- App crashes
- Performance degradation
- Container issues
- Platform issues (Azure-side outages)
```

### Step 4: Check Kudu (advanced)

Kudu is the internal management console for App Service.

Access it at:

```text
https://<your-app-name>.scm.azurewebsites.net
```

Inside Kudu:

```text
Debug console (Bash/CMD)   → browse files, run commands
Process explorer            → see running processes and memory
Log files                   → /LogFiles/
```

# 6. Diagnosing Azure Functions Failures

### Stream function logs:

```bash
az functionapp log tail --name $FUNCAPP --resource-group $RG
```

### Check invocation history in the portal:

```text
Portal → Function App → Functions → <your function> → Monitor
  - Invocations tab: success/failure list with timestamps
  - Click a failed invocation to see the error message and stack trace
```

### Query Application Insights (if connected):

```kql
exceptions
| where timestamp > ago(24h)
| where cloud_RoleName == "my-func-app"
| project timestamp, type, outerMessage, innermostMessage
| order by timestamp desc
```

# 7. Diagnosing VM Failures

### Check if the VM is running:

```bash
az vm show -g $RG -n $VM --show-details --query powerState -o tsv
```

### Serial console output (for boot failures):

```bash
az vm boot-diagnostics get-boot-log \
  --resource-group $RG \
  --name $VM
```

Or enable boot diagnostics in portal:

```text
Portal → VM → Boot diagnostics → Enable → Save
  → Boot diagnostics → Screenshot shows what the VM's console currently displays
```

### Run a command without SSH:

```bash
az vm run-command invoke \
  --resource-group $RG \
  --name $VM \
  --command-id RunShellScript \
  --scripts "systemctl status nginx"
```

### Check agent and extension status:

```bash
az vm get-instance-view \
  --resource-group $RG \
  --name $VM \
  --query "instanceView.extensions"
```

# 8. Resource Health

**Resource Health** tells you if Azure itself is having issues with a specific resource (not a problem with your code or config).

```text
Portal → any resource → Resource health (left sidebar)
```

Statuses:

```text
Available         → resource is healthy
Degraded          → reduced performance, Azure investigating
Unavailable       → resource is not accessible
Unknown           → status cannot be determined
```

If Resource Health shows "Unavailable" and you cannot figure out why, it is likely an Azure platform incident — check:

```text
https://status.azure.com/en-us/status
```

---

# 9. Azure Service Health

**Azure Service Health** shows Azure-wide and regional incidents.

```text
Portal → Monitor → Service Health
  → Active events: current outages and advisory notices
  → Health history: past incidents
  → Health alerts: set up email notifications for incidents affecting your services/regions
```

Subscribe to alerts for your subscription + regions + services:

```text
Service Health → Health alerts → Add service health alert
  → Select: services you use, regions you deploy to
  → Alert on: Incident, Maintenance, Health advisories
  → Action group: email notification
```

# 10. Common Failure Scenarios and Where to Look

| Symptom | Where to look first |
|---|---|
| App returning 500 errors | App Service → Log Stream, Application Insights failures |
| App stopped responding | App Service → Overview charts, Resource Health |
| Someone deleted a resource | Monitor → Activity Log → filter by delete |
| Deployment failed | Activity Log → filter by Failed |
| VM not starting | VM → Boot diagnostics screenshot |
| Permission error in app | App Service → Log Stream (look for 401/403), RBAC role assignments |
| Slow queries | Application Insights → Dependency slowness, SQL Insights |
| Azure-wide issue | Service Health → Active events |

# Quick Recap

| Tool | What It Shows |
|---|---|
| Activity Log | Control-plane operations: who did what to Azure resources |
| Log Stream | Real-time stdout/stderr from App Service or Functions |
| Diagnose and Solve Problems | Built-in App Service diagnostic wizard |
| Kudu | Internal management console for App Service |
| Boot diagnostics | VM console output for boot failures |
| Resource Health | Whether Azure reports the resource as healthy |
| Service Health | Azure-wide incident tracker |
