Every database in Azure is private by default — nothing can connect to it until you explicitly allow it.

This lesson covers how connections work, how to secure them, and the three levels of access control you need to understand.

---

# 1. The Three Layers of Database Access

```text
Layer 1: Network (firewall / VNet)     → can this IP or service reach the database at all?
Layer 2: Authentication                → is the username/password or identity valid?
Layer 3: Authorization                 → what can this user do? (read, write, create tables, etc.)
```

All three must pass. A valid password does not help if the firewall blocks the connection.

---

# 2. Connection Strings

A **connection string** is a single string that tells your app:

```text
- Where the database is (host, port)
- Which database to use
- How to authenticate (username + password, or other)
- Connection options (encryption, timeout, etc.)
```

### Azure SQL Database

```text
Server=tcp:mysqlserver.database.windows.net,1433;
Initial Catalog=myappdb;
User ID=sqladmin;
Password=YourStr0ng!Pass;
Encrypt=True;
Connection Timeout=30;
```

### PostgreSQL Flexible Server

```text
postgresql://sqladmin:YourStr0ng!Pass@myserver.postgres.database.azure.com:5432/myappdb?sslmode=require
```

### Cosmos DB (NoSQL API)

```text
AccountEndpoint=https://myaccount.documents.azure.com:443/;AccountKey=abc123...==;
```

### MySQL Flexible Server

```text
Server=myserver.mysql.database.azure.com;
UserID=sqladmin@myserver;
Password=YourStr0ng!Pass;
Database=myappdb;
SslMode=Required;
```

---

# 3. Where to Store Connection Strings

**Never hardcode connection strings in source code.**

Patterns from most to least safe:

```text
1. Managed Identity (no password at all)          — best
2. Azure Key Vault + Key Vault reference          — very good
3. App Service / Functions Application Settings   — good
4. Environment variables loaded at startup        — acceptable for dev
5. .env file (never commit to Git)                — only local dev
6. Hardcoded in code                              — never
```

### App Service Application Settings

Store the connection string in the portal:

```text
App Service → Configuration → Connection strings → Add
  Name:  DATABASE_URL
  Value: your connection string
  Type:  SQLAzure / PostgreSQL / MySQL / Custom
```

Or via CLI:

```bash
az webapp config connection-string set \
  --resource-group $RG \
  --name $APP \
  --connection-string-type SQLAzure \
  --settings DATABASE_URL="Server=tcp:..."
```

Your app reads it as an environment variable:

```python
import os
conn_str = os.environ["DATABASE_URL"]
```

# 4. SQL Database Firewall Rules

Azure SQL Database has an IP firewall at the **server** level.

Allow your development machine:

```bash
MY_IP=$(curl -s https://ifconfig.me)

az sql server firewall-rule create \
  --resource-group $RG \
  --server $SERVER \
  --name AllowMyIP \
  --start-ip-address $MY_IP \
  --end-ip-address $MY_IP
```

Allow Azure services (App Service, Functions, etc.):

```bash
az sql server firewall-rule create \
  --resource-group $RG \
  --server $SERVER \
  --name AllowAzureServices \
  --start-ip-address 0.0.0.0 \
  --end-ip-address 0.0.0.0
```

List all rules:

```bash
az sql server firewall-rule list \
  --resource-group $RG \
  --server $SERVER \
  --output table
```

Delete a rule:

```bash
az sql server firewall-rule delete \
  --resource-group $RG \
  --server $SERVER \
  --name AllowMyIP
```

---

# 5. PostgreSQL Firewall Rules

Same concept — rules are at the server level:

```bash
az postgres flexible-server firewall-rule create \
  --resource-group $RG \
  --name $SERVER \
  --rule-name AllowMyIP \
  --start-ip-address $MY_IP \
  --end-ip-address $MY_IP
```

Allow Azure services:

```bash
az postgres flexible-server firewall-rule create \
  --resource-group $RG \
  --name $SERVER \
  --rule-name AllowAllAzureServices \
  --start-ip-address 0.0.0.0 \
  --end-ip-address 0.0.0.0
```

# 6. Private Endpoints — Removing the Public Internet Exposure

Both firewall rules above still expose the database to a public IP address. Anyone on the internet can try to connect (though they still need valid credentials to get past auth).

For production, use **Private Endpoints** to remove the public endpoint entirely.

A Private Endpoint gives your database a private IP inside your VNet:

```text
Before private endpoint:
  App Service → public internet → mysqlserver.database.windows.net (public IP)

After private endpoint:
  App Service → VNet → Private Endpoint → mysqlserver.database.windows.net (private IP 10.0.1.4)
```

The database is no longer reachable from the public internet.

Create a private endpoint for SQL Database:

```bash
# Get the SQL server resource ID
SQL_ID=$(az sql server show -g $RG -n $SERVER --query id -o tsv)

# Create a VNet and subnet first (if not already done)
az network vnet create \
  --resource-group $RG \
  --name my-vnet \
  --address-prefix 10.0.0.0/16

az network vnet subnet create \
  --resource-group $RG \
  --vnet-name my-vnet \
  --name db-subnet \
  --address-prefixes 10.0.1.0/24

# Create the private endpoint
az network private-endpoint create \
  --resource-group $RG \
  --name sql-private-endpoint \
  --vnet-name my-vnet \
  --subnet db-subnet \
  --private-connection-resource-id $SQL_ID \
  --group-id sqlServer \
  --connection-name sql-connection
```

After this, disable public access:

```bash
az sql server update \
  --resource-group $RG \
  --name $SERVER \
  --restrict-outbound-network-access true \
  --public-network-access Disabled
```

---

# 7. Service Endpoints vs Private Endpoints

| Feature | Service Endpoint | Private Endpoint |
|---|---|---|
| Private IP in VNet | No | Yes |
| Database visible on internet | Yes (still public IP) | No (fully private) |
| DNS resolution | Public | Private (needs Private DNS Zone) |
| Setup complexity | Low | Medium |
| Best for | Dev/test, small projects | Production, compliance requirements |

Service endpoints restrict which VNets can connect, but the database still has a public address. Private endpoints make the database truly private.

# 8. Managed Identity — No Connection String Password

The safest production approach stores no password anywhere.

Instead, your app's identity (Managed Identity) authenticates directly to the database.

For Azure SQL Database:

```text
1. Enable Managed Identity on your App Service or Function (lesson 05/3)
2. Create a SQL user for the identity inside the database:
```

```sql
-- Run as SQL admin:
CREATE USER [MyAppServiceName] FROM EXTERNAL PROVIDER;
ALTER ROLE db_datareader ADD MEMBER [MyAppServiceName];
ALTER ROLE db_datawriter ADD MEMBER [MyAppServiceName];
```

```text
3. Use a passwordless connection string in your app:
```

For .NET:

```csharp
var conn = new SqlConnection(
    "Server=tcp:mysqlserver.database.windows.net,1433;" +
    "Database=myappdb;Authentication=Active Directory Default;"
);
```

For Python (using azure-identity):

```python
from azure.identity import DefaultAzureCredential
import struct, pyodbc

credential = DefaultAzureCredential()
token = credential.get_token("https://database.windows.net/.default")
token_bytes = token.token.encode("utf-16-le")
token_struct = struct.pack(f"<I{len(token_bytes)}s", len(token_bytes), token_bytes)

conn = pyodbc.connect(
    "Driver={ODBC Driver 18 for SQL Server};"
    "Server=tcp:mysqlserver.database.windows.net,1433;"
    "Database=myappdb;Encrypt=yes;",
    attrs_before={1256: token_struct}
)
```

# 9. Quick Security Checklist

```text
[ ] Connection string is in Application Settings, not in source code
[ ] Firewall rules are limited to known IPs + Azure services
[ ] Public access is disabled (private endpoint) for production
[ ] Managed Identity used instead of passwords in production apps
[ ] No account keys or connection strings committed to Git
[ ] Least-privilege database roles (db_datareader + db_datawriter, not db_owner)
```

---

# Quick Recap

| Topic | Key Point |
|---|---|
| Connection string | Host, database, credentials, options in one string |
| Firewall rules | Must whitelist your IP or Azure services before connecting |
| Service endpoint | Restricts VNet access but database still has public IP |
| Private endpoint | Fully private — no public IP for the database |
| App Settings | Store connection strings in portal/CLI, not source code |
| Managed Identity | Passwordless auth — preferred for production |
