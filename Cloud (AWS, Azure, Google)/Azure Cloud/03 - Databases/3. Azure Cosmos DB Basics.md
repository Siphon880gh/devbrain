Azure Cosmos DB is a globally distributed NoSQL database.

Instead of tables and rows, you store **documents** — JSON objects. There is no fixed schema. Each document can have different fields.

---

# 1. Key Concepts

```text
Cosmos DB Account
└── Database
    └── Container (like a table, but schemaless)
        └── Items (JSON documents)
```

Example item:

```json
{
    "id": "user-001",
    "name": "Alice",
    "email": "alice@example.com",
    "preferences": {
        "theme": "dark",
        "notifications": true
    },
    "tags": ["admin", "beta-tester"]
}
```

No schema required. The next item can have completely different fields.

---

# 2. Partition Key

Every container must have a **partition key**.

The partition key tells Cosmos DB how to distribute data across physical storage partitions for performance and scale.

Example: if your partition key is `/userId`, all items for the same user are stored together.

```json
{
    "id": "order-5001",
    "userId": "user-001",
    "total": 49.99,
    "items": [...]
}
```

```text
Partition key: /userId
Value for this item: "user-001"
```

Rules for choosing a partition key:

```text
- High cardinality (many distinct values — user IDs, order IDs, product IDs)
- Evenly distributed access (no "hot partitions")
- Often included in queries (enables efficient reads)
```

Bad partition keys:

```text
/status    → only a few values (pending, completed, etc.) → hot partitions
/country   → uneven — USA gets 80% of traffic
```

Good partition keys:

```text
/userId
/tenantId
/deviceId
/orderId
```

---

# 3. Request Units (RU/s)

Cosmos DB charges in **Request Units** rather than vCPUs or memory.

```text
1 RU = cost to read a 1 KB document by its partition key and id
```

Approximate costs:

```text
Read 1 KB document by id:     1 RU
Write 1 KB document:          5 RUs
Query with 1 result:          ~2–10 RUs (depends on index use)
Large query across partitions: many more RUs
```

You provision a minimum throughput:

```text
400 RU/s = the minimum (being retired for serverless)
Serverless = pay per RU consumed (no minimum, better for sporadic workloads)
Autoscale = set max RU/s, charged on actual usage (good for variable loads)
```

---

# 4. Free Tier

Cosmos DB has a **free tier**:

```text
1,000 RU/s + 25 GB storage — free forever
One free tier account per Azure subscription
```

For most beginner projects, the free tier is more than enough.

# 5. Create a Cosmos DB Account (CLI)

```bash
RG=rg-cosmos-demo
LOC=eastus
ACCOUNT=mycosmosaccount$RANDOM    # globally unique, lowercase, alphanumeric + dash

az group create -n $RG -l $LOC

az cosmosdb create \
  --name $ACCOUNT \
  --resource-group $RG \
  --locations regionName=$LOC \
  --capabilities EnableServerless
```

> `--capabilities EnableServerless` creates a serverless account (pay per RU, no minimum throughput). Good for learning and sporadic workloads.

Create a database and container:

```bash
DB=mydb
CONTAINER=users

az cosmosdb sql database create \
  --account-name $ACCOUNT \
  --resource-group $RG \
  --name $DB

az cosmosdb sql container create \
  --account-name $ACCOUNT \
  --resource-group $RG \
  --database-name $DB \
  --name $CONTAINER \
  --partition-key-path "/userId"
```

# 6. Interact via the Portal — Data Explorer

The Azure Portal has a built-in **Data Explorer** for Cosmos DB.

```text
Portal → search "Azure Cosmos DB" → click your account
→ Data Explorer
→ expand your database and container
→ click "Items"
→ "New Item" to add a document
```

You can write, query, and delete documents from the browser.

# 7. Query with SQL-Like Syntax

Cosmos DB's NoSQL API supports a SQL-like query language.

```sql
SELECT * FROM c WHERE c.userId = "user-001"

SELECT c.name, c.email FROM c

SELECT * FROM c WHERE c.age > 30 ORDER BY c.name

SELECT VALUE COUNT(1) FROM c
```

Note: `c` is the alias for the container. You must include the partition key in most queries for best performance.

# 8. SDK — Python

Install:

```bash
pip install azure-cosmos azure-identity
```

Connect and write:

```python
from azure.cosmos import CosmosClient, PartitionKey
from azure.identity import DefaultAzureCredential

ACCOUNT_URL = "https://mycosmosaccount1234.documents.azure.com:443/"

client = CosmosClient(url=ACCOUNT_URL, credential=DefaultAzureCredential())
db = client.get_database_client("mydb")
container = db.get_container_client("users")

item = {
    "id": "user-001",
    "userId": "user-001",
    "name": "Alice",
    "email": "alice@example.com"
}

container.upsert_item(item)
```

Read:

```python
result = container.read_item(item="user-001", partition_key="user-001")
print(result["name"])
```

Query:

```python
items = container.query_items(
    query="SELECT * FROM c WHERE c.userId = @uid",
    parameters=[{"name": "@uid", "value": "user-001"}],
    enable_cross_partition_query=False
)
for item in items:
    print(item)
```

# 9. SDK — Node.js

Install:

```bash
npm install @azure/cosmos @azure/identity
```

```javascript
import { CosmosClient } from "@azure/cosmos";
import { DefaultAzureCredential } from "@azure/identity";

const client = new CosmosClient({
    endpoint: "https://mycosmosaccount1234.documents.azure.com:443/",
    aadCredentials: new DefaultAzureCredential()
});

const container = client.database("mydb").container("users");

// Write
await container.items.upsert({
    id: "user-002",
    userId: "user-002",
    name: "Bob",
    email: "bob@example.com"
});

// Read
const { resource } = await container.item("user-002", "user-002").read();
console.log(resource.name);

// Query
const { resources } = await container.items
    .query("SELECT * FROM c WHERE c.userId = 'user-002'")
    .fetchAll();
console.log(resources);
```

# 10. Connection String vs Key vs Managed Identity

**Connection string** (includes account key — only use for local dev):

```bash
az cosmosdb keys list \
  --name $ACCOUNT \
  --resource-group $RG \
  --type connection-strings
```

**Account key** (avoid in production — treat like a password):

```bash
az cosmosdb keys list \
  --name $ACCOUNT \
  --resource-group $RG
```

**Managed Identity** (recommended for production):

```text
Grant your App Service or Function the "Cosmos DB Built-in Data Contributor" role:

az cosmosdb sql role assignment create \
  --account-name $ACCOUNT \
  --resource-group $RG \
  --role-definition-name "Cosmos DB Built-in Data Contributor" \
  --scope "/" \
  --principal-id <managed-identity-object-id>
```

Then use `DefaultAzureCredential()` in your code instead of a key.

# 11. Common Errors

| Error | Likely Cause |
|---|---|
| `Resource Not Found` | Wrong database or container name, or item ID / partition key mismatch |
| `Request rate too large` | Not enough RU/s provisioned; throttled |
| `Cross-partition query disabled` | Pass `enable_cross_partition_query=True` or include partition key |
| `The input content is invalid` | JSON document missing required `id` field |
| Auth error | Wrong key, or RBAC role not assigned for Managed Identity |
