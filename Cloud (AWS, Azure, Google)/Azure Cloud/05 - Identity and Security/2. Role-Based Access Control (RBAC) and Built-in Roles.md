Azure RBAC (Role-Based Access Control) controls what a user, group, or service can do with Azure resources.

There are two separate systems beginners often confuse:

```text
Microsoft Entra ID roles   → control what you can do inside Entra ID
                             (create users, manage tenant settings, etc.)

Azure RBAC roles           → control what you can do with Azure resources
                             (create VMs, read storage, delete resource groups, etc.)
```

This lesson focuses on **Azure RBAC**.

---

# 1. The Three Concepts

```text
Who (Security Principal)   + What (Role Definition)  + Where (Scope)  = Assignment
```

Example:

```text
Alice                      + Contributor              + rg-prod        = Alice can create/edit resources in rg-prod
DevTeam group              + Reader                   + Subscription   = DevTeam can read everything in the subscription
MyAppService               + Storage Blob Data Reader + storage account = MyAppService can read blobs
```

---

# 2. Scopes

RBAC works at four levels:

```text
Management group  (spans subscriptions)
└── Subscription
    └── Resource group
        └── Individual resource
```

Roles assigned at a higher scope inherit down:

```text
Reader assigned at Subscription
  → Applies to all resource groups in that subscription
  → Applies to all resources in those resource groups
```

For learning and least-privilege: assign roles at the **resource group** level.

---

# 3. Built-in Roles

Azure has 70+ built-in roles. The most important ones:

| Role | Can Do | Cannot Do |
|---|---|---|
| Owner | Everything | — |
| Contributor | Create, update, delete resources | Manage access (no role assignments) |
| Reader | View everything | No changes |
| User Access Administrator | Manage access | No resource changes |

Those four are the general-purpose roles.

Data-plane roles for specific services:

| Role | Service | What it allows |
|---|---|---|
| Storage Blob Data Reader | Blob Storage | Read blobs |
| Storage Blob Data Contributor | Blob Storage | Read, write, delete blobs |
| Storage Queue Data Contributor | Queue Storage | Read, write queue messages |
| Cosmos DB Built-in Data Contributor | Cosmos DB | Read, write items |
| SQL DB Contributor | SQL Database | Manage SQL databases |
| Key Vault Secrets User | Key Vault | Read secrets |
| Key Vault Secrets Officer | Key Vault | Read, write, delete secrets |

# 5. Assign a Role (Portal)

```text
Portal → Resource group → Access control (IAM) → Add → Add role assignment
  1. Select role: Contributor
  2. Select principal: search for Alice or DevTeam group
  3. Save
```

---

# 6. Assign a Role (CLI)

Get the IDs you need:

```bash
# Subscription ID
SUB_ID=$(az account show --query id -o tsv)

# Resource group ID (scope for the assignment)
RG=rg-prod
RG_ID=$(az group show --name $RG --query id -o tsv)

# User or group object ID
USER_ID=$(az ad user show --id alice@yourtenant.onmicrosoft.com --query id -o tsv)
```

Assign **Contributor** to Alice on the resource group:

```bash
az role assignment create \
  --assignee $USER_ID \
  --role "Contributor" \
  --scope $RG_ID
```

Assign **Reader** to the DevTeam group on the subscription:

```bash
GROUP_ID=$(az ad group show --group "DevTeam" --query id -o tsv)

az role assignment create \
  --assignee $GROUP_ID \
  --role "Reader" \
  --scope "/subscriptions/$SUB_ID"
```

Assign **Storage Blob Data Contributor** to an identity on a specific storage account:

```bash
STORAGE_ID=$(az storage account show -g $RG -n $STORAGE --query id -o tsv)

az role assignment create \
  --assignee $USER_ID \
  --role "Storage Blob Data Contributor" \
  --scope $STORAGE_ID
```

---

# 7. Remove a Role Assignment

```bash
az role assignment delete \
  --assignee $USER_ID \
  --role "Contributor" \
  --scope $RG_ID
```

# 9. Custom Roles

If no built-in role fits, you can create a custom role.

Define the permissions in JSON:

```json
{
    "Name": "Storage Blob Reader and Lister",
    "Description": "Can read and list blobs but not write or delete",
    "Actions": [],
    "NotActions": [],
    "DataActions": [
        "Microsoft.Storage/storageAccounts/blobServices/containers/blobs/read",
        "Microsoft.Storage/storageAccounts/blobServices/containers/blobs/filter/action"
    ],
    "NotDataActions": [],
    "AssignableScopes": [
        "/subscriptions/<your-subscription-id>"
    ]
}
```

Create the custom role:

```bash
az role definition create --role-definition custom-role.json
```

# 10. Deny Assignments

Azure RBAC has **deny assignments** — explicit denies that override allows.

These are mostly used by Azure Blueprints and managed applications, not manually. You cannot create deny assignments directly in most scenarios.

The typical model is:

```text
No role assigned = denied by default
Assign a role = explicitly grant access
```

---

# 11. Principle of Least Privilege

Best practice:

```text
- Assign the most restrictive role that still allows the work
- Assign at the narrowest scope possible
- Use Contributor for developers on their own resource group, not on the subscription
- Use Reader for auditors and viewers
- Use data-plane roles (Storage Blob Data Contributor, Key Vault Secrets User)
  instead of broad Owner on storage accounts or vaults
- Avoid Owner unless truly necessary
```

Bad assignment:

```text
DevTeam → Owner → Subscription
```

Better assignment:

```text
DevTeam → Contributor → rg-dev
```

# 12. Check What a Role Can Do

```bash
az role definition list --name "Storage Blob Data Contributor" --output json
```

Lists all the `DataActions`, `NotDataActions`, `Actions`, and `NotActions` for that role.

---

# Quick Recap

| Concept | Key Point |
|---|---|
| RBAC | Who (principal) + What (role) + Where (scope) |
| Scope | Management group → Subscription → Resource group → Resource |
| Owner | Full control, can assign roles |
| Contributor | Full resource control, no access management |
| Reader | View only |
| Data-plane roles | Specific to a service (Blob, Cosmos, Key Vault, etc.) |
| Least privilege | Assign narrowest role at narrowest scope |
