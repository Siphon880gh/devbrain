Microsoft Entra ID (formerly called Azure Active Directory or Azure AD) is Microsoft's cloud identity platform.

It answers one question:

```text
Who is this person (or service), and what are they allowed to do?
```

---

# 1. What Entra ID Does

```text
Authentication  → Proves who you are (login with username + password + MFA)
Authorization   → Decides what you can access (roles, policies, app permissions)
Single Sign-On  → One login works for Azure Portal, Microsoft 365, your own apps
App Registration → Register apps so they can log in users or access APIs
```

Entra ID is used for:

```text
- Signing in to the Azure Portal
- Logging users into your web apps (OAuth 2.0 / OpenID Connect)
- Giving VMs, App Services, and Functions identities (Managed Identity)
- Controlling access to Azure resources (RBAC — covered in lesson 2)
```

---

# 2. Terminology

| Term | Meaning |
|---|---|
| Tenant | An instance of Entra ID for one organization |
| Directory | Same as tenant |
| User | A person with a username and password |
| Group | A collection of users |
| Service Principal | An identity for an app or service |
| Managed Identity | A service principal automatically managed by Azure |
| App Registration | Config that lets your app authenticate users |

When you sign up for Azure, a **tenant** is created automatically for you.

Example tenant ID:

```text
xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
```

Your tenant holds your users, groups, roles, and app registrations.

---

# 3. The Two Types of Accounts

```text
Microsoft account (personal):
  - youremail@outlook.com, gmail.com, hotmail.com, live.com
  - Mainly for personal use
  - Works for Azure, Xbox, Microsoft 365 personal

Work or school account (Entra ID identity):
  - you@yourcompany.com
  - Managed by your organization's Entra ID tenant
  - Has IT-controlled policies (MFA requirements, conditional access, etc.)
```

Both can be used to sign in to Azure.

# 4. Add Users to Your Tenant

In the Azure Portal:

```text
Portal → Microsoft Entra ID → Users → New user → Create user
  - Username: alice@yourtenant.onmicrosoft.com
  - Name: Alice Smith
  - Password: auto-generated or set manually
```

Via CLI:

```bash
az ad user create \
  --display-name "Alice Smith" \
  --user-principal-name alice@yourtenant.onmicrosoft.com \
  --password "InitialPass123!" \
  --force-change-password-next-sign-in true
```

# 5. Groups

Groups make it easier to assign permissions to many users at once.

Create a group:

```bash
az ad group create \
  --display-name "DevTeam" \
  --mail-nickname "devteam"
```

Add a user to a group:

```bash
USER_ID=$(az ad user show --id alice@yourtenant.onmicrosoft.com --query id -o tsv)
GROUP_ID=$(az ad group show --group "DevTeam" --query id -o tsv)

az ad group member add \
  --group $GROUP_ID \
  --member-id $USER_ID
```

Now you can assign RBAC roles to the group instead of to each person individually.

---

# 6. App Registration

An **App Registration** is how you enable your web app to log users in using their Microsoft/Entra ID identity.

Create one:

```bash
az ad app create \
  --display-name "My Web App" \
  --sign-in-audience AzureADMyOrg \
  --web-redirect-uris "https://myapp.azurewebsites.net/auth/callback"
```

After registration, your app gets:

```text
Application (client) ID:    xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
Directory (tenant) ID:      xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
```

Use these in your app's authentication library (MSAL).

---

# 7. Microsoft Authentication Library (MSAL)

MSAL is the SDK that handles OAuth 2.0 / OpenID Connect flows for your app.

Install for Node.js:

```bash
npm install @azure/msal-node
```

Minimal example — redirect user to Microsoft login:

```javascript
import { ConfidentialClientApplication } from "@azure/msal-node";

const msalConfig = {
    auth: {
        clientId: process.env.AZURE_CLIENT_ID,
        authority: `https://login.microsoftonline.com/${process.env.AZURE_TENANT_ID}`,
        clientSecret: process.env.AZURE_CLIENT_SECRET
    }
};

const cca = new ConfidentialClientApplication(msalConfig);

const authUrl = await cca.getAuthCodeUrl({
    scopes: ["openid", "profile", "email"],
    redirectUri: "https://myapp.azurewebsites.net/auth/callback"
});

// redirect user to authUrl
```

After the user logs in, Microsoft redirects back to your callback URL with an authorization code. Exchange it for tokens:

```javascript
const tokens = await cca.acquireTokenByCode({
    code: req.query.code,
    scopes: ["openid", "profile", "email"],
    redirectUri: "https://myapp.azurewebsites.net/auth/callback"
});

console.log(tokens.account.username);
```

# 8. Multi-Factor Authentication (MFA)

MFA requires users to provide a second factor after their password — typically a phone prompt or code.

Enable MFA:

```text
Portal → Microsoft Entra ID → Security → MFA → Per-user MFA
  → Enable for specific users or all users
```

Or use **Conditional Access** (Entra ID P1 license or above):

```text
Portal → Entra ID → Security → Conditional Access → New policy
  → Apply to: All users
  → Conditions: Sign-in risk = medium or high
  → Grant: Require MFA
```

For the free Entra ID tier, per-user MFA and Security Defaults (MFA for admins) are available without a license.

---

# 9. Security Defaults

Security Defaults is a free baseline set of security policies.

When enabled:

```text
- All admins must use MFA
- All users must register for MFA (prompted at next login)
- Legacy authentication protocols are blocked
```

Enable via portal:

```text
Entra ID → Properties → Manage Security defaults → Enable: Yes
```

For most small teams without P1/P2 licenses, Security Defaults is the best first step.

---

# 10. Entra ID Editions

| Edition | Cost | Key Features |
|---|---|---|
| Free | Included with Azure | Users, groups, SSO, basic MFA, app registration |
| P1 | ~$6/user/month | Conditional Access, self-service password reset |
| P2 | ~$9/user/month | Identity Protection (risk-based login), PIM |

For most learner and small-business scenarios, the Free tier is sufficient.
