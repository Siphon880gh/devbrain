Azure Key Vault is a secure store for secrets, encryption keys, and TLS certificates.

Instead of putting API keys, database passwords, and connection strings directly in your app's configuration or environment variables, you store them in Key Vault and let your app fetch them at runtime.

---

# 1. What Key Vault Stores

```text
Secrets      → passwords, API keys, connection strings, tokens
Keys         → cryptographic keys for encryption/decryption (RSA, EC)
Certificates → TLS/SSL certificates with automated renewal
```

Most beginners use **secrets**. Keys and certificates are for more advanced cryptographic and PKI scenarios.

---

# 2. Why Use Key Vault

Without Key Vault:

```text
Secrets live in:
  - App Service Application Settings (visible to anyone with Contributor access)
  - Environment variables on VMs (visible in process listings)
  - .env files (at risk of being committed to Git)
  - Hard-coded in source code (extremely bad)
```

With Key Vault:

```text
Secrets live in:
  - Key Vault, with full audit logging of every access
  - Access controlled by RBAC
  - Only your app's identity (Managed Identity) can read the secret
  - Secrets can be versioned
  - Rotation is centralized
```

# 3. Create a Key Vault

```bash
RG=rg-keyvault-demo
LOC=eastus
VAULT=myvault$RANDOM     # must be globally unique, 3-24 chars, alphanumeric + dash

az group create -n $RG -l $LOC

az keyvault create \
  --name $VAULT \
  --resource-group $RG \
  --location $LOC \
  --sku standard
```

---

# 4. Add Secrets

```bash
az keyvault secret set \
  --vault-name $VAULT \
  --name "DatabaseUrl" \
  --value "postgresql://user:pass@myserver.postgres.database.azure.com/mydb?sslmode=require"

az keyvault secret set \
  --vault-name $VAULT \
  --name "ApiKey" \
  --value "sk-abc123..."
```

---

# 5. Read a Secret

```bash
az keyvault secret show \
  --vault-name $VAULT \
  --name "DatabaseUrl" \
  --query value \
  --output tsv
```

List all secrets:

```bash
az keyvault secret list \
  --vault-name $VAULT \
  --output table
```

---

# 6. Grant Your App Access via RBAC

Enable the Managed Identity on your App Service (if not done already):

```bash
az webapp identity assign --resource-group $RG --name $APP
OID=$(az webapp identity show --resource-group $RG --name $APP --query principalId -o tsv)
```

Grant the **Key Vault Secrets User** role:

```bash
VAULT_ID=$(az keyvault show --name $VAULT --query id -o tsv)

az role assignment create \
  --assignee $OID \
  --role "Key Vault Secrets User" \
  --scope $VAULT_ID
```

Now the App Service can read secrets from the vault. Nothing else can.

# 7. Read Secrets in Code

### Python:

```bash
pip install azure-keyvault-secrets azure-identity
```

```python
from azure.keyvault.secrets import SecretClient
from azure.identity import DefaultAzureCredential

vault_url = "https://myvault1234.vault.azure.net/"
credential = DefaultAzureCredential()
client = SecretClient(vault_url=vault_url, credential=credential)

db_url = client.get_secret("DatabaseUrl").value
print(db_url)
```

### Node.js:

```bash
npm install @azure/keyvault-secrets @azure/identity
```

```javascript
import { SecretClient } from "@azure/keyvault-secrets";
import { DefaultAzureCredential } from "@azure/identity";

const client = new SecretClient(
    "https://myvault1234.vault.azure.net/",
    new DefaultAzureCredential()
);

const secret = await client.getSecret("DatabaseUrl");
console.log(secret.value);
```

# 8. Key Vault References (No Code Change)

A **Key Vault reference** lets App Service and Functions read secrets directly from Key Vault without any code changes — the secret value is injected as an environment variable at startup.

Format for an Application Setting value:

```text
@Microsoft.KeyVault(SecretUri=https://myvault1234.vault.azure.net/secrets/DatabaseUrl/)
```

Set via CLI:

```bash
az webapp config appsettings set \
  --resource-group $RG \
  --name $APP \
  --settings DATABASE_URL="@Microsoft.KeyVault(SecretUri=https://myvault1234.vault.azure.net/secrets/DatabaseUrl/)"
```

Your app code reads `os.environ["DATABASE_URL"]` (Python) or `process.env.DATABASE_URL` (Node) as if it were a normal environment variable — but the value comes from Key Vault automatically.

Requirements:

```text
- App Service must have a Managed Identity
- Identity must have "Key Vault Secrets User" role on the vault
```

# 9. Secret Versions

Every time you update a secret, a new **version** is created. The old version is preserved.

Show all versions of a secret:

```bash
az keyvault secret list-versions \
  --vault-name $VAULT \
  --name "DatabaseUrl" \
  --output table
```

Read a specific version:

```bash
az keyvault secret show \
  --vault-name $VAULT \
  --name "DatabaseUrl" \
  --version <version-id> \
  --query value -o tsv
```

Key Vault references without a version always resolve to the current (latest) version automatically.

# 10. Soft Delete and Purge Protection

By default, Key Vault uses **soft delete** — deleted secrets go into a "deleted but recoverable" state for 7–90 days.

Recover a deleted secret:

```bash
az keyvault secret recover \
  --vault-name $VAULT \
  --name "DatabaseUrl"
```

Permanently delete a secret:

```bash
az keyvault secret purge \
  --vault-name $VAULT \
  --name "DatabaseUrl"
```

Enable **purge protection** to prevent permanent deletion until retention period ends:

```bash
az keyvault update \
  --name $VAULT \
  --resource-group $RG \
  --enable-purge-protection true
```

Once purge protection is on, you cannot turn it off — use with care.

---

# 11. Certificates

Key Vault can manage TLS certificates including automated renewal.

Import an existing certificate:

```bash
az keyvault certificate import \
  --vault-name $VAULT \
  --name MyCert \
  --file ./my-cert.pfx \
  --password "cert-password"
```

Create a self-signed certificate (for testing):

```bash
az keyvault certificate create \
  --vault-name $VAULT \
  --name SelfSignedCert \
  --policy "$(az keyvault certificate get-default-policy)"
```

App Service can pull TLS certificates directly from Key Vault for custom domains.

# 12. Pricing

```text
Standard tier:
  - Secrets operations: $0.03 per 10,000 operations
  - Certificate operations: $3 per certificate renewal
  - Very cheap for most applications
```

A small app making 100,000 secret reads per month costs about $0.30 in operations.

# Quick Recap

| Concept | Key Point |
|---|---|
| Key Vault | Secure store for secrets, keys, certificates |
| Secret | Password, API key, connection string |
| RBAC access | Grant "Key Vault Secrets User" to Managed Identity |
| Key Vault reference | `@Microsoft.KeyVault(...)` injects secret as env var — no code change |
| Soft delete | Deleted secrets recoverable for 7–90 days |
| DefaultAzureCredential | Used in SDK code to access vault; works locally (az login) and in production (MI) |
