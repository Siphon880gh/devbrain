Managed Identity solves a specific problem:

```text
My app needs to access Azure services (Storage, Cosmos DB, SQL, Key Vault, etc.).
How does it authenticate without storing a password, connection string, or API key?
```

The answer is Managed Identity — Azure assigns an automatic identity to your app, and you grant that identity RBAC roles. No secrets to store, rotate, or leak.

---

# 1. Why Managed Identity Matters

Without Managed Identity, you store a secret somewhere:

```text
App Service → connection string with password → stored in Application Settings
App Service → storage account key → stored in Application Settings
Function → Cosmos DB key → stored in Application Settings
```

These secrets can be:

```text
- Accidentally committed to Git
- Read by anyone with access to the Application Settings
- Forgotten and never rotated
```

With Managed Identity:

```text
App Service → Azure RBAC role → Storage Account
```

No secret. The identity is managed by Azure itself.

---

# 2. System-Assigned vs User-Assigned

**System-assigned identity:**

```text
- Created automatically when you enable it on a resource
- Tied to that specific resource — when the resource is deleted, the identity is deleted
- One resource, one identity
- Simplest to set up
```

**User-assigned identity:**

```text
- Created as a standalone resource
- Can be assigned to multiple resources
- Lives independently of any specific resource
- Good when multiple services need the same set of permissions
```

For beginners: start with system-assigned. Graduate to user-assigned when you have multiple services sharing the same permissions.

---

# 3. Enable System-Assigned Identity

### App Service:

```bash
az webapp identity assign \
  --resource-group $RG \
  --name $APP
```

### Azure Function:

```bash
az functionapp identity assign \
  --resource-group $RG \
  --name $FUNCAPP
```

### Virtual Machine:

```bash
az vm identity assign \
  --resource-group $RG \
  --name $VM
```

### Container App:

```bash
az containerapp identity assign \
  --resource-group $RG \
  --name $APPNAME \
  --system-assigned
```

Each command prints the identity's **object ID (principal ID)** — you need it to assign RBAC roles.

Get it:

```bash
OID=$(az webapp identity show \
  --resource-group $RG \
  --name $APP \
  --query principalId \
  --output tsv)
```

# 4. Assign an RBAC Role to the Identity

The identity itself does nothing until you grant it permissions.

### Grant access to Blob Storage:

```bash
STORAGE_ID=$(az storage account show -g $RG -n $STORAGE --query id -o tsv)

az role assignment create \
  --assignee $OID \
  --role "Storage Blob Data Contributor" \
  --scope $STORAGE_ID
```

### Grant access to Cosmos DB:

```bash
COSMOS_ID=$(az cosmosdb show -g $RG -n $COSMOS_ACCOUNT --query id -o tsv)

az cosmosdb sql role assignment create \
  --account-name $COSMOS_ACCOUNT \
  --resource-group $RG \
  --role-definition-name "Cosmos DB Built-in Data Contributor" \
  --scope "/" \
  --principal-id $OID
```

### Grant access to Azure SQL Database (via T-SQL):

```sql
-- Run inside your SQL database as admin:
CREATE USER [MyAppServiceName] FROM EXTERNAL PROVIDER;
ALTER ROLE db_datareader ADD MEMBER [MyAppServiceName];
ALTER ROLE db_datawriter ADD MEMBER [MyAppServiceName];
```

### Grant access to Key Vault:

```bash
VAULT_ID=$(az keyvault show -g $RG -n $VAULT --query id -o tsv)

az role assignment create \
  --assignee $OID \
  --role "Key Vault Secrets User" \
  --scope $VAULT_ID
```

# 5. Use the Identity in Code

Azure SDKs use `DefaultAzureCredential` to automatically pick up the Managed Identity when running in Azure.

### Python:

```python
from azure.identity import DefaultAzureCredential
from azure.storage.blob import BlobServiceClient

credential = DefaultAzureCredential()
client = BlobServiceClient(
    account_url="https://mystorageaccount.blob.core.windows.net",
    credential=credential
)
```

### Node.js:

```javascript
import { DefaultAzureCredential } from "@azure/identity";
import { BlobServiceClient } from "@azure/storage-blob";

const credential = new DefaultAzureCredential();
const client = new BlobServiceClient(
    "https://mystorageaccount.blob.core.windows.net",
    credential
);
```

### C# (.NET):

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var client = new BlobServiceClient(
    new Uri("https://mystorageaccount.blob.core.windows.net"),
    new DefaultAzureCredential()
);
```

`DefaultAzureCredential` tries multiple auth methods in order:

```text
1. Environment variables (AZURE_CLIENT_ID, etc.)
2. Workload Identity (AKS)
3. Managed Identity
4. Azure CLI login (`az login`)
5. VS Code login
6. Azure PowerShell login
```

In development, it uses your `az login` session. In production on Azure, it uses the Managed Identity automatically.

# 6. User-Assigned Identity

Create a standalone identity:

```bash
az identity create \
  --resource-group $RG \
  --name my-shared-identity
```

Get its IDs:

```bash
CLIENT_ID=$(az identity show -g $RG -n my-shared-identity --query clientId -o tsv)
PRINCIPAL_ID=$(az identity show -g $RG -n my-shared-identity --query principalId -o tsv)
IDENTITY_ID=$(az identity show -g $RG -n my-shared-identity --query id -o tsv)
```

Assign it to an App Service:

```bash
az webapp identity assign \
  --resource-group $RG \
  --name $APP \
  --identities $IDENTITY_ID
```

Assign the RBAC role to the identity's principal ID (same as system-assigned):

```bash
az role assignment create \
  --assignee $PRINCIPAL_ID \
  --role "Storage Blob Data Contributor" \
  --scope $STORAGE_ID
```

In code with user-assigned identity (specify the client ID):

```python
from azure.identity import ManagedIdentityCredential

credential = ManagedIdentityCredential(client_id="<user-assigned-client-id>")
```

---

# 7. Local Development

When running locally, `DefaultAzureCredential` falls back to your `az login` session.

Make sure you are logged in:

```bash
az login
az account set --subscription $SUB_ID
```

Your local app uses the same RBAC permissions that your personal identity has — which should include the same roles you assigned to the Managed Identity (or broader, since you may have Contributor on the resource group).

---

# 8. Verify the Assignment Works

Quick test from the portal:

```text
Storage Account → Access Control (IAM) → Role assignments
  → Look for your App Service / Function name under "Storage Blob Data Contributor"
```

Or via CLI:

```bash
az role assignment list \
  --assignee $OID \
  --all \
  --output table
```

# Quick Recap

| Concept | Key Point |
|---|---|
| Managed Identity | Identity for an Azure service — no password needed |
| System-assigned | Tied to one resource; easiest to start with |
| User-assigned | Standalone identity; share across multiple resources |
| DefaultAzureCredential | SDK class that auto-selects credential (MI in production, CLI in dev) |
| RBAC role assignment | Grant the identity permission on a resource |
| No secrets | The whole point — nothing to store, rotate, or leak |
