**Infrastructure as Code (IaC)** means defining your Azure resources in files rather than clicking through the portal or running one-off CLI commands.

Benefits:

```text
- Repeatable: deploy the same environment to dev, staging, and production
- Version-controlled: track infrastructure changes in Git
- Reviewable: teammates can review infrastructure changes like code
- Recoverable: recreate deleted resources from the definition file
```

Azure has three main IaC options:

```text
ARM Templates    — Azure's native JSON format (verbose, old)
Bicep            — Azure's modern IaC language (cleaner, compiles to ARM)
Terraform        — cross-cloud IaC by HashiCorp (works on Azure, AWS, GCP, etc.)
```

---

# 1. ARM Templates

**ARM** stands for Azure Resource Manager.

ARM templates are JSON files that describe Azure resources.

Example — create a storage account:

```json
{
    "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentTemplate.json#",
    "contentVersion": "1.0.0.0",
    "parameters": {
        "storageAccountName": {
            "type": "string"
        }
    },
    "resources": [
        {
            "type": "Microsoft.Storage/storageAccounts",
            "apiVersion": "2023-01-01",
            "name": "[parameters('storageAccountName')]",
            "location": "[resourceGroup().location]",
            "sku": { "name": "Standard_LRS" },
            "kind": "StorageV2"
        }
    ]
}
```

Deploy it:

```bash
az deployment group create \
  --resource-group $RG \
  --template-file storage.json \
  --parameters storageAccountName=mystoragelearn123
```

ARM is still widely used, but new projects typically use Bicep instead.

# 2. Bicep

**Bicep** is Azure's own IaC language. It compiles to ARM JSON and has cleaner syntax.

The same storage account in Bicep:

```bicep
param storageAccountName string

resource storageAccount 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: storageAccountName
  location: resourceGroup().location
  sku: {
    name: 'Standard_LRS'
  }
  kind: 'StorageV2'
}
```

### Install Bicep

Bicep is built into the Azure CLI since version 2.20:

```bash
az bicep version
az bicep upgrade
```

VS Code extension:

```text
Install: "Bicep" by Microsoft (provides syntax highlighting + IntelliSense)
```

### Deploy a Bicep file

```bash
az deployment group create \
  --resource-group $RG \
  --template-file storage.bicep \
  --parameters storageAccountName=mystoragelearn123
```

### Convert ARM to Bicep (if you have existing ARM templates)

```bash
az bicep decompile --file storage.json
```

# 3. Bicep — Multi-Resource Example

Deploy a storage account + container together:

```bicep
param storageAccountName string
param containerName string = 'uploads'

resource storageAccount 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: storageAccountName
  location: resourceGroup().location
  sku: {
    name: 'Standard_LRS'
  }
  kind: 'StorageV2'
}

resource blobService 'Microsoft.Storage/storageAccounts/blobServices@2023-01-01' = {
  name: 'default'
  parent: storageAccount
}

resource container 'Microsoft.Storage/storageAccounts/blobServices/containers@2023-01-01' = {
  name: containerName
  parent: blobService
  properties: {
    publicAccess: 'None'
  }
}

output storageAccountId string = storageAccount.id
output blobEndpoint string = storageAccount.properties.primaryEndpoints.blob
```

Deploy:

```bash
az deployment group create \
  --resource-group $RG \
  --template-file infra.bicep \
  --parameters storageAccountName=mystoragelearn123
```

### What-if (dry run)

```bash
az deployment group what-if \
  --resource-group $RG \
  --template-file infra.bicep \
  --parameters storageAccountName=mystoragelearn123
```

Shows what will be created, modified, or deleted — without actually running.

# 4. Bicep Modules

Modules let you split large templates into reusable files.

`modules/storage.bicep`:

```bicep
param name string
param location string = resourceGroup().location

resource account 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: name
  location: location
  sku: { name: 'Standard_LRS' }
  kind: 'StorageV2'
}

output id string = account.id
```

`main.bicep`:

```bicep
module storage './modules/storage.bicep' = {
  name: 'storageDeployment'
  params: {
    name: 'mystoragelearn123'
  }
}

output storageId string = storage.outputs.id
```

# 5. Terraform

**Terraform** is a cross-cloud IaC tool by HashiCorp.

If you use both Azure and AWS or GCP, Terraform lets you manage all of them with the same language (HCL — HashiCorp Configuration Language).

### Install

```bash
# macOS
brew tap hashicorp/tap && brew install hashicorp/tap/terraform

# Windows
winget install Hashicorp.Terraform
```

### Authenticate to Azure

```bash
az login
az account set --subscription $SUB_ID
```

Terraform uses your active `az login` session for local development.

### Example — storage account in Terraform

`main.tf`:

```hcl
terraform {
  required_providers {
    azurerm = {
      source  = "hashicorp/azurerm"
      version = "~> 3.0"
    }
  }
}

provider "azurerm" {
  features {}
}

resource "azurerm_resource_group" "rg" {
  name     = "rg-tf-demo"
  location = "East US"
}

resource "azurerm_storage_account" "storage" {
  name                     = "mystoragelearn123"
  resource_group_name      = azurerm_resource_group.rg.name
  location                 = azurerm_resource_group.rg.location
  account_tier             = "Standard"
  account_replication_type = "LRS"
}
```

### Workflow

```bash
terraform init       # download providers
terraform plan       # show what will change (dry run)
terraform apply      # apply the changes (prompts for confirmation)
terraform destroy    # delete all resources in the state
```

### State file

Terraform tracks what it has deployed in a **state file** (`terraform.tfstate`).

For production, store state remotely:

```hcl
terraform {
  backend "azurerm" {
    resource_group_name  = "rg-terraform-state"
    storage_account_name = "tfstatestorage"
    container_name       = "tfstate"
    key                  = "prod.terraform.tfstate"
  }
}
```

# 6. Which Should I Use?

| Scenario | Use |
|---|---|
| Azure-only infrastructure | Bicep |
| Multi-cloud or existing Terraform org | Terraform |
| Deploying with Azure DevOps pipelines | Bicep or Terraform |
| Converting old ARM templates | Bicep (use `az bicep decompile`) |
| Learning Azure IaC for the first time | Bicep |

Bicep is Azure-native, has full IntelliSense in VS Code, and requires no extra tools beyond the Azure CLI.

# 7. azd — Azure Developer CLI

`azd` is a higher-level tool on top of Bicep that packages an app + its infrastructure together for a repeatable deploy-to-Azure workflow.

```bash
# Install
brew install azure/azd/azd      # macOS
winget install Microsoft.Azd    # Windows

# Initialize a project (creates azure.yaml + infra/ folder)
azd init

# Deploy app + infrastructure together
azd up

# Just redeploy the app code (no infra changes)
azd deploy

# Delete everything
azd down
```

`azd` is covered more in `06 - DevOps / 3. GitHub Actions for Azure Deployments`.

# Quick Recap

| Tool | Language | Scope | Use Case |
|---|---|---|---|
| ARM Templates | JSON | Azure | Legacy; still valid |
| Bicep | HCL-like | Azure | Recommended for Azure-only IaC |
| Terraform | HCL | Multi-cloud | Azure + AWS/GCP together |
| azd | Convention | Azure | Full app + infra deploy workflow |
