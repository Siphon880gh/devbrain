Azure has several services that sit in front of your applications and distribute or route traffic. Beginners often confuse them.

This lesson maps each service to its job.

---

# 1. The Four Traffic Distribution Services

```text
Azure Load Balancer        — Layer 4 (TCP/UDP), within a region, very fast
Application Gateway        — Layer 7 (HTTP/HTTPS), within a region, URL routing + WAF
Azure Front Door           — Layer 7, global, CDN + routing across regions
Azure Traffic Manager      — DNS-based routing, no traffic passes through it
```

---

# 2. Azure Load Balancer

A **Load Balancer** distributes TCP/UDP traffic to a pool of backends.

```text
Internet / VNet → Load Balancer (public or internal) → Backend VM pool
```

It works at Layer 4 — it sees IP addresses and ports, but not HTTP content.

Use it when:

```text
- You have 2+ VMs running the same TCP service
- You want health probes to remove unhealthy VMs from rotation
- You need an internal load balancer to distribute traffic inside a VNet
```

It does not do:

```text
- URL-based routing (send /api to one pool, /static to another)
- TLS termination
- WAF (Web Application Firewall)
```

Two SKUs:

```text
Basic:    Free, being retired, limited features
Standard: ~$0.025/hour (~$18/month) + data processing fee
```

Create a Standard Load Balancer:

```bash
az network lb create \
  --resource-group $RG \
  --name my-lb \
  --sku Standard \
  --public-ip-address my-lb-pip \
  --frontend-ip-name frontendPool \
  --backend-pool-name backendPool
```

Add a health probe:

```bash
az network lb probe create \
  --resource-group $RG \
  --lb-name my-lb \
  --name healthProbe \
  --protocol Tcp \
  --port 80
```

Add a load balancing rule:

```bash
az network lb rule create \
  --resource-group $RG \
  --lb-name my-lb \
  --name lbRule \
  --protocol Tcp \
  --frontend-port 80 \
  --backend-port 80 \
  --frontend-ip-name frontendPool \
  --backend-pool-name backendPool \
  --probe-name healthProbe
```

# 3. Application Gateway

**Application Gateway** is a Layer 7 (HTTP/HTTPS) load balancer with advanced routing.

It understands HTTP — it can route based on URL paths and hostnames.

```text
Request: https://myapp.com/api/*     → route to API backend pool
Request: https://myapp.com/images/*  → route to Image server pool
```

Features:

```text
- URL path-based routing
- Multi-site hosting (different hostnames to different backends)
- TLS termination (HTTPS ends at Application Gateway; backends use HTTP)
- SSL offloading
- Cookie-based session affinity
- Web Application Firewall (WAF) — blocks OWASP Top 10 attacks (SQLi, XSS, etc.)
- Autoscaling (v2 SKU)
- Connection draining during deploys
```

Use Application Gateway when:

```text
- You need URL routing across different backend services
- You need a WAF in front of your app
- Your backends are VMs, App Services, IP addresses, or FQDNs
```

Pricing:

```text
V2 Standard: ~$0.246/hour base + $0.008 per capacity unit (~$180-200+/month)
WAF tier:    Slightly higher than Standard
```

This is expensive. Do not create one for learning unless you need to, and delete it immediately when done.

---

# 4. Azure Front Door

**Azure Front Door** is a global content delivery network (CDN) and HTTP load balancer.

It sits at Azure's edge PoPs (Points of Presence) around the world — not in a single region.

```text
User in Tokyo       → nearest Azure edge → routes to backend
User in London      → nearest Azure edge → routes to backend
User in New York    → nearest Azure edge → routes to backend
```

Features:

```text
- Global HTTP/HTTPS routing
- SSL termination at the edge
- Caching (like a CDN)
- URL rewriting and redirects
- WAF at global scale
- Health probes and failover across backends in different regions
- Custom domains with free TLS certificates
```

Use Front Door when:

```text
- Your users are globally distributed
- You want CDN caching for static assets
- You need failover across regions (active-active or active-passive)
- You want global WAF
```

Not needed for:

```text
- Small apps with users in one region
- Dev/test environments
- Apps already behind Application Gateway in one region
```

Pricing:

```text
Standard tier:  ~$35/month base + data transfer
Premium tier:   ~$330/month base + data transfer (adds WAF, private link to backends)
```

# 5. Azure Traffic Manager

**Traffic Manager** is DNS-based routing — no traffic actually flows through it.

```text
User → DNS lookup for myapp.com
       → Traffic Manager responds with the IP of the best backend
         → User connects directly to that backend
```

Traffic Manager decides which backend based on routing methods:

```text
Performance    → route to the backend with lowest latency for the user
Priority       → use primary, failover to secondary
Weighted       → split traffic 80/20 between backends
Geographic     → route based on user's country/region
```

Use Traffic Manager when:

```text
- You have backends in multiple Azure regions
- You want DNS-level failover
- You want geo-based routing
- You do not need HTTP inspection or caching
```

Pricing:

```text
~$0.54 per million DNS queries
~$0.36 per month per endpoint
Very cheap.
```

---

# 6. Side-By-Side Comparison

| Feature | Load Balancer | Application Gateway | Front Door | Traffic Manager |
|---|---|---|---|---|
| Layer | 4 (TCP/UDP) | 7 (HTTP) | 7 (HTTP) | DNS (no traffic) |
| Scope | Regional | Regional | Global | Global |
| URL routing | No | Yes | Yes | No |
| TLS termination | No | Yes | Yes | No |
| WAF | No | Yes | Yes (Premium) | No |
| CDN / caching | No | No | Yes | No |
| Price | ~$18/month | ~$180+/month | ~$35+/month | Very cheap |

---

# 7. Simple Decision Flowchart

```text
Do you need global routing or CDN?
  └── Yes → Azure Front Door

Do you need URL-path routing or WAF (single region)?
  └── Yes → Application Gateway

Do you have multiple VMs serving the same TCP/UDP service?
  └── Yes → Azure Load Balancer

Do you need DNS failover between regions with no HTTP inspection?
  └── Yes → Traffic Manager
```

---

# 8. Cost Warning

Both Application Gateway and Front Door have significant base costs.

```text
Application Gateway V2: idle with no traffic → ~$180/month
Front Door Standard:    idle with no traffic → ~$35/month
```

For learning:

```text
- Use App Service built-in TLS and load balancing (no extra cost)
- Use Container Apps built-in ingress (no extra cost)
- Only create Application Gateway or Front Door if the lesson requires it
- Delete them immediately when done
```

---

# Quick Recap

| Service | One-line Summary |
|---|---|
| Load Balancer | Distribute TCP/UDP to VMs in a region |
| Application Gateway | HTTP router with WAF in a region |
| Front Door | Global CDN + HTTP router + WAF |
| Traffic Manager | DNS-level routing only, no traffic passes through |

For most beginner projects, App Service and Container Apps have enough built-in routing that you do not need any of these services.
