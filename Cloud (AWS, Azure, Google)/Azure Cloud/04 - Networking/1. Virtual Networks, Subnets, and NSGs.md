When you create a VM or deploy a service that needs network isolation, Azure networking is how you control what can talk to what.

Three building blocks:

```text
Virtual Network (VNet)   — your private network space in Azure
Subnet                   — a segment inside the VNet
Network Security Group   — firewall rules for a subnet or network interface
```

---

# 1. Virtual Network (VNet)

A **VNet** is a logically isolated private network in Azure.

Think of it like your office LAN — it is isolated from the internet and from other Azure customers' networks.

```text
VNet: 10.0.0.0/16
  (this gives you IP addresses from 10.0.0.0 to 10.0.255.255)
```

Resources inside the same VNet can talk to each other by private IP without going over the public internet.

---

# 2. Subnets

A VNet is divided into **subnets**.

```text
VNet: 10.0.0.0/16
├── Subnet: web-tier        10.0.1.0/24  (IPs 10.0.1.0–10.0.1.255)
├── Subnet: app-tier        10.0.2.0/24
└── Subnet: db-tier         10.0.3.0/24
```

Why segment into subnets?

```text
- Apply different firewall rules to each tier
- Put databases in a subnet with no internet access
- Put web servers in a subnet that allows inbound HTTP/HTTPS
- Keep different workloads isolated
```

---

# 3. Create a VNet and Subnets (CLI)

```bash
RG=rg-network-demo
LOC=eastus

az group create -n $RG -l $LOC

az network vnet create \
  --resource-group $RG \
  --name my-vnet \
  --address-prefix 10.0.0.0/16

az network vnet subnet create \
  --resource-group $RG \
  --vnet-name my-vnet \
  --name web-subnet \
  --address-prefixes 10.0.1.0/24

az network vnet subnet create \
  --resource-group $RG \
  --vnet-name my-vnet \
  --name db-subnet \
  --address-prefixes 10.0.2.0/24
```

---

# 4. Network Security Group (NSG)

An **NSG** is a set of allow/deny rules for network traffic.

Rules can filter:

```text
- Inbound traffic (traffic coming into a resource)
- Outbound traffic (traffic leaving a resource)
```

Each rule has:

```text
Priority       → lower number = higher priority (100 runs before 200)
Protocol       → TCP, UDP, Any
Source         → IP range, tag, or "*" (any)
Destination    → IP range, tag, or "*"
Port range     → 80, 443, 22, 3389, or ranges like 8000-8080
Action         → Allow or Deny
```

Azure always adds default rules:

```text
Default inbound:
  - Allow from VNet (traffic from within the same VNet)
  - Allow from Azure Load Balancer
  - Deny everything else

Default outbound:
  - Allow to VNet
  - Allow to internet
  - Deny everything else
```

You add rules to customize on top of these defaults.

---

# 5. Create an NSG and Rules (CLI)

Create the NSG:

```bash
az network nsg create \
  --resource-group $RG \
  --name web-nsg
```

Allow inbound HTTP (port 80):

```bash
az network nsg rule create \
  --resource-group $RG \
  --nsg-name web-nsg \
  --name AllowHTTP \
  --priority 100 \
  --protocol Tcp \
  --direction Inbound \
  --source-address-prefixes "*" \
  --destination-port-ranges 80 \
  --access Allow
```

Allow inbound HTTPS (port 443):

```bash
az network nsg rule create \
  --resource-group $RG \
  --nsg-name web-nsg \
  --name AllowHTTPS \
  --priority 110 \
  --protocol Tcp \
  --direction Inbound \
  --source-address-prefixes "*" \
  --destination-port-ranges 443 \
  --access Allow
```

Allow SSH from a specific IP only:

```bash
az network nsg rule create \
  --resource-group $RG \
  --nsg-name web-nsg \
  --name AllowSSHFromMyIP \
  --priority 120 \
  --protocol Tcp \
  --direction Inbound \
  --source-address-prefixes "203.0.113.10" \
  --destination-port-ranges 22 \
  --access Allow
```

---

# 6. Associate the NSG with a Subnet

```bash
az network vnet subnet update \
  --resource-group $RG \
  --vnet-name my-vnet \
  --name web-subnet \
  --network-security-group web-nsg
```

Now all VMs in `web-subnet` share these rules.

NSGs can also be associated directly with a network interface (NIC) of a specific VM, in addition to the subnet.

---

# 7. Create a VM in a Specific VNet and Subnet

```bash
az vm create \
  --resource-group $RG \
  --name web-vm \
  --image Ubuntu2204 \
  --size Standard_B1s \
  --admin-username azureuser \
  --generate-ssh-keys \
  --vnet-name my-vnet \
  --subnet web-subnet \
  --nsg "" \
  --public-ip-sku Standard
```

> `--nsg ""` skips auto-creating an NSG because we already have one on the subnet.

# 8. NSG for a Database Tier

For the database subnet, you typically want:

```text
- Allow inbound on port 5432 (PostgreSQL) from web-subnet only
- Deny everything else inbound
```

```bash
az network nsg create \
  --resource-group $RG \
  --name db-nsg

az network nsg rule create \
  --resource-group $RG \
  --nsg-name db-nsg \
  --name AllowPostgresFromWebSubnet \
  --priority 100 \
  --protocol Tcp \
  --direction Inbound \
  --source-address-prefixes "10.0.1.0/24" \
  --destination-port-ranges 5432 \
  --access Allow

az network vnet subnet update \
  --resource-group $RG \
  --vnet-name my-vnet \
  --name db-subnet \
  --network-security-group db-nsg
```

Now the database is only reachable from the web tier — not from the internet.

# 9. Service Tags

Instead of hardcoding IP ranges, you can use **service tags** which represent groups of Azure IPs.

Common tags:

```text
Internet            — all public internet addresses
VirtualNetwork      — the VNet and all peered VNets
AzureLoadBalancer   — Azure load balancer health probes
Storage             — Azure Storage service IPs
Sql                 — Azure SQL Database IPs
AzureCloud          — all Azure datacenter IPs
```

Example — allow health probes from the load balancer:

```bash
az network nsg rule create \
  --resource-group $RG \
  --nsg-name web-nsg \
  --name AllowLBProbe \
  --priority 90 \
  --protocol "*" \
  --direction Inbound \
  --source-address-prefixes AzureLoadBalancer \
  --destination-port-ranges "*" \
  --access Allow
```

# 10. VNet Peering

VNets are isolated from each other by default. To connect two VNets, use **VNet peering**.

```bash
az network vnet peering create \
  --resource-group $RG \
  --name vnet1-to-vnet2 \
  --vnet-name my-vnet-1 \
  --remote-vnet my-vnet-2 \
  --allow-vnet-access

az network vnet peering create \
  --resource-group $RG \
  --name vnet2-to-vnet1 \
  --vnet-name my-vnet-2 \
  --remote-vnet my-vnet-1 \
  --allow-vnet-access
```

Peering must be created in both directions.

---

# 11. Common Mistakes

```text
Mistake                                  Result
--------------------------------------   ------------------------------------------
Opening port 22 to *                     SSH exposed to all internet scanners/bots
Using /8 CIDR (too broad)                Huge IP range, harder to manage
No NSG on database subnet                Database reachable from web VM directly
Forgetting to deny explicitly            Default Deny at the end catches most, but
                                         explicit denies make intent clearer
Not associating NSG to subnet            NSG exists but does nothing
```

# Quick Recap

| Concept | Key Point |
|---|---|
| VNet | Private isolated network in Azure |
| Subnet | Segment of a VNet; attach NSGs here |
| NSG | Firewall rules — allow or deny by port, IP, direction |
| Priority | Lower number = evaluated first |
| Service tag | Named group of Azure IPs (e.g., `AzureLoadBalancer`) |
| VNet peering | Connect two VNets so resources can communicate |
