Once you have a Docker container image, Azure gives you two main ways to run it:

```text
Container Apps   — simple, managed, serverless containers
AKS              — full Kubernetes cluster, more control and more complexity
```

This lesson focuses on **Container Apps** for beginners, with an overview of AKS to help you understand when each is appropriate.

---

# 1. What Is a Container?

If you are new to containers, the short version:

```text
A container is a packaged app with everything it needs to run:
- Your code
- Runtime (Node, Python, Java, etc.)
- System libraries
- Config
```

A Docker image is the blueprint. A container is the running instance.

A `Dockerfile` defines how to build the image:

```dockerfile
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]
```

Build it:

```bash
docker build -t my-app:latest .
```

Run it locally:

```bash
docker run -p 3000:3000 my-app:latest
```

---

# 2. Azure Container Apps

Container Apps is a managed service that runs containers with:

```text
- Auto-scaling (including scale to zero)
- No Kubernetes configuration needed
- Built-in HTTPS and domain
- Revision management (deploy new versions, split traffic)
- Event-driven scaling via KEDA
```

Best for:

```text
- Containerized web apps and APIs
- Microservices
- Background workers
- Apps that need to scale to zero to save money
```

---

# 3. Azure Container Registry

Before deploying to Container Apps, you need somewhere to store your image.

**Azure Container Registry (ACR)** is a private Docker registry in Azure.

Create a registry:

```bash
RG=rg-containers-demo
LOC=eastus
ACR=myregistry$RANDOM    # must be globally unique, alphanumeric only

az group create -n $RG -l $LOC

az acr create \
  --resource-group $RG \
  --name $ACR \
  --sku Basic
```

Log in to ACR:

```bash
az acr login --name $ACR
```

Tag and push your image:

```bash
docker build -t my-app:latest .
docker tag my-app:latest ${ACR}.azurecr.io/my-app:latest
docker push ${ACR}.azurecr.io/my-app:latest
```

# 4. Deploy to Container Apps (CLI)

Create a Container Apps environment (the networking and logging layer):

```bash
ENV=my-container-env

az containerapp env create \
  --name $ENV \
  --resource-group $RG \
  --location $LOC
```

Deploy the container:

```bash
APP=my-container-app

az containerapp create \
  --name $APP \
  --resource-group $RG \
  --environment $ENV \
  --image ${ACR}.azurecr.io/my-app:latest \
  --registry-server ${ACR}.azurecr.io \
  --registry-identity system \
  --target-port 3000 \
  --ingress external \
  --min-replicas 0 \
  --max-replicas 5
```

Key flags:

```text
--target-port     The port your container listens on
--ingress         "external" = publicly accessible; "internal" = VNet only
--min-replicas 0  Scale to zero when idle
--max-replicas 5  Scale up to 5 instances under load
```

After deploy, the CLI prints a public URL:

```text
https://my-container-app.nicename-123.eastus.azurecontainerapps.io
```

---

# 5. Update a Container App

Deploy a new image version:

```bash
docker build -t my-app:v2 .
docker tag my-app:v2 ${ACR}.azurecr.io/my-app:v2
docker push ${ACR}.azurecr.io/my-app:v2

az containerapp update \
  --name $APP \
  --resource-group $RG \
  --image ${ACR}.azurecr.io/my-app:v2
```

Each update creates a new **revision**. Old revisions stay available and can receive a percentage of traffic (useful for canary deployments).

# 6. Scale Rules

Container Apps uses **KEDA** (Kubernetes-based Event-Driven Autoscaler) under the hood.

Scale based on HTTP traffic:

```bash
az containerapp update \
  --name $APP \
  --resource-group $RG \
  --min-replicas 0 \
  --max-replicas 10 \
  --scale-rule-name http-rule \
  --scale-rule-type http \
  --scale-rule-http-concurrency 10
```

This means: scale up when more than 10 concurrent HTTP requests are pending.

Scale based on a queue length (Azure Storage Queue):

```text
--scale-rule-type azure-queue
--scale-rule-metadata queueName=my-queue connectionFromEnv=QUEUE_CONNECTION
```

# 7. Environment Variables

```bash
az containerapp update \
  --name $APP \
  --resource-group $RG \
  --set-env-vars DATABASE_URL=secretref:db-url NODE_ENV=production
```

For secrets:

```bash
az containerapp secret set \
  --name $APP \
  --resource-group $RG \
  --secrets db-url="postgresql://..."
```

Then reference with `secretref:db-url`.

# 8. Azure Kubernetes Service (AKS) — Overview

AKS runs a full Kubernetes cluster.

Kubernetes is a platform for scheduling and managing containers across many machines. AKS makes it easier than running Kubernetes yourself, but it is still substantially more complex than Container Apps.

When to use AKS:

```text
- You already know Kubernetes
- You need fine-grained pod scheduling, custom resource definitions, or node pools
- You are running dozens of microservices with complex routing
- You need GPU node pools for AI workloads
- You are deploying a platform others build on top of
```

When to stay with Container Apps:

```text
- You have a few services and want simplicity
- You want scale-to-zero billing
- You do not want to manage Kubernetes cluster upgrades, nodes, and networking
```

A rough cost comparison:

```text
Container Apps (idle, scale to zero):     ~$0/month
AKS with 2 Standard_B2s nodes:            ~$60–$120/month (nodes run 24/7)
```

### Create an AKS cluster (CLI reference)

```bash
az aks create \
  --resource-group $RG \
  --name my-aks-cluster \
  --node-count 1 \
  --node-vm-size Standard_B2s \
  --generate-ssh-keys

az aks get-credentials --resource-group $RG --name my-aks-cluster

kubectl get nodes
```

After `get-credentials`, standard `kubectl` commands work against your AKS cluster.

---

# 9. Side-by-Side: Container Apps vs AKS

| Feature | Container Apps | AKS |
|---|---|---|
| Manages Kubernetes for you | Yes (fully abstracted) | Partially (cluster managed, nodes are yours) |
| Scale to zero | Yes | No (unless using Virtual Nodes) |
| Pricing model | Per vCPU-second | Per node VM hour |
| YAML/Helm required | No | Yes |
| Custom scheduling | No | Yes |
| Complexity | Low | High |
| Best for | Web apps, APIs, workers | Large microservice platforms |

# Quick Recap

| Topic | Key Point |
|---|---|
| Container | Packaged app with all its dependencies |
| ACR | Azure Container Registry — private image store |
| Container Apps | Run containers serverlessly; scale to zero |
| AKS | Full Kubernetes — more control, more cost, more complexity |
| Ingress | `external` = public URL; `internal` = VNet only |
| Revision | Each deploy creates a new version; old ones stay live |
