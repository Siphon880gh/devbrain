Azure App Service is the most common way to host a web app or REST API on Azure without managing servers.

You push code (or a Docker container), and Azure handles:

```text
- The operating system
- Runtime (Node.js, Python, PHP, .NET, Java, etc.)
- TLS/HTTPS
- Load balancing
- Horizontal scaling
- Deployment slots
```

---

# 1. Key Concepts

## App Service Plan

An **App Service Plan** is the underlying compute resource — the virtual machine(s) that run your apps.

```text
App Service Plan
└── App Service (your web app)
└── App Service (another web app, sharing the same plan)
```

You pay for the plan, not for each app individually. Multiple apps can share one plan.

## App Service

An **App Service** is the web app itself. It is what you deploy code to.

Each App Service has:

```text
- A hostname: yourappname.azurewebsites.net
- A runtime (e.g., Node 20, Python 3.11)
- Environment variables
- Deployment configuration
- Logs
```

---

# 2. App Service Plan Tiers

| Tier | Name | Cost | Use Case |
|---|---|---|---|
| F1 | Free | $0 | Learning, toy apps |
| B1 | Basic | ~$13/month | Dev/test, small prod |
| B2 | Basic | ~$27/month | Slightly more RAM |
| S1 | Standard | ~$56/month | Prod, deployment slots, auto-scale |
| P1v3 | Premium v3 | ~$123/month | High perf, large apps |

F1 limitations:

```text
- Shared CPU, 1 GB RAM, 1 GB storage
- No custom domain SSL (no HTTPS on custom domains)
- Apps "sleep" after 20 minutes of inactivity (cold start delay)
- No always-on setting
```

For production, use at least B1. For auto-scaling and deployment slots, use S1 or higher.

# 3. Create an App Service (Portal)

```text
1. Portal → search "App Services" → Create
2. Fill in:
   - Subscription
   - Resource group (create new or use existing)
   - Name: (this becomes yourname.azurewebsites.net — must be globally unique)
   - Publish: Code (or Docker Container)
   - Runtime stack: choose your language (Node.js, Python, PHP, .NET, Java, Ruby)
   - Region: pick a region close to your users
   - Pricing plan: F1 Free or B1 Basic
3. Click "Review + create" → "Create"
```

After a minute, your app is live at:

```text
https://yourname.azurewebsites.net
```

# 4. Create an App Service (CLI)

```bash
RG=rg-webapp-demo
PLAN=plan-webapp-demo
APP=mywebapp$RANDOM        # must be globally unique
LOC=eastus
RUNTIME="NODE:20-lts"      # or PYTHON:3.11, PHP:8.2, DOTNET:8, JAVA:17, etc.

az group create -n $RG -l $LOC

az appservice plan create \
  --name $PLAN \
  --resource-group $RG \
  --sku F1 \
  --is-linux

az webapp create \
  --name $APP \
  --resource-group $RG \
  --plan $PLAN \
  --runtime "$RUNTIME"

echo "App URL: https://${APP}.azurewebsites.net"
```

# 5. Deploy Code

## Option A: Git push (built-in CI)

Configure local Git deployment:

```bash
az webapp deployment source config-local-git \
  --name $APP \
  --resource-group $RG
```

This returns a Git remote URL. Push to it:

```bash
git remote add azure <URL from above>
git push azure main
```

Azure builds and deploys your app.

## Option B: ZIP deploy

Package your project as a zip and push:

```bash
zip -r app.zip . -x "node_modules/*" ".git/*"

az webapp deploy \
  --resource-group $RG \
  --name $APP \
  --src-path app.zip \
  --type zip
```

## Option C: GitHub Actions (recommended for teams)

Link your GitHub repo in the portal:

```text
App Service → Deployment Center → GitHub → connect repo → Save
```

Azure creates a GitHub Actions workflow file in your repo. Every push to main triggers a build and deploy.

## Option D: VS Code

Install the **Azure App Service** extension for VS Code.

```text
Extension sidebar → Azure → App Service → right-click your app → Deploy to Web App
```

# 6. Set Environment Variables

Environment variables are called **Application Settings** in App Service.

Via CLI:

```bash
az webapp config appsettings set \
  --resource-group $RG \
  --name $APP \
  --settings DATABASE_URL="postgresql://..." NODE_ENV="production"
```

These are injected as environment variables at runtime. Your app reads them with `process.env.DATABASE_URL` (Node) or `os.environ["DATABASE_URL"]` (Python).

Do not hardcode secrets in code. Use Application Settings or Key Vault references (lesson 05/4).

# 8. Custom Domain

To add your own domain (e.g., `myapp.example.com`):

```text
1. Buy or control a domain via your registrar
2. App Service → Custom domains → Add custom domain
3. Add a CNAME record at your registrar pointing to yourapp.azurewebsites.net
4. Verify domain ownership (via TXT record)
5. Bind a free managed TLS certificate (available on B1 and above):
   App Service → TLS/SSL settings → Managed Certificate
```

The F1 free plan does not support HTTPS on custom domains.

---

# 9. Scale Up and Scale Out

**Scale up** = move to a bigger plan (more CPU/RAM):

```bash
az appservice plan update \
  --name $PLAN \
  --resource-group $RG \
  --sku B2
```

**Scale out** = add more instances (horizontal):

```text
App Service → Scale out (App Service plan) → Manual scale → set instance count
```

Auto-scale (adds/removes instances based on CPU/memory) is available on Standard (S1) and above:

```text
App Service → Scale out → Custom autoscale → add rules
```

---

# 10. Deployment Slots

Deployment slots let you deploy to a staging environment, test it, then **swap** to production with zero downtime.

Available on Standard (S1) and above:

```text
App Service → Deployment slots → Add slot: "staging"
```

Deploy to staging, verify, then swap:

```bash
az webapp deployment slot swap \
  --resource-group $RG \
  --name $APP \
  --slot staging \
  --target-slot production
```

The old production becomes the new staging. You can instantly swap back if something breaks.

# 11. Common Errors

| Error | Likely Cause |
|---|---|
| `Application Error` on site | Crash at startup — check logs with `az webapp log tail` |
| 404 on `/` | No default document or wrong startup command |
| Slow first load | F1 plan with "sleep" — upgrade to B1 or enable always-on |
| `ENOENT: cannot find module` | Missing `node_modules` — add a build step or include dependencies |
| App won't start after deploy | Wrong runtime version or missing environment variable |

---

# 12. Clean Up

```bash
az group delete --name $RG --yes --no-wait
```

Deletes the resource group, plan, and app together.

# Quick Recap

| Topic | Key Point |
|---|---|
| App Service Plan | The compute resource you pay for |
| App Service | Your web app, deployed on the plan |
| F1 free tier | Shared CPU, limited RAM, no custom domain SSL, sleeps |
| Deploy | ZIP deploy, git push, GitHub Actions, or VS Code |
| Env vars | Application Settings in portal or CLI |
| Scale up | Bigger plan |
| Scale out | More instances (Standard and above for auto-scale) |
| Deployment slots | Stage then swap to production |
