
If you want LiteLLM to call OpenAI models, the flow is straightforward: create your OpenAI project, set budget and limits, generate a project API key, place that key in LiteLLM, and then call LiteLLM through either API requests or a CLI client. OpenAI’s platform supports project-scoped API keys, project budgets, model usage controls, and rate limits. ([OpenAI Help Center](https://help.openai.com/en/articles/9186755-managing-your-work-in-the-api-platform-with-projects?utm_source=chatgpt.com "Managing projects in the API platform"))

## Step 1: Set up your OpenAI organization and project

OpenAI recommends using **Projects** to organize API work rather than creating separate organizations for every effort. Projects let you separate billing, keys, permissions, and model usage more cleanly. ([OpenAI Help Center](https://help.openai.com/en/articles/8991840-can-i-create-an-additional-platform-api-organization?utm_source=chatgpt.com "Can I create an additional Platform API organization?"))

In practical terms:

- create or sign into your OpenAI API organization
    
- create a project for this app or environment
    
- add team members if needed
    
- use project-specific API keys instead of sharing one personal key
    

OpenAI’s help docs state that project API keys are created and managed on the project’s API Keys page. ([OpenAI Help Center](https://help.openai.com/en/articles/9186755-managing-your-work-in-the-api-platform-with-projects?utm_source=chatgpt.com "Managing projects in the API platform"))

## Step 2: Enable billing, budget, and limits

OpenAI documents organization and project limits in the platform. You can set a project monthly budget, a notification threshold, and model usage controls from the project’s Limits section. Their docs also note that rate limits are tracked at the organization and project levels and vary by model. ([OpenAI Help Center](https://help.openai.com/en/articles/9186755-managing-your-work-in-the-api-platform-with-projects?utm_source=chatgpt.com "Managing projects in the API platform"))

So for a clean setup:

- add billing
    
- open the project’s **Limits**
    
- set a monthly budget
    
- set notification thresholds
    
- restrict model access if you want to prevent accidental use of expensive models
    

## Step 3: Generate an API key

From the project’s **API Keys** page, create a new secret key. OpenAI’s help docs state that you can assign permissions such as All, Restricted, or Read Only when creating or editing a key. ([OpenAI Help Center](https://help.openai.com/en/articles/9186755-managing-your-work-in-the-api-platform-with-projects?utm_source=chatgpt.com "Managing projects in the API platform"))

Important security rule: do not put this key in browser code, mobile code, or public repos. OpenAI explicitly recommends keeping API keys server-side and passing them in through environment variables or secret managers. ([OpenAI Developers](https://developers.openai.com/api/docs/guides/production-best-practices "Production best practices | OpenAI API"))

## Step 4: Bring the OpenAI key into LiteLLM

Set the key as an environment variable on the LiteLLM host:

```bash
export OPENAI_API_KEY="sk-..."
```

Then define an OpenAI-backed model in LiteLLM:

```yaml
model_list:
  - model_name: gpt-4o
    litellm_params:
      model: openai/gpt-4o
      api_key: os.environ/OPENAI_API_KEY

general_settings:
  master_key: sk-your-litellm-master-key
```

LiteLLM’s docs show this pattern of using environment variables and mapping a public `model_name` to the provider-specific model string. ([liteLLM](https://docs.litellm.ai/ "liteLLM"))

## Step 5: Start LiteLLM

```bash
litellm --config config.yaml
```

At that point LiteLLM becomes your app-facing endpoint, and OpenAI is just the backend provider behind it. LiteLLM’s proxy is OpenAI-compatible, so your clients can talk to LiteLLM as if it were an OpenAI-style endpoint. ([liteLLM](https://docs.litellm.ai/docs/?utm_source=chatgpt.com "Getting Started | liteLLM"))

## Step 6: Prompt it through the API

Example request to LiteLLM:

```bash
curl http://localhost:4000/v1/chat/completions \
  -H "Authorization: Bearer sk-your-litellm-master-key" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o",
    "messages": [
      {"role": "user", "content": "Write a two-sentence explanation of LiteLLM"}
    ]
  }'
```

That request goes to LiteLLM, and LiteLLM forwards it to OpenAI using the project API key you stored server-side. ([liteLLM](https://docs.litellm.ai/docs/?utm_source=chatgpt.com "Getting Started | liteLLM"))

## Step 7: Prompt it from a CLI or OpenAI-compatible client

Because LiteLLM speaks an OpenAI-style API, most OpenAI-compatible tools can be pointed at LiteLLM by changing:

- the base URL
    
- the API key used to authenticate to LiteLLM
    

That is one of the main reasons teams put LiteLLM in front of providers: existing clients often keep working with minimal changes. ([liteLLM](https://docs.litellm.ai/docs/?utm_source=chatgpt.com "Getting Started | liteLLM"))

## Recommended production habit

Use **project-level keys** and **separate keys per app or environment** where possible. OpenAI recommends unique keys per user and warns against key sharing; LiteLLM can then sit in front as the internal gateway for routing, auditing, and model control. ([OpenAI Help Center](https://help.openai.com/en/articles/5112595-best-practices-for-api-key-safety?utm_source=chatgpt.com "Best Practices for API Key Safety"))

## Bottom line

OpenAI gives you the hosted models and billing controls. LiteLLM gives you the stable gateway in front. That combination is useful when you want OpenAI now but still want the freedom to add Ollama or another provider later without changing your whole app architecture.
