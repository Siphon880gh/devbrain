## Requirement - Model Access

Before adding LiteLLM to your Python app, you need **at least one model backend** available.

That usually means one of these:
- a **remote AI provider** such as OpenAI
- or a **local model server** such as Ollama

LiteLLM does not generate responses by itself.  
Your Python code uses LiteLLM to send requests to a real model backend.

---

## Requirement - uv

Some Python tools recommend using `uv` instead of plain `pip` for project dependency management.

If your project uses `uv`, then adding LiteLLM to the project looks like this:

```bash
uv add litellm
````

First, check whether `uv` is already installed:

```bash
uv --version
```

If you see a version, you are ready.
If you see `command not found`, install it first. Refer to [[_PRIMER - uv (Python)]] for installation.

## What LiteLLM is in a Python app

LiteLLM is a **Python package you import into your code**.

It gives you a **single interface** to talk to different model providers such as:

* OpenAI
* Anthropic
* Azure OpenAI
* Ollama (local models)

Instead of writing different SDK logic for each provider, your app calls LiteLLM in one consistent way.

---

## Why use LiteLLM in Python code

Use LiteLLM when your app needs:

* one interface for multiple LLM providers
* flexibility to switch models later
* support for both remote and local models
* cleaner code without provider-specific SDKs everywhere

Common use cases:

* AI-powered apps
* chat systems
* backend APIs
* agents and automation

## Example usage in Python

```python
from litellm import completion

response = completion(
    model="gpt-4o",
    messages=[
        {"role": "user", "content": "Explain LiteLLM simply"}
    ]
)

print(response["choices"][0]["message"]["content"])
```

## ⚠️ Auto routing in Python

LiteLLM does **not automatically choose models for you based on criteria** in a Python app.

There is no built-in “smart routing” unless you define it.

👉 Routing = **your code deciding the model**
This means you have to manually code the if statements, etc.

### What routing actually looks like

#### Eg. Model based on prompt length

```python
def pick_model(prompt):
    return "gpt-4o-mini" if len(prompt) < 100 else "gpt-4o"
```

#### Eg. Fallback model if error

Helpful especially when OpenAI likes to sunset models

```python
from litellm import completion

try:
    response = completion(
        model="gpt-4o",
        messages=[{"role": "user", "content": "Summarize this"}]
    )
except Exception:
    response = completion(
        model="claude-3-haiku",
        messages=[{"role": "user", "content": "Summarize this"}]
    )
```

### Key takeaway on routing

* LiteLLM **executes requests**
* Your code **decides the model**
* “Auto routing” = simple logic you write (if/else, functions, config)

---

## Mental model

* your Python app → calls LiteLLM
* LiteLLM → forwards to a model
* the model → generates the response

LiteLLM is the **connector**, not the decision-maker.

## When to use this version

Use LiteLLM as a Python dependency when:

* you are building an AI-powered Python app
* you want flexibility inside your code
* you want to avoid provider lock-in

If your app directly calls LLMs in code, this is the version you want.
