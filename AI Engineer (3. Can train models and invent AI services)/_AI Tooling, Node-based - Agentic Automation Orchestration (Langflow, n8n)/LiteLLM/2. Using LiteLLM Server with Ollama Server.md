If you want LiteLLM to use a local model, Ollama is one of the easiest backends to pair with it. Ollama runs a local server, usually on port `11434`, and LiteLLM can forward model requests to it. LiteLLM supports Ollama as a provider, and Ollama exposes local APIs on `localhost:11434`. ([liteLLM](https://docs.litellm.ai/ "liteLLM"))

## If Required - Step 1: Install Ollama

For Linux, Ollama’s official download page shows:

```bash
curl -fsSL https://ollama.com/install.sh | sh
```

For macOS, the official download page provides a macOS installer and notes that it requires macOS 14 Sonoma or later. ([Ollama](https://ollama.com/download?utm_source=chatgpt.com "Download Ollama on Linux"))

## Step 2: Make sure the Ollama server is running

Before you connect LiteLLM to Ollama, verify that Ollama is actually serving requests.

A simple check is:

```bash
curl http://localhost:11434
```

If it is up, Ollama returns:

```text
Ollama is running
```

Ollama’s docs and examples consistently use `localhost:11434` as the local service endpoint. ([Ollama](https://ollama.com/blog/openai-compatibility?utm_source=chatgpt.com "OpenAI compatibility · Ollama Blog"))

If the server is not running yet, start it with:

```bash
ollama serve
```

Some environments run Ollama as a background service already, so if port `11434` is already listening, you may not need to start it manually. The common sign is that your curl check already returns `Ollama is running`. ([GitHub](https://github.com/ollama/ollama/issues/707?utm_source=chatgpt.com "127.0.0.1:11434: bind: address already in use · Issue #707"))

## Step 3: See which models are available

Use:

```bash
ollama list
```

That shows the models available to Ollama on that machine. This is helpful before connecting LiteLLM, because LiteLLM needs a real model name behind the alias you configure. ([Ollama](https://ollama.com/blog/cloud-models?utm_source=chatgpt.com "Cloud models · Ollama Blog"))

## Step 4: Test Ollama directly first

It is smart to test Ollama before bringing LiteLLM into the mix.

For example after you see llama3 in `ollama list`:
- Adjust based on the models in your ollama list:
```bash
ollama run llama3
```

Or test the API directly:

```bash
curl -X POST http://localhost:11434/api/generate -d '{
  "model": "llama3",
  "prompt": "Why is the sky blue?"
}'
```

Ollama’s examples use both the CLI and the local API this way. ([Ollama](https://ollama.com/library/llama2?utm_source=chatgpt.com "llama2"))

## Step 5: Point LiteLLM at Ollama

Add this to your `config.yaml`:
- Note you are prefixing the model with `ollama/` at the `model` field
- Unless you want to have separate sets of available models, you can have the config file at a central point like `~/litellm/config.yaml`
  
```yaml
model_list:
  - model_name: local-llama
    litellm_params:
      model: ollama/llama3
      api_base: http://localhost:11434

general_settings:
  master_key: sk-your-litellm-master-key
```

^ **What these fields mean**
- `model_name`  
    The **name your app uses** when calling LiteLLM.
- `litellm_params.model`  
    The **real model LiteLLM sends the request to**.

^**How it works:**

Your app sends:

```json
{
  "model": "local-llama"
}
```

LiteLLM translates that to while making the conventional OpenAI-style API request:

```yaml
model: ollama/llama3
```

So:
- `local-llama` = alias (what your app sees)
- `ollama/llama3` = actual model running in Ollama
    

---

### What you actually edit

- Change this if you want a different **name in your app**:

```yaml
model_name: local-llama
```

- Change this if you want a different **Ollama model**:
    

```yaml
model: ollama/llama3
```

Make sure the model exists:

```bash
ollama list
```

---

### Run LiteLLM with your config

If your config is at:

```
~/litellm/config.yaml
```

**Start LiteLLM like this:**

```bash
litellm --config ~/litellm/config.yaml
```

LiteLLM does **not automatically detect** config files in your home directory, so you must pass `--config`.

## Step 6: Call Ollama through LiteLLM

Now your app can call LiteLLM using `model: "local-llama"` instead of talking directly to Ollama. LiteLLM’s config docs explain this alias pattern directly. ([liteLLM](https://docs.litellm.ai/docs/proxy/configs "Overview | liteLLM"))

Example:

```bash
curl http://localhost:4000/v1/chat/completions \
  -H "Authorization: Bearer sk-your-litellm-master-key" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "local-llama",
    "messages": [
      {"role": "user", "content": "Why is the grass green?"}
    ]
  }'
```

Because LiteLLM’s proxy is OpenAI-compatible, this looks like a standard OpenAI-style request even though the backend is Ollama. ([liteLLM](https://docs.litellm.ai/docs/?utm_source=chatgpt.com "Getting Started | liteLLM"))
