## 🧠 What LiteLLM actually does

**LiteLLM** is a tool that:

- **Standardizes access** to different LLM providers
    
- Works with both:
    
    - Remote APIs (like OpenAI, Anthropic)
        
    - Local servers (like Ollama running models such as DeepSeek or Llama)
        
- Can **auto-route requests** based on rules you define (model, cost, task, fallback, etc.)
    

👉 Think of it as a **universal adapter + router for LLMs**

---

## ⚠️ Key idea: There are TWO versions of LiteLLM

This is where most people get confused.

LiteLLM can be used in **two completely different ways**:

---

# 1️⃣ LiteLLM as a SERVER (API Gateway)

## What it is

You run LiteLLM as a **central API server**.

It becomes your **single endpoint** for all LLM requests.

---

## What it does

- Exposes an API like:
    
    ```
    http://your-server:4000/v1/chat/completions
    ```
    
- Accepts requests from:
    
    - `curl`
        
    - frontend apps
        
    - backend services
        
    - other languages (Node, Python, etc.)
        

---

## Example usage (curl)

```bash
curl http://localhost:4000/v1/chat/completions \
  -H "Authorization: Bearer sk-..." \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o",
    "messages": [{"role": "user", "content": "Hello"}]
  }'
```

---

## Why use the server version?

- Centralize all LLM usage across your team/app
    
- Hide API keys from clients
    
- Apply:
    
    - routing rules
        
    - fallbacks
        
    - logging
        
    - rate limits
        

---

## Example architecture

```
Frontend / App / curl
        ↓
   LiteLLM Server
        ↓
(OpenAI / Claude / Ollama / etc.)
```

---

## Auto-routing (server)

You can define rules like:

- Short prompts → cheaper model
    
- Long context → Claude
    
- If failure → fallback to another model
    
- Local model first → cloud as backup
    

👉 This is where **auto-routing really shines**

---

# 2️⃣ LiteLLM as a PYTHON LIBRARY (Dependency)

## What it is

You install LiteLLM directly into your Python project:

```bash
uv add litellm
```

---

## What it does

- Gives you a **single Python interface** to call multiple LLMs
    
- Lets your app:
    
    - switch models easily
        
    - use local + remote models
        
    - optionally implement routing logic
        

---

## Example usage (Python)

```python
from litellm import completion

response = completion(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Explain LiteLLM"}]
)

print(response["choices"][0]["message"]["content"])
```

---

## Switching providers (same code)

```python
model="gpt-4o"                  # OpenAI
model="claude-3-haiku"          # Anthropic
model="ollama/llama3"           # Local (Ollama)
```

👉 No SDK switching, no rewrite

---

## Auto-routing (Python)

Yes — **you can implement routing logic here too**, but:

- It’s typically **manual or config-based**
    
- Less centralized than the server version
    

Example:

```python
def route_model(prompt):
    if len(prompt) < 100:
        return "gpt-4o-mini"
    return "claude-3-sonnet"

model = route_model(user_input)
```

---

## Why use the Python version?

- You’re building an **AI-powered backend/app**
    
- You want flexibility inside your code
    
- You don’t need a shared API gateway (yet)
    

---

# 🔥 Server vs Python: When to use each

|Use Case|Use This|
|---|---|
|Multiple apps / team access|✅ Server|
|Central control + logging|✅ Server|
|Quick integration in Python app|✅ Python library|
|Fine-grained logic inside code|✅ Python library|
|Expose API for frontend or curl|✅ Server|

---

## 🧠 Simple mental model

- **Server version** = “LLM gateway for everything”
    
- **Python version** = “LLM helper inside my app”
    

---

## ⚡ Real-world setup (common)

Many teams use **both together**:

```
Python App (LiteLLM library)
        ↓
LiteLLM Server (routing, logging)
        ↓
(OpenAI / Claude / Ollama)
```

---

## ✅ Final takeaway

LiteLLM gives you:

- **Standardized access** to all LLMs
    
- **Support for local + remote models**
    
- **Auto-routing capabilities**
    

But most importantly:

👉 You can use it in **two ways**:

1. **As a server** → centralized API gateway
    
2. **As a Python library** → flexible in-app usage
    

Understanding this split is what unlocks its real power.
